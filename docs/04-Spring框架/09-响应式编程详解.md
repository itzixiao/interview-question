## 一、响应式编程概述

### 1.1 什么是响应式编程

**定义：**
响应式编程（Reactive Programming）是一种面向数据流和变化传播的编程范式。它允许开发者以声明式的方式处理异步数据流，而无需关心底层的线程调度和同步问题。

**核心特点：**

- **异步非阻塞**：不阻塞线程，提高资源利用率
- **事件驱动**：基于事件和回调的组合
- **声明式风格**：代码简洁，易于理解和维护
- **背压支持**：下游控制上游的生产速度

### 1.2 为什么需要响应式编程

#### 传统阻塞 I/O 的问题

```
┌─────────────────────────────────────┐
│   阻塞 I/O 的问题                    │
├─────────────────────────────────────┤
│ ❌ 线程浪费                          │
│    - 每个请求占用一个线程            │
│    - I/O 等待时线程空闲               │
│    - 高并发需要大量线程              │
│                                     │
│ ❌ 资源限制                          │
│    - 线程池大小有限                  │
│    - 上下文切换开销                  │
│    - 内存消耗大                      │
│                                     │
│ ❌ 扩展性差                          │
│    - 难以应对突发流量                │
│    - 资源利用率低                    │
└─────────────────────────────────────┘
```

#### 响应式编程解决方案

```
┌─────────────────────────────────────┐
│   响应式编程的优势                   │
├─────────────────────────────────────┤
│ ✅ 高效资源利用                      │
│    - 少量线程处理大量请求            │
│    - I/O 等待时不阻塞                 │
│    - 减少线程切换开销                │
│                                     │
│ ✅ 弹性伸缩                          │
│    - 轻松应对高并发                  │
│    - 背压保护系统                    │
│    - 故障隔离                        │
│                                     │
│ ✅ 组合性强                          │
│    - 声明式 API                      │
│    - 操作符链式调用                  │
│    - 错误处理统一                    │
└─────────────────────────────────────┘
```

### 1.3 Reactive Streams 规范

**Reactive Streams** 是一个 JVM 响应式流的标准化规范，定义了四个核心接口：

```java
// 发布者（Publisher）
public interface Publisher<T> {
    void subscribe(Subscriber<? super T> s);
}

// 订阅者（Subscriber）
public interface Subscriber<T> {
    void onSubscribe(Subscription s);
    void onNext(T t);
    void onError(Throwable t);
    void onComplete();
}

// 订阅（Subscription）
public interface Subscription {
    void request(long n);  // 请求 n 个元素
    void cancel();         // 取消订阅
}

// 处理器（Processor）
public interface Processor<T, R> extends Subscriber<T>, Publisher<R> {
}
```

## 二、Reactor 模式深入

### 2.1 Reactor 模式原理

**Reactor 模式**是一种事件驱动的设计模式，用于处理多个并发的 I/O 操作。

#### 核心组件

```
┌─────────────────────────────────────────┐
│         Reactor 模式架构                 │
│                                         │
│  ┌─────────┐     ┌──────────────────┐  │
│  │ Acceptor│────▶│  Connection      │  │
│  └─────────┘     │  Handler Pool    │  │
│                  └──────────────────┘  │
│                           │             │
│                  ┌────────┴────────┐   │
│                  │  Event Handler  │   │
│                  │  (业务逻辑)      │   │
│                  └─────────────────┘   │
└─────────────────────────────────────────┘
```

**工作流程：**

1. **Acceptor**：监听并接受新连接
2. **Reactor**：分发事件到对应的 Handler
3. **Handler**：处理具体的 I/O 事件
4. **Business Logic**：执行实际的业务逻辑

### 2.2 Reactor 模式实现对比

| 框架                  | 实现              | 特点                |
|---------------------|-----------------|-------------------|
| **Netty**           | NIO + Reactor   | 高性能网络框架           |
| **Vert.x**          | Event Loop      | 多语言支持             |
| **Node.js**         | libuv           | JavaScript 运行时    |
| **Project Reactor** | Reactor Streams | Spring WebFlux 基础 |

## 三、Project Reactor 核心 API

### 3.1 Mono 和 Flux

#### Mono：发布 0-1 个元素

```java
// 创建 Mono
Mono<String> just = Mono.just("Hello");
Mono<String> empty = Mono.empty();
Mono<String> error = Mono.error(new RuntimeException("Error"));

// 订阅消费
just.subscribe(
    data -> log.info("收到：{}", data),
    error -> log.error("错误：{}", error.getMessage()),
    () -> log.info("完成")
);
```

#### Flux：发布 0-N 个元素

```java
// 创建 Flux
Flux<Integer> range = Flux.range(1, 10);
Flux<String> array = Flux.fromArray(new String[]{"A", "B", "C"});
Flux<String> list = Flux.fromIterable(List.of("X", "Y", "Z"));

// 链式操作
Flux.range(1, 10)
    .filter(x -> x % 2 == 0)
    .map(x -> x * 2)
    .subscribe(System.out::println);
```

### 3.2 常用操作符

#### 转换操作符

```java
// map：一对一转换
Flux.range(1, 5)
    .map(x -> x * 2)
    .subscribe(System.out::println);

// flatMap：一对多转换（扁平化）
Flux.just("hello", "world")
    .flatMap(word -> 
        Flux.fromArray(word.split(""))
    )
    .subscribe(System.out::println);

// concatMap：有序扁平化
Flux.just("hello", "world")
    .concatMap(word -> 
        Flux.fromArray(word.split(""))
    )
    .subscribe(System.out::println);
```

#### 过滤操作符

```java
// filter：条件过滤
Flux.range(1, 10)
    .filter(x -> x > 5)
    .subscribe(System.out::println);

// take：取前 N 个
Flux.range(1, 100)
    .take(5)
    .subscribe(System.out::println);

// distinct：去重
Flux.just(1, 2, 2, 3, 3, 3)
    .distinct()
    .subscribe(System.out::println);
```

#### 组合操作符

```java
// merge：合并多个流（无序）
Flux.merge(
    Flux.interval(Duration.ofMillis(100)).take(3),
    Flux.interval(Duration.ofMillis(150)).take(3)
).subscribe(System.out::println);

// zip：成对组合
Flux.zip(
    Flux.just("A", "B", "C"),
    Flux.just(1, 2, 3)
).subscribe(tuple -> 
    System.out.println(tuple.getT1() + "-" + tuple.getT2())
);

// concat：顺序拼接
Flux.concat(
    Flux.just("A", "B"),
    Flux.just("C", "D")
).subscribe(System.out::println);
```

### 3.3 错误处理

```java
// onErrorReturn：返回默认值
Flux.just(1, 2, 0)
    .map(x -> 10 / x)
    .onErrorReturn(0)
    .subscribe(System.out::println);

// onErrorResume：使用备用流
Flux.just(1, 2, 0)
    .map(x -> 10 / x)
    .onErrorResume(error -> Flux.just(-1))
    .subscribe(System.out::println);

// doOnError：副作用处理
Flux.just(1, 2, 0)
    .map(x -> 10 / x)
    .doOnError(error -> log.error("发生错误", error))
    .subscribe(System.out::println);
```

### 4.3 响应式服务层

```java
@Service
@RequiredArgsConstructor
public class UserService {
    
    private final UserRepository repository;
    
    // 查询单个用户
    public Mono<User> getUserById(Long id) {
        return repository.findById(id);
    }
    
    // 查询所有用户
    public Flux<User> getAllUsers() {
        return repository.findAll();
    }
    
    // 创建用户
    public Mono<User> createUser(User user) {
        return repository.save(user);
    }
    
    // 更新用户
    public Mono<User> updateUser(Long id, User updatedUser) {
        return repository.findById(id)
                .flatMap(existingUser -> {
                    existingUser.setName(updatedUser.getName());
                    existingUser.setEmail(updatedUser.getEmail());
                    return repository.save(existingUser);
                });
    }
    
    // 删除用户
    public Mono<Void> deleteUser(Long id) {
        return repository.deleteById(id);
    }
    
    // 复杂业务：搜索并聚合
    public Flux<Order> getUserOrdersWithDetails(Long userId) {
        return repository.findOrdersByUserId(userId)
                .flatMap(order -> 
                    repository.findOrderDetails(order.getId())
                            .collectList()
                            .map(details -> {
                                order.setDetails(details);
                                return order;
                            })
                );
    }
}
```

## 五、RSocket 协议

### 5.1 什么是 RSocket

**RSocket** 是一个应用层协议，专为响应式编程设计的二进制协议。

**特点：**

- **多交互模式**：支持 4 种通信模式
- **背压支持**：内置背压机制
- **二进制协议**：高效传输
- **连接复用**：单连接多路复用

### 5.2 RSocket 四种交互模式

#### 1. Request-Response（请求 - 响应）

```java
@MessageMapping("user.get")
public Mono<User> getUser(Mono<Long> request) {
    return request.map(id -> new User(id, "User-" + id));
}
```

#### 2. Request-Stream（请求 - 流）

```java
@MessageMapping("users.stream")
public Flux<User> getUsersStream(Mono<Long> count) {
    return count.flatMapMany(n -> 
        Flux.range(1, n.intValue())
            .map(i -> new User((long) i, "User-" + i))
            .delayElements(Duration.ofMillis(500))
    );
}
```

#### 3. Fire-and-Forget（发后即忘）

```java
@MessageMapping("user.log")
public Mono<Void> logUser(User user) {
    log.info("记录用户：{}", user);
    return Mono.empty();  // 不返回任何内容
}
```

#### 4. Channel（双向流）

```java
@MessageMapping("chat")
public Flux<String> chat(Flux<String> clientMessages) {
    return clientMessages
        .doOnNext(msg -> log.info("收到：{}", msg))
        .map(msg -> "Echo: " + msg);
}
```

### 5.3 RSocket 配置

#### 服务端配置

```yaml
spring:
  rsocket:
    server:
      port: 7000
      mapping-path: /rsocket
```

#### 客户端配置

```java
@Configuration
public class RSocketConfig {
    
    @Bean
    public RSocketRequester rSocketRequester() {
        return RSocketRequester.builder()
            .connectTcp("localhost", 7000)
            .block();
    }
}
```

#### 客户端使用

```java
@Service
@RequiredArgsConstructor
public class RSocketClient {
    
    private final RSocketRequester requester;
    
    // Request-Response
    public Mono<User> getUser(Long id) {
        return requester.route("user.get")
                .data(id)
                .retrieveMono(User.class);
    }
    
    // Request-Stream
    public Flux<User> getUsersStream(int count) {
        return requester.route("users.stream")
                .data(count)
                .retrieveFlux(User.class);
    }
    
    // Fire-and-Forget
    public void logUser(User user) {
        requester.route("user.log")
                .data(user)
                .send()
                .subscribe();
    }
}
```

## 六、背压（Backpressure）机制

### 6.1 什么是背压

**背压（Backpressure）** 是响应式编程中的核心概念，指下游（消费者）向上游（生产者）发送信号，控制上游的数据生产速度。

**为什么需要背压？**

- 防止快速生产者压垮慢速消费者
- 避免内存溢出（OOM）
- 保护系统稳定性

### 6.2 背压策略

#### 1. onError（抛出异常）

```java
Flux<Long> fastProducer = Flux.interval(Duration.ofMillis(10));

fastProducer
    .onBackpressureError()  // 默认行为
    .subscribe(x -> {
        try {
            Thread.sleep(100);  // 慢速消费
        } catch (InterruptedException e) {}
        System.out.println("消费：" + x);
    });
// 结果：抛出 BackpressureException
```

#### 2. onBackpressureBuffer（缓冲）

```java
Flux<Long> fastProducer = Flux.interval(Duration.ofMillis(10));

fastProducer
    .onBackpressureBuffer(100)  // 最多缓冲 100 个
    .subscribe(x -> {
        try {
            Thread.sleep(50);
        } catch (InterruptedException e) {}
        System.out.println("消费：" + x);
    });
// 结果：缓冲区满后丢弃新数据
```

#### 3. onBackpressureDrop（丢弃）

```java
Flux<Long> fastProducer = Flux.interval(Duration.ofMillis(10));

fastProducer
    .onBackpressureDrop()  // 丢弃无法处理的数据
    .subscribe(x -> {
        try {
            Thread.sleep(50);
        } catch (InterruptedException e) {}
        System.out.println("消费：" + x);
    });
// 结果：直接丢弃新数据
```

#### 4. onBackpressureLatest（保留最新）

```java
Flux<Long> fastProducer = Flux.interval(Duration.ofMillis(10));

fastProducer
    .onBackpressureLatest()  // 只保留最新的数据
    .subscribe(x -> {
        try {
            Thread.sleep(50);
        } catch (InterruptedException e) {}
        System.out.println("消费：" + x);
    });
// 结果：总是消费最新的数据
```

### 6.3 背压在 WebFlux 中的应用

#### HTTP 层面的背压

```java
@PostMapping("/upload")
public Mono<Void> handleUpload(ServerHttpRequest request) {
    return request.getBody()
            .publishOn(Schedulers.boundedElastic())
            .doOnNext(dataBuffer -> {
                // 处理数据块
                // 如果处理慢，会自动触发背压
            })
            .then();
}
```

#### 数据库背压

```java
@GetMapping("/stream")
public Flux<User> streamUsers() {
    return userRepository.findAll()
            .limitRate(10);  // 每批请求 10 个
}
```

## 七、高频面试题

### 7.1 响应式编程基础（8 道）

**问题 1：什么是响应式编程？解决了什么问题？**

**答：**

响应式编程（Reactive Programming）是一种面向数据流和变化传播的编程范式，通过异步非阻塞的方式处理数据流。

**核心特点：**

1. **异步非阻塞** - 不会阻塞线程等待 I/O 完成
2. **数据流驱动** - 数据变化自动传播到订阅者
3. **背压支持** - 消费者可以控制数据接收速率

**解决的问题：**

- 高并发下的线程资源耗尽问题
- 传统阻塞 I/O 的性能瓶颈
- 复杂异步逻辑的回调地狱

---

**问题 2：响应式编程 vs 命令式编程的区别？**

**答：**

| 对比项 | 命令式编程 | 响应式编程 |
|--------|-----------|-----------|
| 执行方式 | 同步顺序执行 | 异步非阻塞 |
| 数据获取 | 主动拉取（Pull） | 被动推送（Push） |
| 线程使用 | 阻塞等待 I/O | 事件驱动，释放线程 |
| 代码风格 | 逐行执行 | 声明式链式调用 |
| 错误处理 | try-catch | 操作符链式处理 |

**示例对比：**

```java
// 命令式：阻塞等待
User user = userService.getUser(id);  // 阻塞
Order order = orderService.getOrder(user);  // 阻塞

// 响应式：非阻塞链式
Mono<User> userMono = userService.getUser(id);
userMono.flatMap(user -> orderService.getOrder(user))
        .subscribe();
```

---

**问题 3：Reactive Streams 规范的核心接口？**

**答：**

Reactive Streams 是响应式编程的标准规范，定义了 4 个核心接口：

| 接口 | 角色 | 核心方法 |
|------|------|---------|
| **Publisher** | 数据发布者 | `subscribe(Subscriber)` |
| **Subscriber** | 数据订阅者 | `onSubscribe/onNext/onError/onComplete` |
| **Subscription** | 订阅关系 | `request/cancel` |
| **Processor** | 处理器（发布+订阅） | 继承 Publisher 和 Subscriber |

**工作流程：**

```
Publisher → subscribe → Subscriber
              ↓
         Subscription (request n)
              ↓
         onNext(data) × n → onComplete/onError
```

---

**问题 4：什么是背压？为什么需要背压？**

**答：**

**背压（Backpressure）** 是指当生产者产生数据的速度超过消费者处理能力时，消费者向生产者反馈压力，要求降低发送速率或暂停发送的机制。

**为什么需要背压：**

1. **防止内存溢出** - 生产者过快导致缓冲区无限增长
2. **保护消费者** - 避免消费者被大量数据压垮
3. **资源平衡** - 协调生产者和消费者的处理能力

**背压策略：**

| 策略 | 说明 |
|------|------|
| **BUFFER** | 缓冲所有数据（可能 OOM） |
| **DROP** | 丢弃多余数据 |
| **LATEST** | 只保留最新数据 |
| **ERROR** | 直接报错 |

---

**问题 5：Mono 和 Flux 的区别？**

**答：**

Mono 和 Flux 是 Project Reactor 的两个核心发布者类型：

| 特性 | Mono | Flux |
|------|------|------|
| **数据量** | 0 或 1 个元素 | 0 到 N 个元素 |
| **适用场景** | 单对象查询、保存操作 | 列表查询、流式数据 |
| **类比** | `Optional` / `CompletableFuture` | `Stream` / `List` |
| **结束信号** | onNext + onComplete 或 onComplete | 0~N 次 onNext + onComplete |

**代码示例：**

```java
// Mono：单个用户
Mono<User> userMono = userRepository.findById(id);

// Flux：用户列表
Flux<User> userFlux = userRepository.findAll();
```

**问题 6：热流（Hot）vs 冷流（Cold）？**

**答：**

| 特性 | 冷流（Cold） | 热流（Hot） |
|------|-------------|------------|
| **数据产生** | 每个订阅者独立触发 | 独立于订阅者，主动产生 |
| **订阅时机** | 订阅后才发送数据 | 可能错过之前的数据 |
| **数据共享** | 每个订阅者获取完整序列 | 多个订阅者共享同一数据 |
| **典型实现** | `Flux.just()`, `Flux.fromIterable()` | `Flux.share()`, `Flux.publish()` |

**类比：**

- **冷流** = 观看视频点播（每个人从头开始看）
- **热流** = 观看直播（晚来的人错过之前的内容）

---

**问题 7：响应式编程的优缺点？**

**答：**

**优点：**

1. **高吞吐量** - 少量线程处理大量并发连接
2. **资源高效** - 非阻塞 I/O 不占用线程等待
3. **背压保护** - 防止消费者过载
4. **声明式编程** - 代码简洁，易于组合

**缺点：**

1. **学习曲线陡峭** - 思维转换成本高
2. **调试困难** - 异步堆栈追踪复杂
3. **生态限制** - 部分库不支持响应式
4. **阻塞陷阱** - 误用阻塞 API 会抵消优势

---

**问题 8：适用场景有哪些？**

**答：**

**适合场景：**

1. **高并发 Web 服务** - 网关、API 代理
2. **流式数据处理** - 实时日志、股票行情
3. **微服务通信** - 服务间异步调用
4. **I/O 密集型应用** - 数据库、缓存、HTTP 调用

**不适合场景：**

1. **计算密集型任务** - 阻塞线程反而更高效
2. **简单 CRUD** - 引入复杂度得不偿失
3. **遗留系统集成** - 大量阻塞代码难以改造

**问题 9：Project Reactor 的核心组件？**

**答：**

Project Reactor 是 Spring 生态的响应式编程库，核心组件包括：

| 组件 | 说明 |
|------|------|
| **Mono** | 异步单值容器（0 或 1 个元素） |
| **Flux** | 异步多值容器（0 到 N 个元素） |

| **Processor** | 同时实现 Publisher 和 Subscriber |

**核心操作符分类：**

1. **创建操作符**：`just`, `fromIterable`, `create`, `generate`
2. **转换操作符**：`map`, `flatMap`, `filter`, `reduce`
3. **组合操作符**：`merge`, `zip`, `concat`
4. **错误处理**：`onErrorReturn`, `onErrorResume`, `retry`

**问题 10：map 和 flatMap 的区别？**

**答：**

| 特性 | map | flatMap |
|------|-----|---------|
| **返回类型** | 同步转换，返回普通对象 | 异步转换，返回 Publisher |
| **展开层级** | 保持单层 | 自动展平嵌套 Publisher |
| **适用场景** | 简单数据转换 | 涉及异步操作的转换 |
| **执行顺序** | 保持顺序 | 可能乱序（并行时） |

**代码示例：**

```java
// map：同步转换
Flux<String> names = Flux.just("alice", "bob")
    .map(String::toUpperCase);  // ALICE, BOB

// flatMap：异步转换，展平结果
Flux<Order> orders = Flux.just(user1, user2)
    .flatMap(user -> orderService.findByUser(user));  // 展平 List<Order>
```

**选择建议：**

- 简单转换用 `map`
- 涉及数据库/HTTP 调用用 `flatMap`
- 需要保持顺序用 `concatMap`

**问题 11：publishOn 和 subscribeOn 的区别？**

**答：**

| 特性 | subscribeOn | publishOn |
|------|-------------|-----------|
| **影响范围** | 影响整个链（从源头开始） | 只影响下游操作 |
| **执行位置** | 订阅和请求执行 | 数据处理和通知 |
| **使用次数** | 链中第一个生效 | 可多次使用切换线程 |
| **典型用途** | 指定数据源执行线程 | 指定后续处理线程 |

**执行流程：**

```
subscribeOn(Schedulers.boundedElastic())  // 影响源头
    ↓
源数据获取（在 boundedElastic 线程）
    ↓
publishOn(Schedulers.parallel())  // 切换线程
    ↓
后续操作（在 parallel 线程）
```

**常见组合：**

```java
Flux.just(data)
    .subscribeOn(Schedulers.boundedElastic())  // I/O 操作
    .map(this::transform)
    .publishOn(Schedulers.parallel())          // CPU 计算
    .map(this::compute)
    .subscribe();
```

**问题 12：如何处理响应式流的错误？**

**答：**

Project Reactor 提供了多种错误处理策略：

| 操作符 | 作用 |
|--------|------|
| **onErrorReturn** | 发生错误时返回默认值 |
| **onErrorResume** | 发生错误时切换到备用流 |
| **onErrorMap** | 转换错误类型 |
| **doOnError** | 副作用处理（如日志记录） |
| **retry** | 重试指定次数 |
| **retryWhen** | 自定义重试策略 |

**代码示例：**

```java
// 返回默认值
mono.onErrorReturn(e -> e instanceof NotFoundException, defaultUser);

// 切换到备用流
mono.onErrorResume(e -> userService.findById(backupId));

// 重试策略
mono.retryWhen(Retry.backoff(3, Duration.ofSeconds(1))
    .filter(e -> e instanceof TimeoutException));
```

**问题 17：Spring WebFlux vs Spring MVC？**

**答：**

| 特性 | Spring MVC | Spring WebFlux |
|------|-----------|----------------|
| **编程模型** | 命令式（同步阻塞） | 响应式（异步非阻塞） |
| **底层容器** | Servlet 容器（Tomcat/Jetty） | Netty / Undertow / Servlet 3.1+ |
| **线程模型** | 每请求一线程 | 事件驱动，少量线程处理大量请求 |

| **适用场景** | 传统 Web 应用 | 高并发、流式处理 |

**核心区别：**

- **MVC**：`Controller` 返回 `String`/`ModelAndView`
- **WebFlux**：`Controller` 返回 `Mono`/`Flux`

**选择建议：**

- 已有 Spring MVC 项目且无明显性能瓶颈 → 继续使用 MVC
- 网关、代理、高并发场景 → 优先考虑 WebFlux

**问题 18：WebFlux 的两种编程模型？**

**答：**

Spring WebFlux 支持两种编程模型：

**1. 注解式（Annotated Controllers）：**

与 Spring MVC 类似，使用 `@Controller`、`@RequestMapping` 等注解：

```java
@RestController
public class UserController {
    
    @GetMapping("/users/{id}")
    public Mono<User> getUser(@PathVariable String id) {
        return userService.findById(id);
    }
    
    @GetMapping("/users")
    public Flux<User> listUsers() {
        return userService.findAll();
    }
}
```

**2. 函数式（Functional Endpoints）：**

使用 `RouterFunction` 和 `HandlerFunction`：

```java
@Configuration
public class UserRouter {
    
    @Bean
    public RouterFunction<ServerResponse> userRoutes(UserHandler handler) {
        return route(GET("/users/{id}"), handler::getUser)
            .andRoute(GET("/users"), handler::listUsers)
            .andRoute(POST("/users"), handler::createUser);
    }
}

@Component
public class UserHandler {
    public Mono<ServerResponse> getUser(ServerRequest request) {
        String id = request.pathVariable("id");
        return userService.findById(id)
            .flatMap(user -> ServerResponse.ok().bodyValue(user))
            .switchIfEmpty(ServerResponse.notFound().build());
    }
}
```

**对比：**

| 特性 | 注解式 | 函数式 |
|------|--------|--------|
| 学习成本 | 低（与 MVC 一致） | 高 |
| 灵活性 | 受限于注解 | 完全程序化控制 |
| 测试性 | 需要启动容器 | 纯函数，易于单元测试 |
| 适用场景 | 快速开发 | 复杂路由逻辑 |

```java
}
```

**问题 23：WebFlux 的性能优势？**

**答：**

**1. 更高的并发处理能力：**

- 少量线程（如 CPU 核心数）处理大量并发连接
- 避免线程上下文切换开销

**2. 更低的内存占用：**

- 不需要为每个请求分配线程栈（默认 1MB）
- 适合连接数多但计算量小的场景（如网关）

**3. 延迟更低：**

- 非阻塞 I/O，线程不会被等待占用
- 请求处理更及时

**性能对比数据（参考）：**

| 指标 | Spring MVC | Spring WebFlux |
|------|-----------|----------------|
| 并发连接数 | ~1000/线程 | 10000+ |
| 内存占用（1万连接） | ~10GB | ~100MB |
| 延迟（P99） | 较高 | 较低 |

**适用场景：**

- API 网关
- 高并发微服务
- 实时数据推送

**问题 24：WebFlux 的局限性？**

**答：**

**1. 生态兼容性问题：**

- 大量库只提供阻塞 API（如 JDBC、部分 HTTP 客户端）
- 需要使用响应式替代方案（R2DBC、WebClient）

**2. 调试困难：**

- 异步堆栈追踪复杂
- 问题定位难度大

**3. 学习成本高：**

- 思维转换：从命令式到响应式
- 团队培训成本

**4. 不适合的场景：**

- 计算密集型任务（阻塞线程更高效）
- 简单 CRUD 应用（引入复杂度得不偿失）
- 遗留系统大量阻塞代码难以改造

**5. 事务处理复杂：**

```java
// 响应式事务需要特殊处理
@Transactional  // 传统方式无效
public Mono<User> createUser(User user) {

        userRepository.save(user)
```

### 7.4 RSocket（5 道）

**问题 25：RSocket 协议的特点？**

**答：**

RSocket 是一个支持响应式流语义的二进制协议，设计用于微服务通信。

**核心特点：**

| 特性 | 说明 |
|------|------|
| **响应式语义** | 原生支持背压和异步流 |
| **多路复用** | 单连接支持多个交互 |
| **双向通信** | 客户端和服务端都可以发起请求 |
| **连接恢复** | 支持会话恢复，断线重连 |
| **多种传输** | TCP、WebSocket、Aeron 等 |

**与 HTTP 对比优势：**

- 二进制协议，更高效
- 原生流支持，无需 SSE/WebSocket 变通
- 内置背压，避免消费者过载

---

**问题 26：RSocket 的四种交互模式？**

**答：**

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| **Request-Response** | 一发一收 | 传统 RPC 调用 |
| **Request-Stream** | 一发多收 | 实时数据流 |
| **Fire-and-Forget** | 只发不收 | 日志上报、事件通知 |
| **Channel** | 双向流 | 聊天、实时协作 |

**代码示例：**

```java
// Request-Response
rsocket.requestResponse(payload)
    .subscribe(response -> System.out.println(response));

// Request-Stream
rsocket.requestStream(payload)
    .subscribe(data -> System.out.println(data));

// Fire-and-Forget
rsocket.fireAndForget(payload).subscribe();

// Channel（双向流）
rsocket.requestChannel(
    Flux.interval(Duration.ofSeconds(1))  // 发送端
        .map(i -> DefaultPayload.create("Client: " + i))
)
.subscribe(response -> System.out.println(response));
```

**问题 27：RSocket vs HTTP/2？**

**答：**

| 特性 | HTTP/2 | RSocket |
|------|--------|---------|
| **协议层级** | 应用层 | 应用层（在 TCP/WebSocket 之上） |
| **多路复用** | 支持（单连接多流） | 支持（更灵活的流控制） |
| **双向通信** | 服务端推送（Server Push） | 原生支持，更自然 |
| **背压** | 无原生支持 | 原生支持 |
| **连接恢复** | 需应用层实现 | 协议层支持 |
| **流语义** | 请求-响应为主 | 四种交互模式 |

**选择建议：**

- **浏览器客户端** → HTTP/2（兼容性更好）
- **服务间通信** → RSocket（性能更优，语义更丰富）
- **实时流应用** → RSocket（原生流支持）

**问题 28：RSocket 的背压机制？**

**答：**

RSocket 在协议层面实现了背压机制，通过 **LEASE** 帧控制数据流：

**工作原理：**

```
服务端发送 LEASE 帧 → 告知客户端可发送的请求数
客户端根据 LEASE 发送请求
服务端处理完成后，发送新的 LEASE
```

**LEASE 帧包含：**

- **TTL**：租约有效期
- **Number of Requests**：允许发送的请求数

**代码配置：**

```java
RSocketServer.create(new SocketAcceptor() {
    @Override
    public Mono<RSocket> accept(ConnectionSetupPayload setup, RSocket sendingSocket) {
        return Mono.just(new AbstractRSocket() {
            @Override
            public Flux<Payload> requestStream(Payload payload) {
                return Flux.interval(Duration.ofMillis(10))
                    .map(i -> DefaultPayload.create("Data " + i))
                    .onBackpressureBuffer(100);  // 背压缓冲
            }
        });
    }
})
.lease(spec -> spec.maxPendingRequests(100))  // 最大待处理请求
.bind(TcpServerTransport.create(7000))
.subscribe();
```

**问题 29：RSocket 的连接复用？**

**答：**

RSocket 支持单连接多路复用，一个物理连接可以同时处理多个逻辑流：

**优势：**

1. **减少连接开销** - 避免频繁创建/销毁连接
2. **提高资源利用率** - 单连接承载更多请求
3. **简化连接管理** - 连接池规模更小

**实现方式：**

```java
// 客户端保持长连接
RSocket client = RSocketConnector.create()
    .reconnect(Retry.fixedDelay(3, Duration.ofSeconds(1)))  // 自动重连
    .connect(TcpClientTransport.create("localhost", 7000))
    .block();

// 复用同一连接发送多个请求
client.requestResponse(request1).subscribe();
client.requestResponse(request2).subscribe();
client.requestStream(request3).subscribe();
```

**会话恢复（Resumability）：**

```java
RSocketConnector.create()
    .resume(new Resume()
        .sessionDuration(Duration.ofMinutes(5))
        .retry(Retry.fixedDelay(10, Duration.ofSeconds(1))))
    .connect(TcpClientTransport.create("localhost", 7000))
    .subscribe();
```

断线后重连，可恢复之前的会话状态，不丢失数据。

**问题 30：如何设计一个实时推送系统？**

**答：**

**系统架构：**

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   客户端     │◄───►│  WebFlux    │◄───►│  消息队列   │
│  (WebSocket)│     │   Gateway   │     │  (Kafka)    │
└─────────────┘     └──────┬──────┘     └──────┬──────┘
                           │                    │
                    ┌──────▼──────┐     ┌──────▼──────┐
                    │  推送服务    │     │  业务服务   │
                    │ (WebFlux)   │     │ (产生消息)  │
                    └─────────────┘     └─────────────┘
```

**核心实现：**

```java
@Service
public class PushService {
    
    private final Map<String, Sinks.Many<String>> userSinks = new ConcurrentHashMap<>();
    
    // 用户建立连接
    public Flux<String> connect(String userId) {
        Sinks.Many<String> sink = Sinks.many().multicast().onBackpressureBuffer();
        userSinks.put(userId, sink);
        return sink.asFlux()
            .doFinally(signal -> userSinks.remove(userId));
    }
    
    // 推送消息给指定用户
    public void pushToUser(String userId, String message) {
        Sinks.Many<String> sink = userSinks.get(userId);
        if (sink != null) {
            sink.tryEmitNext(message);
        }
    }
    
    // 广播消息
    public void broadcast(String message) {
        userSinks.values().forEach(sink -> sink.tryEmitNext(message));
    }
}

// Controller
@RestController
public class PushController {
    
    @GetMapping(value = "/push/{userId}", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<String> pushStream(@PathVariable String userId) {
        return pushService.connect(userId);
    }
}
```

**关键技术点：**

1. **Sinks** - 用于多播消息到多个订阅者
2. **背压控制** - 使用 `onBackpressureBuffer` 防止内存溢出
3. **心跳检测** - 定期发送心跳保持连接
4. **集群部署** - 使用 Redis Pub/Sub 实现跨节点消息同步

```java
public Mono<User> findById(String id) {
```

**问题 34：微服务架构中 RSocket 的应用场景？**

**答：**

**适用场景：**

| 场景 | RSocket 优势 | 实现方式 |
|------|-------------|---------|
| **实时数据同步** | 双向流、低延迟 | Channel 模式 |

```
└─────────────┘                  └─────────────┘
       │                               │
       │  Fire-and-Forget              │
       ▼                               ▼
┌─────────────┐                  ┌─────────────┐
│  消息服务    │                  │  日志服务    │
└─────────────┘                  └─────────────┘
```

**代码示例 - 实时库存同步：**

```java
// 库存服务（Responder）
@Component
public class InventoryRSocketController {
    
    @MessageMapping("inventory.updates")
    public Flux<InventoryUpdate> streamUpdates() {
        return inventoryService.getUpdateStream()
            .onBackpressureBuffer(100);
    }
    
    @MessageMapping("inventory.reserve")
    public Mono<ReservationResult> reserve(ReservationRequest request) {
        return inventoryService.reserve(request);
    }
}

// 订单服务（Requester）
@Service
public class OrderService {
    
    private final RSocketRequester inventoryRSocket;
    
    public OrderService(RSocketRequester.Builder builder) {
        this.inventoryRSocket = builder
            .tcp("inventory-service", 7000);
    }
    
    // 订阅库存实时更新
    public void subscribeInventoryUpdates() {
        inventoryRSocket.route("inventory.updates")
            .retrieveFlux(InventoryUpdate.class)
            .subscribe(update -> {
                cache.put(update.getProductId(), update.getQuantity());
            });
    }
}
```

1. **性能** - 二进制协议，比 HTTP/JSON 更高效
2. **实时性** - 原生流支持，无需轮询
3. **可靠性** - 内置背压和连接恢复
4. **灵活性** - 四种交互模式适应不同场景

## 💡 实战技巧

### 响应式编程口诀

```
异步非阻塞用 Reactor，
Mono Flux 要分清。
操作符链式来组合，
背压保护不能忘。
WebFlux 建服务，
RSocket 更高效。
阻塞代码要避免，
响应式里全异步。
```

### 常见陷阱

```
1. ⚠️ 在响应式链中使用阻塞调用（.block()）
2. ⚠️ 忘记处理错误情况
3. ⚠️ 不当的线程调度导致性能下降
4. ⚠️ 没有背压保护导致 OOM
5. ⚠️ 过度使用 flatMap 导致无序
```

### 最佳实践

```
✅ 使用 checkstyle 检查 .block() 调用
✅ 统一的错误处理策略
✅ 合理设置背压缓冲区
✅ 使用 StepVerifier 进行测试
✅ 监控响应式流的性能指标
```
