# Redis 缓存与分布式锁

## Redis 缓存

### 缓存策略

#### 1. Cache Aside（旁路缓存）

```
读：先读缓存 → 未命中读数据库 → 写入缓存
写：先更新数据库 → 删除缓存
```

**优点**：简单、容错性好
**缺点**：有短暂不一致

#### 2. Read/Write Through（读写穿透）

读写都经过缓存，缓存与数据库同步。

#### 3. Write Behind（异步回写）

先写缓存，异步批量写入数据库。

### 缓存问题

#### 缓存穿透

**问题**：查询不存在的数据，绕过缓存直达数据库

**解决方案**：
- 缓存空值（设置短过期时间）
- 布隆过滤器
- 参数校验

#### 缓存击穿

**问题**：热点 key 过期，大量请求打崩数据库

**解决方案**：
- 互斥锁（只有一个线程重建缓存）
- 逻辑过期（永不过期，异步更新）
- 热点 key 永不过期

#### 缓存雪崩

**问题**：大量 key 同时过期或 Redis 宕机

**解决方案**：
- 随机过期时间
- 多级缓存（本地 + Redis）
- 熔断降级
- 高可用架构（Cluster + 哨兵）

### 缓存一致性

#### 延迟双删

```
1. 删除缓存
2. 更新数据库
3. 延迟（如 500ms）
4. 再次删除缓存
```

## 分布式锁

### 实现方式对比

| 方式 | 优点 | 缺点 |
|------|------|------|
| Redis SETNX | 性能好 | 需处理续期、主从同步问题 |
| ZooKeeper | 可靠性高 | 性能较低 |
| 数据库 | 简单 | 性能差 |
| etcd | 强一致 | 运维复杂 |

### Redis 分布式锁演进

#### 1. 基础版本（SETNX + EXPIRE）

```java
SETNX lock:key value
EXPIRE lock:key 30
```

**问题**：SETNX 和 EXPIRE 非原子操作

#### 2. 改进版本（SET NX EX）

```java
SET lock:key value NX EX 30
```

**问题**：
- 不可重入
- 主从切换可能丢失锁
- 业务未完成锁过期

#### 3. Redisson（推荐）

```java
RLock lock = redisson.getLock("myLock");
lock.lock();
try {
    // 执行业务
} finally {
    lock.unlock();
}
```

**特性**：
- 可重入
- 看门狗自动续期
- 阻塞等待
- 公平锁

### Redisson 看门狗机制

```
1. 获取锁成功，启动看门狗线程
2. 看门狗每 10 秒检查（过期时间的 1/3）
3. 业务未完成，自动续期到 30 秒
4. 业务完成或宕机，锁最终释放
```

### 锁续期问题

```java
// 错误：锁过期，业务未完成
lock.tryLock(10, 30, TimeUnit.SECONDS);  // 30秒后锁过期

// 正确：使用看门狗（不指定过期时间）
lock.lock();  // 看门狗自动续期
```

### RedLock（红锁）

多节点部署时的分布式锁算法：

```
1. 向 N 个 Redis 节点申请锁
2. 计算申请耗时
3. 多数节点（N/2+1）申请成功且耗时 < 锁过期时间，则获取锁
4. 释放时向所有节点发送释放命令
```

**争议**：RedLock 在分布式系统中有争议，推荐使用单节点 + 持久化或 ZooKeeper。

## 最佳实践

1. **缓存设置过期时间**：避免脏数据永久存在
2. **使用延迟双删**：保证最终一致性
3. **使用 Redisson**：避免自己实现分布式锁的坑
4. **锁粒度要小**：减少竞争
5. **锁内业务要短**：避免锁长时间占用
6. **必须 finally 释放锁**：避免死锁
