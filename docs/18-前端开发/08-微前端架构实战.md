# 微前端架构实战

## 一、微前端核心概念

### 1.1 什么是微前端

微前端（Micro Frontends）是一种将前端应用拆分为更小、更独立单元的架构模式，借鉴了微服务的思想：

```
┌─────────────────────────────────────────────────────────┐
│                    主应用 (Host)                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │   导航栏     │  │   侧边栏     │  │   用户中心    │     │
│  │  (公共组件)  │  │  (公共组件)   │  │  (子应用A)   │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
│  ┌─────────────────────────────────────────────────┐   │
│  │              内容区域 (动态加载)                   │   │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐          │   │
│  │  │ 子应用A  │  │ 子应用B  │  │ 子应用C  │          │   │
│  │  │ (React) │  │  (Vue)  │  │(Angular)│          │   │
│  │  └─────────┘  └─────────┘  └─────────┘          │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 1.2 微前端 vs 单体应用

| 特性       | 单体应用  | 微前端       |
|----------|-------|-----------|
| **代码库**  | 单一仓库  | 多仓库/单仓库多包 |
| **技术栈**  | 统一    | 可异构       |
| **部署**   | 整体部署  | 独立部署      |
| **团队规模** | 适合小团队 | 适合大团队     |
| **复杂度**  | 简单    | 复杂        |
| **性能**   | 可控    | 需额外优化     |
| **共享依赖** | 天然共享  | 需配置共享     |

## 二、微前端方案对比

| 方案                    | 实现方式                    | 技术栈支持       | 特点        |
|-----------------------|-------------------------|-------------|-----------|
| **qiankun**           | JS Sandbox + HTML Entry | 不限          | 阿里开源，生态完善 |
| **Module Federation** | Webpack 5 插件            | 需 Webpack 5 | 模块共享能力强   |
| **single-spa**        | 路由劫持 + 生命周期             | 不限          | 最老牌，社区活跃  |
| **Micro App**         | WebComponent            | 不限          | 京东开源，侵入性低 |
| **wujie**             | iframe + WebComponent   | 不限          | 腾讯开源，隔离性强 |
| **无界**                | iframe 沙箱               | 不限          | 字节开源，性能优秀 |

### 5.2 自定义事件总线

```typescript
// shared/EventBus.ts
class EventBus {
    private events: Map<string, Set<Function>> = new Map();

    // 订阅事件
    on(event: string, callback: Function) {
        if (!this.events.has(event)) {
            this.events.set(event, new Set());
        }
        this.events.get(event)!.add(callback);

        // 返回取消订阅函数
        return () => this.off(event, callback);
    }

    // 取消订阅
    off(event: string, callback: Function) {
        const callbacks = this.events.get(event);
        if (callbacks) {
            callbacks.delete(callback);
        }
    }

    // 触发事件
    emit(event: string, ...args: any[]) {
        const callbacks = this.events.get(event);
        if (callbacks) {
            callbacks.forEach(callback => {
                try {
                    callback(...args);
                } catch (error) {
                    console.error(`Event ${event} handler error:`, error);
                }
            });
        }
    }

    // 只订阅一次
    once(event: string, callback: Function) {
        const onceCallback = (...args: any[]) => {
            this.off(event, onceCallback);
            callback(...args);
        };
        this.on(event, onceCallback);
    }
}

// 创建全局事件总线实例
export const eventBus = new EventBus();

// 挂载到 window 供跨应用访问
(window as any).microAppEventBus = eventBus;

// 使用示例
// 子应用 A 发送消息
import {eventBus} from '@/shared/EventBus';

eventBus.emit('user:login', {userId: '123', name: '张三'});

// 子应用 B 接收消息
eventBus.on('user:login', (userInfo) => {
    console.log('用户登录:', userInfo);
});
```

## 九、高频面试题

**问题 1：微前端解决了什么问题？带来了什么挑战？**

**答：**

**解决的问题：**

1. **技术栈无关** - 不同团队可以使用不同技术栈
2. **独立部署** - 子应用可独立开发、测试、部署
3. **团队自治** - 减少跨团队协调成本
4. **渐进式迁移** - 逐步将老应用改造为微前端

**带来的挑战：**

| 挑战        | 说明        | 解决方案                        |
|-----------|-----------|-----------------------------|
| **样式隔离**  | 子应用样式相互影响 | Shadow DOM、CSS Modules、BEM  |
| **JS 隔离** | 全局变量污染    | JS Sandbox、严格模式             |
| **性能问题**  | 多应用加载慢    | 预加载、共享依赖、懒加载                |
| **通信复杂**  | 跨应用数据同步   | 全局状态、事件总线、URL 参数            |
| **公共依赖**  | 重复加载      | externals、Module Federation |

**问题 2：qiankun 的 JS 沙箱是如何实现的？**

**答：**

qiankun 提供两种沙箱模式：

**1. Snapshot Sandbox（快照沙箱）**

```javascript
// 原理：记录和恢复全局变量
class SnapshotSandbox {
    constructor() {
        this.windowSnapshot = {};
        this.modifyPropsMap = {};
    }
    
    active() {
        // 激活时记录当前全局变量
        this.windowSnapshot = {};
        for (const key in window) {
            this.windowSnapshot[key] = window[key];
        }
        
        // 恢复之前修改的值
        Object.keys(this.modifyPropsMap).forEach(key => {
            window[key] = this.modifyPropsMap[key];
        });
    }
    
    inactive() {
        // 失活时记录修改的值
        for (const key in window) {
            if (window[key] !== this.windowSnapshot[key]) {
                this.modifyPropsMap[key] = window[key];
                window[key] = this.windowSnapshot[key];
            }
        }
    }
}
```

**2. Proxy Sandbox（代理沙箱）**

```javascript
// 原理：使用 Proxy 代理 window
function createProxySandbox() {
    const fakeWindow = {};
    
    return new Proxy(fakeWindow, {
        get(target, key) {
            if (key in target) {
                return target[key];
            }
            return window[key];
        },
        set(target, key, value) {
            target[key] = value;
            return true;
        }
    });
}
```

**问题 3：Module Federation 与 qiankun 有什么区别？**

**答：**

| 特性        | qiankun    | Module Federation |
|-----------|------------|-------------------|
| **实现层级**  | 运行时（应用级）   | 构建时（模块级）          |
| **技术栈要求** | 不限         | 需 Webpack 5       |
| **共享粒度**  | 应用级别       | 模块级别              |
| **共享依赖**  | 需手动配置      | 自动共享              |
| **加载方式**  | HTML Entry | JS Entry          |
| **适用场景**  | 异构技术栈      | 同构技术栈             |
| **生态**    | 国内主流       | 国外主流              |

**选择建议：**

- 技术栈统一 → Module Federation
- 技术栈异构 → qiankun
- 需要细粒度模块共享 → Module Federation

**问题 5：微前端中的公共依赖如何管理？**

**答：**

**方案对比：**

| 方案                    | 优点        | 缺点       |
|-----------------------|-----------|----------|
| **CDN 外链**            | 浏览器缓存、加载快 | 版本控制复杂   |
| **Module Federation** | 自动共享、版本控制 | 需统一构建工具  |
| **npm 依赖**            | 版本明确      | 重复打包、体积大 |
| **import-map**        | 灵活配置      | 需额外工具支持  |

**推荐方案：**

```javascript
// 1. 主应用加载公共依赖
// index.html
<script type="importmap">
{
    "imports": {
        "vue": "https://cdn.jsdelivr.net/npm/vue@3.3.0/dist/vue.esm-browser.js",
        "vue-router": "https://cdn.jsdelivr.net/npm/vue-router@4.2.0/dist/vue-router.esm-browser.js"
    }
}
</script>

// 2. 子应用使用 externals
// vite.config.js
export default {
    build: {
        rollupOptions: {
            external: ['vue', 'vue-router']
        }
    }
};
```
