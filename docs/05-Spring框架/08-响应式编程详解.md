## 一、响应式编程概述

### 1.1 什么是响应式编程

#### 定义与核心思想

**响应式编程（Reactive Programming）** 是一种面向**数据流**和**变化传播**的编程范式。它使用声明式的方式描述数据流的处理逻辑，当数据源发生变化时，变化会自动传播到所有依赖该数据的消费者。

```
┌─────────────────────────────────────────────────────────────────┐
│                    响应式编程核心思想                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   数据源 ──────► 数据流 ──────► 操作符链 ──────► 订阅者          │
│   (Publisher)    (Stream)      (Operators)      (Subscriber)   │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │  关键特性：                                              │  │
│   │  1. 声明式编程 - 描述"做什么"而非"怎么做"                │  │
│   │  2. 数据驱动   - 数据变化自动触发下游处理                 │  │
│   │  3. 组合性     - 通过操作符组合复杂逻辑                   │  │
│   │  4. 异步非阻塞 - 不阻塞线程等待结果                       │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 命令式 vs 响应式编程对比

```java
// ==================== 命令式编程（传统方式）====================
// 特点：主动拉取数据，阻塞等待结果
public User getUserAndOrders(String userId) {
    // 1. 阻塞等待用户数据
    User user = userRepository.findById(userId);  // 阻塞！线程等待
    
    // 2. 阻塞等待订单数据
    List<Order> orders = orderRepository.findByUserId(userId);  // 阻塞！
    
    // 3. 组装数据
    user.setOrders(orders);
    return user;
}
// 问题：每次数据库查询都阻塞线程，高并发时线程资源耗尽

// ==================== 响应式编程（Reactive）====================
// 特点：声明式描述数据流，非阻塞异步执行
public Mono<User> getUserAndOrdersReactive(String userId) {
    return userRepository.findById(userId)  // 返回 Mono<User>
        .flatMap(user -> {
            // 异步获取订单，不阻塞
            return orderRepository.findByUserId(userId)  // 返回 Flux<Order>
                .collectList()  // 将 Flux 转为 Mono<List>
                .map(orders -> {
                    user.setOrders(orders);
                    return user;
                });
        });
}
// 优势：线程不阻塞，可以处理更多并发请求
```

#### 核心特点详解

| 特性 | 说明 | 优势 |
|------|------|------|
| **异步非阻塞** | I/O 操作不阻塞线程，线程可以处理其他任务 | 提高资源利用率，支持高并发 |
| **事件驱动** | 基于事件和回调机制，数据到达时触发处理 | 响应及时，延迟低 |
| **声明式风格** | 描述数据处理流程，而非具体实现步骤 | 代码简洁，易于理解和维护 |
| **背压支持** | 消费者可以控制数据接收速率 | 防止生产者压垮消费者 |

### 1.2 为什么需要响应式编程

#### 传统阻塞 I/O 的痛点分析

```
┌─────────────────────────────────────────────────────────────────┐
│                  传统阻塞 I/O 线程模型                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   请求1 ──► [线程1] ──► 数据库查询 ──► 等待50ms ──► 返回结果    │
│   请求2 ──► [线程2] ──► 数据库查询 ──► 等待50ms ──► 返回结果    │
│   请求3 ──► [线程3] ──► 数据库查询 ──► 等待50ms ──► 返回结果    │
│   请求4 ──► [线程4] ──► 数据库查询 ──► 等待50ms ──► 返回结果    │
│   ...                                                           │
│   请求N ──► [线程N] ──► 数据库查询 ──► 等待50ms ──► 返回结果    │
│                                                                 │
│   ❌ 问题：                                                       │
│   - 1000并发 = 1000线程                                           │
│   - 每个线程栈内存 1MB = 1GB 内存                                 │
│   - 线程上下文切换开销大                                          │
│   - I/O 等待时线程空转，CPU 利用率低                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**具体场景示例：**

```java
// 假设：处理一个请求需要调用3个外部服务，每个服务耗时100ms

// 阻塞方式（Spring MVC）
@RestController
public class OrderController {
    
    @GetMapping("/order/{id}")
    public Order getOrder(@PathVariable String id) {
        // 串行执行，总耗时 = 100 + 100 + 100 = 300ms
        User user = userService.getUser(id);           // 阻塞100ms
        Order order = orderService.getOrder(id);       // 阻塞100ms
        Payment payment = paymentService.getPayment(id); // 阻塞100ms
        
        return combine(user, order, payment);
    }
}
// 1000并发需要1000个线程，每个线程占用1MB栈空间 = 1GB内存

// 响应式方式（Spring WebFlux）
@RestController
public class OrderControllerReactive {
    
    @GetMapping("/order/{id}")
    public Mono<Order> getOrder(@PathVariable String id) {
        // 并行执行，总耗时 ≈ 100ms（最慢的那个）
        Mono<User> userMono = userService.getUser(id);           // 异步
        Mono<Order> orderMono = orderService.getOrder(id);       // 异步
        Mono<Payment> paymentMono = paymentService.getPayment(id); // 异步
        
        // 等待所有结果返回后组合
        return Mono.zip(userMono, orderMono, paymentMono)
            .map(tuple -> combine(tuple.getT1(), tuple.getT2(), tuple.getT3()));
    }
}
// 少量线程（如CPU核心数）即可处理大量并发
```

#### 响应式编程解决方案

```
┌─────────────────────────────────────────────────────────────────┐
│                 响应式编程事件驱动模型                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   事件循环线程（Event Loop）                                      │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │  1. 接收请求1 ──► 发起数据库查询 ──► 注册回调 ──► 处理请求2 │  │
│   │  2. 接收请求2 ──► 发起HTTP调用   ──► 注册回调 ──► 处理请求3 │  │
│   │  3. 接收请求3 ──► 发起Redis查询  ──► 注册回调 ──► 检查事件  │  │
│   │  ...                                                        │  │
│   │  N. 数据库查询完成 ──► 执行回调 ──► 返回响应                │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   ✅ 优势：                                                       │
│   - 少量线程（如 CPU 核心数）处理大量并发                         │
│   - I/O 等待时不占用线程，线程处理其他请求                        │
│   - 无线程上下文切换开销                                          │
│   - 内存占用低（无大量线程栈）                                    │
│                                                                 │
│   📊 性能对比（理论值）：                                          │
│   - 阻塞模型：1000并发 = 1000线程 = 1GB内存                       │
│   - 响应式模型：1000并发 = 4线程 = 4MB内存                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 适用场景总结

| 场景 | 阻塞式（MVC） | 响应式（WebFlux） | 原因 |
|------|--------------|------------------|------|
| 高并发网关/代理 | ❌ 差 | ✅ 优 | 大量连接，少量计算 |
| 微服务聚合层 | ❌ 差 | ✅ 优 | 并行调用多个服务 |
| 实时数据推送 | ❌ 差 | ✅ 优 | SSE/WebSocket 支持 |
| 简单 CRUD | ✅ 优 | ❌ 差 | 引入复杂度无收益 |
| 计算密集型 | ✅ 优 | ❌ 差 | 阻塞线程更高效 |

### 1.3 Reactive Streams 规范详解

#### 什么是 Reactive Streams

**Reactive Streams** 是 JVM 平台上响应式流的**标准化规范**（Java 9 纳入标准库 `java.util.concurrent.Flow`），定义了异步流处理的标准接口和交互规则。

```
┌─────────────────────────────────────────────────────────────────┐
│              Reactive Streams 核心组件关系图                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────┐         subscribe()         ┌─────────────┐  │
│   │  Publisher  │ ───────────────────────────► │  Subscriber │  │
│   │   (发布者)   │                              │   (订阅者)   │  │
│   └─────────────┘                              └──────┬──────┘  │
│                                                        │        │
│                               ┌───────────────────────┘        │
│                               │ onSubscribe(Subscription)        │
│                               ▼                                │
│                        ┌─────────────┐                         │
│                        │ Subscription│ ◄──────────────────────  │
│                        │   (订阅关系) │   request(n) / cancel()  │
│                        └─────────────┘                         │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │  Processor = Subscriber + Publisher（中间处理节点）       │  │
│   │                                                         │  │
│   │   Publisher ──► Processor ──► Processor ──► Subscriber  │  │
│   │   (数据源)      (转换)        (过滤)        (消费者)      │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 四大核心接口详解

```java
/**
 * 1. Publisher（发布者）
 * 职责：生产数据，管理订阅关系
 * 特点：可以订阅多个 Subscriber，但一个 Subscriber 只能订阅一个 Publisher
 */
public interface Publisher<T> {
    /**
     * 订阅方法：Subscriber 调用此方法来订阅数据流
     * 规则：
     * 1. 必须同步调用 onSubscribe，然后才能发送数据
     * 2. 同一个 Subscriber 只能订阅一次
     * 3. 必须处理 null 参数（抛出 NullPointerException）
     */
    void subscribe(Subscriber<? super T> s);
}

/**
 * 2. Subscriber（订阅者）
 * 职责：消费数据，处理背压
 * 调用顺序：onSubscribe → [onNext]* → (onError | onComplete)
 */
public interface Subscriber<T> {
    /**
     * 订阅建立时调用，接收 Subscription 用于控制背压
     * 必须在此方法中调用 subscription.request() 才能接收数据
     */
    void onSubscribe(Subscription s);
    
    /**
     * 接收到数据时调用
     * 规则：
     * 1. Subscription 必须按顺序调用，不能并发
     * 2. 参数不能为 null
     */
    void onNext(T t);
    
    /**
     * 发生错误时调用，流终止
     * 调用后不会再收到 onNext 或 onComplete
     */
    void onError(Throwable t);
    
    /**
     * 数据流正常完成时调用
     * 调用后不会再收到 onNext 或 onError
     */
    void onComplete();
}

/**
 * 3. Subscription（订阅关系）
 * 职责：连接 Publisher 和 Subscriber，控制背压
 */
public interface Subscription {
    /**
     * 请求 n 个元素（背压控制核心）
     * 规则：
     * 1. n > 0，否则 IllegalArgumentException
     * 2. 可以多次调用累加请求数量
     * 3. 如果 Publisher 无法减速，可以忽略此请求
     */
    void request(long n);
    
    /**
     * 取消订阅，释放资源
     * 调用后：
     * 1. Publisher 停止发送数据
     * 2. 最终调用 onComplete 或 onError（可选）
     */
    void cancel();
}

/**
 * 4. Processor（处理器）
 * 职责：既是 Subscriber 又是 Publisher，用于数据转换
 * 典型用途：map、filter、flatMap 等操作符的实现基础
 */
public interface Processor<T, R> extends Subscriber<T>, Publisher<R> {
    // 继承两个接口的方法，实现数据转换逻辑
}
```

#### 完整工作流程示例

```java
/**
 * 自定义 Publisher 实现，展示 Reactive Streams 完整工作流程
 */
public class SimplePublisher implements Publisher<String> {
    
    private final List<String> data = Arrays.asList("A", "B", "C", "D", "E");
    
    @Override
    public void subscribe(Subscriber<? super String> subscriber) {
        // 1. 创建 Subscription
        Subscription subscription = new Subscription() {
            private int index = 0;
            private boolean cancelled = false;
            private long requested = 0;
            
            @Override
            public void request(long n) {
                if (cancelled) return;
                
                // 累加请求数量
                requested += n;
                
                // 发送数据
                while (requested > 0 && index < data.size() && !cancelled) {
                    subscriber.onNext(data.get(index++));
                    requested--;
                }
                
                // 数据发送完毕
                if (index >= data.size() && !cancelled) {
                    subscriber.onComplete();
                }
            }
            
            @Override
            public void cancel() {
                cancelled = true;
            }
        };
        
        // 2. 通知 Subscriber 订阅建立
        subscriber.onSubscribe(subscription);
    }
    
    public static void main(String[] args) {
        SimplePublisher publisher = new SimplePublisher();
        
        // 3. 创建 Subscriber 并订阅
        publisher.subscribe(new Subscriber<String>() {
            private Subscription subscription;
            private int count = 0;
            
            @Override
            public void onSubscribe(Subscription s) {
                this.subscription = s;
                System.out.println("[Subscriber] 订阅建立");
                // 请求 2 个元素（背压控制）
                s.request(2);
            }
            
            @Override
            public void onNext(String s) {
                System.out.println("[Subscriber] 收到数据: " + s);
                count++;
                // 每收到2个数据，再请求2个
                if (count % 2 == 0) {
                    subscription.request(2);
                }
            }
            
            @Override
            public void onError(Throwable t) {
                System.err.println("[Subscriber] 错误: " + t.getMessage());
            }
            
            @Override
            public void onComplete() {
                System.out.println("[Subscriber] 数据流完成");
            }
        });
    }
}

// 输出：
// [Subscriber] 订阅建立
// [Subscriber] 收到数据: A
// [Subscriber] 收到数据: B
// [Subscriber] 收到数据: C
// [Subscriber] 收到数据: D
// [Subscriber] 收到数据: E
// [Subscriber] 数据流完成
```

#### 规范核心规则总结

| 规则类别 | 规则说明 | 违反后果 |
|----------|----------|----------|
| **发送顺序** | `onSubscribe` → `onNext`* → (`onError` \| `onComplete`) | 违反时行为未定义 |
| **信号互斥** | `onError` 和 `onComplete` 只能发生一个 | 违反时行为未定义 |
| **非空约束** | `onNext`、`onError`、`onSubscribe` 参数不能为 null | NullPointerException |
| **背压控制** | Subscriber 必须通过 `request(n)` 请求数据 | 可能 OOM 或数据丢失 |
| **取消响应** | `cancel()` 后 Publisher 应尽快停止发送 | 资源泄漏 |
| **线程安全** | `onNext` 调用必须按顺序，不能并发 | 数据竞争 |

## 二、Reactor 模式深入

### 2.1 Reactor 设计模式详解

#### 什么是 Reactor 模式

**Reactor 模式**是一种**事件驱动**的设计模式，用于处理多个并发 I/O 操作。它将 I/O 事件的**多路分离**和**事件处理**解耦，使用少量线程处理大量并发连接。

```
┌─────────────────────────────────────────────────────────────────┐
│                    Reactor 模式核心架构                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                    Reactor（事件分发器）                  │  │
│   │  ┌─────────────────────────────────────────────────┐   │  │
│   │  │  Selector（多路复用器）                          │   │  │
│   │  │  - 监听多个 Channel 的 I/O 事件                  │   │  │
│   │  │  - 阻塞等待就绪事件（select/poll/epoll）         │   │  │
│   │  └─────────────────────────────────────────────────┘   │  │
│   │                          │                              │  │
│   │                          ▼                              │  │
│   │  ┌─────────────────────────────────────────────────┐   │  │
│   │  │  Dispatcher（事件分发器）                        │   │  │
│   │  │  - 根据事件类型分发到对应 Handler                │   │  │
│   │  │  - ACCEPT 事件 → Acceptor                       │   │  │
│   │  │  - READ 事件   → Handler                        │   │  │
│   │  └─────────────────────────────────────────────────┘   │  │
│   └─────────────────────────────────────────────────────────┘  │
│                              │                                  │
│           ┌──────────────────┼──────────────────┐              │
│           ▼                  ▼                  ▼              │
│   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐         │
│   │  Acceptor   │   │  Handler 1  │   │  Handler N  │         │
│   │  (接受连接)  │   │ (处理业务)  │   │ (处理业务)  │         │
│   └─────────────┘   └─────────────┘   └─────────────┘         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 核心组件详解

| 组件 | 职责 | 类比 |
|------|------|------|
| **Handle（句柄）** | 标识 I/O 资源（如 Socket 描述符） | 文件描述符 FD |
| **Synchronous Event Demultiplexer** | 阻塞等待事件就绪（select/poll/epoll） | 多路复用器 |
| **Initiation Dispatcher** | 注册/移除事件处理器，分发事件 | 事件总线 |
| **Event Handler** | 处理具体事件（连接、读、写） | 回调函数 |
| **Concrete Event Handler** | 业务逻辑实现 | 业务代码 |

#### Reactor 模式工作流程

```
┌─────────────────────────────────────────────────────────────────┐
│                  Reactor 模式工作流程                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   阶段1: 初始化                                                   │
│   ┌─────────┐                                                   │
│   │  启动   │ ──► 创建 Reactor ──► 创建 Acceptor ──► 注册监听    │
│   └─────────┘                                                   │
│                                                                 │
│   阶段2: 事件循环（Event Loop）                                   │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │  while (running) {                                      │  │
│   │      1. selector.select()  // 阻塞等待就绪事件           │  │
│   │      2. 获取就绪的 Channel 集合                          │  │
│   │      3. for each channel:                               │  │
│   │         - 如果是 ACCEPT 事件 → Acceptor.handle()        │  │
│   │         - 如果是 READ 事件   → Handler.handle()         │  │
│   │         - 如果是 WRITE 事件  → Handler.handle()         │  │
│   │  }                                                      │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   阶段3: 处理 ACCEPT 事件                                         │
│   ┌─────────────┐                                               │
│   │ Acceptor    │ ──► 接受新连接 ──► 创建新 Handler            │
│   │ .handle()   │ ──► 注册 READ 事件到 Reactor                │
│   └─────────────┘                                               │
│                                                                 │
│   阶段4: 处理 READ 事件                                           │
│   ┌─────────────┐                                               │
│   │ Handler     │ ──► 读取数据 ──► 业务处理 ──► 注册 WRITE     │
│   │ .handle()   │                                               │
│   └─────────────┘                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 单线程 vs 多线程 Reactor

```
┌─────────────────────────────────────────────────────────────────┐
│  单线程 Reactor（Redis、Node.js）                                │
├─────────────────────────────────────────────────────────────────┤

│                                                                 │

│   缺点：业务逻辑不能阻塞，否则影响整体性能                          │

│   Main Reactor（主线程）                                          │
│   ┌─────────────┐                                               │
│   │   Acceptor  │ ──► 只处理 ACCEPT 事件（建立连接）            │
│   │   线程      │ ──► 将连接分配给 Sub Reactor                  │
│   └─────────────┘                                               │
│                                                                 │
│   Sub Reactor Pool（工作线程池）                                  │
│   ┌─────────┐ ┌─────────┐ ┌─────────┐                          │
│   │Thread 1 │ │Thread 2 │ │Thread N │                          │
│   │Reactor  │ │Reactor  │ │Reactor  │                          │
│   │处理连接1│ │处理连接2│ │处理连接N│                          │
│   └─────────┘ └─────────┘ └─────────┘                          │
│                                                                 │
│   Worker Pool（业务线程池，可选）                                  │
│   ┌─────────┐ ┌─────────┐ ┌─────────┐                          │
│   │Worker 1 │ │Worker 2 │ │Worker N │                          │
│   │业务处理 │ │业务处理 │ │业务处理 │                          │
│   └─────────┘ └─────────┘ └─────────┘                          │
│                                                                 │
│   优点：充分利用多核，业务逻辑可阻塞                               │
│   缺点：实现复杂，需要线程同步                                     │
│   适用：高并发、业务逻辑复杂的场景                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```java
/**
 * Netty 主从多线程 Reactor 模式实现
 */
public class NettyReactorDemo {
    
    public static void main(String[] args) {
        // 主 Reactor：处理连接建立（ACCEPT 事件）
        EventLoopGroup bossGroup = new NioEventLoopGroup(1);
        
        // 从 Reactor：处理 I/O 读写（READ/WRITE 事件）
        EventLoopGroup workerGroup = new NioEventLoopGroup();
        
        try {
            ServerBootstrap bootstrap = new ServerBootstrap();
            bootstrap.group(bossGroup, workerGroup)
                .channel(NioServerSocketChannel.class)
                .childHandler(new ChannelInitializer<SocketChannel>() {
                    @Override
                    protected void initChannel(SocketChannel ch) {
                        ch.pipeline().addLast(
                            // 解码器：处理 READ 事件
                            new StringDecoder(),
                            // 业务 Handler：处理业务逻辑
                            new SimpleChannelInboundHandler<String>() {
                                @Override
                                protected void channelRead0(ChannelHandlerContext ctx, String msg) {
                                    // 业务处理
                                    System.out.println("收到消息: " + msg);
                                    // 发送响应（WRITE 事件）
                                    ctx.writeAndFlush("响应: " + msg);
                                }
                            }
                        );
                    }
                });
            
            ChannelFuture future = bootstrap.bind(8080).sync();
            future.channel().closeFuture().sync();
        } finally {
            bossGroup.shutdownGracefully();
            workerGroup.shutdownGracefully();
        }
    }
}
```

```java
/**
 * Mono 创建方式详解
 */
public class MonoCreationDemo {
    
    public static void main(String[] args) {
        // ==================== 1. just：直接创建 ====================
        // 适用于：已有数据，包装为 Mono
        Mono<String> mono1 = Mono.just("Hello World");
        mono1.subscribe(System.out::println);  // 输出: Hello World
        
        // ⚠️ 注意：just() 会立即捕获值，不要用于异步场景
        // 错误示例：Mono.just(getDataFromDB()); // 这会在创建时阻塞！
        
        // ==================== 2. fromCallable：延迟创建 ====================
        // 适用于：延迟执行，每次订阅都重新执行
        Mono<String> mono2 = Mono.fromCallable(() -> {
            System.out.println("执行数据库查询...");
            return fetchFromDatabase();
        });
        // 此时不会执行，订阅时才执行
        mono2.subscribe(System.out::println);
        
        // ==================== 3. fromSupplier：无参供应商 ====================
        Mono<Double> mono3 = Mono.fromSupplier(() -> Math.random());
        mono3.subscribe(System.out::println);
        
        // ==================== 4. fromFuture：包装 CompletableFuture ====================
        CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
            return "Async Result";
        });
        Mono<String> mono4 = Mono.fromFuture(future);
        mono4.subscribe(System.out::println);
        
        // ==================== 5. empty：空 Mono ====================
        // 适用于：表示"无结果"的操作
        Mono<String> mono5 = Mono.empty();
        mono5.subscribe(
            data -> System.out.println("数据: " + data),
            error -> System.out.println("错误: " + error),
            () -> System.out.println("完成（无数据）")  // 会执行这个
        );
        
        // ==================== 6. error：错误 Mono ====================
        // 适用于：表示操作失败
        Mono<String> mono6 = Mono.error(new RuntimeException("操作失败"));
        mono6.subscribe(
            data -> System.out.println("数据: " + data),
            error -> System.out.println("捕获错误: " + error.getMessage()),
            () -> System.out.println("完成")
        );
        
        // ==================== 7. defer：延迟创建 ====================
        // 每次订阅都重新创建 Mono（与 just 的区别）
        Mono<Long> mono7 = Mono.defer(() -> Mono.just(System.currentTimeMillis()));
        
        mono7.subscribe(t -> System.out.println("第一次订阅: " + t));
        Thread.sleep(100);
        mono7.subscribe(t -> System.out.println("第二次订阅: " + t)); // 时间不同！
        
        // ==================== 8. 组合多个 Mono ====================
        Mono<String> monoA = Mono.just("A");
        Mono<String> monoB = Mono.just("B");
        
        // zip：组合两个 Mono 的结果
        Mono<String> combined = Mono.zip(monoA, monoB, (a, b) -> a + "-" + b);
        combined.subscribe(System.out::println);  // 输出: A-B
        
        // zipWhen：一个 Mono 依赖另一个的结果
        Mono<String> result = monoA.zipWhen(a -> Mono.just(a + "-extra"));
        result.subscribe(System.out::println);  // 输出: [A, A-extra]
    }
    
    private static String fetchFromDatabase() {
        return "Database Data";
    }
}
```

```java
/**

public class FluxCreationDemo {
    
    public static void main(String[] args) throws InterruptedException {
        // ==================== 1. just：直接创建（固定元素）====================
        Flux<String> flux1 = Flux.just("A", "B", "C", "D");
        flux1.subscribe(System.out::println);  // A B C D
        
        // ==================== 2. fromIterable：从集合创建 ====================
        List<String> list = Arrays.asList("X", "Y", "Z");
        Flux<String> flux2 = Flux.fromIterable(list);
        flux2.subscribe(System.out::println);  // X Y Z
        
        // ==================== 3. fromArray：从数组创建 ====================
        Flux<String> flux3 = Flux.fromArray(new String[]{"1", "2", "3"});
        flux3.subscribe(System.out::println);  // 1 2 3
        
        // ==================== 4. range：范围创建 ====================
        Flux<Integer> flux4 = Flux.range(1, 5);  // 1, 2, 3, 4, 5
        flux4.subscribe(System.out::println);
        
        // ==================== 5. interval：定时发射 ====================
        // 每 100ms 发射一个递增的数字
        Flux<Long> flux5 = Flux.interval(Duration.ofMillis(100)).take(5);
        flux5.subscribe(n -> System.out.println("间隔: " + n));
        Thread.sleep(1000);  // 等待发射完成
        
        // ==================== 6. generate：同步生成 ====================
        // 适用于：基于状态的同步生成
        Flux<String> flux6 = Flux.generate(
            () -> 0,  // 初始状态
            (state, sink) -> {
                sink.next("Item-" + state);  // 发射数据
                if (state == 5) {
                    sink.complete();  // 完成
                }
                return state + 1;  // 返回新状态
            }
        );
        flux6.subscribe(System.out::println);  // Item-0 到 Item-5
        
        // ==================== 7. create：异步生成 ====================
        // 适用于：基于回调的异步数据源
        Flux<String> flux7 = Flux.create(sink -> {
            // 模拟异步回调
            new Thread(() -> {
                for (int i = 0; i < 3; i++) {
                    sink.next("Async-" + i);
                    try {
                        Thread.sleep(100);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                }
                sink.complete();
            }).start();
        });
        flux7.subscribe(System.out::println);
        Thread.sleep(500);
        
        // ==================== 8. push：单线程异步生成 ====================
        // 与 create 类似，但只允许一个线程调用 next
        Flux<String> flux8 = Flux.push(sink -> {
            sink.next("Push-1");
            sink.next("Push-2");
            sink.complete();
        });
        flux8.subscribe(System.out::println);
        
        // ==================== 9. empty 和 error ====================
        Flux<String> emptyFlux = Flux.empty();  // 直接完成
        Flux<String> errorFlux = Flux.error(new RuntimeException("错误"));
        
        // ==================== 10. concat：顺序拼接 ====================
        Flux<String> fluxA = Flux.just("A1", "A2");
        Flux<String> fluxB = Flux.just("B1", "B2");
        Flux<String> concatenated = Flux.concat(fluxA, fluxB);
        concatenated.subscribe(System.out::println);  // A1, A2, B1, B2
        
        // ==================== 11. merge：合并（交错）====================
        Flux<String> merged = Flux.merge(fluxA, fluxB);
        // 输出顺序取决于发射速度，可能交错
        
        // ==================== 12. zip：成对组合 ====================
        Flux<String> names = Flux.just("Alice", "Bob", "Charlie");
        Flux<Integer> ages = Flux.just(25, 30, 35);
        Flux<String> zipped = Flux.zip(names, ages, (name, age) -> name + "(" + age + ")");
        zipped.subscribe(System.out::println);  // Alice(25), Bob(30), Charlie(35)
    }
}
```

```java
/**
 * 转换操作符详解

public class TransformOperatorsDemo {
    
    public static void main(String[] args) {
        // ==================== map：同步一对一转换 ====================
        // 特点：同步转换，保持顺序
        Flux<String> names = Flux.just("alice", "bob", "charlie");
        
        Flux<String> upperNames = names.map(String::toUpperCase);
        upperNames.subscribe(System.out::println);  // ALICE, BOB, CHARLIE
        
        // 实际应用：DTO 转换
        Flux<UserDTO> userDTOs = names.map(name -> {
            UserDTO dto = new UserDTO();
            dto.setName(name.toUpperCase());
            return dto;
        });
        
        // ==================== flatMap：异步一对多转换（扁平化）====================
        // 特点：
        // 1. 返回 Publisher（Mono/Flux）
        // 2. 自动扁平化（将嵌套的 Publisher 展开）
        // 3. 可能乱序（并行时）
        // 4. 适用于：涉及异步操作的转换
        
        Flux<String> words = Flux.just("hello", "world");
        
        // 将每个单词拆分为字符
        Flux<String> chars = words.flatMap(word -> 
            Flux.fromArray(word.split(""))
        );
        chars.subscribe(System.out::println);  // h, e, l, l, o, w, o, r, l, d
        
        // 实际应用：并行查询
        Flux<Long> userIds = Flux.just(1L, 2L, 3L);
        Flux<User> users = userIds.flatMap(id -> 
            userRepository.findById(id)  // 返回 Mono<User>
                .subscribeOn(Schedulers.boundedElastic()),
            3  // 并发度：同时处理3个
        );
        
        // ==================== concatMap：有序扁平化 ====================
        // 特点：与 flatMap 类似，但保证顺序
        // 适用：需要保持顺序的场景
        Flux<String> orderedChars = words.concatMap(word -> 
            Flux.fromArray(word.split(""))
        );
        // 输出顺序一定与输入顺序一致
        
        // ==================== switchMap：切换流 ====================
        // 特点：只保留最新的流，取消之前的
        // 适用：搜索框自动补全（只关心最新输入）
        Flux<String> searchInputs = Flux.just("a", "ab", "abc").delayElements(Duration.ofMillis(100));
        
        Flux<List<String>> searchResults = searchInputs.switchMap(keyword -> 
            searchService.search(keyword)  // 如果 "ab" 的结果比 "abc" 慢，会被取消
        );
        
        // ==================== flatMapMany：Mono 转 Flux ====================
        Mono<String> mono = Mono.just("hello");
        Flux<String> flux = mono.flatMapMany(word -> 
            Flux.fromArray(word.split(""))
        );
        flux.subscribe(System.out::println);  // h, e, l, l, o
        
        // ==================== cast：类型转换 ====================
        Flux<Object> objects = Flux.just("A", "B", "C");
        Flux<String> strings = objects.cast(String.class);
        
        // ==================== ofType：过滤并转换类型 ====================
        Flux<Object> mixed = Flux.just("string", 123, "another", 456);
        Flux<String> onlyStrings = mixed.ofType(String.class);
        onlyStrings.subscribe(System.out::println);  // string, another
    }
    
    // 模拟 Repository
    static class UserRepository {
        Mono<User> findById(Long id) {
            return Mono.just(new User(id, "User-" + id));
        }
    }
    
    static class SearchService {
        Flux<List<String>> search(String keyword) {
            return Flux.just(Arrays.asList(keyword + "-result1", keyword + "-result2"));
        }
    }
    
    static class User {
        Long id;
        String name;
        User(Long id, String name) { this.id = id; this.name = name; }
    }
    
    static class UserDTO {
        private String name;
        public void setName(String name) { this.name = name; }
    }
}
```

```java
/**
 * 过滤操作符详解

public class FilterOperatorsDemo {
    
    public static void main(String[] args) {
        // ==================== filter：条件过滤 ====================
        Flux<Integer> numbers = Flux.range(1, 10);
        
        Flux<Integer> evenNumbers = numbers.filter(n -> n % 2 == 0);
        evenNumbers.subscribe(System.out::println);  // 2, 4, 6, 8, 10
        
        // 实际应用：过滤无效数据
        Flux<User> users = getUsers()
            .filter(user -> user.getAge() >= 18)  // 只保留成年人
            .filter(user -> user.getEmail() != null);  // 只保留有邮箱的
        
        // ==================== filterWhen：异步条件过滤 ====================
        // 适用于：过滤条件需要异步判断
        Flux<User> activeUsers = getUsers()
            .filterWhen(user -> userService.isActive(user.getId()));  // 返回 Mono<Boolean>
        
        // ==================== take：取前 N 个 ====================
        Flux<Integer> first5 = numbers.take(5);
        first5.subscribe(System.out::println);  // 1, 2, 3, 4, 5
        
        // takeWhile：满足条件时继续取
        Flux<Integer> whileLessThan5 = numbers.takeWhile(n -> n < 5);
        whileLessThan5.subscribe(System.out::println);  // 1, 2, 3, 4
        
        // takeUntil：直到满足条件（包含该元素）
        Flux<Integer> until5 = numbers.takeUntil(n -> n == 5);
        until5.subscribe(System.out::println);  // 1, 2, 3, 4, 5
        
        // ==================== skip：跳过前 N 个 ====================
        Flux<Integer> skip5 = numbers.skip(5);
        skip5.subscribe(System.out::println);  // 6, 7, 8, 9, 10
        
        // skipWhile：满足条件时跳过
        Flux<Integer> skipWhileLessThan5 = numbers.skipWhile(n -> n < 5);
        skipWhileLessThan5.subscribe(System.out::println);  // 5, 6, 7, 8, 9, 10
        
        // ==================== distinct：去重 ====================
        Flux<String> withDuplicates = Flux.just("A", "B", "A", "C", "B", "D");
        Flux<String> distinct = withDuplicates.distinct();
        distinct.subscribe(System.out::println);  // A, B, C, D
        
        // distinctUntilChanged：去重连续重复
        Flux<String> continuous = Flux.just("A", "A", "B", "B", "B", "C");
        Flux<String> distinctContinuous = continuous.distinctUntilChanged();
        distinctContinuous.subscribe(System.out::println);  // A, B, C
        
        // distinct(Function)：根据属性去重
        Flux<User> usersWithSameName = Flux.just(
            new User(1L, "Alice"),
            new User(2L, "Bob"),
            new User(3L, "Alice")  // 重复名字
        );
        Flux<User> distinctByName = usersWithSameName.distinct(User::getName);
        
        // ==================== elementAt：获取指定位置 ====================
        Mono<Integer> third = numbers.elementAt(2);  // 索引从0开始，获取第3个
        third.subscribe(System.out::println);  // 3
        
        // ==================== single：确保只有一个元素 ====================

        single.subscribe(System.out::println);  // only

        // Flux.empty().single()  // 报错：NoSuchElementException

        // ==================== ignoreElements：忽略所有元素 ====================

        completeSignal.subscribe(
            null,
            null,
            () -> System.out.println("只关心完成信号")

        // ==================== sample：采样 ====================
        // 每 100ms 采样一次
```

```java
public class CombineOperatorsDemo {
    
    public static void main(String[] args) throws InterruptedException {
        // ==================== merge：合并多个流（交错）====================
        // 特点：按时间顺序交错发射，不保证顺序
        Flux<String> stream1 = Flux.interval(Duration.ofMillis(100))
            .map(i -> "A" + i)
            .take(3);
        Flux<String> stream2 = Flux.interval(Duration.ofMillis(150))
            .map(i -> "B" + i)
            .take(3);
        
        Flux<String> merged = Flux.merge(stream1, stream2);
        merged.subscribe(System.out::println);
        Thread.sleep(1000);
        // 可能输出：A0, B0, A1, A2, B1, B2（取决于时间）
        
        // ==================== concat：顺序拼接 ====================
        // 特点：按顺序一个接一个，先等前一个完成
        Flux<String> concat = Flux.concat(stream1, stream2);
        concat.subscribe(System.out::println);
        Thread.sleep(1000);
        // 输出：A0, A1, A2, B0, B1, B2
        
        // ==================== zip：成对组合 ====================
        // 特点：将多个流的对应位置元素组合
        Flux<String> names = Flux.just("Alice", "Bob", "Charlie");
        Flux<Integer> ages = Flux.just(25, 30, 35);
        Flux<String> cities = Flux.just("NY", "LA", "SF");
        
        // 组合两个流
        Flux<String> zipped2 = Flux.zip(names, ages, (name, age) -> name + "(" + age + ")");
        zipped2.subscribe(System.out::println);  // Alice(25), Bob(30), Charlie(35)
        
        // 组合三个流
        Flux<String> zipped3 = Flux.zip(names, ages, cities)
            .map(tuple -> tuple.getT1() + ", " + tuple.getT2() + ", " + tuple.getT3());
        zipped3.subscribe(System.out::println);  // Alice, 25, NY
        
        // ⚠️ zip 以最短流为准
        Flux<String> shortStream = Flux.just("X", "Y");
        Flux<String> longStream = Flux.just("A", "B", "C");
        Flux<String> zippedShort = Flux.zip(shortStream, longStream, (a, b) -> a + b);
        zippedShort.subscribe(System.out::println);  // XA, YB（C被丢弃）
        
        // ==================== zipWith：两两组合 ====================
        Mono<String> mono1 = Mono.just("Hello");
        Mono<String> mono2 = Mono.just("World");
        
        Mono<String> combined = mono1.zipWith(mono2, (a, b) -> a + " " + b);
        combined.subscribe(System.out::println);  // Hello World
        
        // ==================== combineLatest：最新值组合 ====================
        // 当任一流有新值时，用该流最新值与其他流最新值组合
        Flux<String> fast = Flux.interval(Duration.ofMillis(100))
            .map(i -> "Fast" + i)
            .take(5);
        Flux<String> slow = Flux.interval(Duration.ofMillis(300))
            .map(i -> "Slow" + i)
            .take(2);
        
        Flux<String> combinedLatest = Flux.combineLatest(fast, slow, (f, s) -> f + "-" + s);
        combinedLatest.subscribe(System.out::println);
        Thread.sleep(1000);
        // 输出示例：Fast0-Slow0, Fast1-Slow0, Fast2-Slow0, Fast3-Slow1, Fast4-Slow1
        
        // ==================== startWith：在前面添加元素 ====================
        Flux<String> main = Flux.just("B", "C", "D");
        Flux<String> withPrefix = main.startWith("A");
        withPrefix.subscribe(System.out::println);  // A, B, C, D
        
        // 也可以添加另一个流
        Flux<String> withStreamPrefix = main.startWith(Flux.just("X", "Y"));
        withStreamPrefix.subscribe(System.out::println);  // X, Y, B, C, D
        
        // ==================== defaultIfEmpty：默认值 ====================
        Flux<String> empty = Flux.empty();
        Flux<String> withDefault = empty.defaultIfEmpty("default");
        withDefault.subscribe(System.out::println);  // default
        
        Mono<String> emptyMono = Mono.empty();
        Mono<String> monoWithDefault = emptyMono.defaultIfEmpty("default");
        monoWithDefault.subscribe(System.out::println);  // default
        
        // ==================== switchIfEmpty：备用流 ====================
        Flux<String> primary = Flux.empty();
        Flux<String> backup = Flux.just("backup1", "backup2");
        Flux<String> switched = primary.switchIfEmpty(backup);
        switched.subscribe(System.out::println);  // backup1, backup2
        
        // ==================== collect：收集为集合 ====================
        Flux<Integer> numbers = Flux.range(1, 5);
        
        Mono<List<Integer>> listMono = numbers.collectList();
        listMono.subscribe(list -> System.out.println("List: " + list));  // [1, 2, 3, 4, 5]
        
        Mono<Set<Integer>> setMono = numbers.collectSortedSet();
        setMono.subscribe(set -> System.out.println("Set: " + set));
        
        Mono<Map<String, Integer>> mapMono = Flux.just("a", "bb", "ccc")
            .collectMap(String::toUpperCase, String::length);
        mapMono.subscribe(map -> System.out.println("Map: " + map));  // {A=1, BB=2, CCC=3}
        
        // ==================== reduce：归约 ====================
        Mono<Integer> sum = numbers.reduce((a, b) -> a + b);
        sum.subscribe(System.out::println);  // 15 (1+2+3+4+5)

        sumWithInitial.subscribe(System.out::println);  // 25 (10+1+2+3+4+5)

        Flux<Integer> runningSum = numbers.scan((a, b) -> a + b);
```

```java
/**
 * 错误处理详解

public class ErrorHandlingDemo {
    
    public static void main(String[] args) {
        // ==================== onErrorReturn：返回默认值 ====================
        // 发生错误时，用默认值替代，流继续完成
        Flux<Integer> withDefault = Flux.just(1, 2, 0, 4)
            .map(i -> 10 / i)  // 除以0会报错
            .onErrorReturn(0);  // 错误时返回0
        
        withDefault.subscribe(
            data -> System.out.println("数据: " + data),
            error -> System.out.println("错误: " + error),
            () -> System.out.println("完成")
        );
        // 输出：数据: 10, 数据: 5, 数据: 0, 完成
        
        // 条件性返回默认值
        Flux<Integer> conditionalDefault = Flux.just(1, 2, 0, 4)
            .map(i -> 10 / i)
            .onErrorReturn(
                e -> e instanceof ArithmeticException,  // 只处理特定异常
                -1
            );
        
        // ==================== onErrorResume：备用流 ====================
        // 发生错误时，切换到备用流继续
        Flux<String> primary = Flux.just("A", "B")
            .concatWith(Flux.error(new RuntimeException("出错")))
            .concatWith(Flux.just("C", "D"));  // 这不会执行
        
        Flux<String> withBackup = primary
            .onErrorResume(e -> Flux.just("fallback1", "fallback2"));
        
        withBackup.subscribe(System.out::println);  // A, B, fallback1, fallback2
        
        // 条件性备用流
        Flux<String> conditionalResume = primary
            .onErrorResume(
                e -> e instanceof RuntimeException,
                e -> Flux.just("runtime-error-fallback")
            );
        
        // ==================== onErrorContinue：跳过错误继续 ====================
        // 发生错误时，跳过该元素，继续处理后续元素
        Flux<Integer> continueOnError = Flux.just(1, 2, 0, 3, 4)
            .map(i -> 10 / i)
            .onErrorContinue((error, obj) -> {
                System.out.println("跳过错误元素: " + obj + ", 错误: " + error.getMessage());
            });
        
        continueOnError.subscribe(System.out::println);
        // 输出：
        // 10
        // 5
        // 跳过错误元素: 0, 错误: / by zero
        // 3
        // 2
        
        // ==================== onErrorMap：转换错误类型 ====================
        // 将一种异常转换为另一种
        Mono<String> mappedError = Mono.error(new SQLException("数据库错误"))
            .onErrorMap(e -> new BusinessException("业务处理失败", e));
        
        mappedError.subscribe(
            null,
            error -> System.out.println("转换后的错误: " + error.getClass().getSimpleName())
        );
        // 输出：转换后的错误: BusinessException
        
        // ==================== doOnError：副作用处理 ====================
        // 发生错误时执行操作（如记录日志），但不改变错误传播
        Flux<Integer> withLogging = Flux.just(1, 2, 0)
            .map(i -> 10 / i)
            .doOnError(e -> System.err.println("[ERROR] 发生错误: " + e.getMessage()));
        
        withLogging.subscribe(
            data -> System.out.println("数据: " + data),
            error -> System.out.println("最终错误: " + error.getMessage())
        );
        // 输出：
        // 数据: 10
        // 数据: 5
        // [ERROR] 发生错误: / by zero
        // 最终错误: / by zero
        
        // ==================== retry：重试 ====================
        // 发生错误时重新订阅，重试指定次数
        AtomicInteger counter = new AtomicInteger(0);
        
        Mono<String> withRetry = Mono.fromCallable(() -> {
            int count = counter.incrementAndGet();
            System.out.println("尝试第 " + count + " 次");
            if (count < 3) {
                throw new RuntimeException("暂时失败");
            }
            return "成功";
        }).retry(3);  // 最多重试3次
        
        withRetry.subscribe(System.out::println);
        // 输出：
        // 尝试第 1 次
        // 尝试第 2 次
        // 尝试第 3 次
        // 成功
        
        // ==================== retryWhen：自定义重试策略 ====================
        // 使用 Retry 类定义更复杂的重试策略
        Mono<String> withRetryWhen = Mono.fromCallable(() -> {
            if (Math.random() > 0.5) {
                throw new RuntimeException("随机失败");
            }
            return "成功";
        }).retryWhen(Retry.backoff(3, Duration.ofSeconds(1))  // 最多3次，指数退避
            .maxBackoff(Duration.ofSeconds(5))  // 最大间隔5秒
            .filter(e -> e instanceof RuntimeException)  // 只重试特定异常
            .doBeforeRetry(signal -> System.out.println("准备重试..."))
        );
        
        // ==================== timeout：超时处理 ====================
        Mono<String> slowMono = Mono.delay(Duration.ofSeconds(5))
            .map(l -> "结果");
        
        Mono<String> withTimeout = slowMono
            .timeout(Duration.ofSeconds(2))  // 2秒超时
            .onErrorReturn("超时默认值");
        
        withTimeout.subscribe(System.out::println);  // 超时默认值
        
        // 带备用方案的超时
        Mono<String> withTimeoutFallback = slowMono
            .timeout(Duration.ofSeconds(2), Mono.just("备用结果"));
        
        // ==================== using：资源管理 ====================
        // 确保资源正确释放（类似 try-with-resources）
        Flux<String> usingResource = Flux.using(
            () -> new BufferedReader(new FileReader("file.txt")),  // 资源创建
            reader -> Flux.fromStream(reader.lines()),  // 使用资源
            BufferedReader::close  // 资源释放
        );
        
        // ==================== 错误处理最佳实践 ====================
        System.out.println("\n=== 错误处理最佳实践 ===");
        
        // 1. 分层错误处理
        fetchUserData()
            .flatMap(this::validateUser)      // 验证层错误
            .flatMap(this::enrichUserData)    // 数据增强层错误
            .flatMap(this::saveToDatabase)    // 存储层错误
            .doOnError(e -> log.error("操作失败", e))  // 统一日志
            .onErrorResume(e -> {
                if (e instanceof ValidationException) {
                    return Mono.just(createErrorResponse("验证失败", e));
                }
                if (e instanceof DatabaseException) {
                    return Mono.just(createErrorResponse("数据库错误", e));
                }
                return Mono.just(createErrorResponse("系统错误", e));
            });
    }
    
    // 模拟方法
    static Mono<UserData> fetchUserData() { return Mono.just(new UserData()); }
    static Mono<UserData> validateUser(UserData u) { return Mono.just(u); }
    static Mono<UserData> enrichUserData(UserData u) { return Mono.just(u); }
    static Mono<UserData> saveToDatabase(UserData u) { return Mono.just(u); }
    static UserData createErrorResponse(String msg, Throwable e) { return new UserData(); }
    static void log(String msg, Throwable e) { System.err.println(msg + ": " + e); }
    
    static class UserData {}
    static class BusinessException extends RuntimeException {
        BusinessException(String msg, Throwable cause) { super(msg, cause); }
    }
    static class ValidationException extends RuntimeException {}
    static class DatabaseException extends RuntimeException {}
}
```

```java
/**
```

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class UserService {
    
    private final UserRepository userRepository;
    private final OrderServiceClient orderServiceClient;
    private final CacheService cacheService;
    
    // ==================== 基础 CRUD ====================
    
    /**
     * 根据 ID 查询用户
     */
    public Mono<User> getUserById(Long id) {
        return userRepository.findById(id)
            .doOnSubscribe(s -> log.info("查询用户: {}", id))
            .doOnSuccess(user -> log.info("找到用户: {}", user))
            .doOnError(e -> log.error("查询用户失败: {}", id, e));
    }
    
    /**
     * 分页查询所有用户
     */
    public Flux<User> getAllUsers(int page, int size) {
        return userRepository.findAllBy(PageRequest.of(page, size));
    }
    
    /**
     * 创建用户
     */
    public Mono<User> createUser(UserCreateRequest request) {
        return Mono.just(request)
            .map(req -> {
                User user = new User();
                user.setUsername(req.getUsername());
                user.setEmail(req.getEmail());
                user.setAge(req.getAge());
                user.setCreatedAt(LocalDateTime.now());
                return user;
            })
            .flatMap(userRepository::save)
            .doOnSuccess(user -> log.info("创建用户成功: {}", user.getId()));
    }
    
    /**
     * 更新用户
     */
    public Mono<User> updateUser(Long id, UserUpdateRequest request) {
        return userRepository.findById(id)
            .flatMap(existingUser -> {
                if (request.getUsername() != null) {
                    existingUser.setUsername(request.getUsername());
                }
                if (request.getEmail() != null) {
                    existingUser.setEmail(request.getEmail());
                }
                if (request.getAge() != null) {
                    existingUser.setAge(request.getAge());
                }
                existingUser.setUpdatedAt(LocalDateTime.now());
                return userRepository.save(existingUser);
            });
    }
    
    /**
     * 删除用户
     */
    public Mono<Void> deleteUser(Long id) {
        return userRepository.deleteById(id);
    }
    
    // ==================== 复杂业务场景 ====================
    
    /**
     * 搜索用户（带缓存）
     */
    public Flux<User> searchUsers(String keyword) {
        String cacheKey = "user:search:" + keyword;
        
        return cacheService.get(cacheKey)
            .cast(User.class)
            .flux()
            .switchIfEmpty(
                userRepository.findByUsernameContainingIgnoreCase(keyword)
                    .collectList()
                    .doOnNext(users -> cacheService.set(cacheKey, users, Duration.ofMinutes(5)))
                    .flatMapMany(Flux::fromIterable)
            );
    }
    
    /**
     * 获取用户及其订单（服务聚合）
     */
    public Mono<UserWithOrders> getUserWithOrders(Long userId) {
        // 并行获取用户和订单
        Mono<User> userMono = getUserById(userId)
            .switchIfEmpty(Mono.error(new UserNotFoundException(userId)));
        
        Flux<Order> ordersFlux = orderServiceClient.getOrdersByUserId(userId)
            .onErrorResume(e -> {
                log.error("获取订单失败: {}", userId, e);
                return Flux.empty();  // 订单服务失败不影响用户查询
            });
        
        // 组合结果
        return Mono.zip(
            userMono,
            ordersFlux.collectList(),
            (user, orders) -> {
                UserWithOrders result = new UserWithOrders();
                result.setUser(user);
                result.setOrders(orders);
                return result;
            }
        );
    }
    
    /**
     * 批量创建用户（带事务）
     */
    @Transactional
    public Flux<User> batchCreateUsers(List<UserCreateRequest> requests) {
        return Flux.fromIterable(requests)
            .flatMap(this::createUser, 3)  // 并发度3
            .onErrorContinue((error, obj) -> {
                log.error("创建用户失败: {}", obj, error);
            });
    }
    
    /**
     * 获取用户统计信息
     */
    public Mono<UserStatistics> getUserStatistics() {
        return userRepository.count()
            .flatMap(totalCount -> {
                Mono<Long> activeCount = userRepository.countByStatus(UserStatus.ACTIVE);
                Mono<Double> avgAge = userRepository.calculateAverageAge();
                
                return Mono.zip(activeCount, avgAge, (active, avg) -> {
                    UserStatistics stats = new UserStatistics();
                    stats.setTotalCount(totalCount);
                    stats.setActiveCount(active);
                    stats.setAverageAge(avg);
                    return stats;
                });
            });
    }
    
    /**
     * 处理用户导入（大文件流式处理）
     */
    public Flux<ImportResult> importUsers(Flux<UserCreateRequest> userStream) {
        return userStream
            .buffer(100)  // 每100条批量处理
            .flatMap(batch -> 
                Flux.fromIterable(batch)
                    .flatMap(this::createUser)
                    .collectList()
                    .map(createdUsers -> ImportResult.builder()
                        .successCount(createdUsers.size())
                        .build())
            )
            .onErrorResume(e -> Mono.just(ImportResult.builder()
                .error(e.getMessage())
                .build()));
    }
}
```

```java
     * 批量插入（使用 R2DBC 的批量操作）

    public Flux<User> batchInsert(List<User> users) {
        return databaseClient.inConnectionMany(connection -> {
            Statement statement = connection.createStatement(
                "INSERT INTO users (username, email, age) VALUES ($1, $2, $3)"
            );
            
            for (int i = 0; i < users.size(); i++) {
                User user = users.get(i);
                statement.bind(0, user.getUsername())
                    .bind(1, user.getEmail())
                    .bind(2, user.getAge());
                
                if (i < users.size() - 1) {
                    statement.add();
                }
            }
            
            return Flux.from(statement.execute())
                .flatMap(result -> result.map((row, metadata) -> {
                    // 映射返回结果
                    User u = new User();
                    u.setId(row.get("id", Long.class));
                    return u;
                }));
        });
    }

     * 复杂条件动态查询
     */

        if (criteria.getUsername() != null) {
            queryCriteria = queryCriteria.and("username").like("%" + criteria.getUsername() + "%");
        }
        if (criteria.getMinAge() != null) {
            queryCriteria = queryCriteria.and("age").greaterThanOrEquals(criteria.getMinAge());
        }
        if (criteria.getStatus() != null) {
            queryCriteria = queryCriteria.and("status").is(criteria.getStatus());
        }
        
        Query query = Query.query(queryCriteria)
            .sort(Sort.by("created_at").descending())
            .offset(criteria.getOffset())
            .limit(criteria.getLimit());
        
        return databaseClient.select()
            .from("users")
            .matching(query)
            .as(User.class)
            .all();
```

```java
        }
        if (minPrice != null && maxPrice != null) {
            query.addCriteria(Criteria.where("price").gte(minPrice).lte(maxPrice));
        }
        
        // 排序和分页
        query.with(Sort.by(Sort.Direction.DESC, "price"));
        query.skip(0).limit(10);
        
        return mongoTemplate.find(query, Product.class);
    }
    
    /**
     * 流式处理大量数据
     */
    public Mono<Void> processAllProducts() {
        return productRepository.findAll()
            .buffer(100)  // 每100条处理一次
            .flatMap(batch -> {
                // 批量处理逻辑
                return Flux.fromIterable(batch)
                    .flatMap(this::updateProductStock)
                    .then();
            })
            .then();
    }
    
    private Mono<Product> updateProductStock(Product product) {
        // 更新库存逻辑
        return Mono.just(product);
    }
}
```

## 六、背压（Backpressure）机制详解

### 6.1 什么是背压

#### 背压的定义与必要性

**背压（Backpressure）** 是响应式流的核心机制，指**下游消费者**向**上游生产者**反馈处理能力，从而控制数据生产速率，防止消费者被压垮。

```
┌─────────────────────────────────────────────────────────────────┐
│                    背压机制原理图解                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   无背压时（可能 OOM）                                            │
│   ┌─────────────┐                                               │
│   │  生产者     │ ──► 数据1 ──► 数据2 ──► 数据3 ──► ...          │
│   │  (1000/s)   │                                               │
│   └─────────────┘                                               │
│          │                                                      │
│          ▼                                                      │
│   ┌─────────────┐                                               │
│   │   缓冲区    │ [数据1, 数据2, 数据3, ..., 数据10000]          │
│   │  (无限增长) │                                               │
│   └─────────────┘                                               │
│          │                                                      │
│          ▼                                                      │
│   ┌─────────────┐                                               │
│   │  消费者     │ ◄── 处理中... (10/s)                          │
│   │  (慢速)     │                                               │
│   └─────────────┘                                               │
│                                                                 │
│   ❌ 结果：缓冲区无限增长 → 内存溢出（OOM）                       │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   有背压时（系统稳定）                                            │
│   ┌─────────────┐                                               │
│   │  生产者     │ ──► 数据1 ──► 数据2 ──► 暂停生产 ◄──┐         │
│   │  (1000/s)   │                                    │         │
│   └─────────────┘                                    │         │
│          ▲                                           │         │
│          │         request(10)                       │         │
│          └───────────────────────────────────────────┘         │
│   ┌─────────────┐                                               │
│   │   缓冲区    │ [数据1, 数据2, ..., 数据10]                   │
│   │  (固定大小) │                                               │
│   └─────────────┘                                               │
│          │                                                      │
│          ▼                                                      │
│   ┌─────────────┐                                               │
│   │  消费者     │ ◄── 处理完成 ──► request(10) ────────────────►│
│   │  (10/s)     │                                               │
│   └─────────────┘                                               │
│                                                                 │
│   ✅ 结果：消费者控制流速，系统稳定运行                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 背压在 Reactive Streams 中的实现

```
┌─────────────────────────────────────────────────────────────────┐
│              Reactive Streams 背压工作流程                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   阶段1: 订阅建立                                                 │
│   ┌─────────┐    subscribe()     ┌─────────┐                   │
│   │Publisher│ ─────────────────► │Subscriber│                  │
│   └─────────┘                    └────┬────┘                   │
│                                       │                         │
│   阶段2: 请求数据（背压控制点）                                     │
│                                       │ onSubscribe(Subscription)│
│                                       ▼                         │
│                               ┌─────────────┐                   │
│                               │ subscription.request(10)        │
│                               │ "请发送10个数据"                 │
│                               └─────────────┘                   │
│                                       │                         │
│   阶段3: 数据发送（受控）                                          │
│                                       ▼                         │
│   ┌─────────┐    onNext(data1)   ┌─────────┐                   │
│   │Publisher│ ─────────────────► │Subscriber│                  │
│   │         │    onNext(data2)   │         │                  │
│   │         │    ... (共10个)     │         │                  │
│   └─────────┘                    └─────────┘                   │
│                                       │                         │
│   阶段4: 再次请求（循环）                                          │
│                                       │ 处理完成                 │
│                                       ▼                         │
│                               ┌─────────────┐                   │
│                               │ subscription.request(10)        │
│                               │ "再发送10个"                     │
│                               └─────────────┘                   │
│                                                                 │
│   关键规则：                                                      │
│   1. Subscriber 必须通过 request(n) 请求数据                     │
│   2. Publisher 发送的数据量不能超过请求量                         │
│   3. 消费者可以根据处理能力动态调整请求量                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 Project Reactor 背压策略详解

#### 背压策略对比表

| 策略 | 方法 | 行为 | 适用场景 | 注意事项 |
|------|------|------|---------|---------|
| **错误** | `onBackpressureError()` | 缓冲区满时抛出异常 | 不允许数据丢失 | 需要处理异常 |
| **缓冲** | `onBackpressureBuffer()` | 缓冲数据，满时阻塞或丢弃 | 可接受内存增长 | 可能 OOM |
| **丢弃** | `onBackpressureDrop()` | 丢弃无法处理的数据 | 可接受数据丢失 | 可监听丢弃事件 |
| **保留最新** | `onBackpressureLatest()` | 只保留最新数据 | 只关心最新值 | 可能丢失中间值 |

#### 1. onBackpressureError（错误策略）

```java
/**
 * 错误策略：缓冲区满时抛出 BackpressureException
 * 适用场景：数据完整性要求高，不允许丢失
 */
public class BackpressureErrorDemo {
    
    public static void main(String[] args) throws InterruptedException {
        // 快速生产者：每10ms产生一个数据
        Flux<Long> fastProducer = Flux.interval(Duration.ofMillis(10));
        
        // 慢速消费者：每100ms处理一个数据
        CountDownLatch latch = new CountDownLatch(1);
        
        fastProducer
            .onBackpressureError()  // 启用错误策略
            .subscribe(
                data -> {
                    try {
                        Thread.sleep(100);  // 模拟慢速处理
                        System.out.println("处理: " + data);
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    }
                },
                error -> {
                    System.err.println("背压错误: " + error.getMessage());
                    latch.countDown();
                },
                () -> System.out.println("完成")
            );
        
        latch.await();
        // 输出：
        // 处理: 0
        // 处理: 1
        // ...（处理几个后）
        // 背压错误: The receiver is overrun by more signals than expected (bounded queue...)
    }
}
```

```java
 * 缓冲策略：将数据缓冲到队列中
 * 适用场景：短暂的生产高峰，可接受一定延迟

public class BackpressureBufferDemo {

            .onBackpressureBuffer()  // 无界缓冲（⚠️ 可能OOM）
            .subscribe(data -> {
                try {
                    Thread.sleep(50);
                    System.out.println("处理: " + data);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            });
        
        Thread.sleep(10000);
        
        System.out.println("\n=== 有界缓冲 ===");
        fastProducer
            .onBackpressureBuffer(20)  // 最多缓冲20个
            .subscribe(data -> {

                    Thread.sleep(50);
                    System.out.println("处理: " + data);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            },
            error -> System.err.println("缓冲溢出: " + error.getMessage())
            );
        
        Thread.sleep(10000);

            .subscribe(data -> {
                try {
                    Thread.sleep(50);
                    System.out.println("处理: " + data);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            });
        
        Thread.sleep(10000);
        
        System.out.println("\n=== 带回调的缓冲 ===");
        AtomicInteger droppedCount = new AtomicInteger(0);
        
        fastProducer
            .onBackpressureBuffer(
                10,  // 缓冲区大小
                dropped -> {
                    // 监听丢弃的数据
                    droppedCount.incrementAndGet();
                    System.out.println("丢弃数据: " + dropped);
                }
            )
            .subscribe(data -> {
                try {
                    Thread.sleep(50);
                    System.out.println("处理: " + data);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            },
            error -> System.out.println("总共丢弃: " + droppedCount.get() + " 个数据"),
            () -> System.out.println("总共丢弃: " + droppedCount.get() + " 个数据")
            );
        
        Thread.sleep(10000);
```

#### 自定义背压处理

```java
/**
 * 自定义背压处理示例
 */
public class CustomBackpressureDemo {
    
    public static void main(String[] args) throws InterruptedException {
        // 创建一个带背压控制的自定义 Publisher
        Flux<String> customPublisher = Flux.create(emitter -> {
            // 模拟外部数据源（如消息队列）
            MessageQueue queue = new MessageQueue();
            
            // 注册消费者
            queue.consume(message -> {
                // 使用 tryEmitNext 处理背压
                Sinks.EmitResult result = emitter.tryEmitNext(message);
                
                if (result.isSuccess()) {
                    // 发送成功，确认消息
                    queue.ack(message);
                } else if (result == Sinks.EmitResult.FAIL_OVERFLOW) {
                    // 背压溢出，可以：
                    // 1. 丢弃消息
                    // 2. 暂停消费
                    // 3. 存入死信队列
                    System.out.println("背压溢出，消息暂存: " + message);
                    queue.nack(message);
                }
            });
            
        }, FluxSink.OverflowStrategy.BUFFER);  // 使用缓冲策略
        
        // 消费端
        customPublisher
            .onBackpressureBuffer(100, dropped -> {
                System.out.println("丢弃消息: " + dropped);
            })
            .subscribe(message -> {
                try {
                    Thread.sleep(100);  // 慢速处理
                    System.out.println("处理消息: " + message);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            });
        
        Thread.sleep(10000);
    }
    
    // 模拟消息队列
    static class MessageQueue {
        void consume(java.util.function.Consumer<String> handler) {
            // 模拟持续产生消息
            new Thread(() -> {
                int i = 0;
                while (true) {
                    handler.accept("message-" + i++);
                    try {
                        Thread.sleep(10);
                    } catch (InterruptedException e) {
                        break;
                    }
                }
            }).start();
        }
        
        void ack(String message) {
            // 确认消息
        }
        
        void nack(String message) {
            // 否定确认，消息会重新投递
        }
    }
}
```

### 6.4 背压最佳实践

```
┌─────────────────────────────────────────────────────────────────┐
│                    背压最佳实践                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. 选择合适的背压策略                                           │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │ 场景                          │ 推荐策略                 │  │
│   ├───────────────────────────────┼──────────────────────────┤  │

│   │ 日志收集（可丢部分）           │ onBackpressureDrop       │  │
│   │ 实时监控（只关心最新）         │ onBackpressureLatest     │  │
│   │ 批量处理（短暂高峰）           │ onBackpressureBuffer     │  │

│                                                                 │
│   2. 设置合理的缓冲区大小                                          │
│   - 太小：频繁触发背压，性能下降                                   │
│   - 太大：内存占用高，GC 压力大                                    │

│                                                                 │
│   3. 监控背压情况                                                 │
│   - 丢弃数据数量                                                  │
│   - 缓冲区使用率                                                  │
│   - 背压异常次数                                                  │
│                                                                 │
│   4. 结合业务场景优化                                              │
│   - 可采样时：使用 sample() 减少数据量                             │
│   - 可聚合时：使用 buffer() 批量处理                               │
│   - 可延迟时：使用 delayElements() 控制速率                        │
│                                                                 │
│   5. 避免常见陷阱                                                 │
│   ❌ 不要在 subscribe() 中阻塞                                     │
│   ❌ 不要无限缓冲（默认缓冲区可能无限）                             │
│   ❌ 不要忽略背压异常                                              │
│   ✅ 使用 publishOn/subscribeOn 控制线程                           │
```

## 七、高频面试题

### 7.1 响应式编程基础（8 道）

**问题 1：什么是响应式编程？解决了什么问题？**

**答：**

响应式编程（Reactive Programming）是一种面向数据流和变化传播的编程范式，通过异步非阻塞的方式处理数据流。

**核心特点：**

1. **异步非阻塞** - 不会阻塞线程等待 I/O 完成
2. **数据流驱动** - 数据变化自动传播到订阅者
3. **背压支持** - 消费者可以控制数据接收速率

**解决的问题：**

- 高并发下的线程资源耗尽问题
- 传统阻塞 I/O 的性能瓶颈
- 复杂异步逻辑的回调地狱

---

**问题 2：响应式编程 vs 命令式编程的区别？**

**答：**

| 对比项 | 命令式编程 | 响应式编程 |
|--------|-----------|-----------|
| 执行方式 | 同步顺序执行 | 异步非阻塞 |
| 数据获取 | 主动拉取（Pull） | 被动推送（Push） |
| 线程使用 | 阻塞等待 I/O | 事件驱动，释放线程 |
| 代码风格 | 逐行执行 | 声明式链式调用 |
| 错误处理 | try-catch | 操作符链式处理 |

**示例对比：**

```java
// 命令式：阻塞等待
User user = userService.getUser(id);  // 阻塞
Order order = orderService.getOrder(user);  // 阻塞

// 响应式：非阻塞链式
Mono<User> userMono = userService.getUser(id);
userMono.flatMap(user -> orderService.getOrder(user))
        .subscribe();
```

---

**问题 3：Reactive Streams 规范的核心接口？**

**答：**

Reactive Streams 是响应式编程的标准规范，定义了 4 个核心接口：

| 接口 | 角色 | 核心方法 |
|------|------|---------|
| **Publisher** | 数据发布者 | `subscribe(Subscriber)` |
| **Subscriber** | 数据订阅者 | `onSubscribe/onNext/onError/onComplete` |
| **Subscription** | 订阅关系 | `request/cancel` |
| **Processor** | 处理器（发布+订阅） | 继承 Publisher 和 Subscriber |

**工作流程：**

```
Publisher → subscribe → Subscriber
              ↓
         Subscription (request n)
              ↓
         onNext(data) × n → onComplete/onError
```

---

**问题 4：什么是背压？为什么需要背压？**

**答：**

**背压（Backpressure）** 是指当生产者产生数据的速度超过消费者处理能力时，消费者向生产者反馈压力，要求降低发送速率或暂停发送的机制。

**为什么需要背压：**

1. **防止内存溢出** - 生产者过快导致缓冲区无限增长
2. **保护消费者** - 避免消费者被大量数据压垮
3. **资源平衡** - 协调生产者和消费者的处理能力

**背压策略：**

| 策略 | 说明 |
|------|------|
| **BUFFER** | 缓冲所有数据（可能 OOM） |
| **DROP** | 丢弃多余数据 |
| **LATEST** | 只保留最新数据 |
| **ERROR** | 直接报错 |

---

**问题 5：Mono 和 Flux 的区别？**

**答：**

Mono 和 Flux 是 Project Reactor 的两个核心发布者类型：

| 特性 | Mono | Flux |
|------|------|------|
| **数据量** | 0 或 1 个元素 | 0 到 N 个元素 |
| **适用场景** | 单对象查询、保存操作 | 列表查询、流式数据 |
| **类比** | `Optional` / `CompletableFuture` | `Stream` / `List` |
| **结束信号** | onNext + onComplete 或 onComplete | 0~N 次 onNext + onComplete |

**代码示例：**

```java
// Mono：单个用户
Mono<User> userMono = userRepository.findById(id);

// Flux：用户列表
Flux<User> userFlux = userRepository.findAll();
```

---

**问题 6：热流（Hot）vs 冷流（Cold）？**

**答：**

| 特性 | 冷流（Cold） | 热流（Hot） |
|------|-------------|------------|
| **数据产生** | 每个订阅者独立触发 | 独立于订阅者，主动产生 |
| **订阅时机** | 订阅后才发送数据 | 可能错过之前的数据 |
| **数据共享** | 每个订阅者获取完整序列 | 多个订阅者共享同一数据 |
| **典型实现** | `Flux.just()`, `Flux.fromIterable()` | `Flux.share()`, `Flux.publish()` |

**类比：**

- **冷流** = 观看视频点播（每个人从头开始看）
- **热流** = 观看直播（晚来的人错过之前的内容）

---

**问题 7：响应式编程的优缺点？**

**答：**

**优点：**

1. **高吞吐量** - 少量线程处理大量并发连接
2. **资源高效** - 非阻塞 I/O 不占用线程等待
3. **背压保护** - 防止消费者过载
4. **声明式编程** - 代码简洁，易于组合

**缺点：**

1. **学习曲线陡峭** - 思维转换成本高
2. **调试困难** - 异步堆栈追踪复杂
3. **生态限制** - 部分库不支持响应式
4. **阻塞陷阱** - 误用阻塞 API 会抵消优势

---

**问题 8：适用场景有哪些？**

**答：**

**适合场景：**

1. **高并发 Web 服务** - 网关、API 代理
2. **流式数据处理** - 实时日志、股票行情
3. **微服务通信** - 服务间异步调用
4. **I/O 密集型应用** - 数据库、缓存、HTTP 调用

**不适合场景：**

1. **计算密集型任务** - 阻塞线程反而更高效
2. **简单 CRUD** - 引入复杂度得不偿失
3. **遗留系统集成** - 大量阻塞代码难以改造

### 7.2 Project Reactor（8 道）

**问题 9：Project Reactor 的核心组件？**

**答：**

Project Reactor 是 Spring 生态的响应式编程库，核心组件包括：

| 组件 | 说明 |
|------|------|
| **Mono** | 异步单值容器（0 或 1 个元素） |
| **Flux** | 异步多值容器（0 到 N 个元素） |
| **Scheduler** | 调度器，控制执行线程 |
| **Processor** | 同时实现 Publisher 和 Subscriber |

**核心操作符分类：**

1. **创建操作符**：`just`, `fromIterable`, `create`, `generate`
2. **转换操作符**：`map`, `flatMap`, `filter`, `reduce`
3. **组合操作符**：`merge`, `zip`, `concat`
4. **错误处理**：`onErrorReturn`, `onErrorResume`, `retry`

---

**问题 10：map 和 flatMap 的区别？**

**答：**

| 特性 | map | flatMap |
|------|-----|---------|
| **返回类型** | 同步转换，返回普通对象 | 异步转换，返回 Publisher |
| **展开层级** | 保持单层 | 自动展平嵌套 Publisher |
| **适用场景** | 简单数据转换 | 涉及异步操作的转换 |
| **执行顺序** | 保持顺序 | 可能乱序（并行时） |

**代码示例：**

```java
// map：同步转换
Flux<String> names = Flux.just("alice", "bob")
    .map(String::toUpperCase);  // ALICE, BOB

// flatMap：异步转换，展平结果
Flux<Order> orders = Flux.just(user1, user2)
    .flatMap(user -> orderService.findByUser(user));  // 展平 List<Order>
```

**选择建议：**

- 简单转换用 `map`
- 涉及数据库/HTTP 调用用 `flatMap`
- 需要保持顺序用 `concatMap`

---

**问题 11：publishOn 和 subscribeOn 的区别？**

**答：**

| 特性 | subscribeOn | publishOn |
|------|-------------|-----------|
| **影响范围** | 影响整个链（从源头开始） | 只影响下游操作 |
| **执行位置** | 订阅和请求执行 | 数据处理和通知 |
| **使用次数** | 链中第一个生效 | 可多次使用切换线程 |
| **典型用途** | 指定数据源执行线程 | 指定后续处理线程 |

**执行流程：**

```
subscribeOn(Schedulers.boundedElastic())  // 影响源头
    ↓
源数据获取（在 boundedElastic 线程）
    ↓
publishOn(Schedulers.parallel())  // 切换线程
    ↓
后续操作（在 parallel 线程）
```

**常见组合：**

```java
Flux.just(data)
    .subscribeOn(Schedulers.boundedElastic())  // I/O 操作
    .map(this::transform)
    .publishOn(Schedulers.parallel())          // CPU 计算
    .map(this::compute)
    .subscribe();
```

**问题 12：如何处理响应式流的错误？**

**答：**

Project Reactor 提供了多种错误处理策略：

| 操作符 | 作用 |
|--------|------|
| **onErrorReturn** | 发生错误时返回默认值 |
| **onErrorResume** | 发生错误时切换到备用流 |
| **onErrorMap** | 转换错误类型 |
| **doOnError** | 副作用处理（如日志记录） |
| **retry** | 重试指定次数 |
| **retryWhen** | 自定义重试策略 |

**代码示例：**

```java
// 返回默认值
mono.onErrorReturn(e -> e instanceof NotFoundException, defaultUser);

// 切换到备用流
mono.onErrorResume(e -> userService.findById(backupId));

// 重试策略
mono.retryWhen(Retry.backoff(3, Duration.ofSeconds(1))
    .filter(e -> e instanceof TimeoutException));
```

**问题 14：如何测试 Mono 和 Flux？**

**问题 15：如何并行处理多个流？**

**问题 16：如何实现超时和重试？**

**答：**

**超时处理：**

```java
// 简单超时
mono.timeout(Duration.ofSeconds(5))
    .onErrorResume(e -> Mono.just(defaultValue));

// 带备用方案的超时
mono.timeout(Duration.ofSeconds(5), fallbackMono);
```

**重试策略：**

```java
// 固定次数重试
mono.retry(3);

// 指数退避重试
mono.retryWhen(Retry.backoff(3, Duration.ofMillis(100))

mono.retryWhen(errors -> errors
    .zipWith(Flux.range(1, 3), (error, index) -> index)
    .flatMap(index -> Mono.delay(Duration.ofSeconds(index))));
```

### 7.3 Spring WebFlux（8 道）

**问题 17：Spring WebFlux vs Spring MVC？**

**答：**

| 特性 | Spring MVC | Spring WebFlux |
|------|-----------|----------------|
| **编程模型** | 命令式（同步阻塞） | 响应式（异步非阻塞） |
| **底层容器** | Servlet 容器（Tomcat/Jetty） | Netty / Undertow / Servlet 3.1+ |
| **线程模型** | 每请求一线程 | 事件驱动，少量线程处理大量请求 |
| **数据访问** | JPA/JDBC（阻塞） | R2DBC/Mongo Reactive（非阻塞） |
| **适用场景** | 传统 Web 应用 | 高并发、流式处理 |

**核心区别：**

- **MVC**：`Controller` 返回 `String`/`ModelAndView`
- **WebFlux**：`Controller` 返回 `Mono`/`Flux`

**选择建议：**

- 已有 Spring MVC 项目且无明显性能瓶颈 → 继续使用 MVC
- 网关、代理、高并发场景 → 优先考虑 WebFlux

**问题 18：WebFlux 的两种编程模型？**

**答：**

Spring WebFlux 支持两种编程模型：

**1. 注解式（Annotated Controllers）：**

与 Spring MVC 类似，使用 `@Controller`、`@RequestMapping` 等注解：

```java
@RestController
public class UserController {
    
    @GetMapping("/users/{id}")
    public Mono<User> getUser(@PathVariable String id) {
        return userService.findById(id);
    }
    
    @GetMapping("/users")
    public Flux<User> listUsers() {
        return userService.findAll();
    }
}
```

**2. 函数式（Functional Endpoints）：**

使用 `RouterFunction` 和 `HandlerFunction`：

```java
@Configuration
public class UserRouter {
    
    @Bean
    public RouterFunction<ServerResponse> userRoutes(UserHandler handler) {
        return route(GET("/users/{id}"), handler::getUser)
            .andRoute(GET("/users"), handler::listUsers)
            .andRoute(POST("/users"), handler::createUser);
    }
}

@Component
public class UserHandler {
    public Mono<ServerResponse> getUser(ServerRequest request) {
        String id = request.pathVariable("id");
        return userService.findById(id)
            .flatMap(user -> ServerResponse.ok().bodyValue(user))
            .switchIfEmpty(ServerResponse.notFound().build());
    }
}
```

**对比：**

| 特性 | 注解式 | 函数式 |
|------|--------|--------|
| 学习成本 | 低（与 MVC 一致） | 高 |
| 灵活性 | 受限于注解 | 完全程序化控制 |
| 测试性 | 需要启动容器 | 纯函数，易于单元测试 |
| 适用场景 | 快速开发 | 复杂路由逻辑 |

**问题 19：如何选择 Netty vs Servlet 容器？**

**答：**

| 特性 | Netty | Servlet 容器（Tomcat/Jetty） |
|------|-------|------------------------------|
| **性能** | 更高（专为异步设计） | 较低（Servlet 开销） |
| **生态** | 原生支持响应式 | 需要 Servlet 3.1+ |
| **兼容性** | 仅支持 WebFlux | 支持 MVC 和 WebFlux |
| **部署** | 嵌入式为主 | 支持外部容器部署 |

**选择建议：**

- **纯 WebFlux 项目** → Netty（默认，性能最优）
- **需要同时支持 MVC** → Tomcat/Jetty
- **传统应用迁移** → 使用 Tomcat 逐步改造

**配置切换：**

```xml

<dependency>

    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-netty</artifactId>
        </exclusion>
    </exclusions>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-tomcat</artifactId>
</dependency>
```

**问题 20：如何实现 SSE 流式响应？**

**答：**

SSE（Server-Sent Events）是服务器向客户端推送实时数据的技术。

**实现代码：**

```java
@RestController
public class SseController {
    
    @GetMapping(value = "/events", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> streamEvents() {
        return Flux.interval(Duration.ofSeconds(1))
            .map(seq -> ServerSentEvent.<String>builder()
                .id(String.valueOf(seq))
                .event("heartbeat")
                .data("Event " + seq + " at " + LocalTime.now())
                .build());
    }
    
    // 简化写法
    @GetMapping(value = "/simple-events", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<String> simpleEvents() {
        return Flux.interval(Duration.ofSeconds(1))
            .map(seq -> "Event " + seq);
    }
}
```

**前端接收：**

```javascript
const eventSource = new EventSource('/events');

eventSource.onmessage = (event) => {
    console.log('收到数据:', event.data);
};

eventSource.onerror = (error) => {
    console.error('SSE 错误:', error);
};
```

**SSE vs WebSocket：**

| 特性 | SSE | WebSocket |
|------|-----|-----------|
| 通信方向 | 服务器 → 客户端 | 双向 |
| 协议 | HTTP | ws/wss |
| 自动重连 | 原生支持 | 需手动实现 |
| 适用场景 | 股票行情、日志推送 | 聊天、游戏 |

**问题 21：WebFlux 中的跨域处理？**

**答：**

**1. 全局配置（推荐）：**

```java
@Configuration
public class CorsConfig implements WebFluxConfigurer {
    
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
            .allowedOrigins("http://localhost:3000")
            .allowedMethods("GET", "POST", "PUT", "DELETE")
```

**问题 23：WebFlux 的性能优势？**

**答：**

**1. 更高的并发处理能力：**

- 少量线程（如 CPU 核心数）处理大量并发连接
- 避免线程上下文切换开销

**2. 更低的内存占用：**

- 不需要为每个请求分配线程栈（默认 1MB）
- 适合连接数多但计算量小的场景（如网关）

**3. 延迟更低：**

- 非阻塞 I/O，线程不会被等待占用
- 请求处理更及时

**性能对比数据（参考）：**

| 指标 | Spring MVC | Spring WebFlux |
|------|-----------|----------------|
| 并发连接数 | ~1000/线程 | 10000+ |
| 内存占用（1万连接） | ~10GB | ~100MB |
| 延迟（P99） | 较高 | 较低 |

**适用场景：**

- API 网关
- 高并发微服务
- 实时数据推送

**问题 25：RSocket 协议的特点？**

**答：**

RSocket 是一个支持响应式流语义的二进制协议，设计用于微服务通信。

**核心特点：**

| 特性 | 说明 |
|------|------|
| **响应式语义** | 原生支持背压和异步流 |
| **多路复用** | 单连接支持多个交互 |
| **双向通信** | 客户端和服务端都可以发起请求 |
| **连接恢复** | 支持会话恢复，断线重连 |
| **多种传输** | TCP、WebSocket、Aeron 等 |

**与 HTTP 对比优势：**

- 二进制协议，更高效
- 原生流支持，无需 SSE/WebSocket 变通
- 内置背压，避免消费者过载

**问题 26：RSocket 的四种交互模式？**

**问题 28：RSocket 的背压机制？**

**答：**

RSocket 在协议层面实现了背压机制，通过 **LEASE** 帧控制数据流：

**工作原理：**

```
服务端发送 LEASE 帧 → 告知客户端可发送的请求数
客户端根据 LEASE 发送请求
服务端处理完成后，发送新的 LEASE
```

**LEASE 帧包含：**

- **TTL**：租约有效期
- **Number of Requests**：允许发送的请求数

**代码配置：**

```java
RSocketServer.create(new SocketAcceptor() {
    @Override
    public Mono<RSocket> accept(ConnectionSetupPayload setup, RSocket sendingSocket) {
        return Mono.just(new AbstractRSocket() {
            @Override
            public Flux<Payload> requestStream(Payload payload) {
                return Flux.interval(Duration.ofMillis(10))
                    .map(i -> DefaultPayload.create("Data " + i))
                    .onBackpressureBuffer(100);  // 背压缓冲
            }
        });
    }
})
.lease(spec -> spec.maxPendingRequests(100))  // 最大待处理请求
.bind(TcpServerTransport.create(7000))
.subscribe();
```

**问题 29：RSocket 的连接复用？**

**优势：**

1. **减少连接开销** - 避免频繁创建/销毁连接
2. **提高资源利用率** - 单连接承载更多请求
3. **简化连接管理** - 连接池规模更小

**实现方式：**

```java
// 客户端保持长连接
RSocket client = RSocketConnector.create()
    .reconnect(Retry.fixedDelay(3, Duration.ofSeconds(1)))  // 自动重连
    .connect(TcpClientTransport.create("localhost", 7000))
    .block();
```

```java
        .sessionDuration(Duration.ofMinutes(5))

    .subscribe();
```

**核心实现：**

```java
/**
 * 推送服务核心实现
 */
@Service
@Slf4j
public class PushService {
    
    // 用户连接映射：userId -> Sink
    private final ConcurrentHashMap<String, Sinks.Many<PushMessage>> userSinks = new ConcurrentHashMap<>();
    
    // 主题订阅映射：topic -> Set<userId>
    private final ConcurrentHashMap<String, Set<String>> topicSubscribers = new ConcurrentHashMap<>();
    
    /**
     * 用户建立连接
     */
    public Flux<PushMessage> connect(String userId, List<String> topics) {
        // 创建多播 Sink
        Sinks.Many<PushMessage> sink = Sinks.many().multicast()
            .onBackpressureBuffer(100);  // 缓冲100条消息
        
        // 存储用户连接
        userSinks.put(userId, sink);
        
        // 订阅主题
        topics.forEach(topic -> 
            topicSubscribers.computeIfAbsent(topic, k -> ConcurrentHashMap.newKeySet())
                .add(userId)
        );
        
        log.info("用户 {} 连接成功，订阅主题: {}", userId, topics);
        
        return sink.asFlux()
            .doOnCancel(() -> disconnect(userId))  // 取消订阅时清理
            .doOnTerminate(() -> disconnect(userId));  // 终止时清理
    }
    
    /**
     * 断开连接
     */
    public void disconnect(String userId) {
        Sinks.Many<PushMessage> sink = userSinks.remove(userId);
        if (sink != null) {
            sink.tryEmitComplete();
        }
        
        // 从所有主题中移除
        topicSubscribers.values().forEach(set -> set.remove(userId));
        
        log.info("用户 {} 断开连接", userId);
    }
    
    /**
     * 推送消息给指定用户
     */
    public void pushToUser(String userId, PushMessage message) {
        Sinks.Many<PushMessage> sink = userSinks.get(userId);
        if (sink != null) {
            Sinks.EmitResult result = sink.tryEmitNext(message);
            if (result.isFailure()) {
                log.warn("推送消息给用户 {} 失败: {}", userId, result);
            }
        }
    }
    
    /**
     * 广播消息给主题订阅者
     */
    public void broadcast(String topic, PushMessage message) {
        Set<String> subscribers = topicSubscribers.get(topic);
        if (subscribers != null) {
            subscribers.forEach(userId -> pushToUser(userId, message));
        }
    }
    
    /**
     * 广播给所有在线用户
     */
    public void broadcastAll(PushMessage message) {
        userSinks.forEach((userId, sink) -> {
            Sinks.EmitResult result = sink.tryEmitNext(message);
            if (result.isFailure()) {
                log.warn("广播消息给用户 {} 失败", userId);
            }
        });
    }
}

/**
 * 推送消息 DTO
 */
@Data
@Builder
public class PushMessage {
    private String id;
    private String type;        // 消息类型：notification/chat/system
    private String topic;       // 主题
    private Object payload;     // 消息内容
    private long timestamp;
}

/**
 * 推送控制器
 */
@RestController
```

```java
/**
 * 服务聚合层 - 并行调用多个下游服务
 */
@Service
@RequiredArgsConstructor
@Slf4j
public class AggregationService {
    
    private final WebClient.Builder webClientBuilder;
    
    /**
     * 获取用户完整信息（并行调用4个服务）
     * 传统方式：串行调用，总耗时 = 100+100+100+100 = 400ms
     * 响应式方式：并行调用，总耗时 ≈ 100ms（最慢的服务）
     */
    public Mono<UserProfile> getUserProfile(String userId) {
        long startTime = System.currentTimeMillis();
        
        // 并行发起4个调用
        Mono<User> userMono = fetchUser(userId)
            .subscribeOn(Schedulers.boundedElastic());
        
        Mono<List<Order>> ordersMono = fetchOrders(userId)
            .collectList()
            .subscribeOn(Schedulers.boundedElastic());
        
        Mono<Wallet> walletMono = fetchWallet(userId)
            .subscribeOn(Schedulers.boundedElastic());
        
        Mono<List<Coupon>> couponsMono = fetchCoupons(userId)
            .collectList()
            .subscribeOn(Schedulers.boundedElastic());
        
        // 等待所有结果返回后组合
        return Mono.zip(userMono, ordersMono, walletMono, couponsMono)
            .map(tuple -> {
                UserProfile profile = new UserProfile();
                profile.setUser(tuple.getT1());
                profile.setOrders(tuple.getT2());
                profile.setWallet(tuple.getT3());
                profile.setCoupons(tuple.getT4());
                
                long duration = System.currentTimeMillis() - startTime;
                log.info("聚合完成，耗时: {}ms", duration);
                
                return profile;
            })
            .timeout(Duration.ofSeconds(3))  // 整体超时3秒
            .onErrorResume(e -> {
                log.error("聚合用户数据失败: {}", userId, e);
                return Mono.just(createFallbackProfile(userId));
            });
    }
    
    /**
     * 带降级策略的服务调用
     */
    private Mono<User> fetchUser(String userId) {
        return webClientBuilder.build()
            .get()
            .uri("http://user-service/users/{id}", userId)
            .retrieve()
            .bodyToMono(User.class)
            .timeout(Duration.ofSeconds(1))
            .retryWhen(Retry.backoff(2, Duration.ofMillis(100)))
            .onErrorResume(e -> {
                log.warn("获取用户失败，使用缓存: {}", userId);
                return fetchUserFromCache(userId);
            });
    }
    
    private Flux<Order> fetchOrders(String userId) {
        return webClientBuilder.build()
            .get()
            .uri("http://order-service/orders?userId={userId}", userId)
            .retrieve()
            .bodyToFlux(Order.class)
            .timeout(Duration.ofSeconds(1))
            .onErrorResume(e -> Flux.empty());  // 订单服务失败不影响整体
    }
    
    private Mono<Wallet> fetchWallet(String userId) {
        return webClientBuilder.build()
            .get()
            .uri("http://wallet-service/wallets/{userId}", userId)
            .retrieve()
            .bodyToMono(Wallet.class)
            .timeout(Duration.ofSeconds(1));
    }
    
    private Flux<Coupon> fetchCoupons(String userId) {
        return webClientBuilder.build()
            .get()
            .uri("http://coupon-service/coupons?userId={userId}", userId)
            .retrieve()
            .bodyToFlux(Coupon.class)
            .timeout(Duration.ofSeconds(1));
    }
}
```

**问题 34：微服务架构中 RSocket 的应用场景？**

**答：**

**适用场景对比：**

| 场景 | HTTP/REST 问题 | RSocket 优势 | 实现方式 |
|------|---------------|-------------|---------|
| **实时数据同步** | 需要轮询，延迟高 | 双向流、低延迟 | Channel 模式 |
| **服务间推送** | WebSocket 复杂 | 原生支持服务端推送 | Request-Stream |
| **事件总线** | 同步阻塞 | 异步 Fire-and-Forget | 单向通知 |
| **长连接服务** | 连接重建开销大 | 连接恢复、会话保持 | Resume 机制 |
| **高吞吐通信** | HTTP 头部开销大 | 二进制协议高效 | 全双工通信 |

```java
 * 库存服务（RSocket Responder）

@Component
public class InventoryRSocketController {
    
    private final InventoryService inventoryService;
    private final Sinks.Many<InventoryUpdate> inventorySink;
    
    public InventoryRSocketController(InventoryService inventoryService) {
        this.inventoryService = inventoryService;
        this.inventorySink = Sinks.many().multicast().onBackpressureBuffer();
        
        // 订阅库存变更事件
        inventoryService.getInventoryChanges()
            .subscribe(update -> inventorySink.tryEmitNext(update));
    }
    
    /**
     * 实时库存更新流（Request-Stream 模式）
     */
    @MessageMapping("inventory.updates")
    public Flux<InventoryUpdate> streamUpdates() {
        return inventorySink.asFlux()
            .onBackpressureBuffer(100);
    }
    
    /**
     * 库存预留（Request-Response 模式）
     */
    @MessageMapping("inventory.reserve")
    public Mono<ReservationResult> reserve(ReservationRequest request) {
        return inventoryService.reserve(request)
            .doOnSuccess(result -> {
                // 发布库存变更事件
                inventorySink.tryEmitNext(new InventoryUpdate(
                    request.getProductId(),
                    result.getRemainingQuantity()
                ));
            });
    }
    
    /**
     * 双向实时通信（Channel 模式）
     */
    @MessageMapping("inventory.channel")
    public Flux<InventoryResponse> bidirectionalChannel(
            Flux<InventoryRequest> requests) {
        
        return requests.flatMap(request -> {
            if (request.getType() == RequestType.QUERY) {
                return inventoryService.query(request.getProductId())
                    .map(InventoryResponse::new);
            } else if (request.getType() == RequestType.UPDATE) {
                return inventoryService.update(request)
                    .map(InventoryResponse::new);
            }
            return Mono.just(new InventoryResponse("UNKNOWN"));
        });
    }
}

    public void subscribeInventoryUpdates() {

            .retrieveFlux(InventoryUpdate.class)
            .subscribe(update -> {
                // 更新本地缓存
                InventoryCache.put(update.getProductId(), update.getQuantity());
                log.info("库存更新: {} = {}", update.getProductId(), update.getQuantity());
            });
    }
    
    /**
     * 创建订单时预留库存
     */
    public Mono<Order> createOrder(CreateOrderRequest request) {
        // 1. 检查本地缓存
        if (InventoryCache.get(request.getProductId()) < request.getQuantity()) {
            return Mono.error(new InsufficientInventoryException());
        }
        
        // 2. 调用库存服务预留
        return inventoryRSocket.route("inventory.reserve")
            .data(new ReservationRequest(request.getProductId(), request.getQuantity()))
            .retrieveMono(ReservationResult.class)
            .flatMap(result -> {
                if (result.isSuccess()) {
                    // 3. 创建订单
                    return saveOrder(request);
                } else {
                    return Mono.error(new InsufficientInventoryException());
                }
            });
    }
    
    /**

     */
    public Flux<InventoryResponse> syncInventoryBatch(List<InventoryRequest> requests) {
        return inventoryRSocket.route("inventory.channel")
            .data(Flux.fromIterable(requests))
            .retrieveFlux(InventoryResponse.class);
    }
```

**问题 38：响应式编程的常见陷阱和解决方案？**

**答：**

| 陷阱 | 问题描述 | 解决方案 |
|------|---------|---------|
| **阻塞调用** | 在响应式链中调用阻塞方法 | 使用 `subscribeOn(boundedElastic)` 包装 |
| **忘记订阅** | 创建 Mono/Flux 但不订阅 | 确保调用 `subscribe()` 或有其他消费者 |
| **异常吞没** | 错误未处理导致静默失败 | 使用 `doOnError()` 和 `onErrorResume()` |
| **线程混乱** | 在错误的线程执行操作 | 明确使用 `publishOn()` 切换线程 |
| **内存泄漏** | 无限流未取消订阅 | 使用 `take()`、`timeout()` 限制 |
| **背压忽略** | 快速生产者导致 OOM | 使用背压操作符控制流速 |
| **调试困难** | 异步堆栈难以追踪 | 使用 `checkpoint()` 和 `log()` |

```java
/**
 * 常见陷阱示例和正确写法
 */
public class ReactivePitfalls {
    
    // ❌ 陷阱1：阻塞调用
    public Mono<User> badBlocking(String id) {
        return Mono.just(id)
            .map(this::fetchFromDBBlocking);  // 阻塞操作！
    }
    
    // ✅ 正确：使用 subscribeOn
    public Mono<User> goodBlocking(String id) {
        return Mono.fromCallable(() -> fetchFromDBBlocking(id))
            .subscribeOn(Schedulers.boundedElastic());
    }
    
    // ❌ 陷阱2：忘记订阅
    public void badForgetSubscribe() {
        Mono.just("data")
            .map(this::process);  // 没有 subscribe，不会执行！
    }
    
    // ✅ 正确：必须订阅
    public void goodSubscribe() {
        Mono.just("data")
            .map(this::process)
            .subscribe();  // 触发执行
    }
    
    // ❌ 陷阱3：异常吞没
    public Mono<String> badErrorSwallow() {
        return Mono.just("input")
            .map(this::mayThrowException);  // 异常可能未处理
    }
    
    // ✅ 正确：处理异常
    public Mono<String> goodErrorHandling() {
        return Mono.just("input")
            .map(this::mayThrowException)
            .doOnError(e -> log.error("处理失败", e))
            .onErrorResume(e -> Mono.just("default"));
    }
    
    // ❌ 陷阱4：无限流
    public Flux<Long> badInfinite() {
        return Flux.interval(Duration.ofMillis(100));  // 永不停歇！
    }
    
    // ✅ 正确：限制流
    public Flux<Long> goodFinite() {
        return Flux.interval(Duration.ofMillis(100))
            .take(100)  // 只取100个
            .timeout(Duration.ofSeconds(30));  // 30秒超时
    }
    
    // ❌ 陷阱5：调试困难
    public Mono<String> badDebug() {
        return fetchData()
            .map(this::transform)
            .flatMap(this::save);
    }
    
    // ✅ 正确：添加调试点
    public Mono<String> goodDebug() {
        return fetchData()
            .checkpoint("获取数据后")  // 添加检查点
            .log("transform-stage")  // 记录日志
            .map(this::transform)
            .checkpoint("转换后")
            .flatMap(this::save);
    }
}
```

### 响应式编程口诀

```
异步非阻塞用 Reactor，
Mono Flux 要分清。
操作符链式来组合，
背压保护不能忘。
WebFlux 建服务，
RSocket 更高效。
阻塞代码要避免，
响应式里全异步。
```

### 常见陷阱

```
1. ⚠️ 在响应式链中使用阻塞调用（.block()）
2. ⚠️ 忘记处理错误情况
3. ⚠️ 不当的线程调度导致性能下降
4. ⚠️ 没有背压保护导致 OOM
5. ⚠️ 过度使用 flatMap 导致无序
```

