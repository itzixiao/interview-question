### 4.2 缓存更新策略

| 策略                | 描述           | 优点     | 缺点      | 适用场景  |
|-------------------|--------------|--------|---------|-------|
| **Cache Aside**   | 先更新 DB，再删除缓存 | 实现简单   | 存在短暂不一致 | 大多数场景 |
| **Read Through**  | 缓存代理读取 DB    | 应用简单   | 缓存组件复杂  | 读多写少  |
| **Write Through** | 写缓存时同步写 DB   | 数据一致性好 | 写延迟高    | 写入量不大 |
| **Write Behind**  | 异步批量写 DB     | 写入性能高  | 可能丢数据   | 写入量大  |

**Cache Aside 模式实现：**

```java
/**
 * Cache Aside 模式实现
 * 
 * 读流程：
 * 1. 先读缓存
 * 2. 缓存未命中则读数据库
 * 3. 将数据库结果写入缓存
 * 
 * 写流程：
 * 1. 先更新数据库
 * 2. 再删除缓存（不是更新缓存）
 */
public class CacheAsideService {
    
    /**
     * 读取数据
     */
    public User getUser(String userId) {
        // 1. 先读缓存
        User user = redisCache.get("user:" + userId);
        
        if (user != null) {
            return user;  // 缓存命中
        }
        
        // 2. 缓存未命中，读数据库
        user = userRepository.findById(userId);
        
        if (user != null) {
            // 3. 写入缓存
            redisCache.set("user:" + userId, user, 3600);
        } else {
            // 防止缓存穿透：缓存空值
            redisCache.set("user:" + userId, "NULL", 60);
        }
        
        return user;
    }
    
    /**
     * 更新数据
     */
    @Transactional
    public void updateUser(User user) {
        // 1. 先更新数据库
        userRepository.update(user);
        
        // 2. 删除缓存（不是更新）
        // 删除比更新更安全，避免并发写入导致数据不一致
        redisCache.delete("user:" + user.getId());
    }
}
```

### 4.3 缓存问题解决方案

```
┌─────────────────────────────────────────────────────────────────┐
│                    缓存三大问题                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 缓存穿透（Cache Penetration）                                 │
│     问题：查询不存在的数据，缓存和 DB 都没有                           │
│     解决：                                                        │
│     ├─ 布隆过滤器：预先过滤不存在的 key                               │
│     ├─ 空值缓存：缓存 null 值，设置短过期时间                         │
│     └─ 参数校验：过滤非法请求                                       │
│                                                                  │
│  2. 缓存击穿（Cache Breakdown）                                    │
│     问题：热点 key 过期，大量请求同时访问 DB                           │
│     解决：                                                        │
│     ├─ 互斥锁：只允许一个线程重建缓存                                 │
│     ├─ 逻辑过期：不设置 TTL，后台异步更新                             │
│     └─ 热点数据永不过期：配合主动更新策略                              │
│                                                                  │
│  3. 缓存雪崩（Cache Avalanche）                                    │
│     问题：大量缓存同时过期，DB 压力骤增                                │
│     解决：                                                        │
│     ├─ 随机过期时间：避免同时过期                                     │
│     ├─ 多级缓存：L1 + L2 缓存                                       │
│     ├─ 熔断降级：保护数据库                                          │
│     └─ 高可用架构：Redis 哨兵/集群                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 六、高频面试题

**问题 1：Redis 和 Memcached 如何选择？**

**答：**

**选择 Redis 的场景：**

1. 需要复杂数据结构（List、Set、ZSet、Hash）
2. 需要持久化（RDB、AOF）
3. 需要分布式锁、排行榜、消息队列
4. 需要高可用（哨兵、Cluster）
5. Value 较大（超过 1MB）

**选择 Memcached 的场景：**

1. 简单 KV 缓存，数据结构单一
2. 已有成熟的 Memcached 运维体系
3. 多核 CPU 服务器，希望充分利用多核
4. 对简单 KV 性能有极致要求

**问题 2：Redis 和 MongoDB 如何选择？**

**答：**

**选择 Redis 的场景：**

1. 纯缓存场景，数据量不大
2. 需要排行榜、计数器、分布式锁
3. 需要消息队列（简单场景）
4. 对延迟敏感（亚毫秒级）

**选择 MongoDB 的场景：**

1. 需要复杂查询和聚合
2. 数据结构灵活，Schema 经常变化
3. 数据量较大，需要分片
4. 需要持久化存储，数据重要
5. 内容管理、日志分析等场景

**问题 3：如何设计多级缓存架构？**

**答：**

**架构设计：**

1. **L1 本地缓存**：Caffeine/Guava，延迟 ~100ns
2. **L2 分布式缓存**：Redis/Memcached，延迟 ~1ms
3. **L3 数据库**：MySQL/MongoDB，延迟 ~10ms

**关键点：**

1. 缓存一致性：延迟双删、Canal 监听 Binlog
2. 缓存穿透：布隆过滤器、空值缓存
3. 缓存击穿：互斥锁、逻辑过期
4. 缓存雪崩：随机 TTL、熔断降级

**问题 4：缓存和数据库一致性如何保证？**

**答：**

**方案一：Cache Aside（推荐）**

1. 读：先读缓存，未命中读 DB，然后写缓存
2. 写：先更新 DB，再删除缓存

**方案二：延迟双删**

```
1. 删除缓存
2. 更新数据库
3. 延迟一段时间（如 500ms）
4. 再次删除缓存
```

**方案三：Canal 监听 Binlog**

1. 应用只更新数据库
2. Canal 监听 MySQL Binlog
3. 异步更新/删除缓存

**问题 5：如何解决缓存热点问题？**

**答：**

**识别热点 Key：**

1. 监控 QPS 分布
2. Redis MONITOR 命令分析
3. 客户端统计

**解决方案：**

1. **本地缓存**：热点 Key 缓存到应用本地
2. **Key 分片**：将热点 Key 拆分为多个子 Key
3. **多级缓存**：L1 本地缓存 + L2 分布式缓存
4. **限流降级**：保护后端服务
