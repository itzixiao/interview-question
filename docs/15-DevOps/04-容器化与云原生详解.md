## 为什么需要容器化？

### 传统部署 vs 容器化部署

| 对比项   | 传统部署            | 容器化部署          |
|-------|-----------------|----------------|
| 环境一致性 | ❌ 开发、测试、生产环境不一致 | ✅ 一次构建，处处运行    |
| 资源利用率 | ❌ 低（通常<20%）     | ✅ 高（可达 60-80%） |
| 部署速度  | ❌ 分钟级           | ✅ 秒级           |
| 隔离性   | ❌ 进程级隔离         | ✅ 容器级隔离        |
| 弹性伸缩  | ❌ 手动扩缩容         | ✅ 自动扩缩容        |
| 迁移成本  | ❌ 高             | ✅ 低            |

### 容器化的优势

```
┌─────────────────────────────────────────────────┐
│           容器化带来的核心价值                    │
├─────────────────────────────────────────────────┤
│  ✅ 环境一致性：消除"在我机器上能跑"的问题         │
│  ✅ 快速部署：从小时级降到秒级                    │
│  ✅ 资源优化：提高服务器利用率，节省成本          │
│  ✅ 微服务基石：实现服务解耦和独立部署            │
│  ✅ DevOps 支撑：CI/CD的基础设施                  │
│  ✅ 云原生基础：K8s、Service Mesh 的前提          │
└─────────────────────────────────────────────────┘
```

## Docker 基础与镜像构建

### 什么是 Docker？

Docker 是一个开源的应用容器引擎，让开发者可以打包应用以及依赖包到一个可移植的容器中。

**核心概念：**

- **镜像（Image）**：只读模板，包含运行应用所需的代码、库、环境变量和配置文件
- **容器（Container）**：镜像的运行实例
- **Dockerfile**：构建镜像的脚本文件
- **Docker Registry**：存放 Docker 镜像的仓库（如 Docker Hub）

### Dockerfile 详解

#### 基础版示例

```dockerfile
# 使用 OpenJDK 8 作为基础镜像
FROM openjdk:8-jdk-alpine

# 设置工作目录
WORKDIR /app

# 复制 jar 文件
COPY target/interview-containerization-1.0.0.jar app.jar

# 暴露端口
EXPOSE 8080

# 启动应用
ENTRYPOINT ["java", "-jar", "app.jar"]
```

#### 生产优化版（多阶段构建）

```dockerfile
# 第一阶段：构建阶段
FROM maven:3.8-openjdk-8 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn clean package -DskipTests -B

# 第二阶段：运行阶段
FROM openjdk:8-jre-slim

# 创建非 root 用户（安全最佳实践）
RUN groupadd -r appgroup && useradd -r -g appgroup appuser

WORKDIR /app
COPY --from=builder /app/target/interview-containerization-1.0.0.jar app.jar
RUN chown -R appuser:appgroup /app

USER appuser

# JVM 参数优化
ENV JAVA_OPTS="-Xms512m -Xmx512m -XX:+UseG1GC"

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --start-period=60s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1

ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
```

### Docker 常用命令

```bash
# 镜像操作
docker build -t interview-containerization:1.0.0 .          # 构建镜像
docker images                                                  # 查看镜像
docker rmi <image_id>                                          # 删除镜像
docker pull nginx:latest                                       # 拉取镜像
docker push interview-containerization:1.0.0                   # 推送镜像

# 容器操作
docker run -d -p 8080:8080 --name my-app interview-containerization:1.0.0  # 运行容器
docker ps                                                      # 查看运行中的容器
docker logs -f <container_id>                                  # 查看日志
docker exec -it <container_id> sh                              # 进入容器
docker stop <container_id>                                     # 停止容器
docker rm <container_id>                                       # 删除容器

# 网络和存储
docker network ls                                              # 查看网络
docker volume ls                                               # 查看数据卷
```

### Docker 镜像优化技巧

1. **使用多阶段构建**：减少最终镜像大小
2. **选择合适的基础镜像**：Alpine > Slim > Standard
3. **合并 RUN 指令**：减少镜像层数
4. **清理缓存**：`rm -rf /var/cache/apk/*`
5. **使用.dockerignore**：排除不必要的文件

## Docker Compose 多容器编排

### 什么是 Docker Compose？

Docker Compose 是用于定义和运行多容器 Docker 应用程序的工具。通过 YAML 文件配置应用的服务，然后使用一个命令创建所有服务。

### docker-compose.yml 完整示例

```yaml
version: '3.8'

networks:
  app-network:
    driver: bridge

volumes:
  mysql-data:
  redis-data:

services:
  # MySQL 数据库
  mysql:
    image: mysql:8.0
    container_name: interview-mysql
    environment:
      MYSQL_ROOT_PASSWORD: root123
      MYSQL_DATABASE: interview_db
    ports:
      - "3306:3306"
    volumes:
      - mysql-data:/var/lib/mysql
    networks:
      - app-network
    healthcheck:
      test: [ "CMD", "mysqladmin", "ping", "-h", "localhost" ]
      interval: 10s
      timeout: 5s
      retries: 5

  # Redis 缓存
  redis:
    image: redis:7-alpine
    container_name: interview-redis
    ports:
      - "6379:6379"
    volumes:
      - redis-data:/data
    networks:
      - app-network

  # Spring Boot 应用
  app:
    build:
      context: .
      dockerfile: src/main/resources/docker/Dockerfile
    image: interview-containerization:1.0.0
    container_name: interview-app
    depends_on:
      mysql:
        condition: service_healthy
      redis:
        condition: service_healthy
    environment:
      SPRING_PROFILES_ACTIVE: docker
      DB_HOST: mysql
      REDIS_HOST: redis
    ports:
      - "8080:8080"
    networks:
      - app-network
    healthcheck:
      test: [ "CMD", "curl", "-f", "http://localhost:8080/health" ]
      interval: 30s
      timeout: 3s
      start-period: 60s
      retries: 3

  # Nginx 反向代理
  nginx:
    image: nginx:alpine
    container_name: interview-nginx
    depends_on:
      - app
    ports:
      - "80:80"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf
    networks:
      - app-network
```

### Docker Compose 常用命令

```bash
# 启动所有服务
docker-compose up -d

# 查看日志
docker-compose logs -f

# 查看特定服务日志
docker-compose logs -f app

# 停止所有服务
docker-compose down

# 停止并删除数据卷
docker-compose down -v

# 重新构建
docker-compose up -d --build

# 查看服务状态
docker-compose ps

# 进入容器
docker-compose exec app sh

# 扩缩容
docker-compose up -d --scale app=3
```

## Kubernetes 核心概念

### 什么是 Kubernetes？

Kubernetes（简称 K8s）是一个开源的容器编排平台，用于自动化部署、扩展和管理容器化应用。

### K8s 架构图

```
┌──────────────────────────────────────────────────────┐
│                  Kubernetes 集群                      │
├──────────────────────────────────────────────────────┤
│  Control Plane (控制平面)                             │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌─────────┐ │
│  │API Server│ │ Scheduler│ │Controller│ │ etcd    │ │
│  │          │ │          │ │ Manager  │ │         │ │
│  └──────────┘ └──────────┘ └──────────┘ └─────────┘ │
│                                                      │
│  Worker Nodes (工作节点)                             │
│  ┌─────────────────┐  ┌─────────────────┐           │
│  │ Node 1          │  │ Node 2          │           │
│  │ ┌─────────────┐ │  │ ┌─────────────┐ │           │
│  │ │   Kubelet   │ │  │ │   Kubelet   │ │           │
│  │ ├─────────────┤ │  │ ├─────────────┤ │           │
│  │ │ Pod │ Pod   │ │  │ │ Pod │ Pod   │ │           │
│  │ └─────────────┘ │  │ └─────────────┘ │           │
│  └─────────────────┘  └─────────────────┘           │
└──────────────────────────────────────────────────────┘
```

### 1. Pod - 最小调度单元

Pod 是 K8s 中最小的调度和部署单元，包含一个或多个容器。

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: interview-app-pod
  labels:
    app: interview-app
spec:
  containers:
    - name: interview-app
      image: interview-containerization:1.0.0
      ports:
        - containerPort: 8080
      resources:
        requests:
          memory: "512Mi"
          cpu: "250m"
        limits:
          memory: "1Gi"
          cpu: "500m"
      livenessProbe:
        httpGet:
          path: /health
          port: 8080
        initialDelaySeconds: 60
        periodSeconds: 30
```

### 2. Deployment - 无状态应用部署

Deployment 管理 Pod 的副本集，确保期望状态的 Pod 运行。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: interview-app-deployment
spec:
  replicas: 3  # 期望运行 3 个 Pod
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels:
      app: interview-app
  template:
    metadata:
      labels:
        app: interview-app
    spec:
      containers:
        - name: interview-app
          image: interview-containerization:1.0.0
```

**滚动更新过程：**

```
初始状态：[Pod-v1] [Pod-v1] [Pod-v1]
更新开始：[Pod-v1] [Pod-v1] [Pod-v2]  (创建新的 v2)
更新中：  [Pod-v1] [Pod-v2] [Pod-v2]  (删除旧的 v1)
更新完成：[Pod-v2] [Pod-v2] [Pod-v2]
```

### 3. Service - 服务发现与负载均衡

Service 定义了一组 Pod 的访问策略，提供稳定的访问入口。

**Service 类型：**

| 类型           | 说明             | 使用场景        |
|--------------|----------------|-------------|
| ClusterIP    | 集群内部访问（默认）     | 微服务间调用      |
| NodePort     | 节点 IP+ 端口暴露    | 测试环境        |
| LoadBalancer | 云提供商负载均衡       | 生产环境对外服务    |
| Ingress      | HTTP/HTTPS路由规则 | 域名访问、SSL 终止 |

```yaml
apiVersion: v1
kind: Service
metadata:
  name: interview-app-service
spec:
  type: ClusterIP
  selector:
    app: interview-app
  ports:
    - port: 80
      targetPort: 8080
```

### 4. ConfigMap & Secret - 配置管理

**ConfigMap**：存储非敏感配置  
**Secret**：存储敏感信息（Base64 编码）

### 5. HPA - 水平 Pod 自动扩缩容

HPA 根据 CPU/内存使用率或自定义指标自动调整 Pod 副本数。

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: interview-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: interview-app-deployment

  minReplicas: 2
  maxReplicas: 10

  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70  # CPU 使用率超过 70% 扩容

    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80  # 内存使用率超过 80% 扩容

  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容稳定窗口 5 分钟
    scaleUp:
      stabilizationWindowSeconds: 0    # 扩容立即执行
```

**HPA 工作原理：**

```
监控指标 → Metrics Server → HPA Controller → 调整 ReplicaSet → 创建/删除 Pod

CPU 使用率 80% → 触发扩容 → Pod 数量从 3 增加到 5
CPU 使用率 30% → 触发缩容 → Pod 数量从 5 减少到 2
```

## Helm 包管理

### 什么是 Helm？

Helm 是 Kubernetes 的包管理器，类似于 Linux 的 yum/apt。它使用 Chart 来定义、安装和升级 K8s 应用。

### Helm 核心概念

- **Chart**：Helm 应用的打包格式（包含一组 K8s 资源）
- **Repository**：存放 Charts 的仓库
- **Release**：Chart 的运行实例（部署到集群中的应用）

### Chart 目录结构

```
interview-containerization/
├── Chart.yaml              # Chart 元数据
├── values.yaml             # 默认配置值
├── values-prod.yaml        # 生产环境配置
├── charts/                 # 依赖的 Charts
├── templates/              # Kubernetes 资源模板
│   ├── _helpers.tpl        # 模板辅助函数
│   ├── deployment.yaml     # Deployment 模板
│   ├── service.yaml        # Service 模板
│   └── ...
└── .helmignore             # 忽略文件列表
```

### Chart.yaml 示例

```yaml

    condition: mysql.enabled

  - name: redis
    version: 17.11.0
    repository: https://charts.bitnami.com/bitnami
    condition: redis.enabled
```

### Helm 常用命令


## CI/CD流水线实战

### 什么是 CI/CD？

- **CI（持续集成）**：频繁地将代码集成到主干，自动构建和测试
- **CD（持续交付/部署）**：自动将通过测试的代码部署到生产环境

### CI/CD 流程

```
代码提交 → Git 仓库 → CI 触发 → 编译 → 测试 → 打包 → 
构建镜像 → 推送仓库 → CD 触发 → 部署到 K8s → 健康检查
```

### GitLab CI/CD 示例

```yaml
stages:
  - build
  - test
  - package
  - docker-build
  - deploy-dev
  - deploy-prod

variables:
  DOCKER_REGISTRY: registry.example.com
  DOCKER_IMAGE: ${DOCKER_REGISTRY}/interview-containerization

# 构建阶段
build-job:
  stage: build
  image: maven:3.8-openjdk-8
  script:
    - mvn clean compile -DskipTests

# 测试阶段
test-job:
  stage: test
  image: maven:3.8-openjdk-8
  script:
    - mvn test

# 打包阶段
package-job:
  stage: package
  image: maven:3.8-openjdk-8
  script:
    - mvn clean package -DskipTests

# Docker 镜像构建
docker-build-job:
  stage: docker-build
  image: docker:24.0
  services:
    - docker:24.0-dind
  script:
    - docker login -u ${DOCKER_USERNAME} -p ${DOCKER_PASSWORD}
    - docker build -t ${DOCKER_IMAGE}:${CI_COMMIT_SHA} .
    - docker push ${DOCKER_IMAGE}:${CI_COMMIT_SHA}

# 开发环境部署
deploy-dev-job:
  stage: deploy-dev
  image: bitnami/kubectl:latest
  script:
    - kubectl set image deployment/interview-app-deployment \
      interview-app=${DOCKER_IMAGE}:${CI_COMMIT_SHA} -n dev

# 生产环境部署（手动触发）
deploy-prod-job:
  stage: deploy-prod
  image: bitnami/kubectl:latest
  script:
    - helm upgrade --install interview ./helm \
      --namespace production \
      --set image.tag=${CI_COMMIT_SHA}
  when: manual
```

### Jenkins Pipeline 示例

```groovy
pipeline {
    agent any

    tools {
        maven 'Maven-3.8'
        jdk 'JDK-1.8'
    }

    parameters {
        choice(name: 'DEPLOY_ENV', choices: ['dev', 'prod'], description: '部署环境')
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }

        stage('Compile') {
            steps {
                sh 'mvn clean compile -DskipTests'
            }
        }

        stage('Test') {
            steps {
                sh 'mvn test'
            }
        }

        stage('Package') {
            steps {
                sh 'mvn clean package -DskipTests'
            }
        }

        stage('Build Docker Image') {
            steps {
                script {
                    docker.withRegistry('https://registry.example.com', 'credentials-id') {
                        def customImage = docker.build("myapp:${env.BUILD_NUMBER}")
                        customImage.push()
                    }
                }
            }
        }

        stage('Deploy') {
            steps {
                sh '''
                    kubectl set image deployment/myapp myapp=myapp:${env.BUILD_NUMBER}
                '''
            }
        }
    }

    post {
        success {
            echo 'Pipeline completed successfully!'
        }
        failure {
            echo 'Pipeline failed!'
        }
    }
}
```

### 部署策略

#### 1. 蓝绿部署

```
蓝色环境（当前生产）：v1.0
绿色环境（新环境）：v2.0

部署流程：
1. 在绿色环境部署 v2.0
2. 测试绿色环境
3. 切换流量：全部指向绿色环境
4. 保留蓝色环境（快速回滚）
```

**优点**：快速回滚、零停机  
**缺点**：需要双倍资源

#### 2. 金丝雀发布（灰度发布）

```
初始状态：100% 流量 → v1.0

灰度流程：
1. 部署 v2.0 到 1 个 Pod
2. 10% 流量 → v2.0, 90% 流量 → v1.0
3. 观察监控指标（错误率、延迟）
4. 如果正常：50% 流量 → v2.0
5. 继续观察：100% 流量 → v2.0
6. 删除 v1.0
```

**优点**：风险可控、逐步验证  
**缺点**：需要精细的流量控制

#### 3. 滚动更新（K8s 默认）

```
初始：[v1] [v1] [v1]

更新过程：
1. [v1] [v1] [v2]  (创建 v2)
2. [v1] [v2] [v2]  (删除 v1)
3. [v2] [v2] [v2]  (完成更新)
```

**优点**：无需额外资源、平滑过渡  
**缺点**：回滚较慢

**问题 2：如何优化 Docker 镜像大小？**

```dockerfile
# ❌ 不好

RUN apt-get install -y curl
RUN rm -rf /var/cache/apt/*

# ✅ 好
RUN apt-get update && \
    apt-get install -y curl && \
    rm -rf /var/cache/apt/*
```

4. **使用.dockerignore 排除不必要文件**

```

target/
*.md
.env
```

### Kubernetes 相关

**问题 4：Kubernetes 的核心组件有哪些？**

**答：**

**控制平面（Control Plane）：**

1. **API Server**：集群的统一入口，处理 REST 操作
2. **etcd**：分布式键值存储，保存集群状态
3. **Scheduler**：负责 Pod 调度到合适的节点
4. **Controller Manager**：运行各种控制器，维护集群状态

**工作节点（Worker Node）：**

1. **Kubelet**：节点代理，管理 Pod 生命周期
2. **Kube-proxy**：网络代理，实现 Service 负载均衡
3. **Container Runtime**：容器运行时（Docker、containerd）

---

**问题 5：Pod 的生命周期？**

**答：**

```
Pending → Running → Succeeded/Failed
              ↓
           Terminated
```

**阶段说明：**

1. **Pending**：API Server 已创建 Pod，等待调度
2. **Running**：至少一个容器正在运行
3. **Succeeded**：所有容器成功退出
4. **Failed**：至少一个容器异常退出
5. **Terminated**：Pod 被删除

**探针机制：**

- **livenessProbe**：存活探针，失败则重启容器
- **readinessProbe**：就绪探针，失败则从 Service 后端移除
- **startupProbe**：启动探针，失败则不执行其他探针

**问题 7：HPA 的工作原理？**

**答：**

**工作流程：**

```
Metrics Server → 采集指标 → HPA Controller → 
计算目标副本数 → 调整 Deployment → 创建/删除 Pod
```

**计算公式：**

```
目标副本数 = ceil(当前副本数 × 当前指标值 / 目标指标值)

例如：
当前：3 个 Pod，CPU 使用率 90%
目标：CPU 使用率 70%
需要：ceil(3 × 90 / 70) = ceil(3.86) = 4 个 Pod
```

**扩缩容行为：**

- **扩容**：立即执行（stabilizationWindowSeconds=0）
- **缩容**：等待稳定窗口期（默认 300 秒），防止抖动

**问题 9：如何实现灰度发布？**

**答：**

**方案 1：基于权重的流量分配**

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: interview-vs
spec:
  hosts:
    - interview.example.com
  http:
    - route:
        - destination:
            host: interview-app-v1
          weight: 90  # 90% 流量到 v1
        - destination:
            host: interview-app-v2
          weight: 10  # 10% 流量到 v2
```

**方案 2：基于 Header 的灰度**

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: interview-vs
spec:
  hosts:
    - interview.example.com
  http:
    - match:
        - headers:
            version:
              exact: beta  # 匹配特定 Header
      route:
        - destination:
            host: interview-app-v2
    - route:
        - destination:
            host: interview-app-v1
```

**方案 3：K8s + Helm 分阶段发布**

```bash
# 阶段 1：部署 1 个 Pod（10% 流量）
helm upgrade interview ./chart --set replicas=1

# 观察监控指标（错误率、延迟）
kubectl top pods

# 阶段 2：扩展到 3 个 Pod（30% 流量）
helm upgrade interview ./chart --set replicas=3

# 阶段 3：全量发布
helm upgrade interview ./chart --set replicas=10
```
