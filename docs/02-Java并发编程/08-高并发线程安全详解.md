# 高并发线程安全详解

## 一、线程安全问题产生的根本原因

### 1. 原子性问题

多步操作被打断，导致中间状态被其他线程读取或覆盖。

```java
// count++ 不是原子操作，包含三步：
// 1. 读取 count 值到寄存器
// 2. 寄存器值 +1
// 3. 写回内存

// 两个线程同时执行可能的情况：
// 线程1 读取: count = 0
// 线程2 读取: count = 0
// 线程1 写入: count = 1
// 线程2 写入: count = 1  ← 丢失了一次更新！
```

### 2. 可见性问题

线程修改变量后，其他线程看不到最新值。

**原因**：每个线程有自己的工作内存（CPU 缓存），修改后可能不会立即刷新到主内存。

```
线程1         线程2         主内存
┌─────┐      ┌─────┐      ┌─────┐
│ x=1 │      │ x=0 │      │ x=1 │
└─────┘      └─────┘      └─────┘
   ↑            ↑            ↑
 修改了        没看到        实际值
```

### 3. 有序性问题

指令重排序导致执行顺序与代码顺序不一致。

```java
// 代码顺序
int a = 1;  // 1
int b = 2;  // 2

// 可能的执行顺序
int b = 2;  // 2 先执行
int a = 1;  // 1 后执行
```

**编译器和 CPU 为什么要重排序？**
为了优化性能。但在多线程环境下可能导致问题。

## 二、解决方案详解

### 方案1：synchronized

**特点**：

- JVM 内置，自动获取释放
- 可重入（同一线程可多次获取）
- JDK 6+ 性能优化（锁升级）

**三种使用方式**：

```java
// 1. 同步实例方法（锁 this）
public synchronized void method() { }

// 2. 同步静态方法（锁 Class）
public static synchronized void staticMethod() { }

// 3. 同步代码块（锁指定对象）
public void method() {
    synchronized (lockObject) { }
}
```

**锁升级过程**：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁

偏向锁：只有一个线程访问，Mark Word 记录线程ID
轻量级锁：多线程竞争，CAS 自旋
重量级锁：自旋失败，线程阻塞
```

---

### 方案2：Lock

**相比 synchronized 的优势**：

- 可中断：`lockInterruptibly()`
- 可超时：`tryLock(timeout)`
- 可公平：`new ReentrantLock(true)`
- 多条件：多个 `Condition`

```java
ReentrantLock lock = new ReentrantLock();

lock.lock();
try {
    // 临界区
} finally {
    lock.unlock();  // 必须手动释放
}

// 超时获取
if (lock.tryLock(1, TimeUnit.SECONDS)) {
    try {
        // 获取到锁
    } finally {
        lock.unlock();
    }
} else {
    // 超时，执行其他逻辑
}
```

**读写锁**：

```java
ReadWriteLock rwLock = new ReentrantReadWriteLock();

// 读锁（共享）
rwLock.readLock().lock();
try {
    // 读操作
} finally {
    rwLock.readLock().unlock();
}

// 写锁（独占）
rwLock.writeLock().lock();
try {
    // 写操作
} finally {
    rwLock.writeLock().unlock();
}
```

### 方案3：volatile

**作用**：

- 保证可见性：修改立即刷新到主内存
- 禁止重排序：通过内存屏障实现

**内存屏障（Memory Barrier）详解**：

内存屏障是一种 CPU 指令，用于确保指令的执行顺序。JVM 在 volatile 变量的读写操作前后插入特定的内存屏障，防止编译器和 CPU 进行指令重排序。

| 屏障类型 | 指令示例 | 作用 |
|---------|---------|------|
| LoadLoad | Load1; LoadLoad; Load2 | 确保 Load1 在 Load2 之前执行 |
| StoreStore | Store1; StoreStore; Store2 | 确保 Store1 在 Store2 之前执行 |
| LoadStore | Load1; LoadStore; Store2 | 确保 Load1 在 Store2 之前执行 |
| StoreLoad | Store1; StoreLoad; Load2 | 确保 Store1 在 Load2 之前执行（开销最大） |

**volatile 的内存屏障插入策略**：

```
volatile 写操作：
    [普通写操作]
    StoreStore 屏障  // 阻止普通写与 volatile 写重排序
    volatile 写
    StoreLoad 屏障   // 阻止 volatile 写与后续 volatile 读/写重排序

volatile 读操作：
    LoadLoad 屏障    // 阻止 volatile 读与普通读重排序
    LoadStore 屏障   // 阻止 volatile 读与普通写重排序
    volatile 读
    [后续操作]
```

**为什么不能保证原子性**：

`count++` 实际上是三个步骤：读取 → 修改 → 写入。volatile 只能保证单个读/写的原子性，但无法保证复合操作的原子性。


```java
volatile int count = 0;
count++;  // 仍然不是线程安全的
```

**适用场景**：

- 状态标志
- 双重检查锁定（DCL）单例

```java
// 状态标志
volatile boolean running = true;

public void stop() {
    running = false;
}

public void run() {
    while (running) {
        // 工作
    }
}
```

### 方案4：原子类

**分类**：

| 类型   | 类                                        |
|------|------------------------------------------|
| 基本类型 | AtomicInteger, AtomicLong, AtomicBoolean |
| 引用类型 | AtomicReference, AtomicStampedReference  |
| 数组类型 | AtomicIntegerArray, AtomicLongArray      |
| 累加器  | LongAdder, DoubleAdder                   |

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();  // 线程安全的自增
count.compareAndSet(expected, newValue);  // CAS 操作
```

**CAS 原理**：

```
比较内存值 V 与预期值 A
如果 V == A，则 V = B（更新）
否则，重试
```

**LongAdder vs AtomicLong**：

- AtomicLong：所有线程竞争同一个变量
- LongAdder：分散到多个 Cell，减少竞争

---

### 方案5：并发容器

| 容器                    | 特点                       |
|-----------------------|--------------------------|
| ConcurrentHashMap     | JDK8: CAS + synchronized |
| CopyOnWriteArrayList  | 写时复制，读无锁                 |
| BlockingQueue         | 阻塞队列                     |
| ConcurrentLinkedQueue | 无锁队列                     |

```java
ConcurrentHashMap<String, Object> map = new ConcurrentHashMap<>();
map.put("key", "value");
map.computeIfAbsent("key", k -> createValue());

CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();
list.add("item");  // 写时复制整个数组

BlockingQueue<String> queue = new ArrayBlockingQueue<>(100);
queue.put(item);   // 队列满时阻塞
queue.take();      // 队列空时阻塞
```

---

### 方案6：ThreadLocal

**原理**：每个线程有自己的 ThreadLocalMap，实现线程隔离。

```java
ThreadLocal<String> threadLocal = new ThreadLocal<>();
threadLocal.set("value");
String value = threadLocal.get();
threadLocal.remove();  // 避免内存泄漏
```

**内存泄漏问题**：

- key 是弱引用，可能被回收
- value 是强引用，不会被回收
- 解决：使用完调用 `remove()`

---

### 方案7：不可变对象

**特点**：

- 对象创建后状态不能改变
- 所有字段都是 final
- 天然线程安全

```java
public final class ImmutableUser {
    private final String name;
    private final int age;
    
    public ImmutableUser(String name, int age) {
        this.name = name;
        this.age = age;
    }
    
    // 只有 getter，没有 setter
    public String getName() { return name; }
    public int getAge() { return age; }
    
    // 修改返回新对象
    public ImmutableUser withName(String newName) {
        return new ImmutableUser(newName, this.age);
    }
}
```

## 四、高频面试题

**问题 1:synchronized 和 Lock 有什么区别？**

| 特性   | synchronized | Lock              |
|------|--------------|-------------------|
| 实现层面 | JVM 内置       | Java API          |
| 释放方式 | 自动释放         | 必须手动 unlock       |
| 可中断  | 不可中断         | lockInterruptibly |
| 超时获取 | 不支持          | tryLock(timeout)  |
| 公平性  | 只有非公平        | 可配置               |
| 条件队列 | 单个           | 多个 Condition      |

**问题 2:volatile 能保证线程安全吗？**

**不能完全保证**。volatile 只能保证可见性和有序性，不能保证原子性。

`volatile int count; count++` 仍然不是线程安全的。

**适用场景**：状态标志、DCL 单例。

**问题 3:CAS 是什么？有什么问题？**

**CAS (Compare And Swap)**：比较内存值与预期值，相等则更新。

**问题**：

1. **ABA 问题**：值 A→B→A，解决：AtomicStampedReference
2. **自旋消耗 CPU**：解决：LongAdder
3. **只能保证单个变量**：解决：AtomicReference 或加锁

**问题 4：什么是死锁？如何避免？**

**死锁**：多个线程互相等待对方持有的资源。

**四个必要条件**：

1. 互斥
2. 请求与保持
3. 不剥夺
4. 循环等待

**避免方法**：

1. 按固定顺序获取锁
2. 使用 tryLock 超时
3. 使用并发工具类

**问题 5:synchronized 的锁升级过程？**

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

- **偏向锁**：一个线程访问，记录线程ID
- **轻量级锁**：多线程交替，CAS 自旋
- **重量级锁**：竞争激烈，线程阻塞

锁只能升级，不能降级。

**问题 6:ThreadLocal 会导致内存泄漏吗？**

**可能会**。key 是弱引用，value 是强引用。如果 key 被回收，value 无法回收。

**解决**：使用完调用 `remove()`。

**问题 7:ConcurrentHashMap 是如何保证线程安全的？**

**JDK 7**：分段锁，16 个 Segment

**JDK 8**：CAS + synchronized

- 空桶：CAS 插入
- 非空桶：synchronized 锁头节点

**问题 8：什么是可见性？为什么会有可见性问题？**

**可见性**：线程修改变量后，其他线程能立即看到。

**原因**：每个线程有工作内存（CPU缓存），修改可能不会立即刷新到主内存。

**解决**：volatile、synchronized、final。

**问题 9:LongAdder 为什么比 AtomicLong 快？**

**AtomicLong**：所有线程竞争同一个变量，CAS 失败率高。

**LongAdder**：分散到 base + Cell 数组，减少竞争。

**问题 10：如何选择合适的线程安全方案？**

- **简单计数**：AtomicInteger（低并发）/ LongAdder（高并发）
- **复杂临界区**：synchronized / Lock
- **状态标志**：volatile
- **并发集合**：ConcurrentHashMap
- **线程隔离**：ThreadLocal
- **值对象**：不可变对象

---

## 最佳实践

1. **优先使用并发工具类**：比手动加锁更安全
2. **缩小同步范围**：只锁必要的代码
3. **避免嵌套锁**：容易死锁
4. **使用 try-finally**：确保锁释放
5. **ThreadLocal 及时清理**：避免内存泄漏
6. **优先考虑不可变对象**：天然线程安全
