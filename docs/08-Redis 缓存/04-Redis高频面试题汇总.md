# Redis 高频面试题汇总

本文汇总 Redis 相关的高频面试题，涵盖：基础、缓存、分布式锁、持久化、高可用、限流、延时队列等核心主题。

---

## 一、Redis 基础与数据结构

### 1. Redis 支持哪些数据类型？底层实现是什么？

**参考答案：**

| 数据类型 | 底层实现 | 使用场景 |
|----------|----------|----------|
| **String** | SDS（Simple Dynamic String） | 缓存、计数器、分布式锁 |
| **List** | QuickList（双向链表 + ziplist） | 消息队列、最新列表 |
| **Hash** | ziplist / hashtable | 对象存储、购物车 |
| **Set** | intset / hashtable | 标签、共同好友、去重 |
| **ZSet** | ziplist / skiplist + hashtable | 排行榜、延时队列 |
| **Bitmap** | String | 签到、布隆过滤器 |
| **HyperLogLog** | 稀疏/稠密存储 | UV 统计 |
| **Stream** | Radix Tree | 消息队列（Redis 5.0+） |

---

### 2. Redis 为什么这么快？

**参考答案：**

1. **纯内存操作**：数据存储在内存中，读写速度极快
2. **单线程模型**：避免线程切换和锁竞争开销
3. **IO 多路复用**：使用 epoll/kqueue 等高效 IO 模型
4. **高效数据结构**：如 SDS、跳表、压缩列表等
5. **渐进式 rehash**：避免一次性 rehash 导致的卡顿

**注意：** Redis 6.0 引入多线程 IO，但命令执行仍是单线程。

---

### 3. Redis 单线程为什么能支持高并发？

**参考答案：**

1. **内存操作速度快**：单次操作微秒级别
2. **IO 多路复用**：单线程监听多个连接，避免阻塞
3. **避免上下文切换**：无线程切换开销
4. **避免锁竞争**：无需加锁，执行效率高
5. **简单操作为主**：大部分操作 O(1) 或 O(log n)

## 二、缓存问题

### 4. 什么是缓存穿透？如何解决？

**参考答案：**

**定义：** 查询不存在的数据，请求绕过缓存直达数据库。

**解决方案：**

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **缓存空值** | 查询为空时也缓存（设置短过期时间） | 简单，但占用内存 |
| **布隆过滤器** | 在缓存前加一层布隆过滤器 | 效果好，但有误判率 |
| **参数校验** | 在入口处校验参数合法性 | 简单有效 |

---

### 5. 什么是缓存击穿？如何解决？

**参考答案：**

**定义：** 热点 key 过期瞬间，大量并发请求打到数据库。

**解决方案：**

| 方案 | 说明 |
|------|------|
| **互斥锁** | 只允许一个线程重建缓存，其他线程等待 |
| **逻辑过期** | 缓存永不过期，后台异步更新 |
| **热点 key 永不过期** | 对热点数据不设置过期时间 |

**互斥锁示例：**
```java
public Object getData(String key) {
    Object value = redis.get(key);
    if (value == null) {
        if (redis.setnx("lock:" + key, 1, 3, SECONDS)) {
            try {
                value = db.query(key);
                redis.set(key, value, 30, MINUTES);
            } finally {
                redis.del("lock:" + key);
            }
        } else {
            Thread.sleep(50);
            return getData(key); // 重试
        }
    }
    return value;
}
```

---

### 6. 什么是缓存雪崩？如何解决？

**参考答案：**

**定义：** 大量 key 同时过期，或 Redis 宕机，导致请求全部打到数据库。

**解决方案：**

| 方案 | 说明 |
|------|------|
| **随机过期时间** | 在基础过期时间上加随机值 |
| **多级缓存** | 本地缓存（Caffeine）+ Redis |
| **熔断降级** | 使用 Sentinel/Hystrix 熔断 |
| **高可用架构** | Redis Cluster + Sentinel |
| **限流** | 限制到数据库的请求量 |

---

### 7. 如何保证缓存与数据库的一致性？

**参考答案：**

**Cache Aside 模式（推荐）：**
- **读**：先读缓存，未命中读 DB，写入缓存
- **写**：先更新 DB，再删除缓存

**为什么是删除而不是更新缓存？**
- 更新缓存可能导致并发问题
- 删除更简单，下次读取时重建

**延迟双删：**
```
1. 删除缓存
2. 更新数据库
3. 延迟 500ms
4. 再次删除缓存
```

## 三、分布式锁

### 8. Redis 如何实现分布式锁？有什么问题？

**参考答案：**

**基本实现：**
```bash
SET lock:key unique_value NX EX 30
```

**存在的问题：**

| 问题 | 说明 |
|------|------|
| 不可重入 | 同一线程无法多次获取锁 |
| 锁过期业务未完成 | 可能导致并发问题 |
| 主从切换丢锁 | 主节点故障时，锁可能丢失 |
| 无法阻塞等待 | SETNX 获取失败后需自己实现重试 |

---

### 9. Redisson 分布式锁的原理是什么？看门狗机制？

**参考答案：**

**Redisson 锁特点：**
- 可重入（记录线程标识和重入次数）
- 自动续期（看门狗机制）
- 支持阻塞等待
- 支持公平锁、读写锁

**看门狗机制：**
1. 获取锁成功后，启动后台线程（看门狗）
2. 每隔 10 秒（过期时间的 1/3）检查锁是否仍被持有
3. 如果业务未完成，自动将锁续期到 30 秒
4. 业务完成或服务宕机，看门狗停止，锁最终过期释放

**使用示例：**
```java
RLock lock = redisson.getLock("myLock");
lock.lock();  // 不指定过期时间，启用看门狗
try {
    // 执行业务
} finally {
    lock.unlock();
}
```

---

### 10. 什么是 RedLock？有什么争议？

**参考答案：**

**RedLock 原理：**
1. 向 N 个独立的 Redis 节点申请锁
2. 计算获取锁的总耗时
3. 多数节点（N/2+1）成功且耗时 < 过期时间，则获取锁成功
4. 释放时向所有节点发送释放命令

**争议：**
- Martin Kleppmann 指出 RedLock 在时钟漂移等场景下不安全
- Redis 作者 antirez 进行了反驳

**建议：**
- 对一致性要求极高的场景，使用 ZooKeeper 或 etcd
- 一般场景使用单节点 Redis + 持久化即可

## 四、持久化

### 11. RDB 和 AOF 的区别是什么？如何选择？

**参考答案：**

| 特性 | RDB | AOF |
|------|-----|-----|
| 持久化方式 | 定时快照（二进制） | 记录写命令（文本） |
| 数据安全性 | 可能丢失最后一次快照后数据 | 最多丢1秒数据 |
| 文件大小 | 紧凑 | 较大 |
| 恢复速度 | 快 | 慢 |
| 性能影响 | fork 时影响 | 写入时影响 |

**选择建议：**
- **纯缓存**：关闭持久化或仅 RDB
- **数据重要**：混合持久化（推荐）
- **极其重要**：AOF (always)

---

### 12. 什么是混合持久化？为什么推荐？

**参考答案：**

**原理：**
- AOF 重写时，文件开头是 RDB 格式的全量数据
- 后续是 AOF 格式的增量命令

**优点：**
1. 快速恢复（RDB 部分直接加载）
2. 数据安全（AOF 部分，最多丢1秒）
3. 文件较小

**配置：**
```bash
appendonly yes
aof-use-rdb-preamble yes
```

## 五、高可用

### 13. Redis 主从复制的原理是什么？

**参考答案：**

**复制方式：**
1. **全量复制**：首次同步或无法部分复制时，发送完整 RDB
2. **部分复制**：断线重连，只同步增量数据

**全量复制流程：**
1. 从节点发送 PSYNC 命令
2. 主节点执行 BGSAVE 生成 RDB
3. 发送 RDB 给从节点
4. 从节点加载 RDB
5. 主节点发送复制缓冲区的增量命令

---

### 14. Redis Sentinel 如何实现故障转移？

**参考答案：**

**故障转移流程：**
1. **主观下线**：单个 Sentinel 认为主节点故障
2. **客观下线**：多数 Sentinel（≥quorum）确认
3. **选举 Leader**：使用 Raft 算法选举 Leader Sentinel
4. **选择新主节点**：根据优先级、复制偏移量、runid
5. **执行切换**：向新主节点发送 `SLAVEOF NO ONE`
6. **更新配置**：通知其他从节点和客户端

---

### 15. Redis Cluster 如何分片？为什么是 16384 个槽？

**参考答案：**

**分片机制：**
```
slot = CRC16(key) % 16384
```

**为什么是 16384？**
1. 节点数量权衡：支持约 1000 个节点
2. Gossip 消息大小：16384 槽约 2KB，65536 槽约 8KB
3. 2 的幂次方：16384 = 2^14，计算效率高

---

### 16. Redis Cluster 如何处理请求路由？MOVED 和 ASK 的区别？

**参考答案：**

**路由流程：**
1. 客户端计算 key 的槽位
2. 连接对应节点执行命令
3. 如果槽位迁移，收到重定向响应

**MOVED vs ASK：**

| 类型 | 说明 | 处理方式 |
|------|------|----------|
| MOVED | 槽位永久迁移 | 更新本地槽位映射 |
| ASK | 槽位正在迁移 | 仅对当前请求有效 |

## 六、限流与延时队列

### 17. Redis 如何实现限流？常见算法有哪些？

**参考答案：**

**常见限流算法：**

| 算法 | 原理 | 特点 |
|------|------|------|
| **计数器** | 固定窗口计数 | 简单，但有临界问题 |
| **滑动窗口** | 多个小窗口 | 更精确 |
| **令牌桶** | 固定速率产生令牌 | 允许突发流量 |
| **漏桶** | 固定速率处理请求 | 流量整形 |

**令牌桶 Redis 实现（Lua 脚本）：**
```lua
local capacity = tonumber(ARGV[1])
local rate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
-- ... 计算当前令牌数，判断是否允许请求
```

---

### 18. Redis 如何实现延时队列？

**参考答案：**

**Sorted Set 实现：**
```java
// 添加延时任务（score = 执行时间戳）
redis.zadd("delay:queue", executeTime, taskId);

// 获取到期任务
Set<String> tasks = redis.zrangeByScore("delay:queue", 0, now);

// 处理后删除
redis.zrem("delay:queue", taskId);
```

**Redisson 实现：**
```java
RDelayedQueue<String> delayedQueue = redisson.getDelayedQueue("myQueue");
delayedQueue.offer("task:1001", 10, TimeUnit.SECONDS);
```

---

### 19. Redis 如何实现分布式幂等？

**参考答案：**

**Token 机制：**
1. 客户端请求接口前，先获取幂等 Token
2. 请求时携带 Token
3. 服务端使用 Lua 脚本原子校验并删除 Token

```java
// Lua 脚本：校验 Token
String script = 
    "if redis.call('get', KEYS[1]) then " +
    "    redis.call('del', KEYS[1]); return 1; " +
    "else return 0; end";

boolean valid = redis.eval(script, key);
if (valid) {
    // 执行业务
}
```

## 七、其他高频问题

### 20. Redis 的过期键删除策略是什么？

**参考答案：**

**两种策略结合使用：**

| 策略 | 说明 |
|------|------|
| **惰性删除** | 访问 key 时检查是否过期 |
| **定期删除** | 每隔 100ms 随机检查一批 key |

**注意：** 可能存在过期 key 未被及时删除的情况，需配合内存淘汰策略。

---

### 21. Redis 的内存淘汰策略有哪些？

**参考答案：**

| 策略 | 说明 |
|------|------|
| `noeviction` | 不淘汰，内存满时写入报错 |
| `allkeys-lru` | 所有 key 中淘汰最近最少使用的 |
| `volatile-lru` | 设置了过期时间的 key 中淘汰 LRU |
| `allkeys-random` | 所有 key 中随机淘汰 |
| `volatile-random` | 设置了过期时间的 key 中随机淘汰 |
| `volatile-ttl` | 淘汰即将过期的 key |
| `allkeys-lfu` | 所有 key 中淘汰使用频率最低的（Redis 4.0+） |
| `volatile-lfu` | 设置了过期时间的 key 中淘汰 LFU |

**推荐：** 缓存场景使用 `allkeys-lru` 或 `allkeys-lfu`

---

### 22. Redis 事务支持回滚吗？和数据库事务有什么区别？

**参考答案：**

**Redis 事务特点：**
- **不支持回滚**：命令执行失败不会回滚已执行的命令
- **原子性**：事务中的命令要么全部执行，要么全部不执行
- **无隔离级别**：事务中的命令在 EXEC 之前不会被执行

**与数据库事务区别：**

| 特性 | Redis | 数据库 |
|------|-------|--------|
| ACID | 不完全支持 | 完全支持 |
| 回滚 | 不支持 | 支持 |
| 隔离级别 | 无 | 有（四种） |
| 持久性 | 取决于持久化配置 | 默认保证 |

### 23. 如何解决 Redis 大 Key 问题？

**参考答案：**

**大 Key 危害：**
- 内存分配不均
- 删除时阻塞
- 网络传输慢

**解决方案：**
1. **拆分**：将大 Hash/List 拆分成多个小 key
2. **异步删除**：使用 `UNLINK` 代替 `DEL`
3. **批量操作**：分批处理，避免一次性操作
4. **压缩**：对 value 进行压缩

**发现大 Key：**
```bash
redis-cli --bigkeys
redis-cli MEMORY USAGE key
```

### 24. Redis 如何实现消息队列？有什么局限？

**参考答案：**

**实现方式：**

| 方式 | 命令 | 特点 |
|------|------|------|
| List | LPUSH/BRPOP | 简单，但不支持多消费者 |
| Pub/Sub | PUBLISH/SUBSCRIBE | 广播模式，消息不持久化 |
| Stream | XADD/XREAD | 功能完善，支持消费者组（推荐） |

**局限：**
- 不保证消息可靠性（需要自己实现 ACK）
- 不支持延时消息（需配合 ZSet）
- 功能不如专业 MQ（RabbitMQ/Kafka）

---

### 25. Redis 6.0 为什么引入多线程？

**参考答案：**

**背景：**
- Redis 瓶颈不在 CPU，而在网络 IO
- 单线程处理大量网络请求时 CPU 利用率不高

**多线程设计：**
- 仅 IO 线程多线程（读取请求、发送响应）
- **命令执行仍是单线程**
- 避免了加锁，保持简单性

**开启方式：**
```bash
io-threads 4
io-threads-do-reads yes
```
