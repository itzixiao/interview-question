## 💡 高频面试题 Top 10

### 微服务拆分原则（重点）

**问题 1：微服务拆分有哪些基本原则？**

答案：

微服务拆分的核心原则是**高内聚、低耦合**，具体遵循以下原则：

**1. 单一职责原则（SRP）**
```markdown
- 每个服务只负责一个业务领域
- 服务内部功能高度相关
- 避免"大杂烩"式的服务
```

**2. 基于业务领域拆分**
```markdown
✅ 推荐：按业务模块划分
   - 用户服务（User Service）
   - 订单服务（Order Service）
   - 支付服务（Payment Service）
   - 商品服务（Product Service）

❌ 不推荐：按技术层划分
   - DAO 服务、Service 服务、Controller 服务
```

**3. 数据自治原则**
```markdown
- 每个服务拥有独立的数据库
- 服务间不共享数据库表
- 通过 API 进行数据交互
```

**4. 松耦合原则**
```markdown
- 服务间接口清晰明确
- 减少循环依赖
- 避免分布式事务
```

**5. 可独立部署**
```markdown
- 每个服务可独立发布
- 不影响其他服务
- 支持持续交付
```

**问题 2：常见的微服务拆分模式有哪些？**

答案：

**模式 1：按业务领域拆分（推荐）**
```
电商系统：
├── 用户中心（注册、登录、个人信息）
├── 商品中心（商品管理、库存管理）
├── 订单中心（下单、订单查询）
├── 支付中心（支付、退款）
└── 物流中心（发货、物流跟踪）
```

**模式 2：按并发量拆分**
```
高并发服务：
- 秒杀服务（独立部署，专门优化）
- 缓存服务（Redis 集群）

普通服务：
- 后台管理服务
- 报表统计服务
```

**模式 3：按数据敏感性拆分**
```
核心服务（高可用、强一致性）：
- 支付服务
- 账户服务

非核心服务（最终一致性）：
- 积分服务
- 消息通知服务
```

**模式 4：按变更频率拆分**
```
频繁变更的服务：
- 营销活动服务
- 优惠券服务

稳定的服务：
- 基础数据服务
- 字典服务
```

**问题 3：微服务拆分的常见陷阱有哪些？**

答案：

**❌ 陷阱 1：过度拆分**
```markdown
问题：
- 服务粒度过小（如一个 CRUD 一个服务）
- 导致运维复杂度激增
- 网络调用开销过大

案例：
100 个表 → 拆成 100 个服务 → 难以维护

建议：
- 初期适度粗粒度
- 随业务发展逐步细化
- 保持服务数量在可控范围（20-50 个）
```

**❌ 陷阱 2：拆分过晚**
```markdown
问题：
- 单体应用过于庞大
- 代码耦合严重
- 拆分成本极高

建议：
- 在系统复杂度可控时及时拆分
- 当团队规模超过 10 人时考虑拆分
- 当日活达到 10 万 + 时评估拆分
```

**❌ 陷阱 3：共享数据库**
```markdown
问题：
- 多个服务共用一个数据库
- 导致数据耦合
- 无法独立扩展

正确做法：
- 每个服务独立数据库
- 通过 API 或消息队列交互
```

**❌ 陷阱 4：分布式事务滥用**
```markdown
问题：
- 过度使用强一致性分布式事务
- 性能极差
- 系统复杂

建议：
- 优先使用最终一致性
- 采用 Saga、TCC 等模式
- 通过消息队列解耦
```

**❌ 陷阱 5：忽视监控和日志**
```markdown
问题：
- 服务过多后难以排查问题
- 缺乏链路追踪
- 故障定位困难

建议：
- 建设完善的监控体系
- 实现全链路日志追踪
- 配置告警机制
```

**问题 4：如何判断是否需要拆分微服务？**

答案：

**判断维度 1：团队规模**
```
✅ 需要拆分：
- 开发团队 > 10 人
- 多个团队协作开发
- 沟通成本高

❌ 暂不拆分：
- 团队 < 5 人
- 快速迭代验证阶段
```

**判断维度 2：系统复杂度**
```
✅ 需要拆分：
- 代码量 > 50 万行
- 模块间耦合严重
- 编译时间长（>5 分钟）

❌ 暂不拆分：
- 业务逻辑简单
- 代码结构清晰
```

**判断维度 3：性能瓶颈**
```
✅ 需要拆分：
- 部分模块资源紧张
- 需要独立扩展
- 并发量差异大

❌ 暂不拆分：
- 整体负载均衡
- 资源充足
```

**判断维度 4：业务需求**
```
✅ 需要拆分：
- 不同业务线独立运营
- 需要快速迭代
- 部署频率差异大

❌ 暂不拆分：
- 业务相对统一
- 变更频率一致
```

**决策矩阵：**
```
团队规模    系统复杂度    性能需求    是否拆分
─────────────────────────────────────
>10 人      高          高         ✅ 立即拆分
5-10 人     中          中         ⚠️ 评估拆分
<5 人       低          低         ❌ 暂缓拆分
```

### CAP 定理与 BASE 理论（核心理论）

**问题 6：什么是 CAP 定理？在微服务中如何应用？**

答案：

CAP 定理（CAP Theorem）是分布式系统的基石，由 Eric Brewer 在 2000 年提出。

**CAP 三要素：**
```
C（Consistency）一致性：
  - 所有节点在同一时间看到的数据相同
  - 强一致性，类似关系型数据库的 ACID
  
A（Availability）可用性：
  - 每次请求都能获得响应
  - 不保证是最新数据
  
P（Partition tolerance）分区容错性：
  - 网络分区发生时（节点间通信失败），系统仍能运行
  - 分布式系统必须满足
```

**CAP 不可能三角：**
```
          C（一致性）
       /            \
      /              \
     /                \
 A （可用性）—————— P（分区容错性）
   
结论：三者最多同时满足两个！
```

**三种组合策略：**

**1. CP 架构（一致性 + 分区容错性）**
```markdown
特点：
- 保证数据强一致性
- 网络分区时可能不可用
- 适合对数据准确性要求高的场景

典型代表：
✅ HBase
✅ MongoDB（默认配置）
✅ Redis（主从同步时）
✅ ZooKeeper

应用场景：
- 金融支付系统
- 账户余额管理
- 库存扣减
```

**2. AP 架构（可用性 + 分区容错性）**
```markdown
特点：
- 保证高可用
- 数据最终一致性
- 适合对实时性要求不高的场景

典型代表：
✅ Cassandra
✅ DynamoDB
✅ CouchDB
✅ Eureka（服务注册中心）

应用场景：
- 社交媒体的点赞数
- 商品浏览量统计
- 评论系统
```

**3. CA 架构（一致性 + 可用性）**
```markdown
特点：
- 单机数据库架构
- 无法容忍网络分区
- 不是真正的分布式系统

典型代表：
✅ MySQL 单实例
✅ PostgreSQL 单实例
✅ Oracle 单实例

注意：
❌ 分布式系统中不存在 CA 架构
❌ 因为 P（分区容错性）是必须的
```

**微服务中的 CAP 权衡：**

| 场景           | 选择 | 理由                           |
|----------------|------|--------------------------------|
| 支付交易       | CP   | 必须保证数据强一致性           |
| 商品库存       | CP   | 避免超卖                       |
| 用户信息       | CP   | 个人信息准确性要求高           |
| 新闻浏览       | AP   | 短暂不一致可接受               |
| 推荐系统       | AP   | 高优先级，允许最终一致性       |
| 日志收集       | AP   | 允许丢失，保证可用性           |

**问题 8：Redis、MySQL、Zookeeper、RabbitMQ 分别对应哪种 CAP 架构？**

答案：

这是分布式系统选型的经典问题，需要理解每种中间件的 CAP 倾向。

**常见中间件 CAP 分类总览：**

| 中间件        | CAP 选择 | 说明                              |
|---------------|----------|-----------------------------------|
| **ZooKeeper** | ✅ CP     | 强一致性，分区时可能不可用        |
| **Redis**     | ⚠️ CP/AP | 可配置，默认 CP，集群模式偏 AP    |
| **MySQL**     | ✅ CP     | 强一致性（单实例 CA，集群 CP）    |
| **MongoDB**   | ⚠️ CP/AP | 可配置，默认 CP                   |
| **Cassandra** | ✅ AP     | 高可用，最终一致性                |
| **Eureka**    | ✅ AP     | 高可用，节点平等                  |
| **Nacos**     | ⚠️ CP/AP | 支持切换，默认 CP                 |
| **RabbitMQ**  | ✅ AP     | 高可用，消息最终一致              |

### 1. ZooKeeper - CP 架构

```markdown
✅ 选择：CP（一致性 + 分区容错性）

特点：
- 强一致性保证
- Leader-Follower 架构
- ZAB 协议（原子广播）
- 网络分区时，少数派 Follower 不可用

为什么是 CP？
1. Leader 选举期间服务不可用
2. 写操作需要过半节点确认
3. 读取也需要同步最新数据
4. 保证所有节点数据一致

应用场景：
✅ 分布式协调（适合）
✅ 配置中心（适合）
✅ 服务注册发现（适合，但性能一般）
❌ 高并发读写（不适合）
```

---

### 2. Redis - 可配置的 CP/AP 架构

```markdown
⚠️ 选择：可配置，不同模式表现不同

【单机版/主从复制】- 偏 AP
特点：
- 主节点宕机，从节点继续服务
- 允许短暂数据不一致
- 高可用性好

【Redis Sentinel】- 偏 CP
特点：
- 自动故障转移
- 选举新的 Master
- 选举期间不可用（CP 特性）
- 数据可能丢失（异步复制）

【Redis Cluster】- 偏 AP
特点：
- 去中心化多主架构
- 部分节点故障不影响整体
- 数据分片存储
- 允许某些分片不可用

配置示例：
# 强一致性配置（CP）
redis.conf:
  min-slaves-to-write 1      # 至少 1 个从节点确认
  min-slaves-max-lag 10      # 延迟不超过 10 秒

# 高可用配置（AP）
redis.conf:
  min-slaves-to-write 0      # 不需要从节点确认
```

---

### 3. MySQL - CP 架构

```markdown
✅ 选择：CP（一致性 + 分区容错性）

【单实例】- CA 架构（非分布式）
特点：
- 强一致性
- 高可用
- 无法容忍网络分区（不是真正的分布式）

【主从复制】- CP 架构
特点：
- 异步/半同步复制
- 主库写入，从库读取
- 网络分区时保证主库一致性
- 从库可能短暂不一致

【MGR 集群】- CP 架构
特点：
- Multi-Primary 或 Single-Primary
- Paxos 协议保证一致性
- 节点故障时自动选举
- 写入需要多数派确认

为什么是 CP？
1. ACID 事务保证强一致性
2. 锁机制确保并发安全
3. 崩溃恢复保证数据完整
4. 宁可不可用，也不能数据错误

应用场景：
✅ 金融交易系统（适合）
✅ 订单管理（适合）
✅ 账户余额（适合）
❌ 高并发读（需配合缓存）
```

---

### 4. RabbitMQ - AP 架构

```markdown
✅ 选择：AP（可用性 + 分区容错性）

特点：
- 镜像队列保证高可用
- 消息最终一致性
- 网络分区时各节点仍可服务
- 允许消息短暂不一致

为什么是 AP？
1. 镜像队列：消息复制到多个节点
2. 自动故障转移：节点宕机不影响服务
3. 消息确认：消费者 ACK 机制
4. 死信队列：失败消息处理

配置示例：
# 镜像队列配置
{
  "ha-mode": "all",           // 所有节点镜像
  "ha-sync-mode": "automatic" // 自动同步
}

# 消息可靠性
- publisher confirm（发布确认）
- consumer ack（消费确认）
- durable queue（持久化队列）

应用场景：
✅ 异步解耦（适合）
✅ 流量削峰（适合）
✅ 日志收集（适合）
⚠️ 顺序消息（需注意分区）
```

---

### 5. MongoDB - 可配置的 CP/AP 架构

```markdown
⚠️ 选择：可配置，默认 CP

【副本集模式】- CP 架构
特点：
- Primary-Secondary 架构
- 写入需要多数派确认
- 自动故障转移
- 强一致性保证

【分片集群】- AP 架构
特点：
- 数据分片存储
- 每个分片独立服务
- 高可用性
- 最终一致性

配置示例：
// 强一致性写入（CP）
db.collection.insertOne(
  { name: "test" },
  { writeConcern: { w: "majority" } }  // 多数派确认
)

// 高可用读取（AP）
db.collection.find().readPref("secondaryPreferred")
```

### 6. Nacos - 可切换的 CP/AP 架构

```markdown
⚠️ 选择：支持 CP/AP 切换

【AP 模式】- 默认模式
特点：
- Distro 协议（类似 Gossip）
- 节点平等，无 Leader
- 高可用
- 最终一致性

【CP 模式】
特点：
- Raft 协议
- Leader-Follower 架构
- 强一致性
- 选举期间不可用

切换方式：
# application.properties
nacos.core.protocol.distro.data.sync.enable=false  # 关闭 AP
nacos.core.protocol.raft.enable=true               # 开启 CP

应用场景：
✅ AP 模式：服务注册发现（推荐）
✅ CP 模式：配置中心（推荐）
```

| MySQL         | CP       | -        | 核心业务数据      | 强一致         |

| Cassandra     | AP       | -        | 海量数据存储        | 最终一致       |
| Eureka        | AP       | -        | 服务注册发现      | 最终一致       |
| Nacos         | AP       | CP       | 注册中心 + 配置中心 | 弱/强可配      |
| RabbitMQ      | AP       | -        | 消息队列          | 最终一致       |

### 选型建议

**需要强一致性（CP）的场景：**
```markdown
✅ 金融支付 → MySQL + ZooKeeper
✅ 订单系统 → MySQL + Redis（CP 模式）
✅ 配置中心 → Nacos（CP 模式）+ ZooKeeper
✅ 分布式锁 → ZooKeeper + Redis
```

**需要高可用（AP）的场景：**
```markdown
✅ 服务注册 → Eureka + Nacos（AP 模式）
✅ 缓存层 → Redis Cluster + Memcached
✅ 消息队列 → RabbitMQ + Kafka
✅ 日志收集 → Elasticsearch + Cassandra
```

**混合架构（推荐）：**
```markdown
核心业务：
- MySQL（CP）存储核心数据
- Redis（CP 模式）做缓存
- ZooKeeper（CP）做分布式协调

非核心业务：
- Redis（AP 模式）做高速缓存
- RabbitMQ（AP）异步解耦
- Nacos（AP）服务注册发现
```

**问题 7：BASE 理论与 CAP 有什么关系？**

答案：

BASE 理论是对 CAP 理论的延伸，由 eBay 架构师 Dan Pritchett 提出。

**BASE 三要素：**

**1. Basically Available（基本可用）**
```markdown
含义：
- 系统出现不可预知故障时，允许损失部分可用性
- 保证核心功能可用

实现方式：
✅ 降级策略（关闭非核心功能）
✅ 限流熔断
✅ 异步处理
✅ 批量处理

示例：
- 双 11 购物节：关闭评价功能，保证下单流畅
- 高峰期：返回静态缓存数据，而非实时计算
```

**2. Soft State（软状态）**
```markdown
含义：
- 允许系统存在中间状态
- 该状态不影响整体可用性
- 数据可以有一段时间的不一致

对比硬状态：
❌ 硬状态：ACID，数据必须立即一致
✅ 软状态：BASE，允许延迟一致

示例：
- 订单支付后，物流信息稍后更新
- 微博发布后，粉丝稍后看到
```

**3. Eventually Consistent（最终一致性）**
```markdown
含义：
- 经过一段时间后，数据会达到一致
- 不要求实时一致

实现方式：
✅ 消息队列异步同步
✅ 定时任务补偿
✅ 事件驱动架构
✅ 读写分离 + 主从复制

示例：
- 银行转账：跨行转账 T+1 到账
- 积分系统：消费后积分稍后更新
```

**CAP vs BASE 对比：**

| 维度       | CAP              | BASE                  |
|------------|------------------|-----------------------|
| 目标       | 理论指导         | 实践方法              |
| 关注点     | 一致性 vs 可用性 | 高可用性优先          |
| 一致性     | 强一致性或弱一致 | 最终一致性            |
| 适用场景   | 架构设计决策     | 互联网大规模系统      |
| 实现难度   | 较高             | 相对较低              |

**BASE 的核心思想：**
```
通过牺牲强一致性，换取高可用性
采用最终一致性，降低系统复杂度
适合大规模互联网应用
```

**实际应用案例：**

**淘宝购物车（BASE 实践）**
```markdown
场景：
- 用户添加商品到购物车

实现：
1. 基本可用：
   - 即使库存服务挂了，仍可加入购物车
   - 结算时才校验库存

2. 软状态：
   - 购物车数据允许短暂不一致
   - 多端同步有延迟

3. 最终一致性：
   - 通过消息队列同步购物车
   - 定时清理过期数据
```
