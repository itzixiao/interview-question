# 树形数据结构核心知识点详解

## 一、二叉树（Binary Tree）

### 1.1 什么是二叉树？

**定义：** 每个节点最多有两个子节点的树结构

**二叉搜索树（BST）特点：**

- 左子节点 < 父节点 < 右子节点
- 中序遍历得到有序序列
- 查找、插入、删除的时间复杂度：O(h)，h 为树高
- **最坏情况：** 退化为链表，时间复杂度 O(n)

### 1.2 二叉树结构定义

```java
class TreeNode {
    int val;
    TreeNode left;   // 左子节点
    TreeNode right;  // 右子节点
    TreeNode(int x) { val = x; }
}
```

### 1.3 二叉树遍历方式

| 遍历方式              | 顺序        | 特点         |
|-------------------|-----------|------------|
| 前序遍历（Pre-order）   | 根 → 左 → 右 | -          |
| 中序遍历（In-order）    | 左 → 根 → 右 | **得到有序序列** |
| 后序遍历（Post-order）  | 左 → 右 → 根 | -          |
| 层序遍历（Level-order） | 按层从上到下    | 使用队列实现     |

### 1.4 二叉搜索树示例

```
      8
     / \
    3   10
   / \    \
  1   6    14
```

**遍历结果：**

- 前序遍历：8, 3, 1, 6, 10, 14
- 中序遍历：1, 3, 6, 8, 10, 14（**有序！**）
- 后序遍历：1, 6, 3, 14, 10, 8
- 层序遍历：8, 3, 10, 1, 6, 14

### 1.5 二叉搜索树操作

#### 插入操作

```
1. 从根节点开始
2. 比当前节点小 → 往左子树
3. 比当前节点大 → 往右子树
4. 找到空位置，插入新节点
```

#### 查找操作

```
1. 从根节点开始
2. 等于当前节点 → 找到
3. 小于当前节点 → 往左子树
4. 大于当前节点 → 往右子树
5. 到叶子节点还没找到 → 不存在
```

#### 删除操作（最复杂）

**情况 1：叶子节点**

```
直接删除
```

**情况 2：只有一个子节点**

```
子承父业（用子节点替代被删除节点）
```

**情况 3：有两个子节点**

```
找右子树最小值（或左子树最大值）替换，然后删除那个节点
```

## 二、平衡二叉树（AVL Tree）

### 2.1 为什么需要平衡二叉树？

**问题：** 普通二叉搜索树可能退化为链表

**例如：** 插入 1,2,3,4,5,6 → 所有节点都在右边

```
1
 \
  2
   \
    3
     \
      4
       \
        5
         \
          6
```

查找时间复杂度从 O(log n) 退化为 O(n)

**解决：** AVL 树通过旋转保持平衡

### 2.2 AVL 树的定义

**特点：**

1. 是二叉搜索树
2. **任意节点的左右子树高度差 ≤ 1**（平衡因子 ∈ [-1, 0, 1]）
3. 每次插入/删除后自动调整平衡

### 2.3 平衡因子

**公式：** 平衡因子 = 左子树高度 - 右子树高度

**判断：**

- 平衡因子 ∈ {-1, 0, 1} → 平衡
- 平衡因子 > 1 或 < -1 → 不平衡，需要旋转

### 2.4 四种不平衡情况及旋转

#### LL 型（右单旋）

```
    5           3
   /           / \
  3    →      2   5
 /
2
```

#### RR 型（左单旋）

```
  5               7
   \             / \
    7    →      5   9
     \
      9
```

#### LR 型（先左后右双旋）

```
    5           5         4
   /           /         / \
  2    →      4    →    2   5
   \         /
    4       2
```

#### RL 型（先右后左双旋）

```
  2             2           4
   \           / \         / \
    5    →    4   5  →    2   5
   /           \
  4             4
```

### 2.5 AVL 树的优缺点

**优点：**

- ✅ 查询效率极高（严格平衡，O(log n)）
- ✅ 适合查询密集型场景

**缺点：**

- ❌ 插入/删除时需要多次旋转
- ❌ 维护平衡的开销大
- ❌ 不适合频繁插入删除的场景

## 三、红黑树（Red-Black Tree）

### 3.1 红黑树的定义

红黑树是一种自平衡的二叉搜索树，通过节点颜色保证近似平衡

**节点颜色：** 每个节点要么是红色，要么是黑色

### 3.2 红黑树的五大性质

1. **每个节点是红色或黑色**
2. **根节点是黑色**
3. **叶子节点（NIL）是黑色**
4. **红色节点的子节点必须是黑色**（不能有两个连续红色节点）
5. **从任一节点到其叶子的所有路径包含相同数量的黑色节点**

### 3.3 红黑树示例

```
        8(B)
       /    \
     4(R)   12(R)
    /  \    /   \
  2(B) 6(B) 10(B) 14(B)
```

注：(B)=黑色，(R)=红色

### 3.4 红黑树的平衡操作

#### 变色

- 改变节点颜色（红↔黑）
- 用于调整黑色节点数量

#### 左旋

```
  5              7
   \            /
    7    →     5
     \          \
      9          9
```

#### 右旋

```
      7          5
     /            \
    5      →      7
   /                \
  3                  3
```

### 3.5 红黑树 vs AVL 树

| 特性   | 红黑树         | AVL 树     |
|------|-------------|-----------|
| 平衡性  | 弱平衡         | 强平衡       |
| 树高   | ≤ 2log₂(n)  | ≤ log₂(n) |
| 查询效率 | O(log n)    | O(log n)  |
| 插入效率 | 高（最多 3 次旋转） | 低（可能多次旋转） |
| 删除效率 | 高           | 低         |
| 适用场景 | 增删改查频繁      | 查询密集型     |

### 3.6 红黑树的应用

- ✅ **Java TreeMap**：基于红黑树实现
- ✅ **Java HashMap（JDK8+）**：桶数组 + 链表 + 红黑树
- ✅ **C++ STL map**：基于红黑树实现
- ✅ **Linux 内核进程调度**：使用红黑树管理进程

## 四、B 树（B-Tree）

### 4.1 为什么需要 B 树？

**问题：** 数据量巨大时，树很高，磁盘 IO 次数多

**解决：** 增加每个节点的子节点数，降低树高

**目标：** 减少磁盘 IO 次数（数据库索引的核心）

### 4.2 B 树的定义

**m 阶 B 树的特点：**

1. 每个节点最多有 m 个子节点
2. 除根节点外，每个节点至少有 ⌈m/2⌉ 个子节点
3. 根节点至少有 2 个子节点（除非是叶子）
4. 所有叶子节点在同一层
5. 节点包含 k 个关键字和 k+1 个指针

### 4.3 3 阶 B 树示例

```
         [17, 35]
        /    |    \
   [5,10] [20,30] [40,50]
   / | \   / | \   / | \
  1 3 12 18 25 33 38 45 55
```

**特点：**

- 每个节点最多 3 个子节点
- 每个节点最多 2 个关键字
- 所有叶子在同一层

### 4.4 B 树的优势

**矮胖结构：**

- 相比二叉树，B 树更"矮胖"
- **例如：** 100 万个节点
    - 二叉树：树高约 20 层
    - B 树（100 叉）：树高约 3 层
- 减少磁盘 IO 次数

### 4.5 B 树的查找过程

```
1. 从根节点开始
2. 在节点内二分查找关键字
3. 找到 → 返回
4. 没找到 → 根据范围选择子节点
5. 重复步骤 2-4，直到叶子节点
```

### 4.6 B 树的插入与删除

#### 插入

```
1. 找到合适的叶子节点
2. 插入关键字
3. 如果节点关键字数超过上限 → 分裂
4. 中间关键字上移到父节点
```

#### 删除

```
1. 找到要删除的关键字
2. 如果是叶子节点 → 直接删除
3. 如果不是叶子 → 用子节点替代
4. 如果节点关键字数低于下限 → 合并或借位
```

## 五、B+树（B+ Tree）

### 5.1 B+树的定义

B+树是 B 树的变体，更适合文件系统和数据库索引

**核心改进：** 数据只存储在叶子节点，非叶子节点只存索引

### 5.2 B+树 vs B 树

| 特性     | B 树         | B+树      |
|--------|-------------|----------|
| 数据存储位置 | 所有节点        | 只在叶子节点   |
| 非叶子节点  | 存数据 + 索引    | 只存索引     |
| 叶子节点   | 不连接         | 用链表连接    |
| 查询稳定性  | 不稳定（可能在非叶子） | 稳定（都在叶子） |
| 范围查询   | 需中序遍历       | 直接遍历链表   |

### 5.3 B+树示例

**非叶子节点（索引层）：**

```
         [17, 35]
        /    |    \
   [5,10] [20,30] [40,50]
```

**叶子节点（数据层，用链表连接）：**

```
→ [1,3] → [12,18] → [25,33] → [38,45] → [55,60] →
```

**特点：**

- 非叶子节点只存索引（容纳更多）
- 叶子节点存数据 + 指向下一条记录的指针
- 叶子节点形成有序链表

### 5.4 B+树的优势

#### 1. 磁盘读写代价更低

- 非叶子节点不存数据，更小
- 同样大小的磁盘页可以容纳更多索引项
- 树更矮，IO 次数更少

#### 2. 查询效率稳定

- 所有查询都要走到叶子节点
- 查询路径长度相同

#### 3. 范围查询极快

- 叶子节点用链表连接
- 只需找到起点，然后遍历链表
- **例如：** `SELECT * WHERE id > 100`

#### 4. 全表扫描简单

- 直接遍历叶子节点链表即可
- 不需要中序遍历整棵树

### 5.5 B+树在数据库中的应用

#### MySQL InnoDB 引擎

- 聚簇索引：使用 B+树
- 二级索引：使用 B+树
- 叶子节点存储完整数据行

#### 文件系统

- NTFS 文件系统使用 B+树
- HFS+ 文件系统使用 B+树
- 数据库索引普遍使用 B+树

## 六、各种树结构对比

### 6.1 时间复杂度对比

| 数据结构     | 平均查找      | 平均插入      | 平均删除      |
|----------|-----------|-----------|-----------|
| 二叉搜索树    | O(log n)  | O(log n)  | O(log n)  |
| AVL 树    | O(log n)  | O(log n)  | O(log n)  |
| 红黑树      | O(log n)  | O(log n)  | O(log n)  |
| B 树（m 阶） | O(logₘ n) | O(logₘ n) | O(logₘ n) |
| B+树（m 阶） | O(logₘ n) | O(logₘ n) | O(logₘ n) |

### 6.2 空间复杂度对比

| 数据结构  | 空间复杂度 | 说明       |
|-------|-------|----------|
| 二叉搜索树 | O(n)  | -        |
| AVL 树 | O(n)  | 需要存储平衡因子 |
| 红黑树   | O(n)  | 需要存储颜色   |
| B 树   | O(n)  | -        |
| B+树   | O(n)  | -        |

### 6.3 应用场景对比

| 数据结构  | 典型应用场景                            |
|-------|-----------------------------------|
| 二叉搜索树 | 内存中的有序数据                          |
| AVL 树 | 查询密集型（如字典、词典）                     |
| 红黑树   | Java TreeMap/HashMap, C++ STL map |
| B 树   | 文件系统、数据库索引（逐渐被 B+树取代）             |
| B+树   | MySQL InnoDB、文件系统（NTFS、HFS+）      |

### 6.4 选择建议

#### 内存中的数据

- 查询多、插入少 → **AVL 树**
- 增删改查频繁 → **红黑树**
- 简单实现 → **二叉搜索树**

#### 磁盘上的数据（数据库、文件系统）

- **首选 B+树**（范围查询、全表扫描优势明显）
- 特殊场景可考虑 B 树

## 七、高频面试题

**问题 1：二叉树和二叉搜索树的区别？**

**答：**

**二叉树：** 每个节点最多有两个子节点的树

**二叉搜索树：** 在二叉树的基础上，满足：

- 左子节点 < 父节点 < 右子节点
- 中序遍历得到有序序列

**问题 2：二叉树的遍历方式有哪些？**

**答：**

1. **前序遍历**：根 → 左 → 右
2. **中序遍历**：左 → 根 → 右（**得到有序序列**）
3. **后序遍历**：左 → 右 → 根
4. **层序遍历**：按层从上到下遍历

**问题 3：为什么需要平衡二叉树？**

**答：**

普通二叉搜索树可能退化为链表

**例如：** 插入 1,2,3,4,5,6 → 所有节点都在右边

查找时间复杂度从 O(log n) 退化为 O(n)

平衡二叉树通过旋转保持平衡，保证 O(log n)

**问题 4:AVL 树的四种不平衡情况及旋转？**

**答：**

1. **LL 型（右单旋）**：左子树的左子树导致不平衡
2. **RR 型（左单旋）**：右子树的右子树导致不平衡
3. **LR 型（先左后右双旋）**：左子树的右子树导致不平衡
4. **RL 型（先右后左双旋）**：右子树的左子树导致不平衡

**问题 5：红黑树的五大性质？**

**答：**

1. 每个节点是红色或黑色
2. 根节点是黑色
3. 叶子节点（NIL）是黑色
4. 红色节点的子节点必须是黑色
5. 从任一节点到叶子的路径包含相同数量的黑色节点

**问题 6：红黑树如何保证平衡？**

**答：**

通过以下操作保证近似平衡：

1. **变色**：改变节点颜色
2. **左旋**：顺时针旋转
3. **右旋**：逆时针旋转

树高 ≤ 2log₂(n)，查询效率 O(log n)

**问题 7：红黑树 vs AVL 树的区别？**

**答：**

**平衡性：**

- AVL 树：严格平衡（左右子树高度差 ≤ 1）
- 红黑树：弱平衡（最长路径 ≤ 最短路径的 2 倍）

**性能：**

- AVL 树：查询更快，插入删除慢（旋转多）
- 红黑树：查询稍慢，插入删除快（旋转少）

**应用：**

- AVL 树：字典、词典等查询密集型场景
- 红黑树：Java TreeMap/HashMap、C++ STL map

**问题 8：为什么数据库索引使用 B+树而不是二叉树？**

**答：**

**1. 磁盘 IO 优化：**

- 数据量大时，二叉树很高，IO 次数多
- B+树是"矮胖"结构，IO 次数少

**2. 范围查询优势：**

- B+树叶节点用链表连接
- 范围查询只需遍历链表

**3. 全表扫描简单：**

- 直接遍历叶子节点链表
- 不需要中序遍历整棵树

**问题 9:B 树和 B+树的区别？**

**答：**

**1. 数据存储位置：**

- B 树：所有节点都存数据
- B+树：只有叶子节点存数据

**2. 非叶子节点：**

- B 树：存数据 + 索引
- B+树：只存索引

**3. 叶子节点：**

- B 树：不连接
- B+树：用链表连接

**4. 查询稳定性：**

- B 树：不稳定（可能在非叶子节点找到）
- B+树：稳定（都要到叶子节点）

**问题 10：为什么 B+树比 B 树更适合数据库索引？**

**答：**

**1. 磁盘读写代价更低：**

- 非叶子节点不存数据，更小
- 同样大小的磁盘页容纳更多索引项
- 树更矮，IO 次数更少

**2. 范围查询极快：**

- 叶子节点用链表连接
- 只需找到起点，然后遍历链表

**3. 全表扫描简单：**

- 直接遍历叶子节点链表即可

**问题 11:B 树的阶数 m 如何选择？**

**答：**

**原则：** 让一个节点刚好占满一个磁盘页

**例如：**

- 磁盘页大小：4KB
- 关键字：8 字节，指针：4 字节
- m 阶 B 树：m*4 + (m-1)*8 ≤ 4096
- 解得：m ≈ 170

**问题 12:Java的 TreeMap 和 HashMap 底层实现？**

**答：**

**TreeMap：**

- 基于红黑树实现
- 元素有序
- 操作时间复杂度：O(log n)

**HashMap（JDK8+）：**

- 数组 + 链表 + 红黑树
- 链表长度 > 8 且数组长度 ≥ 64 → 转红黑树
- 链表长度 < 6 → 转回链表
- 操作时间复杂度：O(1) ~ O(log n)

**问题 13:MySQL索引为什么不用红黑树？**

**答：**

**1. 树高问题：**

- 红黑树是二叉树，数据量大时树很高
- 每次查询需要多次磁盘 IO

**2. 范围查询劣势：**

- 红黑树需要中序遍历
- B+树只需遍历叶子节点链表

**3. B+树优势：**

- 矮胖结构，IO 次数少
- 范围查询快
- 全表扫描快

**问题 14：什么场景下使用 AVL 树？**

**答：**

**适合查询密集型场景：**

- 字典、词典（频繁查询，很少修改）
- 内存数据库的索引
- 对查询性能要求极高的场景

**不适合：**

- 频繁插入删除的场景
- 数据量巨大的场景（应该用 B+树）

**问题 15：设计一个海量数据的索引系统，你会选择什么数据结构？**

**答：**

**选择 B+树，原因：**

1. **磁盘 IO 优化**：矮胖结构，减少 IO 次数
2. **范围查询高效**：叶子节点链表连接
3. **全表扫描简单**：遍历叶子节点即可
4. **成熟稳定**：MySQL、PostgreSQL 都在用
5. **可扩展**：支持动态扩容、分区等

```
│   ├── AVL 树（严格平衡）
│   └── 旋转操作（LL、RR、LR、RL）
│
├── 红黑树（弱平衡）
│   ├── 五大性质
│   └── 平衡操作（变色、左旋、右旋）
│
└── 多路搜索树
    ├── B 树（所有节点存数据）
    └── B+树（只有叶子存数据，数据库首选）
```

### 8.2 学习建议

1. **理解演进过程**：二叉树 → AVL → 红黑树 → B 树 → B+树
2. **掌握各自特点**：每种树解决的问题和优化方向
3. **结合应用场景**：内存数据 vs 磁盘数据
4. **多画图理解**：树的结构、旋转操作、插入删除过程
