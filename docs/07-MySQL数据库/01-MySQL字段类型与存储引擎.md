# MySQL 字段类型与存储引擎详解

## 一、MySQL 字段类型

### 1.1 数值类型

| 类型           | 字节 | 有符号范围            | 无符号范围       | 说明          |
|--------------|----|------------------|-------------|-------------|
| TINYINT      | 1  | -128 ~ 127       | 0 ~ 255     | 状态标识（0/1）   |
| SMALLINT     | 2  | -32,768 ~ 32,767 | 0 ~ 65,535  | 较小整数        |
| MEDIUMINT    | 3  | ±8 百万            | 0~16 百万     | 中等整数        |
| INT/INTEGER  | 4  | ±21 亿            | 0~42 亿      | 常用整数        |
| BIGINT       | 8  | ±9×10^18         | 0~1.8×10^19 | 超大整数（订单 ID） |
| FLOAT        | 4  | 单精度浮点数           | -           | 近似值         |
| DOUBLE       | 8  | 双精度浮点数           | -           | 近似值         |
| DECIMAL(M,D) | 变长 | 精确小数             | -           | **金额存储**    |

**选型建议：**

- ✅ 金额用 `DECIMAL`，不用 `FLOAT/DOUBLE`（避免精度丢失）
- ✅ 状态标识用 `TINYINT(1)` 表示 false/true
- ✅ 主键自增用 `BIGINT UNSIGNED`

### 1.2 字符串类型

| 类型         | 最大长度      | 说明         | 适用场景    |
|------------|-----------|------------|---------|
| CHAR(N)    | 255 字符    | 固定长度，不足补空格 | 手机号、身份证 |
| VARCHAR(N) | 65,535 字节 | 可变长度，节省空间  | 用户名、地址  |
| TINYTEXT   | 255 字节    | 小文本        | 短描述     |
| TEXT       | 65,535 字节 | 中等文本       | 文章内容    |
| MEDIUMTEXT | 16MB      | 大文本        | 博客文章    |
| LONGTEXT   | 4GB       | 超大文本       | 日志记录    |
| BLOB       | -         | 二进制对象      | 图片、视频   |

**CHAR vs VARCHAR：**

- CHAR：定长，查询效率高（无需计算长度）
- VARCHAR：变长，节省存储空间

### 1.3 日期时间类型

| 类型        | 格式                  | 范围                      | 特点        |
|-----------|---------------------|-------------------------|-----------|
| DATE      | YYYY-MM-DD          | 1000-01-01 ~ 9999-12-31 | 只存日期      |
| TIME      | HH:MM:SS            | -838:59:59 ~ 838:59:59  | 只存时间      |
| DATETIME  | YYYY-MM-DD HH:MM:SS | 1000-01-01 ~ 9999-12-31 | **与时区无关** |
| TIMESTAMP | YYYY-MM-DD HH:MM:SS | 1970-01-01 ~ 2038-01-19 | 受时区影响     |
| YEAR      | YYYY                | 1901 ~ 2155             | 只存年份      |

**推荐：** 使用 `DATETIME`（与时区无关，范围更广）

## 二、存储引擎

### 2.1 InnoDB（默认引擎）

**核心特性：**

- ✅ 支持事务（ACID）
- ✅ 支持行级锁（并发性能好）
- ✅ 支持外键约束
- ✅ 支持 MVCC（多版本并发控制）
- ✅ 聚簇索引（数据和索引在一起）
- ✅ 崩溃恢复能力强

**适用场景：** 高并发、事务安全、频繁更新

### 2.2 MyISAM

**核心特性：**

- ❌ 不支持事务
- ❌ 只支持表级锁（并发差）
- ❌ 不支持外键
- ✅ 查询速度快（读多写少）
- ✅ 占用空间小
- ✅ 支持全文索引（5.6 前）

**适用场景：** 读多写少、不需要事务

### 2.3 Memory

**核心特性：**

- ✅ 数据存储在内存中（速度极快）
- ❌ 重启后数据丢失
- ❌ 不支持大字段（TEXT/BLOB）
- ❌ 只支持表级锁

**适用场景：** 临时表、缓存表

### 2.4 Archive

**核心特性：**

- ✅ 高压缩比（节省空间）
- ✅ 只支持 INSERT 和 SELECT
- ❌ 不支持 UPDATE 和 DELETE

**适用场景：** 日志归档、历史数据存储

### 2.5 引擎对比表

| 特性   | InnoDB | MyISAM | Memory | Archive |
|------|--------|--------|--------|---------|
| 事务支持 | ✓      | ✗      | ✗      | ✗       |
| 行级锁  | ✓      | ✗      | ✗      | ✗       |
| 外键支持 | ✓      | ✗      | ✗      | ✗       |
| MVCC | ✓      | ✗      | ✗      | ✗       |
| 崩溃恢复 | 强      | 弱      | 无      | 弱       |
| 并发性能 | 高      | 低      | 中      | 低       |
| 存储空间 | 较大     | 小      | 内存     | 最小      |

## 三、高频面试题

**问题 1:VARCHAR(50) 的 50 表示什么？**

**答：**

- 表示最多存储 **50 个字符**
- 实际占用空间 = 字符数 + 长度标识（1-2 字节）
- UTF8 编码下，一个汉字占 3 个字节
- 例如：`VARCHAR(50)` 存储"你好"占用 6 字节（2 个汉字 × 3）+ 1 字节长度标识

**问题 2:CHAR 和 VARCHAR 的区别？**

**答：**

| 对比项  | CHAR      | VARCHAR |
|------|-----------|---------|
| 长度   | 固定长度      | 可变长度    |
| 填充   | 不足补空格     | 不填充     |
| 存储效率 | 高（无需计算长度） | 较低      |
| 空间占用 | 可能浪费      | 节省      |
| 适用场景 | 手机号、身份证   | 用户名、地址  |

**示例：**

```sql
CREATE TABLE users (
    phone CHAR(11),        -- 固定 11 位
    name VARCHAR(50)       -- 可变长度
);
```

**问题 3:InnoDB 和 MyISAM 的区别？**

**答：**

1. **事务支持**：InnoDB 支持事务，MyISAM 不支持
2. **锁粒度**：InnoDB 支持行级锁，MyISAM 只支持表级锁
3. **外键支持**：InnoDB 支持外键，MyISAM 不支持
4. **MVCC**：InnoDB 支持 MVCC，MyISAM 不支持
5. **索引结构**：InnoDB 是聚簇索引，MyISAM 是非聚簇索引
6. **COUNT(*)**：MyISAM 内部维护计数器，InnoDB 需要全表扫描
7. **崩溃恢复**：InnoDB 强（有 Redo Log），MyISAM 弱

**问题 4：为什么金额要用 DECIMAL？**

**答：**

- `FLOAT/DOUBLE` 是浮点数，存在精度丢失问题
- `DECIMAL` 是精确的小数类型，以字符串形式存储
- 例如：`DECIMAL(10,2)` 可以精确表示 99999999.99

**示例：**

```sql
-- 错误示范
CREATE TABLE orders (
    amount FLOAT(10,2)  -- 可能出现 0.01 精度误差
);

-- 正确示范
CREATE TABLE orders (
    amount DECIMAL(10,2)  -- 精确表示金额
);
```

**问题 5:DATETIME 和 TIMESTAMP 的区别？**

**答：**

| 对比项  | DATETIME    | TIMESTAMP    |
|------|-------------|--------------|
| 存储范围 | 1000-9999 年 | 1970-2038 年  |
| 时区影响 | 无           | 有（自动转换）      |
| 占用空间 | 8 字节        | 4 字节         |
| 自动更新 | 不支持         | 支持 ON UPDATE |

**建议：** 推荐使用 `DATETIME`（与时区无关，范围更广）

## 四、最佳实践

### 4.1 字段类型选择原则

1. **越小越好**：在满足需求的前提下，选择最小的类型
2. **简单为好**：避免复杂类型，如 JSON（除非必要）
3. **避免 NULL**：设置 `NOT NULL DEFAULT ''`

### 4.2 引擎选择建议

- **默认选择 InnoDB**（支持事务、高并发）
- **只读场景**可考虑 MyISAM（压缩比高）
- **临时表**使用 Memory（速度快）
- **日志归档**使用 Archive（节省空间）
