# 微服务架构基础

## 一、微服务概述

### 1.1 什么是微服务？

微服务是一种架构风格，将单体应用拆分为一组小型服务，每个服务：

- 运行在独立的进程中
- 围绕业务能力构建
- 通过轻量级通信机制（HTTP/REST）交互
- 可独立部署和扩展
- 由小团队维护

### 1.2 微服务 vs 单体架构

| 维度    | 单体架构 | 微服务架构 |
|-------|------|-------|
| 部署    | 整体部署 | 独立部署  |
| 扩展    | 整体扩展 | 按需扩展  |
| 技术栈   | 统一   | 可异构   |
| 故障影响  | 全局   | 局部    |
| 开发复杂度 | 低    | 高     |
| 运维复杂度 | 低    | 高     |

## 二、微服务拆分原则

### 2.1 核心原则

**高内聚、低耦合** 是微服务拆分的核心原则，具体包括：

#### 1. 单一职责原则（SRP）

```markdown
- 每个服务只负责一个业务领域
- 服务内部功能高度相关
- 避免"大杂烩"式的服务
```

#### 2. 基于业务领域拆分

```markdown
✅ 推荐：按业务模块划分
   - 用户服务（User Service）
   - 订单服务（Order Service）
   - 支付服务（Payment Service）
   - 商品服务（Product Service）

❌ 不推荐：按技术层划分
   - DAO 服务、Service 服务、Controller 服务
```

#### 3. 数据自治原则

```markdown
- 每个服务拥有独立的数据库
- 服务间不共享数据库表
- 通过 API 进行数据交互
```

#### 4. 松耦合原则

```markdown
- 服务间接口清晰明确
- 减少循环依赖
- 避免分布式事务
```

#### 5. 可独立部署

```markdown
- 每个服务可独立发布
- 不影响其他服务
- 支持持续交付
```

### 2.2 常见拆分模式

**模式 1：按业务领域拆分（推荐）**

```
电商系统：
├── 用户中心（注册、登录、个人信息）
├── 商品中心（商品管理、库存管理）
├── 订单中心（下单、订单查询）
├── 支付中心（支付、退款）
└── 物流中心（发货、物流跟踪）
```

**模式 2：按并发量拆分**

```
高并发服务：
- 秒杀服务（独立部署，专门优化）
- 缓存服务（Redis 集群）

普通服务：
- 后台管理服务
- 报表统计服务
```

**模式 3：按数据敏感性拆分**

```
核心服务（高可用、强一致性）：
- 支付服务
- 账户服务

非核心服务（最终一致性）：
- 积分服务
- 消息通知服务
```

### 2.3 拆分决策矩阵

```
团队规模    系统复杂度    性能需求    是否拆分
─────────────────────────────────────
>10 人      高          高         ✅ 立即拆分
5-10 人     中          中         ⚠️ 评估拆分
<5 人       低          低         ❌ 暂缓拆分
```

### 2.4 常见陷阱

| 陷阱          | 问题              | 建议            |
|-------------|-----------------|---------------|
| **过度拆分**    | 服务粒度过小，运维复杂度激增  | 初期适度粗粒度，逐步细化  |
| **拆分过晚**    | 单体应用过于庞大，拆分成本极高 | 团队 >10 人时考虑拆分 |
| **共享数据库**   | 多服务共用数据库，导致数据耦合 | 每个服务独立数据库     |
| **分布式事务滥用** | 过度使用强一致性事务，性能极差 | 优先使用最终一致性     |
| **忽视监控**    | 服务过多后难以排查问题     | 建设完善的监控体系     |

## 三、CAP 定理

### 3.1 定义

CAP 定理是分布式系统的基石，由 Eric Brewer 在 2000 年提出。

**CAP 三要素：**

```
C（Consistency）一致性：
  - 所有节点在同一时间看到的数据相同
  - 强一致性，类似关系型数据库的 ACID
  
A（Availability）可用性：
  - 每次请求都能获得响应
  - 不保证是最新数据
  
P（Partition tolerance）分区容错性：
  - 网络分区发生时（节点间通信失败），系统仍能运行
  - 分布式系统必须满足
```

**CAP 不可能三角：**

```
          C（一致性）
       /            \
      /              \
     /                \
 A （可用性）—————— P（分区容错性）
   
结论：三者最多同时满足两个！
```

### 3.2 三种组合策略

#### CP 架构（一致性 + 分区容错性）

```markdown
特点：
- 保证数据强一致性
- 网络分区时可能不可用
- 适合对数据准确性要求高的场景

典型代表：ZooKeeper、HBase、MongoDB（默认）

应用场景：
- 金融支付系统
- 账户余额管理
- 库存扣减
```

#### AP 架构（可用性 + 分区容错性）

```markdown
特点：
- 保证高可用
- 数据最终一致性
- 适合对实时性要求不高的场景

典型代表：Cassandra、DynamoDB、Eureka

应用场景：
- 社交媒体的点赞数
- 商品浏览量统计
- 评论系统
```

#### CA 架构（一致性 + 可用性）

```markdown
特点：
- 单机数据库架构
- 无法容忍网络分区
- 不是真正的分布式系统

典型代表：MySQL 单实例、PostgreSQL 单实例

注意：
❌ 分布式系统中不存在 CA 架构
❌ 因为 P（分区容错性）是必须的
```

### 3.3 微服务中的 CAP 权衡

| 场景   | 选择 | 理由           |
|------|----|--------------|
| 支付交易 | CP | 必须保证数据强一致性   |
| 商品库存 | CP | 避免超卖         |
| 用户信息 | CP | 个人信息准确性要求高   |
| 新闻浏览 | AP | 短暂不一致可接受     |
| 推荐系统 | AP | 高优先级，允许最终一致性 |
| 日志收集 | AP | 允许丢失，保证可用性   |

## 四、BASE 理论

### 4.1 定义

BASE 理论是对 CAP 理论的延伸，由 eBay 架构师 Dan Pritchett 提出。

### 4.2 三要素

#### 1. Basically Available（基本可用）

```markdown
含义：
- 系统出现不可预知故障时，允许损失部分可用性
- 保证核心功能可用

实现方式：
✅ 降级策略（关闭非核心功能）
✅ 限流熔断
✅ 异步处理

示例：
- 双 11 购物节：关闭评价功能，保证下单流畅
- 高峰期：返回静态缓存数据，而非实时计算
```

#### 2. Soft State（软状态）

```markdown
含义：
- 允许系统存在中间状态
- 该状态不影响整体可用性
- 数据可以有一段时间的不一致

示例：
- 订单支付后，物流信息稍后更新
- 微博发布后，粉丝稍后看到
```

#### 3. Eventually Consistent（最终一致性）

```markdown
含义：
- 经过一段时间后，数据会达到一致
- 不要求实时一致

实现方式：
✅ 消息队列异步同步
✅ 定时任务补偿
✅ 事件驱动架构
✅ 读写分离 + 主从复制
```

### 4.3 CAP vs BASE

| 维度   | CAP        | BASE     |
|------|------------|----------|
| 目标   | 理论指导       | 实践方法     |
| 关注点  | 一致性 vs 可用性 | 高可用性优先   |
| 一致性  | 强一致性或弱一致   | 最终一致性    |
| 适用场景 | 架构设计决策     | 互联网大规模系统 |

## 五、中间件 CAP 分类

### 5.1 总览表

| 中间件           | CAP 选择 | 说明                 |
|---------------|--------|--------------------|
| **ZooKeeper** | CP     | 强一致性，分区时可能不可用      |
| **Redis**     | CP/AP  | 可配置，默认 CP，集群模式偏 AP |
| **MySQL**     | CP     | 强一致性（单实例 CA，集群 CP） |
| **MongoDB**   | CP/AP  | 可配置，默认 CP          |
| **Cassandra** | AP     | 高可用，最终一致性          |
| **Eureka**    | AP     | 高可用，节点平等           |
| **Nacos**     | CP/AP  | 支持切换，默认 AP         |
| **RabbitMQ**  | AP     | 高可用，消息最终一致         |

### 5.2 详细分析

#### ZooKeeper - CP 架构

```markdown
特点：
- 强一致性保证
- Leader-Follower 架构
- ZAB 协议（原子广播）
- 网络分区时，少数派 Follower 不可用

为什么是 CP？
1. Leader 选举期间服务不可用
2. 写操作需要过半节点确认
3. 读取也需要同步最新数据
```

#### Redis - 可配置的 CP/AP

```markdown
【单机版/主从复制】- 偏 AP
- 主节点宕机，从节点继续服务
- 允许短暂数据不一致

【Redis Sentinel】- 偏 CP
- 自动故障转移
- 选举期间不可用

【Redis Cluster】- 偏 AP
- 去中心化多主架构
- 部分节点故障不影响整体
```

#### Nacos - 可切换的 CP/AP

```markdown
【AP 模式】- 默认模式
- Distro 协议（类似 Gossip）
- 节点平等，无 Leader
- 高可用，最终一致性

【CP 模式】
- Raft 协议
- Leader-Follower 架构
- 强一致性，选举期间不可用

应用场景：
✅ AP 模式：服务注册发现（推荐）
✅ CP 模式：配置中心（推荐）
```

### 5.3 选型建议

**需要强一致性（CP）的场景：**

```markdown
✅ 金融支付 → MySQL + ZooKeeper
✅ 订单系统 → MySQL + Redis（CP 模式）
✅ 配置中心 → Nacos（CP 模式）
✅ 分布式锁 → ZooKeeper + Redis
```

**需要高可用（AP）的场景：**

```markdown
✅ 服务注册 → Eureka + Nacos（AP 模式）
✅ 缓存层 → Redis Cluster
✅ 消息队列 → RabbitMQ + Kafka
✅ 日志收集 → Elasticsearch
```

## 六、微服务拆分步骤

### 步骤 1：业务分析

```markdown
1. 梳理业务流程
2. 识别业务领域边界
3. 确定核心业务和辅助业务
4. 绘制业务架构图
```

### 步骤 2：数据梳理

```markdown
1. 分析数据库表关系
2. 识别数据聚合根
3. 确定数据所有权
4. 规划数据迁移方案
```

### 步骤 3：接口设计

```markdown
1. 定义服务间接口
2. 确定通信协议（HTTP/RPC）
3. 设计数据格式（DTO/VO）
4. 制定版本管理策略
```

### 步骤 4：渐进式拆分

```markdown
第一阶段：
- 剥离边缘服务（如消息通知）
- 验证拆分方案
- 积累运维经验

第二阶段：
- 拆分核心业务（如订单、用户）
- 完善监控和日志
- 优化性能

第三阶段：
- 全面微服务化
- 自动化运维
- 持续优化
```

### 步骤 5：测试验证

```markdown
1. 单元测试确保功能正确
2. 集成测试验证服务协作
3. 性能测试评估系统表现
4. 灰度发布降低风险
```
