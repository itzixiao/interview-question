# 等级保护二级详解

## 概述

等级保护二级（等保二级）是指信息系统受到破坏后，会对公民、法人和其他组织的合法权益产生损害，但不损害国家安全、社会秩序和公共利益。属于"
指导保护级"，需要进行公安备案。

## 一、等级保护概述

### 1.1 等保制度背景

|      法规/标准      | 发布时间  | 核心内容              |
|:---------------:|:-----:|:------------------|
|     《网络安全法》     | 2017年 | 明确等级保护制度为网络安全基本制度 |
| GB/T 22239-2019 | 2019年 | 等保2.0核心标准，扩展保护对象  |
| GB/T 22240-2020 | 2020年 | 定级指南，明确系统定级方法     |
| 《信息安全等级保护管理办法》  | 2007年 | 等保工作具体实施办法        |

### 1.2 等保等级划分

|   等级   |   保护级别   | 破坏后果        | 监管方式          |
|:------:|:--------:|:------------|:--------------|
|   一级   |   自主保护   | 损害个人权益      | 无需备案，自主保护     |
| **二级** | **指导保护** | **损害组织权益**  | **公安备案，建议测评** |
|   三级   |   监督保护   | 损害社会秩序/公共利益 | 公安备案，强制测评     |
|   四级   |   强制保护   | 损害国家安全      | 公安备案，强制测评     |
|   五级   |   专控保护   | 严重损害国家安全    | 专门机构监管        |

## 二、二级等保定级标准

### 2.1 定级依据

根据 GB/T 22240-2020《信息安全技术 网络安全等级保护定级指南》：

**二级系统特征：**

1. **侵害客体**：公民、法人和其他组织的合法权益
2. **侵害程度**：造成严重损害
3. **典型系统**：
    - 一般企业信息系统
    - 非敏感政务系统
    - 普通电商平台
    - 中小型教育系统

### 2.2 定级流程

```
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
```

### 3.4 安全计算环境

#### 3.4.1 身份鉴别

|   要求项    | 具体内容                    |
|:--------:|:------------------------|
|   身份标识   | 用户具有唯一标识                |
| 鉴别信息复杂度  | 密码长度≥8位，包含大小写字母、数字、特殊字符 |
| 鉴别信息定期更换 | 密码定期更换，建议90天            |
|  登录失败处理  | 配置登录失败处理功能（锁定账户、结束会话）   |
|  管理默认账户  | 修改或禁用默认账户               |

**身份鉴别配置示例：**

```java
/**
 * 用户身份鉴别服务
 */
@Service
public class AuthenticationService {

    // 密码复杂度校验
    public boolean validatePasswordComplexity(String password) {
        // 至少8位，包含大小写字母、数字、特殊字符
        String pattern = "^(?=.*[a-z])(?=.*[A-Z])(?=.*\\d)(?=.*[@$!%*?&])[A-Za-z\\d@$!%*?&]{8,}$";
        return password.matches(pattern);
    }

    // 登录失败锁定
    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    private static final int MAX_LOGIN_ATTEMPTS = 5;
    private static final long LOCK_TIME = 30 * 60; // 30分钟

    public void handleLoginFailure(String username) {
        String key = "login:failed:" + username;
        Long attempts = redisTemplate.opsForValue().increment(key);

        if (attempts == 1) {
            redisTemplate.expire(key, LOCK_TIME, TimeUnit.SECONDS);
        }

        if (attempts >= MAX_LOGIN_ATTEMPTS) {
            // 锁定账户
            lockAccount(username);
            throw new AccountLockedException("账户已锁定，请30分钟后重试");
        }
    }

    // 双因子认证（可选，三级强制要求）
    public boolean verifyTwoFactor(String username, String code) {
        String storedCode = redisTemplate.opsForValue()
                .get("2fa:" + username);
        return code.equals(storedCode);
    }
}
```

#### 3.4.2 访问控制

|  要求项   | 具体内容                        |
|:------:|:----------------------------|
| 访问控制策略 | 依据安全策略控制用户访问                |
| 最小权限原则 | 仅授予用户所需最小权限                 |
|  权限分离  | 管理用户权限分离（系统管理员、安全管理员、审计管理员） |

**访问控制实现：**

```java
/**
 * 基于RBAC的访问控制
 */
@Service
public class AccessControlService {

    // 角色权限配置
    private Map<String, Set<String>> rolePermissions = Map.of(
            "SYSTEM_ADMIN", Set.of("system:manage", "user:create", "user:delete"),
            "SECURITY_ADMIN", Set.of("security:manage", "audit:view"),
            "AUDIT_ADMIN", Set.of("audit:view", "audit:export"),
            "USER", Set.of("data:read", "data:write")
    );

    // 权限检查
    public boolean hasPermission(String username, String permission) {
        Set<String> roles = getUserRoles(username);
        return roles.stream()
                .flatMap(role -> rolePermissions.getOrDefault(role, Set.of()).stream())
                .anyMatch(permission::equals);
    }

    // 权限分离检查
    public void checkRoleSeparation(String username, String newRole) {
        Set<String> currentRoles = getUserRoles(username);

        // 系统管理员不能同时是审计管理员
        if (newRole.equals("AUDIT_ADMIN") && currentRoles.contains("SYSTEM_ADMIN")) {
            throw new RoleConflictException("系统管理员不能兼任审计管理员");
        }
    }
}
```

#### 3.4.3 安全审计

|  要求项   | 具体内容               |
|:------:|:-------------------|
|  审计范围  | 覆盖所有用户行为和重要安全事件    |
|  审计内容  | 事件日期、时间、用户、事件类型、结果 |
| 审计日志保护 | 防止审计日志被篡改、删除       |
| 审计日志保存 | 保存期限≥6个月           |

**安全审计实现：**

```java
/**
 * 安全审计日志服务
 */
@Service
public class SecurityAuditService {

    @Autowired
    private AuditLogRepository auditLogRepository;

    // 审计事件类型
    public enum AuditEventType {
        LOGIN_SUCCESS,      // 登录成功
        LOGIN_FAILURE,      // 登录失败
        LOGOUT,             // 登出
        PERMISSION_DENIED,  // 权限拒绝
        DATA_ACCESS,        // 数据访问
        DATA_MODIFY,        // 数据修改
        CONFIG_CHANGE,      // 配置变更
        ACCOUNT_CREATE,     // 账户创建
        ACCOUNT_DELETE,     // 账户删除
        PASSWORD_CHANGE     // 密码修改
    }

    // 记录审计日志
    public void logAuditEvent(String username, AuditEventType eventType,
                              String resource, String result, String detail) {
        AuditLog log = AuditLog.builder()
                .eventTime(LocalDateTime.now())
                .username(username)
                .eventType(eventType.name())
                .resource(resource)
                .result(result)
                .detail(detail)
                .clientIp(getClientIp())
                .userAgent(getUserAgent())
                .build();

        auditLogRepository.save(log);
    }

    // AOP切面自动记录审计日志
    @Around("@annotation(auditLog)")
    public Object auditLogAspect(ProceedingJoinPoint pjp, AuditLogAnnotation auditLog)
            throws Throwable {
        String username = getCurrentUsername();
        String operation = auditLog.operation();
        long startTime = System.currentTimeMillis();

        try {
            Object result = pjp.proceed();
            logAuditEvent(username, AuditEventType.DATA_MODIFY,
                    operation, "SUCCESS", null);
            return result;
        } catch (Exception e) {
            logAuditEvent(username, AuditEventType.DATA_MODIFY,
                    operation, "FAILURE", e.getMessage());
            throw e;
        }
    }
}
```

## 六、高频面试题

**问题 1：什么是网络安全等级保护？为什么要实施等保？**

**答：**

网络安全等级保护是我国网络安全的基本制度，根据信息系统的重要程度和遭到破坏后的危害程度，将信息系统划分为五个安全保护等级，对不同等级实施不同强度的安全保护。

**实施等保的原因：**

1. **法律要求** - 《网络安全法》明确规定网络运营者应当按照等级保护制度要求履行安全保护义务
2. **安全保障** - 通过系统化的安全建设，提升信息系统整体安全防护能力
3. **合规要求** - 满足行业监管和主管部门的合规要求
4. **风险控制** - 识别系统安全风险，采取针对性措施降低风险

**问题 2：等保二级和三级的主要区别是什么？**

**答：**

|  对比项  |    二级    |     三级     |
|:-----:|:--------:|:----------:|
| 保护级别  |  指导保护级   |   监督保护级    |
| 侵害客体  |  组织合法权益  | 社会秩序/公共利益  |
| 备案要求  |   公安备案   |    公安备案    |
| 测评要求  |   建议测评   | 强制测评（每年一次） |
| 身份鉴别  |  单因子认证   |   双因子认证    |
| 安全审计  | 覆盖主要用户行为 |  覆盖所有用户行为  |
| 数据保密性 |  敏感数据加密  |  重要数据加密存储  |
| 测评周期  | 建议每2年一次  |    每年一次    |

**问题 3：等保测评的流程是怎样的？**

**答：**

等保测评流程分为四个阶段：

1. **测评准备阶段**
    - 收集系统资料
    - 进行系统调研
    - 确定测评范围

2. **方案编制阶段**
    - 编制测评方案
    - 确定测评对象和内容
    - 制定测评实施计划

3. **现场测评阶段**
    - 实施现场测评
    - 记录测评证据
    - 进行差距分析

4. **报告编制阶段**
    - 编写测评报告
    - 提出整改建议
    - 给出测评结论

**问题 4：二级等保对身份鉴别有哪些具体要求？**

**答：**

二级等保身份鉴别要求：

1. **身份标识唯一性** - 每个用户具有唯一标识
2. **鉴别信息复杂度** - 密码长度≥8位，包含大小写字母、数字、特殊字符
3. **定期更换密码** - 建议每90天更换一次
4. **登录失败处理** - 配置登录失败锁定功能（如5次失败锁定30分钟）
5. **管理默认账户** - 修改或禁用系统默认账户
6. **鉴别信息保护** - 密码加密存储（如BCrypt）

**注意：** 二级不强制要求双因子认证，三级必须支持双因子认证。

**问题 5：如何满足等保的安全审计要求？**

**答：**

安全审计实现要点：

1. **审计范围覆盖**
    - 登录/登出行为
    - 权限变更操作
    - 敏感数据访问
    - 系统配置变更

2. **审计内容完整**
   ```json
   {
     "eventTime": "2026-03-23T10:30:00",
     "username": "admin",
     "eventType": "DATA_MODIFY",
     "resource": "/api/user/123",
     "result": "SUCCESS",
     "clientIp": "192.168.1.100",
     "detail": "修改用户信息"
   }
   ```

3. **日志保护措施**
    - 日志存储加密
    - 日志访问权限控制
    - 日志完整性校验

4. **保存期限要求**
    - 二级：至少6个月
    - 三级：至少12个月

**问题 6：等保二级系统定级的依据是什么？**

**答：**

根据 GB/T 22240-2020 定级指南：

1. **侵害客体识别**
    - 公民、法人和其他组织的合法权益
    - 社会秩序、公共利益
    - 国家安全

2. **侵害程度评估**
    - 一般损害
    - 严重损害
    - 特别严重损害

3. **二级定级标准**
    - 侵害客体：公民、法人和其他组织的合法权益
    - 侵害程度：造成严重损害

4. **典型系统示例**
    - 中小型企业信息系统
    - 非敏感政务系统
    - 普通电商平台

**问题 8：等保2.0相比1.0有哪些主要变化？**

**答：**

| 对比项  |          等保1.0          |          等保2.0           |
|:----:|:-----------------------:|:------------------------:|
| 标准名称 | 《信息安全技术 信息系统安全等级保护基本要求》 |  《信息安全技术 网络安全等级保护基本要求》   |
| 保护对象 |          信息系统           |   云计算、移动互联、物联网、工业控制系统等   |
| 控制层面 |     物理、网络、主机、应用、数据      | 物理环境、通信网络、区域边界、计算环境、管理中心 |
| 安全要求 |         侧重技术防护          |         技术与管理并重          |
| 测评方法 |         侧重合规检查          |          强调风险分析          |

**核心变化：**

1. 扩展保护对象范围
2. 强化安全管理要求
3. 增加新技术应用安全要求
4. 完善测评方法体系
