## 一、算法概述

### 1.1 什么是一致性哈希

一致性哈希（Consistent Hashing）由David Karger等人在1997年提出，主要用于**解决分布式系统中数据分布和负载均衡问题**。

**核心思想**：
```
传统取模哈希：
hash(key) % N → 节点编号
问题：N变化时，几乎所有key的位置都变化

一致性哈希：
1. 将节点和数据都映射到一个哈希环上（0 ~ 2³²-1）
2. 数据存储在顺时针方向的第一个节点上
3. 节点增减时只影响相邻节点的数据

哈希环示意：
        0 (2^32)
        |
   NodeA    NodeB
      \      /
       \    /
        \  /
    key1  \/  key2
         /\
        /  \
   NodeC    NodeD
        |
      2^31
```

### 1.2 解决的问题

| 问题 | 传统取模哈希 | 一致性哈希 | 改进效果 |
|------|--------------|------------|----------|
| **节点变化影响** | 大量数据重新映射 | 只影响相邻节点 | 迁移量减少80%+ |
| **数据迁移量** | 几乎全量迁移 | 少量迁移 | 仅1/N数据迁移 |
| **负载均衡** | 均匀 | 需虚拟节点优化 | 可调节 |
| **可扩展性** | 差 | 好 | 动态扩缩容 |

## 二、算法原理

### 2.1 哈希环

```
        0 (2^32-1)
        |
   NodeA    NodeB
      \      /
       \    /
        \  /
    key1  \/  key2
         /\
        /  \
   NodeC    NodeD
        |
      2^31
```

### 2.2 数据定位

```
1. 计算数据的哈希值：hash(key)
2. 在哈希环上顺时针查找第一个节点
3. 数据存储在该节点上

示例：
hash("user:1001") = 12345678 → 存储在NodeA
hash("user:2048") = 87654321 → 存储在NodeB
```

## 三、Java实现

### 3.1 基础一致性哈希

```java
public class ConsistentHashRing {
    private final TreeMap<Long, Node> ring;  // 哈希环
    private final int virtualNodes;          // 虚拟节点数
    
    public ConsistentHashRing(int virtualNodes) {
        this.ring = new TreeMap<>();
        this.virtualNodes = virtualNodes;
    }
    
    // 添加节点
    public void addNode(String nodeName) {
        for (int i = 0; i < virtualNodes; i++) {
            String virtualName = nodeName + "#" + i;
            long hash = hash(virtualName);
            ring.put(hash, new Node(virtualName, nodeName));
        }
    }
    
    // 移除节点
    public void removeNode(String nodeName) {
        for (int i = 0; i < virtualNodes; i++) {
            ring.remove(hash(nodeName + "#" + i));
        }
    }
    
    // 获取数据对应的节点
    public String getNode(String key) {
        if (ring.isEmpty()) return null;
        
        long hash = hash(key);
        Map.Entry<Long, Node> entry = ring.ceilingEntry(hash);
        
        // 顺时针绕回
        if (entry == null) {
            entry = ring.firstEntry();
        }
        
        return entry.getValue().physicalNode;
    }
}
```

### 3.2 带权重的哈希环

```java
public class WeightedConsistentHash {
    private final TreeMap<Long, String> ring;
    private final Map<String, Integer> nodeWeights;
    private final int defaultVirtualNodes;
    
    public void addNode(String nodeName, int weight) {
        nodeWeights.put(nodeName, weight);
        int virtualNodes = defaultVirtualNodes * weight;
        
        for (int i = 0; i < virtualNodes; i++) {
            String virtualName = nodeName + "#" + i;
            ring.put(hash(virtualName), nodeName);
        }
    }
}
```

## 四、虚拟节点

### 4.1 为什么需要虚拟节点

**问题**：
- 节点数量少时，数据分布不均匀
- 某些节点可能分配到的数据很少

**解决方案**：
- 每个物理节点对应多个虚拟节点
- 虚拟节点越多，分布越均匀

### 4.2 虚拟节点数量选择

| 虚拟节点数 | 标准差 | 负载均衡性 |
|------------|--------|------------|
| 0 | 高 | 差 |
| 50 | 中 | 一般 |
| 150 | 低 | 好 |
| 300 | 很低 | 很好 |

**推荐**：150-200个虚拟节点/物理节点

## 五、一致性哈希 vs 传统哈希

### 5.1 数据迁移对比

```
场景：3个节点 → 4个节点，10000个key

传统取模哈希：
- 数据迁移量：约75%
- 原因：hash % 3 → hash % 4，大部分key位置变化

一致性哈希：
- 数据迁移量：约25%
- 原因：只有新节点附近的key需要迁移
```

### 5.2 性能对比

| 指标 | 传统哈希 | 一致性哈希 |
|------|----------|------------|
| 查找时间 | O(1) | O(log V)，V为虚拟节点数 |
| 节点变化影响 | 全局 | 局部 |
| 负载均衡 | 好 | 需虚拟节点优化 |
| 实现复杂度 | 简单 | 较复杂 |

## 六、应用场景

### 6.1 分布式缓存

```
Redis Cluster使用一致性哈希：
- 数据分片存储在多个Redis节点
- 节点扩容时只迁移部分数据
- 客户端缓存哈希环，直接定位节点
```

### 6.2 数据库分片

```
用户ID分片：
user_1001 → DB-Shard-01
user_2048 → DB-Shard-02
user_3392 → DB-Shard-03

添加新分片时，只迁移部分用户数据
```

### 6.3 负载均衡

```
Nginx一致性哈希负载均衡：
- 根据客户端IP计算哈希值
- 相同IP的请求路由到同一服务器
- 服务器增减时保持会话粘性
```

## 七、面试高频题详解

### 题1：一致性哈希解决了什么问题？⭐⭐⭐⭐⭐

**传统取模哈希的问题**：
```
场景：3个节点，10000个key

hash(key) % 3 → 节点编号

添加第4个节点后：
hash(key) % 4 → 节点编号

影响分析：
- 原来在节点0的key：hash % 3 = 0
- 新位置：hash % 4 可能为0,1,2,3
- 只有hash % 4 = 0的key还在节点0
- 迁移比例：约75%的key需要迁移！
```

**一致性哈希的优势**：
```
添加新节点NodeD：
- 只影响NodeD顺时针方向的下一个节点
- 该节点上部分数据迁移到NodeD
- 迁移比例：约1/N（N为节点数）
- 3节点→4节点：仅约25%数据迁移
```

### 题2：虚拟节点的作用？⭐⭐⭐⭐⭐

**问题**：节点数量少时，数据分布不均匀
```
场景：2个节点在哈希环上

NodeA在位置100
NodeB在位置200

如果大部分key哈希到150-200之间：
- NodeB负载很高
- NodeA负载很低
```

**解决方案：虚拟节点**
```
每个物理节点对应多个虚拟节点：

NodeA → A#1, A#2, A#3, ..., A#150
NodeB → B#1, B#2, B#3, ..., B#150

效果：
- 300个虚拟节点均匀分布在环上
- 数据分布更均匀
- 负载均衡性提高
```

**虚拟节点数量选择**：
| 数量 | 标准差 | 推荐场景 |
|------|--------|----------|
| 50 | 高 | 测试环境 |
| 150 | 中 | 生产环境（推荐） |
| 300 | 低 | 高负载场景 |

### 题3：一致性哈希的时间复杂度？⭐⭐⭐⭐

| 操作 | 时间复杂度 | 说明 |
|------|------------|------|
| **查找节点** | O(log V) | V为虚拟节点数，TreeMap查询 |
| **添加节点** | O(V log V) | 添加V个虚拟节点 |
| **删除节点** | O(V log V) | 删除V个虚拟节点 |
| **空间复杂度** | O(V) | 存储哈希环 |

**优化**：
- 使用跳表或二分查找可优化到O(log V)
- 客户端缓存哈希环，避免重复计算

### 题4：一致性哈希 vs 传统哈希？⭐⭐⭐⭐⭐

| 对比维度 | 传统取模哈希 | 一致性哈希 |
|----------|--------------|------------|
| **查找时间** | O(1) | O(log V) |
| **节点变化影响** | 全局 | 局部 |
| **数据迁移量** | 几乎全量 | 约1/N |
| **负载均衡** | 均匀 | 需虚拟节点优化 |
| **实现复杂度** | 简单 | 较复杂 |
| **适用场景** | 节点固定 | 动态扩缩容 |

### 题5：实际应用中的注意事项？⭐⭐⭐⭐

1. **虚拟节点数量**：150-200个/物理节点
2. **节点权重**：根据机器性能配置不同权重
3. **数据迁移策略**：
   - 渐进式迁移，避免流量突增
   - 双写策略：新旧节点同时写入
   - 迁移完成后切换读取
4. **哈希函数选择**：
   - MD5/SHA-256：分布均匀
   - 避免使用简单取模
5. **监控**：
   - 节点负载监控
   - 数据分布监控
