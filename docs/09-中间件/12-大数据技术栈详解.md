## 一、大数据技术栈概述

### 1.1 大数据 4V 特征

**Volume（大量）**：TB/PB 级别的数据量
**Velocity（高速）**：实时或近实时的数据处理
**Variety（多样）**：结构化、半结构化、非结构化数据
**Value（价值）**：低价值密度，需要挖掘

### 1.2 技术选型对比

| 场景 | 技术选型 | 延迟 | 吞吐量 |
|------|---------|------|--------|
| **实时计算** | Flink | 毫秒级 | 高 |
| **准实时** | Spark Streaming | 秒级 | 很高 |
| **离线批处理** | Spark/Hive | 分钟/小时级 | 极高 |
| **OLAP 查询** | ClickHouse/Doris | 秒级 | 中 |
| **数据存储** | HDFS/S3 | - | - |

### 1.3 典型架构演进

```
┌─────────────────────────────────────┐
│   第一代：Lambda 架构                │
│   批处理 + 流处理双链路              │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   第二代：Kappa 架构                 │
│   统一流处理                        │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   第三代：Lakehouse                  │
│   数据湖 + 数据仓库融合              │
└─────────────────────────────────────┘
```

## 二、Flink 实时计算

### 2.1 Flink 核心特点

**真正的流处理**：
- 基于事件驱动，非微批处理
- 原生支持流式计算

**低延迟高吞吐**：
- 毫秒级延迟
- 百万级 TPS

**Exactly-Once 语义**：
- Checkpoint 机制保证精确一次
- End-to-End 一致性

**事件时间处理**：
- Watermark 解决乱序问题
- Window 窗口聚合

**状态管理**：
- 本地/ RocksDB 状态后端
- 增量 Checkpoint

### 2.2 Flink 编程模型

#### DataStream API

```java
// 创建环境
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

// 读取数据源
DataStream<String> text = env.socketTextStream("localhost", 9999);

// 转换处理
DataStream<Tuple2<String, Integer>> wordCounts = text
    .flatMap(new Tokenizer())
    .keyBy(value -> value.f0)
    .window(TumblingEventTimeWindows.of(Time.seconds(5)))
    .sum(1);

// 输出结果
wordCounts.print();

// 执行任务
env.execute("Flink WordCount");
```

#### 窗口操作

**滚动窗口（Tumbling Window）**：
```java
.keyBy(keySelector)
.window(TumblingEventTimeWindows.of(Time.minutes(1)))
```

**滑动窗口（Sliding Window）**：
```java
.window(SlidingEventTimeWindows.of(
    Time.minutes(1),  // 窗口大小
    Time.seconds(10)  // 滑动步长
))
```

**会话窗口（Session Window）**：
```java
.window(EventTimeSessionWindows.withGap(Time.minutes(10)))
```

### 2.3 Flink SQL

```sql
-- 创建 Kafka 源表
CREATE TABLE user_behavior (
    user_id BIGINT,
    item_id BIGINT,
    behavior STRING,
    ts TIMESTAMP(3),
    WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH (
    'connector' = 'kafka',
    'topic' = 'user_behavior',
    'properties.bootstrap.servers' = 'localhost:9092',
    'format' = 'json'
);

-- 实时聚合查询
SELECT 
    TUMBLE_START(ts, INTERVAL '1' MINUTE) AS window_start,
    behavior,
    COUNT(*) AS behavior_count
FROM user_behavior
GROUP BY TUMBLE(ts, INTERVAL '1' MINUTE), behavior;
```

### 2.4 生产实践要点

**Checkpoint 配置**：
```java
env.enableCheckpointing(60000); // 每分钟 checkpoint
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30000);
env.setStateBackend(new RocksDBStateBackend("hdfs://path"));
```

**背压处理**：
- 自动背压机制
- 监控背压级别

**容错恢复**：
- Savepoint 保存点
- FromSavepoint 恢复

## 三、Spark 离线分析

### 3.1 Spark 核心概念

**RDD（弹性分布式数据集）**：
- 不可变、可分区的元素集合
- 支持并行操作

**Transformation（转换）**：
- map、filter、flatMap、reduceByKey
- 惰性求值

**Action（行动）**：
- collect、count、reduce、saveAsTextFile
- 触发实际计算

### 3.2 Spark RDD 示例

```java
JavaSparkContext sc = new JavaSparkContext("local[*]", "SparkDemo");

// 创建 RDD
List<Integer> data = Arrays.asList(1, 2, 3, 4, 5);
JavaRDD<Integer> rdd = sc.parallelize(data);

// Transformation
JavaRDD<Integer> squared = rdd.map(x -> x * x);
JavaRDD<Integer> filtered = squared.filter(x -> x > 10);

// Action
long count = filtered.count();
int sum = filtered.reduce((a, b) -> a + b);

sc.close();
```

### 3.3 Spark SQL DataFrame

```java
SparkSession spark = SparkSession.builder()
        .appName("SparkSQL")
        .master("local[*]")
        .getOrCreate();

// 创建 DataFrame
List<User> users = Arrays.asList(
    new User(1L, "张三", 25),
    new User(2L, "李四", 30)
);

Dataset<Row> df = spark.createDataFrame(users, User.class);

// DataFrame 操作
df.filter(df.col("age").gt(25)).show();
df.groupBy("city").count().show();

// SQL 查询
df.createOrReplaceTempView("users");
spark.sql("SELECT city, AVG(age) FROM users GROUP BY city").show();

spark.stop();
```

### 3.4 Spark vs Flink

| 对比项 | Spark | Flink |
|--------|-------|-------|
| **处理模型** | 微批处理 | 真正流处理 |
| **延迟** | 秒级 | 毫秒级 |
| **吞吐量** | 更高 | 高 |
| **容错** | Checkpoint | Checkpoint |
| **窗口支持** | 丰富 | 更灵活 |
| **SQL 支持** | 成熟 | 完善中 |

## 四、Hadoop 生态概览

### 4.1 Hadoop 核心组件

#### HDFS（分布式文件系统）

**架构**：
```
┌─────────────────┐
│   NameNode      │  ← 主节点（管理元数据）
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
┌───┴──┐ ┌───┴──┐
│Data  │ │Data  │  ← 从节点（存储数据）
│Node  │ │Node  │
└──────┘ └──────┘
```

**特点**：
- 高容错性
- 高吞吐量
- 适合大规模数据集
- 流式数据访问

#### MapReduce（分布式计算框架）

**工作流程**：
```
Input → Split → Map → Shuffle → Reduce → Output
```

**词频统计示例**：
```
输入：
  "hello world"
  "hello hadoop"

Map 阶段：
  ("hello", 1), ("world", 1)
  ("hello", 1), ("hadoop", 1)

Shuffle 阶段：
  "hello" -> [1, 1]
  "world" -> [1]
  "hadoop" -> [1]

Reduce 阶段：
  ("hello", 2), ("world", 1), ("hadoop", 1)
```

#### YARN（资源调度系统）

**组件**：
- ResourceManager：全局资源管理
- NodeManager：节点资源管理
- ApplicationMaster：应用管理

### 4.2 Hadoop 生态圈

```
┌─────────────────────────────────────┐
│          数据采集层                  │
│  Flume(日志)  Sqoop(关系型)  Kafka  │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│          数据存储层                  │
│     HDFS    HBase    S3             │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│          数据计算层                  │
│  MapReduce  Spark  Flink  Hive     │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│          数据查询层                  │
│     Hive    Impala    Presto       │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│          数据应用层                  │
│   BI 报表    AI 模型    数据服务     │
└─────────────────────────────────────┘
```

## 五、ClickHouse OLAP

### 5.1 ClickHouse 特点

**列式存储**：
- 只读取需要的列
- 高压缩率（10:1）

**向量化执行**：
- SIMD 指令集优化
- CPU 缓存友好

**SQL 支持**：
- 标准 SQL 语法
- 丰富的函数库

**实时查询**：
- 亚秒级响应
- 高并发支持

### 5.2 表引擎

**MergeTree 系列**：
```sql
-- 基础 MergeTree
CREATE TABLE events (
    event_id UInt64,
    user_id UInt64,
    event_time DateTime
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(event_time)
ORDER BY (user_id, event_time)
SETTINGS index_granularity = 8192;
```

**SummingMergeTree**：
```sql
-- 自动聚合
CREATE TABLE stats (
    date Date,
    category String,
    amount UInt64
) ENGINE = SummingMergeTree(amount)
ORDER BY (date, category);
```

**Distributed**：
```sql
-- 分布式表
CREATE TABLE events_all AS events
ENGINE = Distributed(cluster_name, default, events, rand());
```

### 5.3 OLAP 查询实战

#### 聚合分析
```sql
SELECT 
    event_type,
    count() AS event_count,
    sum(amount) AS total_amount
FROM user_events
WHERE event_time >= today() - INTERVAL 7 DAY
GROUP BY event_type
ORDER BY event_count DESC;
```

#### 漏斗分析
```sql
SELECT 
    event_type,
    count(DISTINCT user_id) AS unique_users
FROM user_events
WHERE event_time >= today() - INTERVAL 30 DAY
  AND event_type IN ('view', 'click', 'add_cart', 'purchase')
GROUP BY event_type;
```

#### 留存分析
```sql
WITH yesterday_users AS (
    SELECT DISTINCT user_id
    FROM user_events
    WHERE event_time >= today() - INTERVAL 1 DAY
      AND event_time < today()
)
SELECT 
    count(DISTINCT e.user_id) / count(DISTINCT y.user_id) AS retention_rate
FROM yesterday_users y
LEFT JOIN user_events e 
    ON y.user_id = e.user_id
    AND e.event_time >= today()
    AND e.event_time < today() + INTERVAL 1 DAY;
```

## 六、Data Lake 数据湖

### 6.1 数据湖 vs 数据仓库

| 对比项 | 数据仓库 | 数据湖 |
|--------|---------|--------|
| **数据类型** | 结构化 | 所有类型 |
| **Schema** | Schema-on-Write | Schema-on-Read |
| **成本** | 高 | 低 |
| **使用者** | BI 分析师 | 数据科学家 |
| **灵活性** | 低 | 高 |

### 6.2 数据湖架构

```
┌─────────────────────────────────────┐
│          数据应用层                  │
│   (BI 报表、AI 模型、实时查询)         │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│          数据服务层                  │
│   (数据 API、查询引擎、计算引擎)     │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│          数据存储层                  │
│   (HDFS、S3、OSS、MinIO)            │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│          数据采集层                  │
│   (Kafka、Flume、Sqoop、CDC)        │
└─────────────────────────────────────┘
```

### 6.3 Lambda vs Kappa 架构

**Lambda 架构**：
```
应用层
   ↙     ↘
批处理层  速度层
   ↘     ↙
    服务层
```
- ✅ 兼顾准确性和实时性
- ❌ 维护两套代码

**Kappa 架构**：
```
应用层
   ↑
流处理层
   ↑
Kafka（日志存储）
```
- ✅ 只需维护一套代码
- ❌ 对流处理要求高

### 6.4 Lakehouse（湖仓一体）

**特点**：
- 结合数据湖和数据仓库优势
- 支持 ACID 事务
- 支持 Schema 演化
- 支持增量更新

**代表技术**：
- Delta Lake（Databricks）
- Apache Hudi
- Apache Iceberg

### 6.5 数据仓库分层

**ODS 层（原始数据层）**：
- 保持数据原貌
- 不做清洗加工

**DWD 层（明细数据层）**：
- 清洗和规范化
- 维度退化

**DWS 层（服务数据层）**：
- 轻度聚合
- 按主题汇总

**ADS 层（应用数据层）**：
- 高度聚合
- 面向应用场景

**DIM 层（维度层）**：
- 存储维度信息

