## 一、OAuth2.0 协议详解

### 1.1 OAuth2.0 四种授权模式

#### 1. 授权码模式（Authorization Code）⭐⭐⭐⭐⭐

**最安全的模式，适用于 Web 应用**

```
┌─────────┐                    ┌──────────────┐                    ┌─────────┐
│  Client │                    │ Authorization│                    │ Resource│
│  (Web)  │                    │    Server    │                    │  Server │
└────┬────┘                    └──────┬───────┘                    └────┬────┘
     │                                │                                 │
     │ 1. 重定向到授权页面             │                                 │
     │───────────────────────────────>│                                 │
     │                                │                                 │
     │ 2. 用户登录并授权               │                                 │
     │<───────────────────────────────│                                 │
     │                                │                                 │
     │ 3. 返回授权码（Code）           │                                 │
     │<───────────────────────────────│                                 │
     │                                │                                 │
     │ 4. 用授权码换取 Token           │                                 │
     │───────────────────────────────>│                                 │
     │                                │                                 │
     │ 5. 返回 Access Token            │                                 │
     │<───────────────────────────────│                                 │
     │                                │                                 │
     │ 6. 携带 Token 访问资源          │────────────────────────────────>│
     │                                │                                 │
```

**代码示例：**

```java
// 1. 构建授权 URL
String authorizationUrl = "https://auth-server.com/oauth/authorize?" +
        "client_id=interview-client" +
        "&redirect_uri=http://localhost:8080/callback" +
        "&response_type=code" +
        "&scope=read,write" +
        "&state=random-state-string";

// 2. 回调处理 - 用授权码换取 Token
@PostMapping("/callback")
public ResponseEntity<?> handleCallback(@RequestParam String code) {
    // 准备请求参数
    MultiValueMap<String, String> params = new LinkedMultiValueMap<>();
    params.add("grant_type", "authorization_code");
    params.add("code", code);
    params.add("client_id", clientId);
    params.add("client_secret", clientSecret);
    params.add("redirect_uri", redirectUri);
    
    // 发送 POST 请求换取 Token
    RestTemplate restTemplate = new RestTemplate();
    HttpHeaders headers = new HttpHeaders();
    headers.setContentType(MediaType.APPLICATION_FORM_URLENCODED);
    
    HttpEntity<MultiValueMap<String, String>> request = 
            new HttpEntity<>(params, headers);
    
    ResponseEntity<Map> response = restTemplate.postForEntity(
            tokenUri, request, Map.class);
    
    return ResponseEntity.ok(response.getBody());
}
```

#### 2. 简化模式（Implicit）❌

**已废弃，不安全！**

```
客户端直接获取 Token，没有中间码
适用于：纯前端应用（现已不推荐使用）
```

#### 3. 密码模式（Resource Owner Password Credentials）⭐⭐

**仅适用于受信任的应用**

```java
// 用户提供用户名密码给客户端
MultiValueMap<String, String> params = new LinkedMultiValueMap<>();
params.add("grant_type", "password");
params.add("username", "user@example.com");
params.add("password", "password123");
params.add("client_id", clientId);
params.add("client_secret", clientSecret);

// 直接换取 Token
ResponseEntity<Map> response = restTemplate.postForEntity(tokenUri, params, Map.class);
```

**应用场景：**

- 公司内部系统
- 第一方客户端（官方 App）

#### 4. 客户端模式（Client Credentials）⭐⭐⭐⭐

**机器对机器通信**

```java
// 不需要用户参与，客户端自己获取 Token
MultiValueMap<String, String> params = new LinkedMultiValueMap<>();
params.add("grant_type", "client_credentials");
params.add("client_id", clientId);
params.add("client_secret", clientSecret);

ResponseEntity<Map> response = restTemplate.postForEntity(tokenUri, params, Map.class);
```

**应用场景：**

- 微服务之间调用
- 后台任务定时执行

## 二、JWT Token 机制

### 2.1 JWT 结构

```
Header.Payload.Signature

Header:
{
  "alg": "HS256",
  "typ": "JWT"
}

Payload:
{
  "sub": "1234567890",
  "name": "John Doe",
  "iat": 1516239022,
  "exp": 1516242622,
  "authorities": ["ROLE_USER", "ORDER_READ"]
}

Signature:
HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret
)
```

### 2.2 JWT 最佳实践

```java
@Component
public class JwtTokenProvider {
    
    @Value("${security.jwt.secret}")
    private String jwtSecret;
    
    @Value("${security.jwt.expiration:3600}")
    private long jwtExpiration; // 1 小时
    
    /**
     * 生成 JWT Token
     */
    public String generateToken(String username, Map<String, Object> claims) {
        Date now = new Date();
        Date expiryDate = new Date(now.getTime() + jwtExpiration * 1000);
        
        return Jwts.builder()
                .setClaims(claims != null ? claims : new HashMap<>())
                .setSubject(username)
                .setIssuedAt(now)
                .setExpiration(expiryDate)
                .signWith(secretKey, SignatureAlgorithm.HS256)
                .compact();
    }
    
    /**
     * 验证 Token
     */
    public boolean validateToken(String token) {
        try {
            Jwts.parserBuilder()
                    .setSigningKey(secretKey)
                    .build()
                    .parseClaimsJws(token);
            return true;
        } catch (SignatureException e) {
            log.error("Invalid JWT signature");
        } catch (ExpiredJwtException e) {
            log.error("Expired JWT token");
        }
        return false;
    }
    
    /**
     * 刷新 Token
     */
    public String refreshToken(String token) {
        Claims claims = getClaimsFromToken(token);
        Date now = new Date();
        Date expiryDate = new Date(now.getTime() + jwtExpiration * 1000);
        
        return Jwts.builder()
                .setClaims(claims)
                .setIssuedAt(now)
                .setExpiration(expiryDate)
                .signWith(secretKey, SignatureAlgorithm.HS256)
                .compact();
    }
}
```

### 2.3 Access Token vs Refresh Token

| 类型            | 有效期            | 用途              | 存储位置              |
|---------------|----------------|-----------------|-------------------|
| Access Token  | 短（15 分钟 -1 小时） | 访问资源接口          | 内存/SessionStorage |
| Refresh Token | 长（7 天 -30 天）   | 刷新 Access Token | HttpOnly Cookie   |

## 三、Spring Security 实战

### 3.1 Spring Security 配置

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf().disable()
            .cors().and()
            .sessionManagement()
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            .and()
            .authorizeHttpRequests()
                .antMatchers("/api/auth/**").permitAll()
                .antMatchers("/api/public/**").permitAll()
                .antMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            .and()
            .addFilterBefore(jwtAuthenticationFilter(), 
                    UsernamePasswordAuthenticationFilter.class);
        
        return http.build();
    }
    
    @Bean
    public AuthenticationManager authenticationManager(
            AuthenticationConfiguration config) throws Exception {
        return config.getAuthenticationManager();
    }
}
```

## 五、RBAC 权限模型

### 5.1 RBAC 核心概念

```
用户（User） ──→ 角色（Role） ──→ 权限（Permission）
   ↑                                    ↓
   └────────────────────────────────────┘
              （通过角色间接获得权限）
```

### 5.2 数据库设计

```sql
-- 用户表
CREATE TABLE sys_user (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50) UNIQUE,
    password VARCHAR(100),
    status TINYINT
);

-- 角色表
CREATE TABLE sys_role (
    id BIGINT PRIMARY KEY,
    code VARCHAR(50) UNIQUE,
    name VARCHAR(50)
);

-- 权限表
CREATE TABLE sys_permission (
    id BIGINT PRIMARY KEY,
    code VARCHAR(50),
    name VARCHAR(50),
    resource_type VARCHAR(20),
    resource_path VARCHAR(200)
);

-- 用户角色关联表
CREATE TABLE sys_user_role (
    user_id BIGINT,
    role_id BIGINT,
    PRIMARY KEY (user_id, role_id)

CREATE TABLE sys_role_permission (
    role_id BIGINT,
    permission_id BIGINT,
    PRIMARY KEY (role_id, permission_id)
);
```

### 6.2 网关过滤器实现

```java
@Component
public class AuthenticationFilter implements WebFilter {
    
    private static final String[] WHITE_LIST = {
            "/api/auth/login",
            "/api/auth/register",
            "/api/public/**"
    };
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        String path = exchange.getRequest().getPath().value();
        
        // 白名单放行
        if (isWhiteList(path)) {
            return chain.filter(exchange);
        }
        
        // 获取并验证 Token
        String token = extractToken(exchange);
        if (!validateToken(token)) {
            return unauthorized(exchange, "Invalid token");
        }
        
        return chain.filter(exchange);
    }
}
```

## 七、常见攻击防护

### 7.1 XSS（跨站脚本攻击）

**攻击原理：**

- 注入恶意脚本到网页
- 其他用户浏览时执行
- 窃取 Cookie、会话令牌

**防护方案：**

```java
@Component
public class XssFilter {
    
    public String sanitize(String input) {
        if (input == null) return null;
        
        // 移除<script>标签
        String sanitized = input.replaceAll("<script.*?>.*?</script>", "");
        
        // 移除事件处理器
        sanitized = sanitized.replaceAll("\\s+on\\w+\\s*=", "");
        
        // HTML 实体编码
        return htmlEncode(sanitized);
    }
    
    private String htmlEncode(String input) {
        return input.replace("<", "&lt;")
                   .replace(">", "&gt;")
                   .replace("&", "&amp;")
                   .replace("\"", "&quot;");
    }
}
```

### 7.2 CSRF（跨站请求伪造）

**攻击原理：**

- 诱导用户点击恶意链接
- 以用户身份执行操作
- 转账、修改密码等

**防护方案：**

```java
@Configuration
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.csrf().csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse());
        return http.build();
    }
}
```

### 7.3 SQL 注入

**攻击原理：**

- 在输入中注入 SQL 语句
- 欺骗服务器执行恶意 SQL

**防护方案：**

```java
// ❌ 错误示例 - 字符串拼接
String sql = "SELECT * FROM users WHERE username = '" + username + "'";

// ✅ 正确示例 - 使用预编译
String sql = "SELECT * FROM users WHERE username = ?";
PreparedStatement pstmt = connection.prepareStatement(sql);
pstmt.setString(1, username);
```

### 7.4 限流防刷

**固定窗口算法：**

```java
@Component
public class RateLimiter {
    
    private final Cache<String, AccessCounter> cache = 
            CacheBuilder.newBuilder()
                .maximumSize(10000)
                .expireAfterWrite(1, TimeUnit.MINUTES)
                .build();
    
    public boolean tryAcquire(String userId, int limit, long windowSeconds) {
        AccessCounter counter = cache.getIfPresent(userId);
        
        if (counter == null || counter.isExpired(windowSeconds)) {
            counter = new AccessCounter();
            cache.put(userId, counter);
        }
        
        synchronized (counter) {
            counter.count++;
            return counter.count <= limit;
        }
    }
}
```

## 八、高频面试题

### 8.1 OAuth2.0 相关（10 道）

**问题 1：OAuth2.0 的四种授权模式？分别适用于什么场景？**

**答：**

| 授权模式 | 适用场景 | 特点 |
|---------|---------|------|
| **授权码模式** | 第三方Web应用 | 最安全，先获取code再换token |
| **简化模式** | 单页应用SPA | 直接返回token，适合无后端应用 |
| **密码模式** | 受信任的应用 | 用户直接提供账号密码 |
| **客户端模式** | 服务间调用 | 无用户参与，纯后端交互 |

**选择建议：**

- 有后端服务的第三方应用 → 授权码模式
- 纯前端单页应用 → 简化模式
- 自家APP/官方应用 → 密码模式
- 微服务内部调用 → 客户端模式

---

**问题 2：授权码模式为什么最安全？**

**答：**

**安全原因：**

1. **Token 不经过浏览器**：Access Token 直接由服务端交换，不暴露在浏览器中
2. **Code 一次性使用**：授权码只能使用一次，即使被截获也无法重复使用
3. **Code 绑定客户端**：交换 Token 时需要 client_id 和 client_secret 验证
4. **支持 PKCE**：移动端可以额外使用 PKCE 防止授权码拦截攻击

**流程对比：**

```
简化模式：浏览器 ← 直接获取 Token（易被截获）
授权码模式：浏览器 ← Code → 服务端交换 Token（安全）
```

---

**问题 3：Access Token 和 Refresh Token 的区别？**

**答：**

| 特性 | Access Token | Refresh Token |
|------|-------------|---------------|
| **有效期** | 短（15分钟-2小时） | 长（7天-30天） |
| **用途** | 访问受保护资源 | 换取新的 Access Token |
| **存储位置** | 内存/前端 | 安全存储（HttpOnly Cookie） |
| **泄露风险** | 低（有效期短） | 高（需重点保护） |

**双令牌优势：**

- Access Token 泄露影响有限（快速过期）
- Refresh Token 减少用户登录次数
- 支持 Token 刷新时更新权限信息

**问题 4：JWT Token 的结构？**

**答：**

JWT 由三部分组成，用 `.` 分隔：

```
xxxxx.yyyyy.zzzzz
│      │      │
│      │      └── Signature（签名）
│      └───────── Payload（负载）
└──────────────── Header（头部）
```

**Header：**
```json
{
  "alg": "HS256",  // 签名算法
  "typ": "JWT"     // Token类型
}
```

**Payload（Claims）：**
```json
{
  "sub": "user123",    // 主题（用户ID）
  "iat": 1516239022,   // 签发时间
  "exp": 1516242622,   // 过期时间
  "role": "admin"      // 自定义声明
}
```

**Signature：**
```
HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret
)
```

**问题 5：如何防止 JWT Token 被篡改？**

**答：**

**防篡改机制：**

1. **数字签名**：服务端用密钥对 Header + Payload 签名
2. **签名验证**：每次请求时重新计算签名，与 Token 中的签名对比
3. **密钥保密**：密钥只保存在服务端，攻击者无法伪造签名

**验证过程：**

```java
// 1. 分割 Token
String[] parts = token.split("\\.");

// 2. 重新计算签名
String signature = HMACSHA256(parts[0] + "." + parts[1], secret);

// 3. 对比签名
if (!signature.equals(parts[2])) {
    throw new InvalidTokenException("Token被篡改");
}
```

**问题 6：JWT Token 过期了如何处理？**

**答：**

**方案一：Refresh Token（推荐）**

```
1. Access Token 过期（401）
2. 用 Refresh Token 请求 /refresh 接口
3. 服务端验证 Refresh Token 有效
4. 返回新的 Access Token + Refresh Token
```

**方案二：自动续期（Sliding Session）**

```
1. 每次请求时检查 Token 过期时间
2. 如果即将过期（如剩余5分钟），自动签发新 Token
3. 在响应头中返回新 Token
```

**方案三：Redis 存储 Token 状态**

```
1. Token 本身不过期或长期有效
2. 在 Redis 中存储 Token 黑名单/有效期
3. 每次请求查 Redis 验证 Token 状态
```

**问题 7：OAuth2.0 有哪些安全隐患？**

**答：**

| 风险 | 描述 | 防护方案 |
|------|------|---------|
| **授权码拦截** | 攻击者拦截授权码 | 使用 PKCE、HTTPS |
| **CSRF 攻击** | 伪造授权请求 | 使用 state 参数 |
| **Token 泄露** | Access Token 被窃取 | 短有效期、HTTPS |
| **重放攻击** | 重复发送请求 | 时间戳、Nonce |
| **Redirect URI 劫持** | 篡改回调地址 | 严格校验白名单 |

**最佳实践：**

- 所有通信使用 HTTPS
- 授权码一次性使用
- Token 设置合理过期时间
- 定期轮换密钥

**问题 8：如何实现 Token 黑名单？**

**答：**

**场景：** 用户登出、Token 泄露、权限变更时需要使 Token 失效。

**实现方案（Redis）：**

```java
@Service
public class TokenBlacklistService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    // 加入黑名单
    public void addToBlacklist(String token, long expiration) {
        String jti = getJtiFromToken(token);  // JWT ID
        long ttl = getExpirationDate(token).getTime() - System.currentTimeMillis();
        redisTemplate.opsForValue().set("blacklist:" + jti, "1", ttl, TimeUnit.MILLISECONDS);
    }
    
    // 检查是否在黑名单
    public boolean isBlacklisted(String token) {
        String jti = getJtiFromToken(token);
        return Boolean.TRUE.equals(redisTemplate.hasKey("blacklist:" + jti));
    }
}
```

**优化：**

- 使用 JWT ID (jti) 作为黑名单键，减少存储
- 设置过期时间自动清理
- 高并发场景使用布隆过滤器优化

**问题 9：单点登录（SSO）的原理？**

**答：**

**核心原理：** 多个应用共享一个认证中心，登录状态在中心维护。

**实现流程：**

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  应用A   │     │  应用B   │     │ 认证中心 │
└────┬────┘     └────┬────┘     └────┬────┘
     │               │               │
```

```java
@Component
public class RateLimitAspect {
    
    @Around("@annotation(rateLimit)")
    public Object around(ProceedingJoinPoint point, RateLimit rateLimit) {

        Long count = redisTemplate.opsForValue().increment(key);

        return point.proceed();
    }
}
```

**问题 23：常见的限流算法有哪些？**

**答：**

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **计数器** | 固定窗口计数 | 简单 | 边界突发 |
| **滑动窗口** | 滑动时间窗口 | 平滑 | 内存占用 |
| **令牌桶** | 匀速产生令牌 | 允许突发 | 实现复杂 |
| **漏桶** | 匀速流出请求 | 绝对平滑 | 无突发能力 |

**令牌桶实现：**

```java
public class TokenBucket {
    private final long capacity;      // 桶容量
    private final long rate;          // 产生速率（个/秒）
    private long tokens;              // 当前令牌数
    private long lastTime;            // 上次更新时间
    
    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 计算新增令牌
        tokens = Math.min(capacity, 
            tokens + (now - lastTime) * rate / 1000);
        lastTime = now;
        
        if (tokens > 0) {
            tokens--;
            return true;
        }
        return false;
    }
}
```

**Guava RateLimiter：**

```java
RateLimiter limiter = RateLimiter.create(100);  // 每秒100个
if (limiter.tryAcquire()) {
    // 处理请求
} else {
    // 限流
}
```

**问题 25：HTTPS 的工作原理？**

**答：**

**TLS 握手过程：**

```
客户端                                    服务端
  │                                        │
  │  1. ClientHello（支持的加密套件）         │
  │ ──────────────────────────────────────>│
  │                                        │
  │  2. ServerHello（选定加密套件）           │
  │     Certificate（服务端证书）            │
  │<────────────────────────────────────── │
  │                                        │
  │  3. 验证证书 → 生成预主密钥               │
  │     ClientKeyExchange（加密后的预主密钥） │
  │ ──────────────────────────────────────>│
  │                                        │
  │  4. 双方生成会话密钥                      │
  │     ChangeCipherSpec（后续加密通信）      │
  │<─────────────────────────────────────> │
  │                                        │
  │  5. 加密数据传输                          │
  │<══════════════════════════════════════>│
```

**关键概念：**

- **非对称加密**：握手阶段（RSA/ECDHE）
- **对称加密**：数据传输阶段（AES-GCM）
- **数字证书**：CA 签发，验证服务器身份
- **前向安全**：ECDHE 即使私钥泄露，历史会话也无法解密

```java
public boolean verifyRequest(RequestDTO request) {

}
```

```java
    }
    
    public Object getAttribute(String sessionId, String key) {
        return redisTemplate.opsForHash().get("session:" + sessionId, key);
    }
}
```

```java
@Service
public class SecureCaptchaService {
    
    // 1. 生成验证码
    public Captcha generate() {
        String code = RandomStringUtils.randomNumeric(6);
        String captchaId = UUID.randomUUID().toString();
        
        // 存储：限制频率（同一手机号1分钟1次）
        String limitKey = "captcha:limit:" + phone;
        if (!rateLimiter.tryAcquire(limitKey)) {
            throw new RateLimitException("请求过于频繁");
        }
        
        // 存储验证码（5分钟有效）
        redisTemplate.opsForValue().set(
            "captcha:" + captchaId, 
            code, 
            5, TimeUnit.MINUTES
        );
        
        return new Captcha(captchaId, generateImage(code));
    }
    
    // 2. 验证验证码
    public boolean verify(String captchaId, String inputCode) {
        String key = "captcha:" + captchaId;
        String storedCode = redisTemplate.opsForValue().get(key);
        
        // 无论成功与否，验证后立即删除（防暴力破解）
        redisTemplate.delete(key);
        
        if (storedCode == null) {
            return false;  // 已过期
        }
        
        // 常量时间比较（防时序攻击）
        return MessageDigest.isEqual(
            storedCode.getBytes(), 
            inputCode.getBytes()
        );
    }
}
```

**防护措施：**

1. **频率限制**：同一手机号/IP 限制发送频率
2. **错误次数限制**：连续错误5次锁定30分钟
3. **图形验证码**：短信验证码前先验证图形码
4. **行为验证**：滑动验证、点击验证等无感验证
5. **防止遍历**：验证码错误时不提示具体错误（统一提示"验证码错误"）

### Token 双令牌机制

```java
@Service
public class AuthService {
    
    public JwtToken login(String username, String password) {
        // 验证用户
        User user = authenticate(username, password);
        
        // 生成 Access Token（1 小时）
        String accessToken = jwtProvider.generateToken(
                user.getUsername(), 
                buildClaims(user)
        );
        
        // 生成 Refresh Token（7 天）
        String refreshToken = jwtProvider.generateRefreshToken(
                user.getUsername()
        );
        
        // 保存 Refresh Token 到 Redis
        redisTemplate.opsForValue().set(
                "refresh:" + user.getId(), 
                refreshToken, 
                7, TimeUnit.DAYS
        );
        
        return JwtToken.of(accessToken, refreshToken, 3600);
    }
    
    public JwtToken refresh(String refreshToken) {
        // 验证 Refresh Token
        if (!jwtProvider.validateToken(refreshToken)) {
            throw new BusinessException("Refresh token expired");
        }
        
        // 从 Redis 验证是否存在
        String storedToken = redisTemplate.opsForValue()
                .get("refresh:" + getUserId(refreshToken));
        
        if (!refreshToken.equals(storedToken)) {
            throw new BusinessException("Invalid refresh token");
        }
        
        // 生成新的 Token 对
        String username = jwtProvider.getUsernameFromToken(refreshToken);
        return loginWithoutPassword(username);
    }
}
```
