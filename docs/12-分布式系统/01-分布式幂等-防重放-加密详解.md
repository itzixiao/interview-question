# 分布式幂等、接口防重放、敏感数据加密详解

## 概述

在分布式系统中，安全性是至关重要的。本文详细讲解三个核心安全主题：

- **分布式幂等**：防止业务重复执行
- **接口防重放**：防止请求被恶意重发

## 第一部分：分布式幂等

### 1.1 什么是幂等性

- **数学定义**：f(f(x)) = f(x)，多次执行结果与一次执行相同
- **编程定义**：同一操作多次执行，结果与执行一次相同
- **HTTP幂等**：GET/PUT/DELETE 幂等，POST 非幂等

### 1.2 为什么需要幂等

| 场景     | 描述         |
|--------|------------|
| 网络超时重试 | 客户端超时后重发请求 |
| 消息队列重试 | 消费失败后重新投递  |
| 表单重复提交 | 用户快速多次点击   |
| 定时任务执行 | 分布式任务多节点触发 |
| 第三方回调  | 支付回调可能重复调用 |

### 1.3 幂等实现方案对比

| 方案          | 适用场景     | 优缺点           |
|-------------|----------|---------------|
| **Token机制** | 表单提交、下单  | 通用性强，需要两次请求   |
| **请求指纹去重**  | 无Token场景 | 一次请求，需合理选指纹字段 |
| **数据库唯一索引** | 插入操作     | 依赖数据库，高并发有锁竞争 |
| **状态机幂等**   | 状态流转     | 最优雅，需业务支持     |
| **乐观锁**     | 更新操作     | 需要版本字段，重试逻辑复杂 |

### 1.4 Token机制详解（推荐）

#### 执行流程

```
客户端                        服务端                         Redis
  │                            │                              │
  │  1.请求获取Token             │                              │
  │ ─────────────────────────> │                              │
  │                            │  2.生成Token存入Redis         │
  │                            │ ─────────────────────────────>
  │                            │  SET idempotent:token:xxx 1  │
  │                            │  EX 600                      │
  │  3.返回Token                │                              │
  │ <───────────────────────── │                              │
  │                            │                              │
  │  4.携带Token提交业务         │                              │
  │ ─────────────────────────> │                              │
  │                            │  5.Lua脚本原子删除Token        │
  │                            │ ─────────────────────────────>
  │                            │  if GET key then DEL key     │
  │                            │                              │
  │  7.执行/拒绝业务             │                              │
  │ <───────────────────────── │                              │
```

#### Lua脚本（保证原子性）

```lua
-- 原子性校验并删除 Token
local key = KEYS[1]
local token = redis.call('GET', key)
if token then
    redis.call('DEL', key)
    return 1  -- 第一次请求，删除成功
end
return 0      -- 重复请求，Token不存在或已被消费
```

#### Java实现代码

```java
@Service
public class IdempotentTokenService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final String TOKEN_PREFIX = "idempotent:token:";
    
    // 生成Token
    public String generateToken(String userId) {
        String token = UUID.randomUUID().toString().replace("-", "");
        String key = TOKEN_PREFIX + token;
        redisTemplate.opsForValue().set(key, userId, 10, TimeUnit.MINUTES);
        return token;
    }
    
    // 验证并消费Token
    public boolean verifyAndConsumeToken(String token) {
        String luaScript = 
            "if redis.call('get', KEYS[1]) then " +
            "   redis.call('del', KEYS[1]) " +
            "   return 1 " +
            "else " +
            "   return 0 " +
            "end";
        
        DefaultRedisScript<Long> script = new DefaultRedisScript<>();
        script.setScriptText(luaScript);
        script.setResultType(Long.class);
        
        Long result = redisTemplate.execute(script, 
            Collections.singletonList(TOKEN_PREFIX + token));
        return result != null && result == 1;
    }
}
```

## 第二部分：接口防重放

### 2.1 什么是重放攻击

攻击者截获合法请求后，原封不动地重复发送，达到非法目的。

**危害**：

- 重复转账
- 重复下单
- 绕过验证

### 2.2 防重放三要素

| 要素        | 作用     | 实现              |
|-----------|--------|-----------------|
| **时间戳**   | 拒绝过期请求 | 校验时间差 < 5分钟     |
| **Nonce** | 拒绝重复请求 | Redis SETNX 去重  |
| **签名**    | 防止参数篡改 | MD5/HMAC-SHA256 |

**三者缺一不可**：

- 无时间戳：Nonce永久存储，内存爆炸
- 无Nonce：时间窗口内可重放
- 无签名：攻击者可伪造时间戳和Nonce

### 2.3 请求参数结构

```json
{
    "appId": "app_10001",
    "timestamp": 1709827200000,
    "nonce": "a1b2c3d4e5f6",
    "sign": "D4B5E6F7...",
    "data": {
        "orderNo": "ORD001",
        "amount": 99.99
    }
}
```

### 2.4 签名算法

```java
// MD5签名
String signContent = "appId=" + appId
    + "&data=" + data
    + "&nonce=" + nonce
    + "&timestamp=" + timestamp
    + "&key=" + secretKey;

String sign = DigestUtils.md5DigestAsHex(signContent.getBytes()).toUpperCase();
```

### 2.5 完整校验代码

```java
public boolean verifyRequest(RequestDTO request, String secret) {
    long serverTime = System.currentTimeMillis();
    
    // 1. 校验时间戳（5分钟内有效）
    if (Math.abs(serverTime - request.getTimestamp()) > 300000) {
        throw new BusinessException("请求已过期");
    }
    
    // 2. 校验 Nonce（Redis SETNX）
    String nonceKey = "replay:nonce:" + request.getNonce();
    Boolean success = redis.opsForValue()
        .setIfAbsent(nonceKey, "1", 5, TimeUnit.MINUTES);
    if (!Boolean.TRUE.equals(success)) {
        throw new BusinessException("请求已被处理");
    }
    
    // 3. 校验签名
    String expectedSign = generateSign(request, secret);
    if (!expectedSign.equals(request.getSign())) {
        throw new BusinessException("签名验证失败");
    }
    
    return true;
}
```

## 第三部分：敏感数据加密

### 3.1 敏感数据分类

| 类型  | 处理方式      | 示例      |
|-----|-----------|---------|
| 密码类 | 哈希存储，永不解密 | 登录密码    |
| 证件类 | 可逆加密，需要查询 | 身份证、银行卡 |
| 展示类 | 脱敏显示      | 手机号、邮箱  |

### 3.2 加密算法选型

| 算法                  | 特点           | 适用场景      |
|---------------------|--------------|-----------|
| **MD5/SHA-256 + 盐** | 哈希不可逆，加盐防彩虹表 | 密码存储      |
| **BCrypt/Argon2**   | 专用密码哈希，自带盐   | 密码存储（推荐）  |
| **AES-256-GCM**     | 对称加密，高性能     | 身份证、银行卡加密 |
| **RSA-2048**        | 非对称，安全但慢     | 密钥交换、数字签名 |

### 3.3 AES-GCM加密实现

```java
public class AesGcmUtil {
    
    // 加密
    public static String encrypt(String plainText, SecretKey secretKey) throws Exception {
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        
        // 生成12字节随机IV
        byte[] iv = new byte[12];
        new SecureRandom().nextBytes(iv);
        
        GCMParameterSpec parameterSpec = new GCMParameterSpec(128, iv);
        cipher.init(Cipher.ENCRYPT_MODE, secretKey, parameterSpec);
        
        byte[] encrypted = cipher.doFinal(plainText.getBytes(StandardCharsets.UTF_8));
        
        // IV + 密文一起存储
        byte[] result = new byte[iv.length + encrypted.length];
        System.arraycopy(iv, 0, result, 0, iv.length);
        System.arraycopy(encrypted, 0, result, iv.length, encrypted.length);
        
        return Base64.getEncoder().encodeToString(result);
    }
    
    // 解密
    public static String decrypt(String encryptedText, SecretKey secretKey) throws Exception {
        byte[] decoded = Base64.getDecoder().decode(encryptedText);
        
        // 分离IV和密文
        byte[] iv = Arrays.copyOfRange(decoded, 0, 12);
        byte[] encrypted = Arrays.copyOfRange(decoded, 12, decoded.length);
        
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        GCMParameterSpec parameterSpec = new GCMParameterSpec(128, iv);
        cipher.init(Cipher.DECRYPT_MODE, secretKey, parameterSpec);
        
        return new String(cipher.doFinal(encrypted), StandardCharsets.UTF_8);
    }
}
```

### 3.4 混合加密方案（推荐）

```
┌─────────────────────────────────────────────────────────────┐
│  1. 客户端生成随机 AES 密钥                                  │
│  2. 用 AES 密钥加密业务数据（高效）                          │
│  3. 用服务端 RSA 公钥加密 AES 密钥（安全）                   │
│  4. 发送: RSA(AES_KEY) + AES(DATA)                          │
│  5. 服务端用 RSA 私钥解密得到 AES 密钥                       │
│  6. 用 AES 密钥解密业务数据                                  │
└─────────────────────────────────────────────────────────────┘
```

## 高频面试题

**问题 1:什么是接口幂等性？哪些场景需要保证幂等？

**答**：

幂等性是指同一操作多次执行，结果与执行一次相同。

需要保证幂等的场景：

1. 网络超时重试 - 客户端超时后重发请求
2. 消息队列重试 - 消费失败后重新投递
3. 表单重复提交 - 用户快速多次点击
4. 定时任务执行 - 分布式任务多节点触发
5. 第三方回调 - 支付回调可能重复调用

---

**问题 2:如何实现分布式幂等？对比各方案优缺点？

**答**：

| 方案      | 优点   | 缺点         |
|---------|------|------------|
| Token机制 | 通用性强 | 需要两次请求     |
| 指纹去重    | 一次请求 | 需合理选指纹字段   |
| 数据库唯一索引 | 简单可靠 | 依赖数据库，有锁竞争 |
| 状态机幂等   | 最优雅  | 需业务支持      |
| 乐观锁     | 无死锁  | 需版本字段，重试复杂 |

---

**问题 3:Token幂等方案为什么要用Lua脚本？

**答**：

为了保证"查询Token是否存在"和"删除Token"两个操作的原子性。

如果分开执行：

```
线程A: GET token → 存在
线程B: GET token → 存在（还未删除）
线程A: DEL token → 成功
线程B: DEL token → 成功（但实际是重复请求）
```

Lua脚本在Redis中原子执行，不会被其他命令打断。

---

**问题 4:什么是重放攻击？如何防止？

**答**：

重放攻击是攻击者截获合法请求后，原封不动地重复发送。

防止方案（三要素）：

1. **时间戳** - 拒绝超时请求（如5分钟外），缩小攻击窗口
2. **Nonce** - 随机数，服务端缓存已用，拒绝重复
3. **签名** - 参数+时间戳+Nonce+密钥计算签名，防篡改

三者缺一不可！

---

**问题 5:幂等和防重放的区别？

**答**：

| 维度   | 幂等          | 防重放          |
|------|-------------|--------------|
| 目的   | 防止业务重复执行    | 防止请求被恶意重发    |
| 触发场景 | 网络重试、消息重投   | 中间人攻击、抓包重放   |
| 关注点  | 业务数据一致性     | 接口安全性        |
| 实现手段 | Token去重、状态机 | 时间戳+Nonce+签名 |

---

**问题 6:AES和RSA的区别？实际项目中如何选择？

**答**：

| 维度   | AES（对称加密） | RSA（非对称加密） |
|------|-----------|------------|
| 密钥   | 加解密同一密钥   | 公钥加密，私钥解密  |
| 性能   | 快，适合大数据量  | 慢，适合小数据    |
| 密钥管理 | 密钥需安全传输   | 公钥可公开      |
| 典型场景 | 数据库字段加密   | 密钥交换、数字签名  |

**实际项目推荐「混合加密」**：

1. 用RSA加密AES密钥（安全传输）
2. 用AES加密业务数据（高效处理）

**问题 7:为什么密码要加盐？彩虹表攻击是什么？

**答**：

彩虹表是预先计算好的「明文→哈希值」对照表。攻击者获取数据库后，直接查表就能反推原始密码。

加盐原理：

```
原始: hash = SHA256(password)          → 可查表破解
加盐: hash = SHA256(password + salt)   → 彩虹表失效
```

最佳实践：

1. 每个用户使用不同的随机盐值
2. 盐值和哈希一起存储: salt:hash
3. 推荐 BCrypt/Argon2，自带盐和计算成本

---

**问题 8:AES的ECB和GCM模式有什么区别？

**答**：

**ECB（电子密码本模式）**：

- 相同明文块加密出相同密文块
- 可通过密文模式推测明文结构
- **不安全，不推荐使用**

**GCM（Galois/Counter模式）**：

- 每次加密使用随机IV，相同明文密文不同
- 自带认证标签（Authentication Tag），可检测篡改
- **推荐使用，HTTPS/TLS默认采用**

**问题 9:如何实现接口级别的数据脱敏？

**答**：

推荐使用自定义注解 + Jackson序列化器：

1. 定义脱敏注解 `@Sensitive`
2. 定义脱敏类型枚举：PHONE/ID_CARD/BANK_CARD/EMAIL
3. 实现 `JsonSerializer<String>`，根据类型脱敏
4. VO字段加注解，返回时自动脱敏

优点：

- 统一管理脱敏规则
- 字段级别控制
- 与业务代码解耦

**问题 10:分布式环境下如何保证密钥安全？

**答**：

1. **密钥管理服务（KMS）**
    - 使用阿里云KMS、AWS KMS等托管服务
    - 密钥不落地，由云服务管理

2. **配置中心加密**
    - Nacos/Apollo支持配置加密
    - 密钥从配置中心动态获取

3. **环境变量注入**
    - K8s Secret、Docker Secret
    - 启动时注入，不写入代码和配置文件

4. **密钥轮换**
    - 定期更换密钥
    - 支持新旧密钥并存过渡期

**问题 11:如何设计一个通用的幂等框架？

**答**：

核心设计：

1. **注解驱动**: `@Idempotent(key="SpEL", expire=300)`
2. **AOP拦截**: 方法执行前后处理幂等逻辑
3. **Key生成**: SpEL表达式支持从参数取值
4. **存储抽象**: 支持Redis、数据库等多种存储

```java
@Around("@annotation(idempotent)")
public Object around(ProceedingJoinPoint pjp) {
    String key = parseKey(idempotent.key(), pjp);
    if (!tryLock(key, idempotent.expire())) {
        throw new RepeatRequestException();
    }
    try {
        return pjp.proceed();
    } finally {
        // 根据策略决定是否释放锁
    }
}
```

**问题 12:生产环境遇到幂等失效怎么排查？

**答**：

排查思路：

1. **确认Key生成逻辑**
    - 是否包含了唯一标识？
    - 参数是否可能为空导致Key相同？

2. **检查Redis连接**
    - Redis是否可用？
    - 是否存在网络分区？

3. **检查过期时间**
    - 过期时间是否设置过短？
    - 业务执行时间是否超过过期时间？

4. **检查Lua脚本执行**
    - 是否被其他命令打断？
    - 集群模式下Key是否在同一slot？

5. **日志追踪**
    - 记录幂等Key、请求参数、执行结果
    - 便于定位重复请求来源
