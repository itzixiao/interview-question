## 一、DDD 领域驱动设计

### 1.1 核心概念

**DDD（Domain-Driven Design）** 是一种以业务领域为核心的软件设计方法论，强调通过深入理解业务来指导软件设计。

#### 三大核心要素

```
┌─────────────────────────────────────────┐
│           DDD 核心要素                   │
├─────────────────────────────────────────┤
│  实体（Entity）   - 有唯一标识的对象     │
│  值对象（Value Object） - 无标识，不可变 │
│  聚合根（Aggregate Root） - 一致性边界   │
└─────────────────────────────────────────┘
```

### 1.2 值对象（Value Object）

**特点：**
- ✅ 无唯一标识性（通过属性值判断相等）
- ✅ 不可变性（创建后不能修改）
- ✅ 自验证（包含业务规则校验）

**示例代码：** [`Money.java`](interview-microservices-parent/interview-system-design/src/main/java/cn/itzixiao/interview/systemdesign/domain/valueobject/Money.java)

```java
package cn.itzixiao.interview.systemdesign.domain.valueobject;

@Value
@Builder
public class Money {
    String currency;        // 币种
    BigDecimal amount;      // 金额数值
    
    public Money {
        // 值对象的自验证逻辑
        if (amount == null || amount.compareTo(BigDecimal.ZERO) < 0) {
            throw new IllegalArgumentException("金额不能为负数");
        }
    }
    
    // 值对象的业务行为
    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalStateException("不同币种不能相加");
        }
        return Money.builder()
                .currency(this.currency)
                .amount(this.amount.add(other.amount))
                .build();
    }
}
```

**使用场景：**
- 金额、地址、时间范围、坐标等

### 1.3 实体（Entity）

**特点：**
- ✅ 有唯一标识（ID）
- ✅ 可变性（状态可以改变）
- ✅ 有生命周期（创建、修改、删除）

**示例代码：** [`User.java`](interview-microservices-parent/interview-system-design/src/main/java/cn/itzixiao/interview/systemdesign/domain/entity/User.java)

```java
package cn.itzixiao.interview.systemdesign.domain.entity;

@Getter
@Setter
public class User {
    private Long id;          // 唯一标识 ID
    private String username;
    private String email;
    private Address address;  // 值对象
    
    // 实体的业务行为
    public void updateEmail(String newEmail) {
        if (newEmail == null || !newEmail.matches("^[A-Za-z0-9+_.-]+@(.+)$")) {
            throw new IllegalArgumentException("邮箱格式不正确");
        }
        this.email = newEmail;
    }
    
    @Override
    public boolean equals(Object obj) {
        // 实体通过 ID 判断相等
        if (this == obj) return true;
        if (!(obj instanceof User)) return false;
        User other = (User) obj;
        return this.id != null && this.id.equals(other.id);
    }
}
```

### 1.4 聚合根（Aggregate Root）

**特点：**
- ✅ 聚合的入口点，外部只能通过聚合根访问聚合内对象
- ✅ 维护聚合内的业务规则和一致性边界
- ✅ 负责发布领域事件

**示例代码：** [`Order.java`](interview-microservices-parent/interview-system-design/src/main/java/cn/itzixiao/interview/systemdesign/domain/aggregate/Order.java)

```java
package cn.itzixiao.interview.systemdesign.domain.aggregate;

@Getter
public class Order {
    private final String orderId;       // 订单 ID（聚合根标识）
    private final Long userId;
    private Money totalAmount;
    private OrderStatus status;
    private final List<OrderItem> items;
    
    /**
     * 静态工厂方法 - 创建订单
     */
    public static Order createOrder(String orderId, Long userId, 
                                    ApplicationEventPublisher publisher) {
        Order order = new Order(orderId, userId);
        
        // 发布订单创建事件
        order.publishEvent(new OrderCreatedEvent(
                order, orderId, userId, 0.0, 0));
        
        return order;
    }
    
    /**
     * 添加订单项
     */
    public void addItem(OrderItem item) {
        if (this.status != OrderStatus.CREATED) {
            throw new IllegalStateException("只有 CREATED 状态的订单才能添加商品");
        }
        this.items.add(item);
        recalculateTotal();
    }
    
    /**
     * 支付订单
     */
    public void pay(String paymentId, String paymentMethod) {
        if (this.status != OrderStatus.CREATED) {
            throw new IllegalStateException("只有 CREATED 状态的订单才能支付");
        }
        
        this.status = OrderStatus.PAID;
        
        // 发布订单支付事件
        publishEvent(new OrderPaidEvent(
                this, orderId, paymentId, 
                totalAmount.getAmount().doubleValue(), paymentMethod));
    }
}
```

### 1.5 领域事件（Domain Event）

**特点：**
- ✅ 描述过去发生的事情（使用过去时态）
- ✅ 不可变
- ✅ 包含事件发生时的完整上下文信息

**示例代码：** [`OrderCreatedEvent.java`](interview-microservices-parent/interview-system-design/src/main/java/cn/itzixiao/interview/systemdesign/domain/event/OrderCreatedEvent.java)

```java
package cn.itzixiao.interview.systemdesign.domain.event;

@Getter
public class OrderCreatedEvent extends ApplicationEvent {
    private final String orderId;
    private final Long userId;
    private final Double totalAmount;
    private final Integer itemCount;
    private final Instant occurredOn;  // 事件发生时间
    
    public OrderCreatedEvent(Object source, String orderId, Long userId, 
                             Double totalAmount, Integer itemCount) {
        super(source);
        this.orderId = orderId;
        this.userId = userId;
        this.totalAmount = totalAmount;
        this.itemCount = itemCount;
        this.occurredOn = Instant.now();
    }
}
```

**领域事件的作用：**
1. 解耦微服务之间的通信
2. 实现最终一致性
3. 记录业务审计日志
4. 支持事件溯源

┌────────────────────────────────────────────┐

├────────────────────────────────────────────┤
│  1. 单一职责原则 - 每个服务只负责一个领域   │

## 三、数据库拆分策略

### 3.1 读写分离

**适用场景：**
- 读多写少（如：查询订单、浏览商品）
- 主库压力大（写操作影响读性能）
- 需要水平扩展读能力

**示例代码：** [`ReadWriteSeparationStrategy.java`](interview-microservices-parent/interview-system-design/src/main/java/cn/itzixiao/interview/systemdesign/database/readwrite/ReadWriteSeparationStrategy.java)

```java
@Slf4j
@Component
public class ReadWriteSeparationStrategy {
    
    public enum DataSourceType {
        MASTER,   // 主库（写）
        SLAVE_1,  // 从库 1（读）
        SLAVE_2,  // 从库 2（读）
        SLAVE_3   // 从库 3（读）
    }
    
    private static final ThreadLocal<DataSourceType> CONTEXT_HOLDER = 
            new ThreadLocal<>();
    
    /**
     * 设置为主库（写操作前调用）
     */
    public void useMaster() {
        CONTEXT_HOLDER.set(DataSourceType.MASTER);
        log.debug("切换到主库（写模式）");
    }
    
    /**
     * 设置为从库（读操作前调用）
     */
    public void useSlave() {
        int index = (int) (System.currentTimeMillis() % 3);
        DataSourceType slaveType = DataSourceType.values()[index + 1];
        CONTEXT_HOLDER.set(slaveType);
        log.debug("切换到从库：{}（读模式）", slaveType);
    }
}
```

### 3.2 分库分表

**适用场景：**
- 数据量大（单表超过 500 万）
- 写并发高（单库写入遇到瓶颈）
- 查询性能下降（索引效率降低）

**分片策略：**
1. **哈希分片** - 按 ID 取模
2. **范围分片** - 按时间/区域
3. **地理分片** - 按地域

**示例代码：** [`DatabaseShardingStrategy.java`](interview-microservices-parent/interview-system-design/src/main/java/cn/itzixiao/interview/systemdesign/database/sharding/DatabaseShardingStrategy.java)

```java
@Slf4j
@Component
public class DatabaseShardingStrategy {
    
    private static final int DATABASE_COUNT = 4;
    private static final int TABLE_COUNT_PER_DB = 4;
    
    /**
     * 计算应该路由到哪个数据库
     */
    public int getDatabaseIndex(Long userId) {
        return Math.abs(userId.hashCode()) % DATABASE_COUNT;
    }
    
    /**
     * 计算应该路由到哪个表
     */
    public int getTableIndex(Long userId) {
        return Math.abs(userId.hashCode()) % TABLE_COUNT_PER_DB;
    }
    
    /**
     * 获取实际的表名
     */
    public String getTableName(Long userId) {
        int dbIndex = getDatabaseIndex(userId);
        int tableIndex = getTableIndex(userId);
        return "t_order_" + dbIndex + "_" + tableIndex;
    }
    
    /**
     * 演示分片路由
     */
    public void demonstrateSharding() {
        Long[] userIds = {1001L, 1002L, 1003L, 2001L, 2002L, 3001L};
        
        for (Long userId : userIds) {
            String dataSource = getDataSourceName(userId);
            String tableName = getTableName(userId);
            
            log.info("userId={} -> 数据库：{}，表：{}", 
                    userId, dataSource, tableName);
        }
    }
}
```

**输出示例：**
```
userId=1001 -> 数据库：ds_1，表：t_order_1_1
userId=1002 -> 数据库：ds_2，表：t_order_2_2
userId=1003 -> 数据库：ds_3，表：t_order_3_3
```

## 四、CQRS架构模式

### 4.1 核心思想

**CQRS（Command Query Responsibility Segregation）** - 命令查询职责分离

```
┌─────────────────────────────────────────────┐
│          CQRS 架构模式                       │
├─────────────────────────────────────────────┤
│  写模型（Command）                           │
│  - 负责业务逻辑和数据验证                   │
│  - 保证数据一致性                           │
│  - 发布领域事件                             │
├─────────────────────────────────────────────┤
│  读模型（Query）                             │
│  - 负责高效查询和展示                       │
│  - 扁平化数据结构                          │
│  - 可使用 NoSQL、ES 等优化查询              │
└─────────────────────────────────────────────┘
```

### 4.4 适用场景

- 开发资源有限的小项目

## 五、事件溯源EventSourcing

### 5.1 核心思想

**Event Sourcing** - 将状态变更表示为一系列事件的架构模式

```
┌──────────────────────────────────────────────┐
│          事件溯源 vs 传统方式                 │
├──────────────────────────────────────────────┤
│  传统方式：                                   │
│  - 只保存当前状态                            │
│  - 丢失历史变更信息                          │
│  - 难以追溯数据变化过程                      │
├──────────────────────────────────────────────┤
│  事件溯源：                                   │
│  - 保存所有历史事件                          │
│  - 当前状态由事件推导而来                    │
│  - 可以回放历史状态                          │
│  - 完整的审计日志                            │
└──────────────────────────────────────────────┘
```

### 5.2 事件基类

### 5.4 聚合根实现

**账户聚合根：** [`Account.java`](interview-microservices-parent/interview-system-design/src/main/java/cn/itzixiao/interview/systemdesign/eventsourcing/aggregate/Account.java)

```java
@Getter
public class Account {
    private final String accountId;
    private String username;
    private Double balance;
    private Long version;
    private final List<DomainEvent> uncommittedEvents;
    
    /**
     * 静态工厂方法 - 创建账户
     */
    public static Account createAccount(String accountId, String username, 
                                        Double initialBalance) {
        Account account = new Account(accountId);
        
        AccountCreatedEvent event = new AccountCreatedEvent(
                accountId, 1L, username, initialBalance);
        
        account.applyEvent(event);
        account.uncommittedEvents.add(event);
        
        return account;
    }
    
    /**
     * 存款
     */
    public void deposit(Double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("存款金额必须大于 0");
        }
        
        Long nextVersion = version + 1;
        Double newBalance = this.balance + amount;
        
        MoneyDepositedEvent event = new MoneyDepositedEvent(
                accountId, nextVersion, amount, newBalance);
        
        applyEvent(event);
        uncommittedEvents.add(event);
    }
    
    /**
     * 取款
     */
    public void withdraw(Double amount, String reason) {
        if (amount > this.balance) {
            throw new IllegalStateException("余额不足");
        }
        
        Long nextVersion = version + 1;
        Double newBalance = this.balance - amount;
        
        MoneyWithdrawnEvent event = new MoneyWithdrawnEvent(
                accountId, nextVersion, amount, reason, newBalance);
        
        applyEvent(event);
        uncommittedEvents.add(event);
    }
    
    /**
     * 从事件流重建状态（事件回放）
     */
    public static Account reconstructFromHistory(List<DomainEvent> events) {
        Account account = new Account(events.get(0).getAggregateId());
        
        // 按顺序应用所有事件，重建状态
        for (DomainEvent event : events) {
            account.applyEvent(event);
        }
        
        return account;
    }
}
```

### 5.6 事件溯源的优势

✅ **优势：**
1. **完整审计** - 所有变更都有记录
2. **可追溯性** - 可以查看任意时间点的状态
3. **临时查询** - 可以从事件派生新的读模型
4. **高并发写入** - 追加写性能更好

⚠️ **挑战：**
1. **复杂性增加** - 需要处理事件版本控制
2. **查询复杂** - 需要 CQRS 配合
3. **事件迁移** - 业务变化时事件结构可能变化

## 七、学习建议

### 学习路线

```
基础阶段（3 天）
├── DDD 核心概念（1 天）
│   ├── 实体、值对象、聚合根
│   └── 领域事件
├── 微服务拆分（1 天）
│   ├── 拆分原则
│   └── 限界上下文
└── 数据库拆分（1 天）
    ├── 读写分离
    └── 分库分表

进阶阶段（3 天）
├── CQRS 架构（1.5 天）
│   ├── 命令端
│   └── 查询端
└── 事件溯源（1.5 天）
    ├── 事件建模
    └── 事件存储与回放

实战阶段（2 天）
└── 综合案例实践
    ├── 电商订单系统
    └── 银行账户系统
```

### 实践建议

1. **从简单开始** - 先理解实体、值对象、聚合根
2. **注重业务** - DDD 的核心是理解业务领域
3. **循序渐进** - 不要一开始就追求完美的领域模型
4. **结合项目** - 在实际项目中应用和实践
5. **持续重构** - 领域模型需要不断演进
