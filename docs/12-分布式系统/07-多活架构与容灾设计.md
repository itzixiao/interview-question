# 多活架构与容灾设计

## 概述

多活架构是保障系统高可用的重要手段，包括同城双活、异地多活等方案，涉及数据同步、流量调度、故障切换等核心技术。

---

## 一、多活架构分类

### 1.1 架构演进路线

```
单机房 → 同城双活 → 异地多活
  ↓         ↓          ↓
RTO:小时   RTO:分钟    RTO:秒级
RPO:小时   RPO:分钟    RPO:近零
成本:低    成本:中      成本:高
```

### 1.2 三种架构对比

| 架构       | RTO | RPO | 适用场景  | 复杂度 |
|----------|-----|-----|-------|-----|
| **同城双活** | 分钟级 | 分钟级 | 金融、电商 | 中   |
| **异地冷备** | 小时级 | 小时级 | 中小应用  | 低   |
| **异地多活** | 秒级  | 近零  | 大型互联网 | 高   |

## 二、同城双活架构

### 2.1 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                         同城双活架构                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│    ┌─────────────┐              ┌─────────────┐             │
│    │   机房A      │◄────────────►│   机房B      │             │
│    │  (主)        │   专线同步    │  (备)        │             │
│    └──────┬──────┘              └──────┬──────┘             │
│           │                            │                    │
│    ┌──────▼──────┐              ┌──────▼──────┐             │
│    │  负载均衡    │              │  负载均衡    │             │
│    │  (SLB)      │              │  (SLB)      │             │
│    └──────┬──────┘              └──────┬──────┘             │
│           │                            │                    │
│    ┌──────▼──────┐              ┌──────▼──────┐             │
│    │  应用集群    │              │  应用集群    │             │
│    └──────┬──────┘              └──────┬──────┘             │
│           │                            │                    │
│    ┌──────▼──────┐              ┌──────▼──────┐             │
│    │  数据库主   │◄────────────►│  数据库从   │             │
│    │  (Master)  │   同步复制    │  (Slave)   │             │
│    └─────────────┘              └─────────────┘             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 数据同步方案

| 方案                   | 同步方式     | 延迟  | 适用场景  |
|----------------------|----------|-----|-------|
| **MySQL主从复制**        | 异步/半同步   | 毫秒级 | 同城双活  |
| **Oracle DataGuard** | 同步/异步    | 毫秒级 | 企业级应用 |
| ** canal + Kafka**   | 基于Binlog | 秒级  | 异构同步  |

### 2.3 流量调度策略

```
1. DNS轮询
   → 用户1 → 机房A
   → 用户2 → 机房B
   
2. 地理位置调度
   → 北京用户 → 北京机房
   → 上海用户 → 上海机房
   
3. 权重调度
   → 机房A(70%) + 机房B(30%)
```

## 三、异地多活架构

### 3.1 单元化架构

```
┌─────────────────────────────────────────────────────────────┐
│                        异地多活架构                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐      │
│   │   北京单元   │   │   上海单元   │   │   广州单元   │      │
│   │  (华北用户)  │   │  (华东用户)  │   │  (华南用户)  │      │
│   └──────┬──────┘   └──────┬──────┘   └──────┬──────┘      │
│          │                  │                  │            │
│          └──────────────────┼──────────────────┘            │
│                             │                               │
│                      ┌──────▼──────┐                        │
│                      │  全局路由网关 │                        │
│                      │  (GSLB)     │                        │
│                      └─────────────┘                        │
│                                                              │
│   数据分区：                                                 │
│   - 用户ID % 3 = 0 → 北京                                   │
│   - 用户ID % 3 = 1 → 上海                                   │
│   - 用户ID % 3 = 2 → 广州                                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 数据分区策略

| 策略       | 说明          | 示例       |
|----------|-------------|----------|
| **哈希取模** | user_id % N | 用户数据均匀分布 |
| **地理分区** | 按地域划分       | 华北、华东、华南 |
| **时间分区** | 按时间划分       | 历史数据归档   |
| **业务分区** | 按业务划分       | 订单、支付、物流 |

### 3.3 数据同步挑战

```
挑战1：跨单元数据访问
→ 解决方案：全局表（每个单元全量复制）

挑战2：数据一致性
→ 解决方案：最终一致性 + 补偿机制

挑战3：单元故障迁移
→ 解决方案：路由切换 + 数据回放
```

## 四、容灾设计

### 4.1 RTO与RPO

| 指标      | 定义               | 目标值   |
|---------|------------------|-------|
| **RTO** | 恢复时间目标（故障到恢复的时间） | < 5分钟 |
| **RPO** | 恢复点目标（允许丢失的数据量）  | < 1分钟 |

### 4.2 故障切换流程

```
1. 故障检测
   → 健康检查、心跳检测、告警触发
   
2. 故障确认
   → 人工确认或自动确认（避免误切）
   
3. 流量切换
   → DNS切换、路由切换、负载均衡调整
   
4. 数据校验
   → 确认数据一致性
   
5. 业务恢复
   → 逐步放开流量，监控指标
```

### 4.3 容灾演练

| 演练类型     | 频率  | 内容           |
|----------|-----|--------------|
| **切换演练** | 每季度 | 模拟主库故障，切换到备库 |
| **容灾演练** | 每半年 | 完整灾备流程演练     |
| **混沌工程** | 每月  | 随机注入故障       |

## 五、高频面试题

**问题 1：同城双活和异地多活有什么区别？**

**答：**

| 对比项      | 同城双活        | 异地多活         |
|----------|-------------|--------------|
| **距离**   | 同城（<100km）  | 异地（>1000km）  |
| **延迟**   | 网络延迟低（<5ms） | 网络延迟高（>30ms） |
| **数据同步** | 同步复制        | 异步复制         |
| **一致性**  | 强一致性        | 最终一致性        |
| **成本**   | 中           | 高            |
| **适用**   | 金融、电商       | 大型互联网        |

---

**问题 2：多活架构下的数据一致性如何保证？**

**答：**

**同城双活：**

- 使用半同步复制，确保至少一个从库收到数据
- 牺牲一定性能换取一致性

**异地多活：**

- 采用最终一致性模型
- 使用CRDT（无冲突复制数据类型）
- 冲突检测与自动合并
- 业务层补偿机制

---

**问题 3：如何设计一个高可用的容灾系统？**

**答：**

1. **分层容灾**
    - 应用层：多实例部署
    - 数据层：主从复制、备份
    - 网络层：多线路接入

2. **故障检测**
    - 健康检查、心跳检测
    - 多维度监控告警

3. **自动切换**
    - 基于规则的自动切换
    - 人工确认机制（关键系统）

4. **数据保护**
    - 定期备份
    - 异地备份
    - 备份恢复演练

---

**问题 4：单元化架构的优势和挑战是什么？**

**答：**

**优势：**

- 水平扩展能力强
- 故障隔离性好
- 就近访问，延迟低

**挑战：**

- 跨单元数据访问复杂
- 数据一致性保障困难
- 运维复杂度高
