## 一、CAP/BASE 理论深入

### 1.1 CAP 定理

#### CAP 三要素

```
C（Consistency）一致性：
- 所有节点在同一时间具有相同的数据
- 操作后能立即读取到最新数据

A（Availability）可用性：
- 保证每个请求不管成功或者失败都有响应
- 系统一直可用，不会无限期等待

P（Partition tolerance）分区容错性：
- 系统中任意信息的丢失或失败不会影响系统的继续运作
- 网络分区发生时系统仍能运行
```

#### CAP 不可能三角

```
     C
    / \
   /   \
  A-----P
  
  无法同时满足三点，最多只能满足两点
```

#### 实际选择方案

| 方案 | 特点 | 代表系统 | 应用场景 |
|------|------|----------|----------|
| **CP** | 强一致性 + 分区容错<br>（放弃可用性） | Zookeeper<br>HBase<br>Redis | 分布式锁<br>元数据管理<br>金融核心账务 |
| **AP** | 高可用 + 分区容错<br>（放弃强一致性） | Eureka<br>Cassandra<br>DynamoDB | 服务发现<br>日志收集<br>商品浏览 |
| **CA** | 强一致性 + 高可用<br>（放弃分区容错） | 传统单机数据库<br>Oracle/MySQL | 单机房部署<br>不跨网络分区 |

#### CP vs AP 对比示例

**CP 系统 - Zookeeper：**
```
场景：Leader 节点宕机

处理流程：
1. Leader 宕机
2. 集群进入选举阶段（暂停服务 ❌）
3. 选出新 Leader（约 30-120 秒）
4. 恢复服务，数据保持一致 ✅

特点：宁可不可用，也要保证数据一致
```

**AP 系统 - Eureka：**
```
场景：节点 A 宕机

处理流程：
1. 节点 A 宕机
2. 节点 B、C 继续提供服务 ✅
3. 可能存在短暂数据不一致 ⚠️
4. 通过心跳机制最终达到一致 ✅

特点：先保证可用，数据可以最终一致
```

### 1.2 BASE 理论

#### BASE 三要素

```
Basically Available（基本可用）
- 分布式系统出现故障时，允许损失部分可用性
- 如：降级、熔断、限流

Soft state（软状态）
- 允许系统存在中间状态
- 该状态不影响系统整体可用性

Eventually consistent（最终一致性）
- 系统中的所有数据副本经过一段时间后
- 最终能够达到一致的状态
```

#### BASE 实践案例

**1. 电商大促降级（基本可用）**
```java
// 核心业务：下单、支付
if (systemLoad > threshold) {
    // 降级非核心业务
    disableRecommendation();  // 推荐系统
    disableCommentSystem();   // 评论系统
    // 保证核心交易链路可用
}
```

**2. 订单状态流转（软状态）**
```
订单状态：待支付 → 支付中 → 已支付 → 已完成
           ↑
      中间状态（允许短暂存在）
```

**3. 缓存与数据库一致性（最终一致性）**
```java
// 1. 更新数据库
updateDatabase(data);

// 2. 删除缓存（延迟双删）
deleteCache();

// 3. 延迟 500ms 再次删除
Thread.sleep(500);
deleteCache();  // 防止主从同步期间的脏数据

// 最终缓存会重新加载最新数据
```

## 二、Seata 框架实战

### 2.1 Seata 架构概览

```
┌─────────┐      ┌─────────┐      ┌─────────┐
│   TM    │      │   RM    │      │   TC    │
│Transaction│      │Resource │      │Transaction│
│ Manager │      │ Manager │      │Coordinator│
└────┬────┘      └────┬────┘      └────┬────┘
     │                │                │
     │ 1.开启事务     │                │
     │───────────────>│                │
     │                │                │
     │ 2.注册分支     │                │
     │────────────────────────────────>│
     │                │                │
     │ 3.执行 SQL     │                │
     │───────────────>│                │
     │                │                │
     │ 4.提交/回滚    │                │
     │<───────────────────────────────│
```

**角色说明：**
- **TM（Transaction Manager）**：事务管理器，开启全局事务
- **RM（Resource Manager）**：资源管理器，管理本地资源
- **TC（Transaction Coordinator）**：事务协调器，协调全局事务

### 2.2 AT 模式原理

#### AT 模式执行流程

**一阶段（Prepare）：**
```
1. 解析 SQL，找出更新前后的数据
2. 保存 before image 和 after image 到 undo_log
3. 提交本地事务
4. 返回成功给 TC
```

**二阶段（Commit）：**
```
1. TC 收集所有分支成功
2. 发送提交指令给各 RM
3. 异步批量删除 undo_log
4. 完成
```

**二阶段（Rollback）：**
```
1. TC 检测到某个分支失败
2. 发送回滚指令给各 RM
3. 根据 undo_log 生成反向 SQL
4. 恢复数据到 before image 状态
5. 提交本地事务
6. 报告回滚结果
```

#### AT 模式代码示例

```java
@Service
public class OrderService {
    
    @Autowired
    private AccountService accountService;
    
    @Autowired
    private StockService stockService;
    
    /**
     * 创建订单（使用 Seata AT 模式）
     */
    @GlobalTransactional  // 开启全局事务
    public void createOrder(Order order) {
        // 1. 创建订单
        orderMapper.insert(order);
        
        // 2. 扣减库存
        stockService.deduct(order.getProductId(), order.getCount());
        
        // 3. 扣减余额
        accountService.decrease(order.getUserId(), order.getAmount());
        
        // 如果任何一步失败，全部回滚
    }
}
```

#### undo_log 表结构

```sql
CREATE TABLE `undo_log` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `branch_id` BIGINT NOT NULL,
  `xid` VARCHAR(100) NOT NULL,
  `context` VARCHAR(128) NOT NULL,
  `rollback_info` LONGBLOB NOT NULL,  -- before/after image
  `log_status` INT NOT NULL,
  `log_created` DATETIME NOT NULL,
  `log_modified` DATETIME NOT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `ux_id_branch_xid` (`branch_id`, `xid`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**rollback_info JSON 结构：**
```json
{
  "beforeImage": {
    "rows": [{"id": 1, "stock": 100}]
  },
  "afterImage": {
    "rows": [{"id": 1, "stock": 99}]
  }
}
```

### 2.3 TCC 模式实战

#### TCC 模式原理

```
Try：尝试执行
- 完成所有业务检查
- 预留必需业务资源
- 只检查资源，不真正执行

Confirm：确认执行
- 真正执行业务
- 使用 Try 阶段预留的资源
- 满足幂等性

Cancel：取消执行
- 释放 Try 阶段预留的资源
- 回滚业务状态
- 满足幂等性
```

#### TCC 代码示例

```java
@LocalTCC
public interface StockService {
    
    /**
     * Try 阶段 - 冻结库存
     */
    boolean freezeStock(Long productId, Integer count);
    
    /**
     * Confirm 阶段 - 扣减冻结的库存
     */
    boolean deductFrozenStock(Long productId, Integer count);
    
    /**
     * Cancel 阶段 - 解冻库存
     */
    boolean unfreezeStock(Long productId, Integer count);
}
```

```java
@Service
public class StockServiceImpl implements StockService {
    
    @Override
    public boolean freezeStock(Long productId, Integer count) {
        // 1. 检查库存是否充足
        int stock = getStock(productId);
        if (stock < count) {
            return false;
        }
        
        // 2. 冻结库存
        updateFrozenStock(productId, count);
        
        return true;
    }
    
    @Override
    public boolean deductFrozenStock(Long productId, Integer count) {
        // 1. 扣减冻结库存
        decreaseFrozenStock(productId, count);
        
        // 2. 扣减实际库存
        decreaseStock(productId, count);
        
        return true;
    }
    
    @Override
    public boolean unfreezeStock(Long productId, Integer count) {
        // 1. 解冻库存
        increaseFrozenStock(productId, count);
        
        return true;
    }
}
```

## 三、本地消息表方案

### 3.1 方案设计

#### 核心思想

将分布式事务拆分为两个本地事务：
1. 业务执行 + 写入消息表（同一本地事务）
2. 异步监听消息表并执行

#### 表结构设计

```sql
-- 业务表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    amount DECIMAL(10,2),
    status VARCHAR(20)
);

-- 本地消息表
CREATE TABLE local_message (
    id BIGINT PRIMARY KEY,
    business_id BIGINT,          -- 业务 ID
    message_type VARCHAR(50),    -- 消息类型
    content TEXT,                -- 消息内容
    status VARCHAR(20),          -- WAITING/SENT/FAILED
    retry_count INT DEFAULT 0,   -- 重试次数
    next_retry_time DATETIME,    -- 下次重试时间
    created_time DATETIME
);
```

### 3.2 实现代码

```java
@Service
public class LocalMessageService {
    
    @Autowired
    private JdbcTemplate jdbcTemplate;
    
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    /**
     * 事务性发送消息
     */
    @Transactional
    public void sendMessage(Message message) {
        // 1. 执行业务逻辑
        doBusinessLogic();
        
        // 2. 写入消息表（同一事务）
        insertMessage(message);
    }
    
    /**
     * 定时任务扫描消息表
     */
    @Scheduled(fixedRate = 5000)  // 每 5 秒执行一次
    public void scanAndSend() {
        // 1. 查询待发送消息
        List<Message> messages = queryWaitingMessages();
        
        // 2. 发送到 MQ
        for (Message msg : messages) {
            try {
                rocketMQTemplate.send(msg.getTopic(), msg.getContent());
                
                // 3. 更新状态为已发送
                updateMessageStatus(msg.getId(), "SENT");
            } catch (Exception e) {
                // 4. 更新重试次数
                incrementRetryCount(msg.getId());
            }
        }
    }
}
```

## 四、RocketMQ 事务消息

### 4.1 事务消息原理

#### 执行流程

```
1. Producer 发送 half 消息（半消息）
   ↓
2. MQ Server 存储 half 消息（对消费者不可见）
   ↓
3. Producer 执行本地事务
   ↓
4. 根据本地事务结果提交 Commit/Rollback
   ↓
5. MQ Server 根据提交结果决定投递或删除
```

#### 异常处理

```
情况 1：Producer 提交 Commit 失败
→ MQ Server 定期回查 Producer 本地事务状态
→ 根据回查结果提交或回滚

情况 2：Producer 执行本地事务失败
→ Producer 主动回滚
→ MQ Server 删除 half 消息

情况 3：Producer 挂掉
→ MQ Server 定期回查（默认 60 秒）
→ 询问其他 Producer 实例
```

### 4.2 代码实现

```java
@Component
@RocketMQTransactionListener(txProducerGroup = "tx-order-group")
public class TransactionListenerImpl implements TransactionListener {
    
    @Autowired
    private OrderService orderService;
    
    /**
     * 执行本地事务
     */
    @Override
    public LocalTransactionState executeLocalTransaction(
            Message msg, Object arg) {
        try {
            // 1. 执行本地事务
            orderService.createOrder(msg.getBody());
            
            // 2. 提交事务
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            // 3. 回滚事务
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }
    
    /**
     * 事务回查
     */
    @Override
    public LocalTransactionState checkLocalTransaction(
            ExtendedMessageQueueMsgInfo msg) {
        // 1. 根据 msgId 查询本地事务状态
        Order order = orderService.queryOrder(msg.getMsgId());
        
        if (order != null && order.getStatus() == "CREATED") {
            return LocalTransactionState.COMMIT_MESSAGE;
        } else if (order == null) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        } else {
            return LocalTransactionState.UNKNOW;  // 继续等待回查
        }
    }
}
```

## 五、最大努力通知

### 5.1 方案设计

#### 核心思想

- 主动通知：定期调用对方接口
- 失败重试：指数退避策略
- 人工介入：多次失败后转人工

#### 重试策略

```java
@Component
public class MaxEffortNotifyService {
    
    /**
     * 最大努力通知
     */
    @Scheduled(fixedDelay = 60000)  // 每分钟执行一次
    public void notifyPaymentResult() {
        // 1. 查询待通知记录
        List<PaymentNotify> notifies = queryPendingNotifies();
        
        for (PaymentNotify notify : notifies) {
            try {
                // 2. 调用第三方接口
                callThirdPartyApi(notify);
                
                // 3. 标记成功
                markSuccess(notify.getId());
                
            } catch (Exception e) {
                // 4. 更新重试次数
                int retryCount = notify.getRetryCount() + 1;
                
                if (retryCount >= 10) {
                    // 5. 超过最大重试次数，转人工处理
                    markManualProcess(notify.getId());
                } else {
                    // 6. 计算下次重试时间（指数退避）
                    long nextRetryTime = calculateBackoff(retryCount);
                    updateRetryInfo(notify.getId(), retryCount, nextRetryTime);
                }
            }
        }
    }
    
    /**
     * 指数退避算法
     * 第 1 次：1 分钟后
     * 第 2 次：2 分钟后
     * 第 3 次：4 分钟后
     * ...
     * 第 n 次：2^(n-1) 分钟后
     */
    private long calculateBackoff(int retryCount) {
        long delay = (long) Math.pow(2, retryCount - 1) * 60 * 1000;
        return System.currentTimeMillis() + delay;
    }
}
```

## 六、分布式事务监控

### 6.1 监控指标

#### 关键指标

| 指标 | 阈值 | 说明 |
|------|------|------|
| 事务成功率 | > 99.9% | 低于阈值需要告警 |
| 平均耗时 | < 1s | 超过阈值性能优化 |
| 回滚率 | < 1% | 过高说明业务问题 |
| 超时率 | < 0.1% | 网络或服务问题 |
| 死锁检测 | 0 | 发现立即处理 |

### 6.2 监控平台搭建

#### Prometheus + Grafana

```yaml
# Prometheus 配置
scrape_configs:
  - job_name: 'seata'
    static_configs:
      - targets: ['localhost:8091']
    metrics_path: '/metrics'
```

#### 监控 Dashboard

```
1. 全局事务概览
   - 总事务数
   - 成功/失败数
   - 平均耗时
   
2. 分支事务详情
   - 各服务事务数
   - 回滚统计
   - 超时统计
   
3. 实时告警
   - 失败事务告警
   - 超时事务告警

```

## 💡 实战技巧

### 分布式事务选型口诀

```
业务简单用 AT，无侵入性能好
业务复杂用 TCC，灵活控制强
长事务用 Saga，分段补偿好
要求不高本地表，简单可靠稳
MQ 事务高并发，削峰填谷强
最大努力通知，异步解耦好
```

### Seata 调优建议

```
1. 合理设置全局事务超时时间（默认 60s）
2. 避免大事务，拆分为小事务
3. 减少全局锁竞争，热点数据隔离
4. undo_log 定期清理
5. TC 集群部署，提高可用性
```
