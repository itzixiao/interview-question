# Redis 持久化详解

## 一、Redis 持久化概述

Redis 是内存数据库，数据存储在内存中。为了防止数据丢失，Redis 提供了两种持久化机制：

| 持久化方式 | 原理 | 适用场景 |
|------------|------|----------|
| **RDB** | 定时生成内存快照 | 数据备份、灾难恢复 |
| **AOF** | 记录所有写操作命令 | 数据安全性要求高 |
| **混合持久化** | RDB + AOF 结合 | 兼顾性能和安全性（推荐） |

---

## 二、RDB 持久化（Redis Database）

### 2.1 RDB 原理

RDB 持久化是将某一时刻的内存数据以**二进制快照**的形式保存到磁盘上（默认文件名：`dump.rdb`）。

```
┌─────────────────────────────────────────────────────────────┐
│                     RDB 持久化流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Redis 主进程                                              │
│       │                                                     │
│       │  fork()                                             │
│       ▼                                                     │
│   ┌─────────────┐    Copy-On-Write     ┌─────────────────┐ │
│   │  主进程     │ ←─────────────────────│  子进程         │ │
│   │  继续处理   │                       │  生成 RDB 文件  │ │
│   │  请求       │                       │  ↓              │ │
│   └─────────────┘                       │  dump.rdb       │ │
│                                         └─────────────────┘ │
│                                                             │
│   特点：fork 后使用 Copy-On-Write 技术，子进程与父进程      │
│         共享内存页，只有被修改的页才会复制                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 RDB 触发方式

#### 手动触发

| 命令 | 说明 | 是否阻塞 |
|------|------|----------|
| `SAVE` | 同步保存，阻塞主进程 | 是（生产环境禁用） |
| `BGSAVE` | 后台保存，fork 子进程 | 仅 fork 时短暂阻塞 |

#### 自动触发

```bash
# redis.conf 配置
save 900 1      # 900秒内至少1个key变化
save 300 10     # 300秒内至少10个key变化
save 60 10000   # 60秒内至少10000个key变化
```

**其他自动触发场景：**
- 执行 `FLUSHALL` 命令
- 执行 `SHUTDOWN` 命令
- 主从复制时，主节点自动触发

### 2.3 RDB 配置参数

```bash
# RDB 文件名
dbfilename dump.rdb

# RDB 文件保存路径
dir /var/lib/redis

# 后台保存失败时是否停止写入
stop-writes-on-bgsave-error yes

# 是否压缩 RDB 文件（LZF 算法）
rdbcompression yes

# 是否校验 RDB 文件完整性
rdbchecksum yes
```

### 2.4 RDB 优缺点

| 优点 | 缺点 |
|------|------|
| 文件紧凑，适合备份 | 可能丢失最后一次快照后的数据 |
| 恢复速度快（直接加载） | 大数据集时 fork() 可能耗时较长 |
| 对性能影响小（子进程处理） | fork 时内存占用可能翻倍（最坏情况） |

## 三、AOF 持久化（Append Only File）

### 3.1 AOF 原理

AOF 持久化记录所有**写操作命令**（Redis 协议格式），恢复时重新执行这些命令。

```
┌─────────────────────────────────────────────────────────────┐
│                     AOF 持久化流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   客户端写命令 → Redis 执行 → 写入 AOF 缓冲区 → 同步到磁盘 │
│                                                             │
│   ┌──────────────────────────────────────────────────────┐  │
│   │  appendonly.aof 文件内容示例：                       │  │
│   │                                                      │  │
│   │  *3                                                  │  │
│   │  $3                                                  │  │
│   │  SET                                                 │  │
│   │  $4                                                  │  │
│   │  name                                                │  │
│   │  $5                                                  │  │
│   │  redis                                               │  │
│   │                                                      │  │
│   │  表示：SET name redis                                │  │
│   └──────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 AOF 同步策略

| 策略 | 说明 | 数据安全 | 性能 |
|------|------|----------|------|
| `always` | 每次写入都同步到磁盘 | 最高（不丢数据） | 最低 |
| `everysec` | 每秒同步一次（**推荐**） | 高（最多丢1秒） | 较好 |
| `no` | 由操作系统决定何时同步 | 低 | 最高 |

```bash
# AOF 配置
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec
```

### 3.3 AOF 重写机制

随着运行时间增长，AOF 文件会越来越大。AOF 重写可以**压缩**文件体积。

**重写原理：**
- 不是读取旧 AOF 文件，而是根据当前内存数据生成最小命令集
- 重写期间新命令写入重写缓冲区，重写完成后追加

**重写配置：**
```bash
# AOF 文件增长超过 100% 时自动重写
auto-aof-rewrite-percentage 100

# AOF 文件最小达到 64MB 才触发重写
auto-aof-rewrite-min-size 64mb

# 重写期间是否继续同步
no-appendfsync-on-rewrite no
```

**手动触发重写：**
```bash
BGREWRITEAOF
```

### 3.4 AOF 优缺点

| 优点 | 缺点 |
|------|------|
| 数据安全性高，最多丢1秒数据 | 文件体积比 RDB 大 |
| AOF 文件可读，可手动修复 | 恢复速度比 RDB 慢 |
| 写入采用追加方式，不影响历史数据 | 写入性能开销比 RDB 大 |

## 四、混合持久化（Redis 4.0+）

### 4.1 混合持久化原理

混合持久化结合了 RDB 和 AOF 的优点：
- AOF 文件**开头**是 RDB 格式的全量数据
- **后续**是 AOF 格式的增量命令

```
┌─────────────────────────────────────────────────────────────┐
│                   混合持久化 AOF 文件结构                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  RDB 格式（全量数据快照）                           │   │
│   │  - 二进制格式                                       │   │
│   │  - 恢复速度快                                       │   │
│   └─────────────────────────────────────────────────────┘   │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  AOF 格式（增量命令）                               │   │
│   │  - 文本格式                                         │   │
│   │  - 记录快照后的写命令                               │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   优点：                                                    │
│   - 快速恢复（RDB 部分）                                   │
│   - 数据安全（AOF 部分，最多丢1秒）                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 配置方式

```bash
# 开启 AOF
appendonly yes

# 开启混合持久化
aof-use-rdb-preamble yes
```

## 五、持久化策略选择

### 5.1 场景对比

| 场景 | 推荐方案 | 说明 |
|------|----------|------|
| 纯缓存，允许丢数据 | 关闭持久化或仅 RDB | 性能最优 |
| 数据重要，允许少量丢失 | 混合持久化（推荐） | 兼顾性能和安全 |
| 数据极其重要 | AOF (always) | 性能较差但最安全 |
| 只需要定期备份 | RDB | 适合冷备份 |

### 5.2 生产环境推荐配置

```bash
# 混合持久化（推荐）
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

# 同时保留 RDB 作为备份
save 900 1
save 300 10
save 60 10000
```

## 六、主从复制与持久化

### 6.1 主从复制流程

```
┌─────────────────────────────────────────────────────────────┐
│                     主从复制流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   从节点发送 PSYNC 命令                                     │
│          │                                                  │
│          ▼                                                  │
│   ┌──────────────────────────────────┐                     │
│   │  判断是否可以部分复制             │                     │
│   │  (检查 replication ID 和 offset) │                     │
│   └──────────────┬───────────────────┘                     │
│                  │                                          │
│        ┌─────────┴─────────┐                               │
│        ▼                   ▼                               │
│   全量复制            部分复制                              │
│   (首次同步)          (断线重连)                            │
│        │                   │                               │
│        ▼                   ▼                               │
│   主节点 BGSAVE        发送复制缓冲区                       │
│   生成 RDB             中的增量数据                         │
│        │                                                    │
│        ▼                                                    │
│   发送 RDB 给从节点                                         │
│        │                                                    │
│        ▼                                                    │
│   发送复制缓冲区的新命令                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 主从复制配置

```bash
# 从节点配置
replicaof 192.168.1.100 6379   # 主节点地址
masterauth password             # 主节点密码

# 从节点是否响应读请求（同步期间）
replica-serve-stale-data yes

# 从节点是否只读
replica-read-only yes

# 复制缓冲区大小（用于部分复制）
repl-backlog-size 1mb

# 无磁盘复制（直接通过网络发送 RDB）
repl-diskless-sync no
```

## 七、高频面试题

**问题 1:RDB 和 AOF 的区别是什么？如何选择？**

**参考答案：**

| 特性 | RDB | AOF |
|------|-----|-----|
| 持久化方式 | 定时快照（二进制） | 记录写命令（文本） |
| 数据安全性 | 可能丢失最后一次快照后数据 | 最多丢1秒数据（everysec） |
| 文件大小 | 紧凑 | 较大（重写后可压缩） |
| 恢复速度 | 快 | 慢（需重放命令） |
| 对性能影响 | fork 时有影响 | 写入时有影响 |

**选择建议：**
- 纯缓存场景：关闭持久化或仅 RDB
- 数据重要：使用混合持久化（推荐）
- 数据极其重要：AOF (always)

---

**问题 2:RDB 的 fork 操作会阻塞主进程吗？Copy-On-Write 是什么？**

**参考答案：**

**fork 是否阻塞：**
- fork() 本身会短暂阻塞主进程
- 阻塞时间与内存中的数据量有关
- 通常几毫秒到几十毫秒

**Copy-On-Write（写时复制）：**
- fork 后，父子进程共享相同的物理内存页
- 只有当某一方修改数据时，才会复制该内存页
- 优点：减少内存占用，加快 fork 速度
- 最坏情况：如果所有页都被修改，内存占用翻倍

---

**问题 3:AOF 重写的原理是什么？会阻塞 Redis 吗？**

**参考答案：**

**重写原理：**
1. fork 子进程执行重写
2. 子进程根据**当前内存数据**生成新 AOF 文件（不读旧文件）
3. 重写期间，主进程将新命令同时写入：
   - AOF 缓冲区（写入旧 AOF 文件）
   - AOF 重写缓冲区
4. 子进程完成后，主进程将重写缓冲区内容追加到新 AOF 文件
5. 原子替换旧 AOF 文件

**是否阻塞：**
- fork 时短暂阻塞
- 最后追加重写缓冲区时短暂阻塞
- 其他时间不阻塞

---

**问题 4:Redis 宕机后如何恢复数据？恢复的优先级是什么？**

**参考答案：**

**恢复优先级：**
1. 如果开启了 AOF，优先加载 AOF 文件
2. 如果没有 AOF 或 AOF 损坏，加载 RDB 文件

**恢复流程：**
```
Redis 启动
    │
    ▼
检查是否开启 AOF？
    │
  ┌─┴─┐
  是   否
  │    │
  ▼    ▼
加载 AOF  加载 RDB
```

**AOF 文件损坏时：**
```bash
# 使用 redis-check-aof 工具修复
redis-check-aof --fix appendonly.aof
```

---

**问题 5：混合持久化的原理是什么？为什么推荐使用？**

**参考答案：**

**原理：**
- AOF 重写时，将当前数据以 RDB 格式写入 AOF 文件开头
- 后续增量命令以 AOF 格式追加

**优点：**
1. **快速恢复**：RDB 部分直接加载，比纯 AOF 快很多
2. **数据安全**：AOF 部分保证最多丢1秒数据
3. **文件较小**：比纯 AOF 文件小

**配置：**
```bash
appendonly yes
aof-use-rdb-preamble yes
```

---

**问题 6:Redis持久化会影响性能吗？如何优化？**

**参考答案：**

**性能影响点：**
1. **RDB fork**：大数据集时 fork 耗时可能较长
2. **AOF fsync**：always 模式每次写入都同步，性能最差
3. **AOF 重写**：fork 和追加缓冲区时有短暂阻塞

**优化建议：**
1. 使用 `appendfsync everysec`（平衡性能和安全）
2. 在从节点执行持久化，主节点关闭
3. 避免大 key，减少 fork 时间
4. 配置合理的重写触发条件
5. 使用 SSD 存储持久化文件

---

**问题 7：什么是 Redis 的无磁盘复制？有什么优缺点？**

**参考答案：**

**无磁盘复制（Diskless Replication）：**
- 主节点不先保存 RDB 到磁盘，而是直接通过网络发送给从节点
- 适用于磁盘 IO 较慢的场景

**配置：**
```bash
repl-diskless-sync yes
repl-diskless-sync-delay 5  # 等待5秒让更多从节点连接
```

**优点：**
- 减少磁盘 IO
- 网络带宽充足时同步更快

**缺点：**
- 如果网络中断，需要重新传输全部数据
- 不会在主节点保留 RDB 备份

---

**问题 8：如何保证 Redis 主从数据一致性？**

**参考答案：**

**Redis 主从是最终一致性**，不保证强一致。

**一致性保障措施：**

1. **复制缓冲区**：记录主节点的增量命令
   ```bash
   repl-backlog-size 1mb
   ```

2. **复制偏移量**：主从各自维护 offset，用于部分复制

3. **心跳检测**：从节点定期汇报 offset

4. **min-replicas 配置**：
   ```bash
   min-replicas-to-write 1    # 至少1个从节点
   min-replicas-max-lag 10    # 延迟不超过10秒
   ```

**注意：** 如果需要强一致性，应使用分布式事务或其他方案。

---

**问题 9:Redis Sentinel 如何实现高可用？故障转移流程是怎样的？**

**参考答案：**

**Sentinel 功能：**
1. 监控主从节点健康状态
2. 自动故障转移
3. 通知客户端新的主节点地址

**故障转移流程：**
1. **主观下线**：单个 Sentinel 认为主节点故障
2. **客观下线**：多数 Sentinel（≥quorum）确认主节点故障
3. **选举 Leader Sentinel**：使用 Raft 算法
4. **选择新主节点**：根据优先级、复制偏移量、runid
5. **执行故障转移**：`SLAVEOF NO ONE`
6. **更新配置**：通知其他从节点和客户端

**配置示例：**
```bash
sentinel monitor mymaster 192.168.1.100 6379 2
sentinel down-after-milliseconds mymaster 5000
```

---

**问题 10:RDB 和 AOF 可以同时开启吗？Redis 如何处理？**

**参考答案：**

**可以同时开启！**

**Redis 处理方式：**
1. 启动时优先加载 AOF（数据更完整）
2. RDB 作为备份手段

**推荐配置（混合持久化）：**
```bash
# 开启 AOF
appendonly yes
appendfsync everysec

# 开启混合持久化
aof-use-rdb-preamble yes

# 同时保留 RDB 备份
save 900 1
save 300 10
save 60 10000
```

**注意：** 如果 AOF 文件损坏，可以删除 AOF 文件，从 RDB 恢复（会丢失部分数据）。
