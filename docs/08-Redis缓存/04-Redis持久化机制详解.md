# Redis持久化机制详解

## 一、Redis持久化概述

Redis 是内存数据库，数据存储在内存中。为了防止数据丢失，Redis 提供了多种持久化机制。

### 1.1 持久化方式对比

| 持久化方式     | 原理           | 适用场景         |
|-----------|--------------|--------------|
| **RDB**   | 定时生成内存快照     | 数据备份、灾难恢复    |
| **AOF**   | 记录所有写操作命令    | 数据安全性要求高     |
| **混合持久化** | RDB + AOF 结合 | 兼顾性能和安全性（推荐） |

---

## 二、RDB 持久化（Redis Database）

### 2.1 RDB 原理

RDB 持久化是将某一时刻的内存数据以**二进制快照**的形式保存到磁盘上（默认文件名：`dump.rdb`）。

```
┌─────────────────────────────────────────────────────────────┐
│                     RDB 持久化流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Redis 主进程                                              │
│       │                                                     │
│       │  fork()                                             │
│       ▼                                                     │
│   ┌─────────────┐    Copy-On-Write     ┌─────────────────┐ │
│   │  主进程     │ ←─────────────────────│  子进程         │ │
│   │  继续处理   │                       │  生成 RDB 文件  │ │
│   │  请求       │                       │  ↓              │ │
│   └─────────────┘                       │  dump.rdb       │ │
│                                         └─────────────────┘ │
│                                                             │
│   **Copy-On-Write 关键细节**：                              │
│   - fork 后，父子进程共享物理内存页，内核通过页表映射实现    │
│   - 主进程执行写操作时，会复制被修改的内存页（写时复制）     │
│   - 子进程仅读取内存生成 RDB，不执行写操作，因此不会触发复制 │
│   - 注意：fork 时会复制父进程的页表，数据量越大，页表复制    │
│     耗时越长（这是 fork 阻塞的主要原因）                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

### 2.2 RDB 触发方式

#### 手动触发

| 命令       | 说明            | 是否阻塞         |
|----------|---------------|--------------|
| `SAVE`   | 同步保存，阻塞主进程    | 是（生产环境禁用）    |
| `BGSAVE` | 后台保存，fork 子进程 | 仅 fork 时短暂阻塞 |

#### 自动触发

```bash
# redis.conf 配置
save 900 1      # 900 秒内至少 1 个 key 变化
save 300 10     # 300 秒内至少 10 个 key 变化
save 60 10000   # 60 秒内至少 10000 个 key 变化
```

**其他自动触发场景：**

- 执行 `SHUTDOWN` 命令（非 `SHUTDOWN NOSAVE`）时，Redis 会先执行 `BGSAVE` 生成 RDB，再关闭
- 主从复制时，主节点接收到从节点的 `SYNC` 命令后，会触发 `BGSAVE` 生成 RDB 并发送给从节点
- 执行 `FLUSHALL`/`FLUSHDB` 时，若配置了 `save` 规则，会触发 RDB 生成（但生成的 RDB 文件为空，生产环境需注意）

---

### 2.3 RDB 配置参数

```bash
# RDB 文件名
dbfilename dump.rdb

# RDB 文件保存路径
dir /var/lib/redis

# 后台保存失败时是否停止写入
stop-writes-on-bgsave-error yes

# 是否压缩 RDB 文件（LZF 算法）
rdbcompression yes

# 是否校验 RDB 文件完整性
rdbchecksum yes
```

---

### 2.4 RDB 优缺点

| 优点             | 缺点                                                         |
|----------------|------------------------------------------------------------|
| 文件紧凑，适合备份      | 可能丢失最后一次快照后的数据                                             |
| 恢复速度快（直接加载二进制） | 大数据集时 fork() 可能耗时较长（阻塞主进程）                                 |
| 对性能影响小（子进程处理）  | fork 后虚拟内存占用翻倍（物理内存通过 Copy-On-Write 共享，仅修改页会复制，最坏情况物理内存翻倍） |

## 三、AOF 持久化（Append Only File）

### 3.1 AOF 原理

AOF 持久化记录所有**写操作命令**（Redis 协议格式），恢复时重新执行这些命令。

```
┌─────────────────────────────────────────────────────────────┐
│                     AOF 持久化流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   客户端写命令 → Redis 执行 → 写入 AOF 缓冲区 → 同步到磁盘 │
│                                                             │
│   ┌──────────────────────────────────────────────────────┐  │
│   │  appendonly.aof 文件内容示例：                       │  │
│   │                                                      │  │
│   │  *3                                                  │  │
│   │  $3                                                  │  │
│   │  SET                                                 │  │
│   │  $4                                                  │  │
│   │  name                                                │  │
│   │  $5                                                  │  │
│   │  redis                                               │  │
│   │                                                      │  │
│   │  表示：SET name redis                                │  │
│   └──────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

### 3.2 AOF 同步策略

| 策略         | 说明                   | 数据安全       | 性能（TPS）       | 适用场景          |
|------------|----------------------|------------|---------------|---------------|
| `always`   | 每次写入都调用 fsync 同步到磁盘  | 最高（不丢数据）   | 最低（约 1w TPS）  | 金融、支付等极致安全场景  |
| `everysec` | 每秒调用一次 fsync（**推荐**） | 高（最多丢 1 秒） | 较好（约 10w TPS） | 绝大多数业务场景      |
| `no`       | 由操作系统决定何时同步（通常 30s）  | 低（可能丢大量数据） | 最高（接近纯内存）     | 纯缓存、允许大量丢数据场景 |

```bash
# AOF 配置
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec
```

---

### 3.3 AOF 重写机制

随着运行时间增长，AOF 文件会越来越大。AOF 重写可以**压缩**文件体积。

**重写原理**：

- 不是读取旧 AOF 文件，而是根据当前内存数据生成最小命令集
- 重写期间新命令写入重写缓冲区，重写完成后追加

**重写配置**：

```bash
# AOF 文件增长超过 100% 时自动重写（相对于上次重写后的大小）
auto-aof-rewrite-percentage 100

# AOF 文件最小达到 64MB 才触发重写（避免小文件频繁重写）
auto-aof-rewrite-min-size 64mb

# 重写期间是否暂停 AOF 同步（默认 no）
# yes：重写期间暂停 fsync，提升重写速度，但可能丢失这期间的数据
# no：重写期间继续 fsync，数据安全但重写速度稍慢（推荐）
no-appendfsync-on-rewrite no
```

**手动触发重写**：

```bash
BGREWRITEAOF
```

---

### 3.4 AOF 优缺点

| 优点               | 缺点            |
|------------------|---------------|
| 数据安全性高，最多丢 1 秒数据 | 文件体积比 RDB 大   |
| AOF 文件可读，可手动修复   | 恢复速度比 RDB 慢   |
| 写入采用追加方式，不影响历史数据 | 写入性能开销比 RDB 大 |

## 四、混合持久化（Redis 4.0+）

### 4.1 混合持久化原理

**混合持久化原理**：
Redis 4.0+ 引入，仅在 **AOF 重写** 时触发：

- 重写生成的新 AOF 文件**开头**是 RDB 格式的全量数据（二进制，恢复快）
- **后续**是 AOF 格式的增量命令（文本，记录重写后的写操作）
- 日常写入的 AOF 仍是纯命令格式，仅重写后的 AOF 是混合格式

```
┌─────────────────────────────────────────────────────────────┐
│                   混合持久化 AOF 文件结构                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  RDB 格式（全量数据快照）                           │   │
│   │  - 二进制格式                                       │   │
│   │  - 恢复速度快                                       │   │
│   └─────────────────────────────────────────────────────┘   │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  AOF 格式（增量命令）                               │   │
│   │  - 文本格式                                         │   │
│   │  - 记录快照后的写命令                               │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   优点：                                                    │
│   - 快速恢复（RDB 部分）                                   │
│   - 数据安全（AOF 部分，最多丢 1 秒）                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

### 4.2 配置方式

```bash
# 开启 AOF
appendonly yes

# 开启混合持久化（Redis 4.0+ 支持，默认 yes；Redis 6.0+ 默认开启）
aof-use-rdb-preamble yes

# 验证是否开启成功
127.0.0.1:6379> CONFIG GET aof-use-rdb-preamble
1) "aof-use-rdb-preamble"
2) "yes"
```

## 五、持久化策略选择

### 5.1 场景对比

| 场景          | 推荐方案         | 说明       |
|-------------|--------------|----------|
| 纯缓存，允许丢数据   | 关闭持久化或仅 RDB  | 性能最优     |
| 数据重要，允许少量丢失 | 混合持久化（推荐）    | 兼顾性能和安全  |
| 数据极其重要      | AOF (always) | 性能较差但最安全 |
| 只需要定期备份     | RDB          | 适合冷备份    |

---

### 5.2 生产环境推荐配置

```bash
# 混合持久化（推荐）
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

# 同时保留 RDB 作为备份
save 900 1
save 300 10
save 60 10000
```

## 六、高频面试题

### 问题 1:RDB 和 AOF 的区别是什么？如何选择？

**参考答案**：

| 特性    | RDB           | AOF                 |
|-------|---------------|---------------------|
| 持久化方式 | 定时快照（二进制）     | 记录写命令（文本）           |
| 数据安全性 | 可能丢失最后一次快照后数据 | 最多丢 1 秒数据（everysec） |
| 文件大小  | 紧凑            | 较大（重写后可压缩）          |
| 恢复速度  | 快             | 慢（需重放命令）            |
| 对性能影响 | fork 时有影响     | 写入时有影响              |

**选择建议**：

- 纯缓存场景：关闭持久化或仅 RDB
- 数据重要：使用混合持久化（推荐）
- 数据极其重要：AOF (always)

---

### 问题 2:RDB 的 fork 操作会阻塞主进程吗？Copy-On-Write 是什么？

**参考答案**：

**fork 是否阻塞**：

- fork() 本身会短暂阻塞主进程
- 阻塞时间与内存中的数据量有关
- 通常几毫秒到几十毫秒

**Copy-On-Write（写时复制）**：

- fork 后，父子进程共享相同的物理内存页
- 只有当某一方修改数据时，才会复制该内存页
- 优点：减少内存占用，加快 fork 速度
- 最坏情况：如果所有页都被修改，内存占用翻倍

---

### 问题 3:AOF 重写的原理是什么？会阻塞 Redis 吗？

**参考答案**：

**重写原理**：

1. fork 子进程执行重写
2. 子进程根据**当前内存数据**生成新 AOF 文件（不读旧文件）
3. 重写期间，主进程将新命令同时写入：
    - AOF 缓冲区（写入旧 AOF 文件）
    - AOF 重写缓冲区
4. 子进程完成后，主进程将重写缓冲区内容追加到新 AOF 文件
5. 原子替换旧 AOF 文件

**是否阻塞**：

- fork 时短暂阻塞
- 最后追加重写缓冲区时短暂阻塞
- 其他时间不阻塞

---

### 问题 4:Redis 宕机后如何恢复数据？恢复的优先级是什么？

**参考答案**：

**恢复优先级**：

1. 如果开启了 AOF **且 AOF 文件有效**，优先加载 AOF 文件（数据更完整）
2. 如果 AOF 未开启、AOF 文件不存在或损坏且无法修复，加载 RDB 文件
3. 如果两者都不存在，启动后 Redis 为空

**恢复流程**：

```
Redis 启动
    │
    ▼
检查是否开启 AOF？
    │
  ┌─┴─┐
  是   否
  │    │
  ▼    ▼
加载 AOF  加载 RDB
```

**AOF 文件损坏时**：

```bash
# 使用 redis-check-aof 工具修复
redis-check-aof --fix appendonly.aof

# 模拟 AOF 损坏后的恢复流程
# 1. 停止 Redis
redis-cli shutdown

# 2. 修复 AOF 文件
redis-check-aof --fix /data/redis/appendonly.aof

# 3. 若修复失败，删除 AOF 文件，从 RDB 恢复
rm /data/redis/appendonly.aof

# 4. 启动 Redis
redis-server /etc/redis/redis.conf
```

---

### 问题 5：什么是混合持久化？为什么推荐使用？

**参考答案**：

**原理**：

- AOF 重写时，将当前数据以 RDB 格式写入 AOF 文件开头
- 后续增量命令以 AOF 格式追加

**优点**：

1. **快速恢复**：RDB 部分直接加载，比纯 AOF 快很多
2. **数据安全**：AOF 部分保证最多丢 1 秒数据
3. **文件较小**：比纯 AOF 文件小

**配置**：

```bash
appendonly yes
aof-use-rdb-preamble yes
```

---

### 问题 6:Redis持久化会影响性能吗？如何优化？

**参考答案**：

**性能影响点**：

1. **RDB fork**：大数据集时 fork 耗时可能较长（页表复制开销）
2. **AOF fsync**：always 模式每次写入都同步，性能最差
3. **AOF 重写**：fork 和追加缓冲区时有短暂阻塞

**优化建议**：

1. **AOF 同步策略**：优先使用 `appendfsync everysec`（平衡性能和安全）
2. **从节点持久化**：主节点关闭 RDB/AOF，仅在从节点执行（主节点专注处理请求）
3. **减少 fork 耗时**：
    - 避免大 key（如 100MB 以上的 String、百万级 List），拆分小 key
    - 控制 Redis 内存大小（建议单实例不超过 8GB），减少页表数量
4. **重写优化**：
    - 调大 `auto-aof-rewrite-min-size`（如 128MB），减少重写频率
    - 低峰期手动触发 `BGSAVE`/`BGREWRITEAOF`
5. **存储优化**：使用 SSD 存储持久化文件（提升 fsync 速度）
6. **内核优化**：设置 `vm.overcommit_memory = 1`（避免 fork 因内存不足失败）

---

### 问题 7:RDB 和 AOF 可以同时开启吗？Redis 如何处理？

**参考答案**：

**可以同时开启！**

**Redis 处理方式**：

1. 启动时优先加载 AOF（数据更完整）
2. RDB 作为备份手段

**推荐配置（混合持久化）**：

```bash
# 开启 AOF
appendonly yes
appendfsync everysec

# 开启混合持久化
aof-use-rdb-preamble yes

# 同时保留 RDB 备份
save 900 1
save 300 10
save 60 10000
```

**注意**：如果 AOF 文件损坏，可以删除 AOF 文件，从 RDB 恢复（会丢失部分数据）。

## 七、最佳实践

### 7.1 生产环境配置建议

```bash
# 基础配置
bind 0.0.0.0
port 6379
daemonize yes
pidfile /var/run/redis_6379.pid
logfile /var/log/redis/redis.log

# 内存管理（根据业务场景选择）
maxmemory 4gb
# 推荐配置：
# - 纯缓存场景（如电商商品缓存）：allkeys-lru（淘汰最少使用的 key）
# - 混合场景（缓存 + 持久化）：volatile-lru（仅淘汰带过期时间的 key）
# - 禁止淘汰（数据重要）：noeviction（内存满时拒绝写操作）
maxmemory-policy volatile-lru

# 混合持久化（推荐，Redis 4.0+）
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
no-appendfsync-on-rewrite no

# RDB 备份（压缩字符串类型，列表/哈希等结构不压缩）
save 900 1
save 300 10
save 60 10000
dbfilename dump.rdb
dir /data/redis
rdbcompression yes  # 压缩仅针对字符串类型，压缩耗时少，建议开启
rdbchecksum yes     # 校验 RDB 文件完整性

# 网络优化
tcp-keepalive 300
timeout 0
```

**补充：持久化文件备份策略**

```bash
# 1. 定时备份 RDB 文件（示例：每天凌晨 3 点）
0 3 * * * cp /data/redis/dump.rdb /data/redis/backup/dump_$(date +%Y%m%d).rdb

# 2. 保留 7 天备份，清理旧文件
0 4 * * * find /data/redis/backup -name "dump_*.rdb" -mtime +7 -delete

# 3. AOF 文件定期校验
0 5 * * * redis-check-aof --check-only /data/redis/appendonly.aof
```

### 7.2 监控指标

```bash
# 查看持久化状态
INFO persistence

# 关键指标
rdb_changes_since_last_save        # 距离上次 RDB 保存的变更次数
rdb_last_bgsave_time_sec           # 上次 BGSAVE 耗时（秒），-1 表示失败，0 表示未执行

aof_last_rewrite_time_sec          # 上次 AOF 重写耗时（秒）
aof_last_bgrewriteaof_status       # 上次 AOF 重写状态（ok/err）
aof_current_size                   # 当前 AOF 文件大小（字节）
```

## 八、总结

| 知识点   | 核心内容                                                                 |
|-------|----------------------------------------------------------------------|
| RDB   | 定时快照，文件紧凑，恢复快，可能丢数据；fork 时虚拟内存翻倍，物理内存通过 COW 共享                       |
| AOF   | 记录写命令，数据安全，文件大，恢复慢；everysec 推荐，最多丢 1 秒                               |
| 混合持久化 | Redis 4.0+，仅在 AOF 重写时触发；RDB+ AOF 结合，兼顾性能和安全（推荐）                      |
| 性能优化  | everysec 同步、从节点持久化、SSD 存储、避免大 key、内核参数优化                             |
| 恢复优先级 | AOF（有效）优先于 RDB；损坏时可修复或删除 AOF 从 RDB 恢复                                |
| 监控指标  | rdb_last_bgsave_status、aof_last_bgrewriteaof_status、aof_current_size |
