## 一、JVM 调优实战

### 1.1 GC 日志分析

#### JVM 参数配置

```bash
# 生产环境推荐配置（G1 垃圾回收器）
java -Xms4g -Xmx4g \
     -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \
     -XX:+ParallelRefProcEnabled \
     -XX:+UseStringDeduplication \
     -Xloggc:/var/log/gc.log \
     -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -XX:+UseGCLogFileRotation \
     -XX:NumberOfGCLogFiles=5 \
     -XX:GCLogFileSize=10M \
     -jar application.jar
```

#### GC 日志分析工具

1. **GCViewer** - 可视化 GC 日志
2. **GCEasy.io** - 在线分析工具
3. **IBM GC Analyzer** - 专业分析工具

#### GC 日志示例分析

```
2026-03-15T10:30:15.123+0800: [GC (Allocation Failure) 
2026-03-15T10:30:15.123+0800: [ParNew: 512000K->64000K(589824K), 0.0523456 secs]
5760000K->5312000K(6291456K), 0.0524123 secs]
[Times: user=0.21 sys=0.03, real=0.05 secs]
```

**关键指标：**
- **GC 原因**：Allocation Failure（年轻代空间不足）
- **GC 时间**：0.052 秒（52ms，正常）
- **回收效果**：从 512MB 回收到 64MB（效果良好）

### 1.2 MAT/JProfiler 工具使用

#### MAT（Memory Analyzer Tool）

**使用步骤：**

1. **生成 Heap Dump**
```bash
# JVM 参数配置自动 dump
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/heapdump.hprof

# 手动生成 dump
jmap -dump:format=b,file=heap.hprof <pid>
```

2. **导入 MAT 分析**
   - File -> Heap Dump -> Open
   - 查看 Histogram（内存直方图）
   - 查看 Dominator Tree（支配树）

3. **常见问题定位**
   - **内存泄漏**：查找 GC Roots 引用链
   - **大对象**：按大小排序找出 Top 10
   - **重复对象**：使用 Duplicate Classes 查询

#### JProfiler 使用

**功能：**
- CPU  profiling（方法调用耗时）
- Memory profiling（内存分配）
- Thread profiling（线程状态）

**典型场景：**
```
1. 接口响应慢 -> CPU Profiling -> 找出热点方法
2. 内存占用高 -> Memory Profiling -> 分析对象分配
3. 死锁问题 -> Thread Profiling -> 查看线程状态
```

### 1.3 内存泄漏排查

**常见内存泄漏场景：**

1. **静态集合类**
```java
// ❌ 错误示例 - 静态 List 无限增长
private static final List<byte[]> MEMORY_LEAK = new ArrayList<>();

public void addData() {
    MEMORY_LEAK.add(new byte[1024 * 1024]); // 每次添加 1MB
}
```

2. **未关闭的资源**
```java
// ❌ 错误示例 - 数据库连接未关闭
Connection conn = dataSource.getConnection();
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery("SELECT * FROM users");
// 忘记关闭资源
```

3. **监听器未注销**
```java
// ❌ 错误示例 - 注册监听器后未注销
applicationContext.addListener(eventListener);
// 应用生命周期内未移除
```

**排查步骤：**
1. 监控内存使用率（持续上升）
2. 生成 Heap Dump
3. MAT 分析 GC Roots
4. 定位泄漏代码位置
5. 修复并验证

## 二、MySQL 调优

### 2.1 Explain 执行计划详解

#### 执行计划关键字段

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND status = 'PENDING';
```

**结果说明：**

| 字段 | 含义 | 优化目标 |
|------|------|----------|
| id | 查询序号 | - |
| select_type | 查询类型 | - |
| table | 表名 | - |
| **type** | 连接类型 | 至少达到 range 级别 |
| **key** | 使用的索引 | 不为 NULL |
| **rows** | 扫描行数 | 越少越好 |
| **Extra** | 额外信息 | 避免 Using filesort/temporary |

#### type 连接类型（从优到差）

```
system > const > eq_ref > ref > range > index > ALL

- system：表中只有一行数据
- const：主键或唯一索引查询
- eq_ref：主键或唯一索引关联查询
- ref：非唯一索引查询
- range：索引范围扫描
- index：全索引扫描
- ALL：全表扫描（最差）
```

#### Extra 关键信息

```
✅ Using index        - 覆盖索引（性能好）
✅ Using where        - 使用了 WHERE 过滤
❌ Using filesort    - 需要文件排序（需优化）
❌ Using temporary   - 使用了临时表（需优化）
```

### 2.2 慢查询优化

#### 开启慢查询日志

```ini
# my.cnf 配置
[mysqld]
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 2  # 超过 2 秒的 SQL 记录
log_queries_not_using_indexes = 1  # 记录未使用索引的 SQL
```

#### 慢查询分析工具

```bash
# 查看慢查询数量
mysqldumpslow -s c /var/log/mysql/slow.log

# 查看返回记录最多的 SQL
mysqldumpslow -s r /var/log/mysql/slow.log

# 查看前 10 个最慢的 SQL
mysqldumpslow -t 10 -s t /var/log/mysql/slow.log
```

#### 优化案例

**原始 SQL（慢）：**
```sql
SELECT * FROM orders 
WHERE YEAR(create_time) = 2026 
  AND MONTH(create_time) = 3;
-- 执行时间：5.2 秒
-- 问题：对字段进行函数操作，导致索引失效
```

**优化后（快）：**
```sql
SELECT * FROM orders 
WHERE create_time >= '2026-03-01' 
  AND create_time < '2026-04-01';
-- 执行时间：0.05 秒
-- 优化：建立 create_time 索引，范围查询
```

### 2.3 锁等待分析

#### 查询当前事务

```sql
-- 查看活跃事务
SELECT * FROM information_schema.INNODB_TRX;

-- 查看正在执行的锁
SELECT * FROM performance_schema.data_locks;

-- 查看锁等待情况
SELECT * FROM performance_schema.data_lock_waits;
```

#### 锁等待处理

```sql
-- 查找阻塞其他事务的事务
SELECT 
    blocking_trx.trx_id AS blocking_trx_id,
    blocked_trx.trx_id AS blocked_trx_id,
    blocked_trx.trx_query AS blocked_query
FROM 
    information_schema.INNODB_TRX blocking_trx,
    information_schema.INNODB_TRX blocked_trx,
    information_schema.INNODB_LOCK_WAITS w
WHERE 
    blocking_trx.trx_id = w.blocking_engine_transaction_id
    AND blocked_trx.trx_id = w.requesting_engine_transaction_id;

-- 杀掉阻塞事务
KILL <trx_id>;
```

## 三、Redis 性能优化

### 3.1 大 Key 处理

#### 大 Key 定义与危害

**定义：** Value 大小超过 10KB 的 Key

**危害：**
- 网络阻塞（单次读取占用大量带宽）
- 内存分布不均（集群中某些节点内存爆满）
- 迁移慢（集群扩容时迁移时间长）

#### 大 Key 检测方法

```bash
# 方法 1：redis-cli --bigkeys
redis-cli -h localhost -p 6379 --bigkeys

# 方法 2：MEMORY USAGE 命令
redis-cli MEMORY USAGE user:profile:123

# 方法 3：定期扫描（代码实现）
Set<String> keys = redisTemplate.keys("*");
for (String key : keys) {
    Long size = redisTemplate.opsForValue().size(key);
    if (size > 10240) { // 大于 10KB
        log.warn("发现大 Key: {}", key);
    }
}
```

#### 大 Key 优化方案

**方案 1：拆分法**
```java
// ❌ 错误示例 - 单个大 Key
String bigKey = "user:profile:123";
UserProfile profile = getProfile(); // 包含 100 个字段
redisTemplate.opsForValue().set(bigKey, profile);

// ✅ 正确示例 - 拆分为多个小 Key
redisTemplate.opsForValue().set("user:profile:123:basic", basicInfo);
redisTemplate.opsForValue().set("user:profile:123:contact", contactInfo);
redisTemplate.opsForValue().set("user:profile:123:preferences", preferences);
```

**方案 2：Hash 结构**
```java
// 使用 Hash 结构存储对象
Map<String, Object> hash = new HashMap<>();
hash.put("name", "John");
hash.put("age", 25);
hash.put("email", "john@example.com");

redisTemplate.opsForHash().putAll("user:profile:123", hash);

// 获取单个字段（不需要获取整个对象）
String name = (String) redisTemplate.opsForHash().get("user:profile:123", "name");
```

### 3.2 热 Key 处理

#### 热 Key 定义与危害

**定义：** 访问频率 > 1000 次/秒 的 Key

**危害：**
- 单节点负载过高（所有请求打到同一台 Redis）
- 网络拥塞（单机带宽被打满）
- 影响其他 Key 的访问

#### 热 Key 检测方法

```bash
# 方法 1：Redis 4.0+ hotkeys 功能
redis-cli -h localhost -p 6379 --hotkeys

# 方法 2：监控工具
# - Redis Monitor
# - Prometheus + Grafana
# - 阿里云 Redis 监控

# 方法 3：日志分析
# 分析 access.log 统计访问频率
```

#### 热 Key 优化方案

**方案 1：本地缓存**
```java
// 使用 Caffeine 本地缓存
Cache<String, Object> localCache = Caffeine.newBuilder()
    .maximumSize(1000)
    .expireAfterWrite(5, TimeUnit.SECONDS)
    .build();

public Object getData(String key) {
    // 先查本地缓存
    Object value = localCache.getIfPresent(key);
    if (value != null) {
        return value;
    }
    
    // 再查 Redis
    value = redisTemplate.opsForValue().get(key);
    if (value != null) {
        localCache.put(key, value);
    }
    
    return value;
}
```

**方案 2：多副本**
```java
// 为热 Key 创建多个副本
String hotKey = "product:1001";
Object value = getProduct(hotKey);

// 写入多个副本
for (int i = 0; i < 3; i++) {
    redisTemplate.opsForValue().set(hotKey + ":copy:" + i, value);
}

// 随机读取副本
Random random = new Random();
int copyIndex = random.nextInt(3);
return redisTemplate.opsForValue().get(hotKey + ":copy:" + copyIndex);
```

## 四、Tomcat 调优

### 4.1 线程池配置

#### 核心参数

```yaml
server:
  port: 8080
  
tomcat:
  threads:
    max: 200          # 最大工作线程数
    min-spare: 10     # 最小空闲线程数
  max-connections: 8192    # 最大连接数
  accept-count: 100        # 接受队列长度
  connection-timeout: 20000 # 连接超时时间（毫秒）
```

#### 参数调优原则

**max（最大线程数）：**
- CPU 密集型应用：CPU 核数 + 1
- IO 密集型应用：CPU 核数 * 2 或更高
- 计算公式：`线程数 = CPU 核数 * (1 + 等待时间/计算时间)`

**min-spare（最小空闲线程）：**
- 保持一定数量的空闲线程应对突发流量
- 建议值：10-50

**accept-count（接受队列）：**
- 当所有线程都在忙时的排队队列
- 设置过大导致响应慢，过小导致拒绝连接
- 建议值：100-500

### 4.2 NIO 连接器优化

#### NIO vs BIO

```
BIO（Blocking IO）：
- 一个连接一个线程
- 适合连接数少且固定的场景

NIO（Non-blocking IO）：
- 多路复用，一个线程管理多个连接
- 适合高并发短连接场景

NIO2（Asynchronous IO）：
- 异步非阻塞
- 性能最优但配置复杂
```

#### NIO 配置参数

```xml
<!-- server.xml 配置 -->
<Connector 
    port="8080" 
    protocol="org.apache.coyote.http11.Http11NioProtocol"
    connectionTimeout="20000"
    redirectPort="8443"
    
    <!-- 线程池 -->
    maxThreads="200"
    minSpareThreads="10"
    
    <!-- 连接数 -->
    maxConnections="8192"
    acceptCount="100"
    
    <!-- NIO 特定参数 -->
    selectorTimeout="5000"
    socket.soKeepAlive="true"
    socket.tcpNoDelay="true"
/>
```

### 5.1 CPU 性能分析

#### 常用命令

```bash
# 查看 CPU 使用率
top
htop

# 查看每个进程的 CPU 使用
ps aux --sort=-%cpu | head

# 查看 CPU 详细信息
cat /proc/cpuinfo

# 查看 CPU 负载
uptime
```

#### CPU 使用率分析

```
top 命令输出：
%Cpu(s):  5.2 us,  2.1 sy,  0.0 ni, 92.5 id,  0.2 wa,  0.0 hi,  0.0 si,  0.0 st

- us（user）：用户空间占用 5.2%（正常）
- sy（system）：内核空间占用 2.1%（正常）
- id（idle）：空闲 92.5%（充足）
- wa（iowait）：等待 IO 0.2%（很低，好）
```

**优化建议：**
- us 过高：优化应用程序代码
- sy 过高：减少系统调用
- wa 过高：检查磁盘 IO 瓶颈

### 5.4 网络参数优化

#### 内核参数调优

```bash
# /etc/sysctl.conf 配置

# TCP 连接相关
net.ipv4.tcp_tw_reuse = 1              # 允许重用 TIME_WAIT socket
net.ipv4.tcp_fin_timeout = 30          # FIN-WAIT-2 超时时间
net.ipv4.tcp_keepalive_time = 1200     # keepalive 探测间隔
net.ipv4.tcp_max_syn_backlog = 8192    # SYN 队列长度
net.core.somaxconn = 65535             # 接受队列最大长度

# 内存优化
net.core.rmem_max = 16777216           # 接收缓冲区最大值
net.core.wmem_max = 16777216           # 发送缓冲区最大值
net.ipv4.tcp_rmem = 4096 87380 16777216  # TCP 接收缓冲区
net.ipv4.tcp_wmem = 4096 65536 16777216  # TCP 发送缓冲区

# 端口范围
net.ipv4.ip_local_port_range = 1024 65535  # 本地端口范围

# 应用配置
sysctl -p  # 使配置生效
```

```bash

netstat -an | grep TIME_WAIT | wc -l

netstat -n | awk '/^tcp/ {++S[$NF]} END {for(a in S) print a, S[a]}'

# 输出示例：
# LISTEN 10
# ESTABLISHED 256
# TIME_WAIT 128
# CLOSE_WAIT 5
```

## 💡 实战技巧

### JVM 调优口诀

```
内存配置看应用，堆大堆小看数据
年轻年老比例配，G1 CMS 看需求
GC 日志要分析，停顿时间是关键
内存泄漏 MAT 查，CPU 高用 jstack
```

### MySQL 优化口诀

```
Explain 要看 type，range 以上才合格
索引不是万能药，选择性低不如扫
慢查询日志开启，定位问题分析之
最左前缀要遵守，函数运算索引失
```

### Redis 优化口诀

```
大 Key 要拆分，热 Key 做缓存
集群监控不能少，命中率要高才好
持久化看场景，RDB AOF 灵活配
内存淘汰策略选，LRU LFU 看需求
```
