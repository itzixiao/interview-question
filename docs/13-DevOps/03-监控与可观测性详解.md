### 1.2 指标类型详解

**示例代码：** [`BusinessMetricsCollector.java`](interview-microservices-parent/interview-observability/src/main/java/cn/itzixiao/interview/observability/monitoring/BusinessMetricsCollector.java)

```java
package cn.itzixiao.interview.observability.monitoring;

@Component
public class BusinessMetricsCollector {
    
    private final Counter orderCreatedCounter;
    private final Timer orderProcessTimer;
    
    public BusinessMetricsCollector(MeterRegistry meterRegistry) {
        // Counter：只增不减的计数器
        this.orderCreatedCounter = Counter.builder("order.created.total")
                .description("Total number of orders created")
                .tag("service", "order-service")
                .register(meterRegistry);
        
        // Timer：耗时统计
        this.orderProcessTimer = Timer.builder("order.process.time")
                .description("Time taken to process order")
                .tag("service", "order-service")
                .register(meterRegistry);
    }
    
    // 记录订单创建
    public void recordOrderCreated(String orderType) {
        orderCreatedCounter.increment();
    }
    
    // 记录订单处理耗时
    public void processOrder(String orderId) {
        orderProcessTimer.record(() -> {
            // 业务逻辑
        });
    }
}
```

**四大指标类型：**

| 类型 | 说明 | 使用场景 |
|------|------|---------|
| **Counter** | 只增不减的计数器 | 请求总数、错误数 |
| **Gauge** | 可增可减的度量值 | CPU 使用率、内存占用 |
| **Timer** | 耗时统计 | 接口响应时间 |
| **Summary** | 分位数统计 | P95、P99 延迟 |

### 1.3 PromQL 查询语言

**常用查询示例：**

```promql
# 1. 基础查询
# 查询订单创建总数
order_created_total

# 2. 速率查询
# 查询每秒订单创建数
rate(order_created_total[1m])

# 3. 分位数查询
# 查询 P95 响应时间
histogram_quantile(0.95, rate(http_server_requests_seconds_bucket[5m]))

# 4. 聚合查询
# 查询所有实例的总 QPS
sum(rate(http_server_requests_seconds_count[1m]))

# 5. 错误率查询
# 查询 HTTP 5xx 错误率
sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m])) 
/ 
sum(rate(http_server_requests_seconds_count[5m])) * 100

# 6. JVM 内存查询
# 查询堆内存使用率
jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} * 100
```

### 1.4 告警规则配置

**示例文件：** [`alerting-rules.yml`](interview-microservices-parent/interview-observability/src/main/resources/prometheus/alerting-rules.yml)

```yaml
groups:
  - name: application_alerts
    rules:
      # 应用宕机告警
      - alert: ApplicationDown
        expr: up{job="interview-observability"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "应用 {{ $labels.instance }} 已宕机"
      
      # CPU 使用率过高告警
      - alert: HighCPUUsage
        expr: process_cpu_seconds_total > 0.8
        for: 5m
        labels:
          severity: warning
      
      # 错误率过高告警
      - alert: HighErrorRate
        expr: rate(http_server_requests_seconds_count{status=~"5.."}[5m]) 
              / rate(http_server_requests_seconds_count[5m]) > 0.05
        for: 5m
        labels:
          severity: critical
```

## 三、SkyWalkingAPM链路追踪

### 3.1 链路追踪原理

```
Trace ID: abc123...
├─ Span 1: Controller.queryOrder() [0ms - 230ms]
│  └─ Span 2: UserService.queryUserInfo() [50ms - 100ms]
│  └─ Span 3: OrderService.queryOrderDetail() [100ms - 180ms]
│  └─ Span 4: RecommendationService.calculate() [180ms - 280ms]
```

**核心概念：**
- **Trace**：整个请求链路
- **Span**：单个操作单元
- **Parent Span**：父子操作关系

### 3.2 SkyWalking 配置

**示例文件：** [`agent.config`](interview-microservices-parent/interview-observability/src/main/resources/skywalking/agent.config)

```properties
# Agent 服务名称
agent.service_name=${SW_AGENT_NAME:interview-observability}

# OAP Server 地址
collector.backend_service=${SW_COLLECTOR_BACKEND_SERVICE:127.0.0.1:11800}

# 采样率
agent.sample_n_per_3_secs=${SW_AGENT_SAMPLE:-1}

# 慢调用阈值
agent.slow_query_threshold=${SW_AGENT_SLOW_QUERY_THRESHOLD:1000}
```

### 3.3 链路追踪演示

**示例代码：** [`TracingDemoController.java`](interview-microservices-parent/interview-observability/src/main/java/cn/itzixiao/interview/observability/tracing/TracingDemoController.java)

```java
@RestController
@RequestMapping("/api/tracing")
public class TracingDemoController {
    
    @GetMapping("/order/{orderId}")
    public String queryOrder(@PathVariable String orderId) {
        // Span 1: 查询用户信息
        String userInfo = queryUserInfo(orderId);
        
        // Span 2: 查询订单详情
        String orderDetail = queryOrderDetail(orderId);
        
        // Span 3: 计算推荐商品
        String recommendation = calculateRecommendation(orderId);
        
        return String.format("订单 %s 查询完成", orderId);
    }
}
```

## 四、ELK日志平台

### 4.1 ELK 架构

```
┌─────────────┐
│   Filebeat  │  日志采集
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Logstash   │  日志处理、过滤、格式化
└──────┬──────┘
       │
       ▼
┌─────────────┐
│Elasticsearch│  日志存储、索引
└──────┬──────┘
       │
       ▼
┌─────────────┐
│    Kibana   │  日志可视化、搜索
└─────────────┘
```

### 4.2 Filebeat 配置

**示例文件：** [`filebeat.yml`](interview-microservices-parent/interview-observability/src/main/resources/elk/filebeat.yml)

```yaml
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/interview-observability/*.log
    
    # JSON 日志解析
    json.keys_under_root: true
    json.message_key: message
    
    # 多行日志合并（Java 堆栈）
    multiline.pattern: '^\s'
    multiline.match: after
    multiline.max_lines: 500
    
    fields:
      service: interview-observability
      environment: dev

output.logstash:
  hosts: ["logstash:5044"]
```

### 4.3 Logstash 管道配置

**示例文件：** [`logstash-pipeline.conf`](interview-microservices-parent/interview-observability/src/main/resources/elk/logstash-pipeline.conf)

```ruby
input {
  beats {
    port => 5044
  }
}

filter {
  # JSON 解析
  json {
    source => "message"
    target => "json_data"
  }
  
  # 时间戳格式化
  date {
    match => ["timestamp", "ISO8601"]
    timezone => "Asia/Shanghai"
  }
  
  # 日志级别分类
  if [json_data][level] in ["error", "fatal"] {
    mutate {
      add_tag => ["error_log"]
    }
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "interview-logs-%{+YYYY.MM.dd}"
  }
}
```

## 五、健康检查与自愈

### 5.1 健康检查类型

```
┌────────────────────────────────────────────┐
│         Kubernetes 三种探针                 │
├────────────────────────────────────────────┤
│ Liveness Probe（存活探针）                  │
│ - 检测容器是否存活                         │
│ - 失败则重启容器                           │
├────────────────────────────────────────────┤
│ Readiness Probe（就绪探针）                 │
│ - 检测是否准备好处理请求                   │
│ - 失败则从 Service 移除                      │
├────────────────────────────────────────────┤
│ Startup Probe（启动探针）                   │
│ - 检测应用是否完成启动                     │
│ - 启动期间不执行其他探测                   │
└────────────────────────────────────────────┘
```

### 5.2 自定义健康检查

**示例代码：** [`DatabaseHealthIndicator.java`](interview-microservices-parent/interview-observability/src/main/java/cn/itzixiao/interview/observability/health/DatabaseHealthIndicator.java)

```java
@Component
public class DatabaseHealthIndicator implements HealthIndicator {
    
    @Override
    public Health health() {
        int activeConnections = checkActiveConnections();
        double usageRatio = (double) activeConnections / 100;
        
        if (usageRatio >= 0.9) {
            return Health.down()
                    .withDetail("status", "CRITICAL")
                    .build();
        }
        
        return Health.up()
                .withDetail("usageRatio", String.format("%.2f%%", usageRatio * 100))
                .build();
    }
}
```

### 5.3 K8s 探针配置

**示例文件：** [`deployment.yml`](interview-microservices-parent/interview-observability/src/main/resources/k8s/deployment.yml)

```yaml
containers:
- name: interview-observability
  image: interview-observability:1.0.0
  
  # Liveness Probe
  livenessProbe:
    httpGet:
      path: /actuator/health/liveness
      port: 8080
    initialDelaySeconds: 60
    periodSeconds: 30
    failureThreshold: 3
  
  # Readiness Probe
  readinessProbe:
    httpGet:
      path: /actuator/health/readiness
      port: 8080
    initialDelaySeconds: 30
    periodSeconds: 10
    failureThreshold: 3
  
  # Startup Probe
  startupProbe:
    httpGet:
      path: /actuator/health/startup
      port: 8080
    failureThreshold: 30
```

### 5.4 自愈机制

```
应用故障 → 探针检测 → K8s 响应
    │
    ├─ Liveness 失败 → 重启 Pod
    ├─ Readiness 失败 → 从 Service 移除
    └─ HPA 触发 → 自动扩容
```
