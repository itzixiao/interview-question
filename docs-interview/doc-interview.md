# Java 反射面试题

## 1. 什么是 Java 反射？

**答案：**

Java 反射是 Java 提供的一种运行时动态获取类信息、动态操作对象的机制。它允许程序在运行时：

- 获取类的完整结构（包名、父类、接口、构造方法、成员方法、成员变量、注解等）
- 动态创建对象
- 动态调用方法
- 动态修改属性

反射打破了 Java 静态编译限制，让代码更灵活、可扩展。

---

## 2. 反射主要有哪些应用场景？

**答案：**

反射是 Java 框架的核心技术，典型场景有 4 个：

1. **框架底层**
   Spring IOC、AOP、依赖注入、Bean 初始化都基于反射。

2. **动态加载类**
   如 JDBC 驱动加载：`Class.forName("com.mysql.cj.jdbc.Driver")`。

3. **ORM 框架**
   MyBatis、Hibernate 通过反射将数据库结果集自动映射成 Java 对象。

4. **工具类与动态功能**
   调试工具、热部署、通用序列化/反序列化、通用接口调用等。

---

## 3. 反射的核心 API 有哪些？

**答案：**

反射核心 API 主要在 `java.lang.reflect` 包下：

| API           | 说明                                                                  |
|---------------|---------------------------------------------------------------------|
| `Class`       | 代表一个类/类型，是反射入口，可通过 `Class.forName()`、`obj.getClass()`、`类名.class` 获取 |
| `Constructor` | 代表构造方法，用于动态创建对象                                                     |
| `Method`      | 代表成员方法，用于动态调用方法                                                     |
| `Field`       | 代表成员变量，用于获取/修改属性                                                    |

这四个是反射最基础、最常用的类。

---

## 4. 使用反射有哪些缺点？

**答案：**

反射是一把双刃剑，主要有 3 个缺点：

| 缺点              | 说明                                                  |
|-----------------|-----------------------------------------------------|
| **性能较差**        | 反射涉及动态类型解析，比直接方法调用慢很多，不适合高频/高性能场景                   |
| **破坏封装性**       | 可以通过 `setAccessible(true)` 访问私有成员，违背面向对象封装原则，存在安全风险 |
| **代码可读性、可维护性差** | 基于字符串匹配，编译期无法检查错误，难以调试、重构                           |

---

## 5. 如何优化反射的性能？

**答案：**

| 优化方式           | 说明                                             |
|----------------|------------------------------------------------|
| **缓存反射对象**     | 把 Class、Method、Field 对象缓存起来，重复使用，不要反复获取        |
| **关闭访问检查**     | 调用 `setAccessible(true)`，跳过权限检查，可提升 20%~30% 性能 |
| **减少反射调用次数**   | 能直接调用就不用反射，避免在循环/高频逻辑中大量使用反射                   |
| **使用更高效的 API** | Java 7+ 的 MethodHandle（方法句柄）性能更接近原生调用，可替代传统反射  |

# spring 面试题

## 1. 如何理解 spring， IOC 和 AOP 的思想？

**答案：**

理解 Spring IOC 和 AOP，可以从 “解耦” 和 “增强” 两个核心点切入。
IOC 是控制反转，简单说就是把对象的创建、依赖管理从代码里 “交出去”，让 Spring 容器来负责，比如你不用自己 new
Service、Dao，容器会自动实例化并注入依赖，这样类之间只依赖接口，不依赖具体实现，耦合度就低了，维护起来更灵活。
AOP 是面向切面编程，它能把日志、事务、权限这些重复的 “横切逻辑” 抽离出来，做成独立的切面，在不修改业务代码的情况下，动态织入到目标方法的前后，比如给
Service 的增删改方法统一加事务，既保持了业务代码的纯净，又实现了功能复用。
IOC 是 Spring 的 “骨架”，负责对象的生命周期和依赖；AOP 是 “肌肉”，负责在不侵入的情况下增强功能，两者结合让 Spring
既能管理对象，又能灵活扩展，这也是 Spring 能简化开发的核心原因。
